# C++语言基础与对象模型

## new/delete 和 malloc/free 有什么区别？
**速答：**  
`new/delete` 是 C++ 运算符，会分配/释放内存并调用构造/析构函数；`malloc/free` 是 C 函数，只管理裸内存，不处理对象生命周期。  
`new` 失败默认抛 `std::bad_alloc`，`malloc` 失败返回 `nullptr`。  
二者不能混用，现代 C++ 更推荐 RAII、容器和智能指针。

**追问速答：**

- `new` 失败和 `malloc` 失败有什么不同？  
  `new` 默认抛异常，`malloc` 返回 `nullptr`。
- 为什么不建议手写 `new/delete`？  
  容易泄漏、重复释放、异常路径漏释放，RAII 和智能指针更安全。
- `malloc` 出来的内存能用 `delete` 吗？  
  不能，分配释放方式必须匹配，否则未定义行为。

---

## 什么是 RAII？为什么 C++ 特别强调它？
**速答：**  
RAII 是“资源获取即初始化”，把资源绑定到对象生命周期：构造拿资源，析构释放资源。  
这样即使函数提前返回或抛异常，也能自动清理。  
锁、文件、socket、堆内存、数据库连接都适合 RAII 管理。

**追问速答：**
- `lock_guard` 为什么是 RAII？  
  构造时加锁，析构时解锁，锁和作用域绑定。
- RAII 只能管理内存吗？  
  不只，凡是有获取和释放动作的资源都能管理。
- RAII 和异常安全有什么关系？  
  异常栈展开会调用局部对象析构函数，所以能自动释放资源。

---

## 深拷贝、浅拷贝，以及为什么会引出 Rule of Three/Five/Zero？
**速答：**  
**浅拷贝只复制成员值，裸指针会导致多个对象指向同一资源；**

**深拷贝会复制资源本身。**  
类自己管理资源时，要处理析构、拷贝构造、拷贝赋值；C++11 后还要考虑移动构造和移动赋值。  
更推荐 Rule of Zero：用 `string/vector/unique_ptr` 等标准库成员管理资源。

**追问速答：**
- 什么情况下会 double free？  
  浅拷贝导致两个对象共享同一堆资源，析构时释放两次。
- Rule of Five 比 Rule of Three 多什么？  
  多了移动构造和移动赋值。
- 为什么推崇 Rule of Zero？  
  资源交给标准库成员管理，少写特殊成员函数，更安全。

---

## 指针和引用有什么区别？什么时候更推荐引用？
**速答：**  

引用是对象别名，必须初始化，不能改绑，语义上通常不为空；

指针是独立对象，可以为空、可以改指向。  

函数参数如果要求对象一定存在，优先引用；

如果要表达可空、可改指向、所有权或底层间接访问，用指针。  

引用语法更自然，指针表达能力更底层。

**追问速答：**
- 为什么常用 `const T&` 传参？  
  避免大对象拷贝，同时表达不修改参数。
- 引用底层是不是指针？  
  实现上常借助指针，但语言语义上引用不是对象。
- 返回引用要注意什么？  
  **不能返回局部变量引用**，否则悬空；只能返回生命周期仍有效的对象。

---

## C 和 C++ 有什么区别？
**速答：**  
C 是面向过程语言，C++ 在兼容 C 的基础上增加了面向对象、模板、STL、RAII、异常、引用、函数重载等机制。  
C++ 不只是“C 加类”，更强调类型安全、资源生命周期和抽象能力。  
它既保留底层控制能力，又提供更高层的工程抽象。

**追问速答：**
- C++ 为什么兼容 C？  
  复用大量 C 代码、库和系统接口。
- 面向对象三大特性是什么？  
  封装、继承、多态。
- 什么是零成本抽象？  
  抽象尽量在编译期完成，运行时开销接近手写底层代码。

---

## struct 和 class 有什么区别？
**速答：**  
`struct` 默认 `public`，`class` 默认 `private`。  
语法能力基本一样，简单数据聚合常用 `struct`，强调封装行为常用 `class`。

**追问速答：**

- C++ 的 `struct` 和 C 的一样吗？  
  不一样，C++ 的 `struct` 可以有函数、构造析构、继承和虚函数。
- 什么时候用 `struct`？  
  简单数据结构、字段公开、弱封装时。
- POD 是什么？  
  接近 C 风格的简单数据类型，适合二进制拷贝和 C 接口交互。

---

## const 的各种用法怎么区分？
**速答：**  
记住“`const` 修饰谁，谁不能改”。  
`const int* p` 不能通过 p 改值；`int* const p` 不能改 p 的指向；`const int* const p` 两者都不能改。  
`const` 成员函数承诺不修改普通成员变量，`const` 对象只能调用 `const` 成员函数。

**追问速答：**

- `const` 成员函数里能改成员变量吗？  
  普通成员不能改，`mutable` 成员可以改。
- `mutable` 是什么？  
  允许成员在 `const` 函数中被修改，常用于缓存、计数等逻辑不变字段。
- `const int*` 和 `int const*` 一样吗？  
  一样，都是指向常量 int 的指针。

---

## const 和 constexpr 有什么区别？
**速答：**  
`const` 表示只读，不一定是编译期常量；`constexpr` 表示可以在编译期求值。  
运行期初始化的 `const` 只是不能改，不是编译期常量。  
`constexpr` 适合数组大小、模板参数、编译期计算，也比宏更安全。

**追问速答：**

- `constexpr` 函数只能编译期调用吗？  
  不是，也可以运行期调用。
- `constexpr` 比宏好在哪？  
  有类型检查、作用域规则，调试更友好。
- `const` 一定是编译期常量吗？  
  不一定，取决于初始化表达式是否能编译期确定。

---

## static 有哪些用法？
**速答：**  
`static` 修饰局部变量：生命周期到程序结束，只初始化一次；

修饰全局变量/函数：限制文件可见性。  

修饰类成员变量：所有对象共享一份；

修饰静态成员函数：属于类，没有 `this` 指针。  
总结：`static` **要么改变生命周期，要么改变链接属性，要么表示属于类而不是对象。**

**追问速答：**
- **静态局部变量什么时候初始化？  
  第一次执行到定义语句时初始化。**
- C++11 后局部静态初始化线程安全吗？  
  是线程安全的。
- 静态成员函数为什么不能直接访问普通成员？  
  因为没有 `this` 指针。

---

## inline 和宏有什么区别？
**速答：**  
宏是预处理文本替换，没有类型检查；`inline` 是函数机制，有类型检查、作用域和调试信息。  
`inline` 只是建议内联，不保证一定展开。  
`inline` 还允许函数定义放头文件中，被多个翻译单元包含时不重复定义。

**追问速答：**
- `inline` 不一定会内联，由编译器决定。
- **类内定义成员函数默认是 `inline`**  
- 宏没有类型检查，可能有优先级问题和重复求值。

---

## explicit 的作用是什么？
**速答：**  
`explicit` 禁止构造函数参与隐式类型转换。  
常用于单参数构造函数，避免 `A a = 10;` 这种不明显转换。它让构造意图更明确，减少隐藏 bug。

**追问速答：**
- **哪些构造函数容易隐式转换？  
  单参数构造函数，或除第一个参数外都有默认值的构造函数。**
- `explicit` 可以用于转换函数吗？  
  可以，如 `explicit operator bool()`。
- 为什么单参数构造函数建议加 `explicit`？  
  防止编译器自动转换导致语义错误。

---

## nullptr 和 NULL 有什么区别？
**速答：**  
`NULL` 通常是 `0` 或 `0L`，可能被当成整数；

`nullptr` 是 C++11 空指针字面量，类型是 `std::nullptr_t`。  

`nullptr` 可以转成任意指针类型，但不会转成普通整数。  

现代 C++ 表示空指针优先用 `nullptr`。

**追问速答：**

- `nullptr` 的类型是什么？  
  `std::nullptr_t`。
- 为什么 `NULL` 有重载歧义？  
  因为它可能是整数 0，可能匹配 `int` 重载。
- **为什么不用 0 表示空指针？  
  语义不清晰，模板和重载中容易出错**。

---

## sizeof 和 strlen 有什么区别？
**速答：**  
`sizeof` 是运算符，返回对象或类型占用字节数；

`strlen` 是函数，计算 C 字符串 `\0` 前的字符数。  

`sizeof(char arr[]="abc")` 是 4，`strlen(arr)` 是 3。  

数组在函数参数中会退化成指针，此时 `sizeof` 得到指针大小。

**追问速答：**

- `sizeof(std::string)` 是字符串长度吗？  
  不是，是 string 对象本身大小。
- 数组传参后为什么大小丢失？  
  因为数组退化成首元素指针，当数组名作为函数实参时，不会传递整个数组，只会传递数组首元素的内存地址。

---

## volatile 有什么作用？
**速答：**  
`volatile` 告诉编译器变量可能被外部因素修改，不要过度优化读写。  
常用于硬件寄存器、中断变量、信号相关变量。  
它不保证原子性和线程同步，不能替代 `atomic` 或锁。

**追问速答：**

- `volatile` 能替代锁吗？  
  不能。
- `volatile` 和 `atomic` 的区别？  
  `volatile` 限制优化，`atomic` 保证原子操作和内存序。

---

# 指针、数组与声明补充

## 指针数组和数组指针怎么区分？
**速答：**  
`int* arr[5]` 是指针数组：arr 是数组，元素是 `int*`。  
`int (*p)[5]` 是数组指针：p 是指针，指向 `int[5]`。  
判断复杂声明：从变量名开始，先看和谁结合，括号优先。

**追问速答：**

- `int (*p)[5]` 怎么访问元素？  
  `(*p)[i]`。
- 二维数组传参为什么要写列数？  
  第一维可退化，第二维必须知道每行跨度。
- 数组名什么时候不退化？  
  `sizeof(arr)`、`&arr`、`decltype(arr)`。

---

## 数组名和指针完全一样吗？
**速答：**  
不一样。数组名在多数表达式中会退化为首元素指针，但数组名本身不是普通指针。  
`sizeof(arr)` 得到整个数组大小，`sizeof(p)` 得到指针大小。  
`arr` 类型通常退化为 `int*`，`&arr` 是指向整个数组的指针 `int (*)[N]`。

**追问速答：**
- `arr` 和 `&arr` 有什么区别？  
  地址值可能相同，但类型不同，指针加一跨度不同。
- 函数参数里数组大小为什么丢失？  
  因为退化成指针。
- 如何保留数组大小？  
  传长度、用模板引用、`std::array` 或 `vector`。

---

## 野指针和悬空指针是什么？
**速答：**  
野指针是未初始化或指向未知地址的指针；

悬空指针是对象已销毁但指针仍保存旧地址。解引用二者都是未定义行为。  

避免方式：初始化为 `nullptr`，释放后置空，优先智能指针和容器。

**追问速答：**
- `delete p` 后为什么 `p=nullptr`？  
  `delete` 不会自动改指针值，置空避免误用。
- 返回局部变量地址为什么危险？  
  局部变量函数结束后销毁，地址悬空。
- 如何避免野指针？  
  初始化、置空、使用 RAII/智能指针。

---

## 函数指针怎么写？和回调有什么关系？
**速答：**  
函数指针保存函数入口地址，例如 `int (*fp)(int,int)`。  
回调本质是把函数传给另一个模块，由对方在合适时机调用。  
现代 C++ 回调还常用 lambda、函数对象、`std::function`。

**追问速答：**

- 函数指针和 `std::function` 区别？  
  函数指针只存普通函数；`std::function` 可包装 lambda/仿函数，但有开销。
- lambda 能转函数指针吗？  
  无捕获 lambda 可以，有捕获不行。
- 成员函数指针和普通函数指针一样吗？  
  不一样，成员函数依赖对象和 `this`。

---

# 智能指针与资源管理

## unique_ptr、shared_ptr、weak_ptr 有什么区别？（所有权模型）怎么选？
**速答：**  
`unique_ptr` 独占所有权，不能拷贝只能移动，默认首选；

`shared_ptr` 共享所有权，通过引用计数释放。  

`weak_ptr` 不拥有对象，不增加强引用，主要用于打破循环引用和观察对象。  

工程上能独占就不要共享，共享意味着生命周期更复杂。

**追问速答：**
- 为什么默认优先 `unique_ptr`？  
  语义清晰、开销小、生命周期好推理。
- `weak_ptr` 为什么不能直接访问？  
  不拥有对象，对象可能已释放，必须先 `lock()`。
- `shared_ptr` 开销来自哪里？  
  控制块、引用计数、可能的原子操作。

---

## shared_ptr 为什么会有循环引用问题？怎么解决？
**速答：**  
两个对象互相持有 `shared_ptr` 会让强引用计数永远不归零，导致析构不发生。  
解决方法是把其中一边改成 `weak_ptr`，表达“观察不拥有”。  
本质是先判断谁拥有谁，回指针通常不该是强拥有关系。

**追问速答：**
- `weak_ptr` 怎么访问对象？  
  用 `lock()` 得到临时 `shared_ptr` 后访问。
- 双向关系都要 `shared_ptr + weak_ptr` 吗？  
  不一定，关键看所有权关系。
- 裸指针能解决循环引用吗？  
  能表达非拥有，但生命周期要自己保证。

---

## make_shared 和 shared_ptr(new T) 有什么区别？
**速答：**  
`make_shared` 通常一次分配对象和控制块，效率更高、异常安全更好。  
`shared_ptr<T>(new T)` 通常对象和控制块分开分配，可能两次分配。  
但 `make_shared` 不方便自定义 deleter，且弱引用长期存在时整块内存可能不能立即归还。

**追问速答：**
- `make_shared` 为什么更高效？  
  对象和控制块通常一次性分配。
- `make_shared` 缺点是什么？  
  控制块不释放时，合并分配的内存可能不能立刻归还。
- 什么时候不用 `make_shared`？  
  需要自定义 deleter 或对象内存和控制块分开管理时。

---

## enable_shared_from_this 是什么？
**速答：**  
`enable_shared_from_this` 允许对象在成员函数中安全获得指向自己的 `shared_ptr`。  
它避免直接 `shared_ptr<T>(this)` 产生新的控制块，导致重复释放。  
前提是对象已经被某个 `shared_ptr` 管理。

**追问速答：**
- 为什么不能直接 `shared_ptr(this)`？  
  会创建独立控制块，可能 double delete。
- `shared_from_this()` 什么时候出错？  
  对象还没被 `shared_ptr` 管理时可能抛 `bad_weak_ptr`。
- 常用场景是什么？  
  异步回调中延长对象生命周期。

---

## weak_ptr 的 expired() 和 lock() 有什么区别？
**速答：**  
`expired()` 只检查对象是否已释放；`lock()` 尝试提升为 `shared_ptr`。  
访问对象时应优先用 `lock()`，成功后访问期间对象不会释放。  
多线程下 `expired()` 检查完对象可能马上被释放，所以不能靠它直接访问。

**追问速答：**
- `weak_ptr` 能直接解引用吗？  
  不能，必须先 `lock()`。
- `expired()==false` 后一定安全吗？  
  不一定，多线程下状态可能立刻变化。
- `weak_ptr` 会延长对象生命周期吗？  
  不会，不增加强引用计数。

---

# 面向对象、多态与虚函数

## C++ 多态的实现原理是什么？虚函数表是什么？
**速答：**  
运行时多态通常靠虚函数表实现：类有 vtable，对象有隐藏 vptr。  
通过基类指针/引用调用虚函数时，根据对象真实类型的 vptr 找到对应函数。  
多态成立通常需要继承、virtual、基类指针或引用调用。

**追问速答：**
- 为什么必须基类指针/引用调用才体现多态？  
  静态类型和动态类型不一致时才需要运行时分派。
- 虚函数开销在哪？  
  间接调用、可能不易内联、对象多 vptr。
- 纯虚函数是什么？  
  `=0` 声明的虚函数，用于定义接口。

---

## 为什么基类析构函数通常要声明成 virtual？
**速答：**  
通过基类指针删除派生类对象时，基类析构函数必须是虚函数，才能完整调用派生类析构链。  
否则行为未定义，可能只析构基类部分，导致资源泄漏。  
只要类作为多态基类使用，析构函数通常应为 virtual。

**追问速答：**
- 构造函数为什么不能是虚函数？  
  构造时对象动态类型还没完整建立。
- 不打算被继承还要虚析构吗？  
  不需要，虚函数会带来布局和调用成本。
- 只有析构函数是 virtual 可以吗？  
  可以，至少保证多态删除正确。

---

## 构造函数和析构函数的调用顺序是什么？
**速答：**  
构造顺序：基类 -> 成员变量 -> 派生类构造函数体。  
析构顺序相反：派生类析构函数体 -> 成员变量逆序析构 -> 基类析构。  
成员初始化顺序由成员声明顺序决定，不由初始化列表书写顺序决定。

**追问速答：**
- 成员变量初始化顺序由什么决定？  
  类中声明顺序。
- 初始化列表顺序重要吗？  
  实际不决定顺序，但建议和声明顺序一致。
- 构造里调用虚函数有多态吗？  
  不表现为完整派生类多态。

---

## 构造函数和析构函数中可以调用虚函数吗？
**速答：**  
语法上可以，但不建议。  
构造时派生类部分还没构造完成，析构时派生类部分已经销毁，虚调用不会按完整派生类分派。  
不要在构造/析构中依赖虚函数多态。

**追问速答：**
- 构造函数为什么不能是虚函数？  
  虚机制依赖完整对象，构造时动态类型不完整。
- 析构函数为什么可以是虚函数？  
  删除派生类对象时需要按动态类型执行析构链。
- 析构中调用虚函数会怎样？  
  通常调用当前析构层级版本。

---

## 重载、重写、隐藏有什么区别？
**速答：**  
重载：同一作用域函数名相同、参数不同。  
重写：派生类重新实现基类虚函数，签名要匹配。  
隐藏：派生类同名函数隐藏基类同名函数，即使参数不同。

**追问速答：**
- 重载看返回值吗？  
  不看，不能只靠返回值重载。
- 重写必须完全一致吗？  
  函数名、参数、const 等要匹配，建议加 `override`。
- 为什么派生类会隐藏基类函数？  
  名字查找先找派生类作用域，同名即隐藏。

---

## override 和 final 有什么作用？
**速答：**  
`override` 让编译器检查派生类函数是否真的重写基类虚函数。  
`final` 修饰类表示不能再继承，修饰虚函数表示不能再重写。  
二者都能减少多态使用中的隐藏错误。

**追问速答：**
- 为什么推荐写 `override`？  
  防止签名写错导致没有真正重写。
- `final` 对优化有帮助吗？  
  可能有，编译器可做去虚化。
- `final` 修饰类和函数区别？  
  类不能继承；函数不能继续重写。

---

## 虚继承解决什么问题？
**速答：**  
虚继承解决菱形继承中公共基类被重复继承的问题。  
它保证最终派生类中只有一份虚基类子对象。  
代价是对象布局更复杂，可能有额外指针或偏移信息。

**追问速答：**
- 菱形继承为什么二义性？  
  最终派生类中有多份公共基类子对象。
- 虚继承影响对象布局吗？  
  会，引入虚基类指针/偏移信息。
- 虚基类由谁构造？  
  最底层最终派生类。

---

## 空类大小为什么不是 0？
**速答：**  
空类对象大小至少为 1 字节，因为 C++ 要保证不同对象有不同地址。  
如果有虚函数，通常会有 vptr，64 位常见大小是 8 字节。  
静态成员不属于对象，不增加对象大小。

**追问速答：**
- 类有虚函数后大小是多少？  
  通常至少一个 vptr 大小。
- 静态成员变量增加对象大小吗？  
  不增加。
- padding 是什么？  
  编译器为满足对齐插入的空字节。

---

## this 指针是什么？
**速答：**  
`this` 是非静态成员函数隐含的指针，指向当前对象。  
成员函数区分不同对象，就是靠隐式传入的 `this`。  
静态成员函数属于类本身，没有 `this`，不能直接访问普通成员。

**追问速答：**
- `this` 可以是 nullptr 吗？  
  正常不应该，空指针调用非静态成员函数是未定义行为。
- 静态成员函数为什么没有 `this`？  
  它不属于具体对象。
- 成员函数怎么访问当前对象数据？  
  通过隐式 `this` 指针。

---

## friend 友元是什么？
**速答：**  
`friend` 允许某个函数或类访问当前类的私有/保护成员。  
友元函数不是成员函数，只是被授权访问。  
常用于 `operator<<` 等需要访问私有成员但左操作数不是本类的场景。

**追问速答：**
- 友元函数是不是成员函数？  
  不是。
- 友元关系能继承吗？  
  不能，也不是双向的。
- 为什么 `operator<<` 常写友元？  
  左操作数是 `ostream`，不能作为本类成员实现 `cout << obj`。

---

## 纯虚函数和抽象类是什么？
**速答：**  
纯虚函数是 `virtual f() = 0`，表示基类只定义接口。  
含纯虚函数的类是抽象类，不能直接实例化。  
派生类实现所有纯虚函数后才能成为具体类。

**追问速答：**
- 抽象类能有构造函数吗？  
  能，派生类构造时仍要构造基类部分。
- 纯虚函数能有函数体吗？  
  能，但类仍是抽象类。
- 抽象类能有成员变量吗？  
  能，抽象类不一定只有接口。

---

## 纯虚析构函数为什么也要有定义？
**速答：**  
纯虚析构函数能让类变抽象类，但析构链中基类析构函数最终一定会被调用。  
所以即使声明为纯虚，也必须提供定义，否则链接错误。  
它常用于“接口基类 + 多态删除”场景。

**追问速答：**
- 纯虚析构函数会让类抽象吗？  
  会。
- 为什么纯虚析构还会被调用？  
  派生类析构后必须析构基类子对象。
- 多态基类析构为什么要 virtual？  
  保证通过基类指针删除时析构链完整。

---

## 运算符重载要注意什么？
**速答：**  
运算符重载不能改变优先级、结合性、操作数个数，也不能创造新运算符。  
重载要符合原运算符语义，避免代码难懂。  
`operator<<` 常写成普通函数/友元，`operator=` 通常返回 `A&`。

**追问速答：**
- 哪些运算符不能重载？  
  `.`、`.*`、`::`、`?:`、`sizeof` 等。
- `operator=` 返回什么？  
  通常返回 `*this` 的引用，支持连续赋值。
- 前置/后置 `++` 怎么区分？  
  前置 `operator++()`，后置 `operator++(int)`。

---

# 内存管理补充

## 栈上对象和堆上对象有什么区别？
**速答：**  
栈对象自动分配和释放，生命周期随作用域结束。  
堆对象由 `new/malloc` 动态分配，需要手动释放或交给智能指针。  
堆生命周期灵活，但更容易内存泄漏、重复释放和悬空指针。

**追问速答：**
- 栈为什么快？  
  通常只是移动栈指针。
- 堆为什么有碎片？  
  不同大小内存块频繁申请释放。
- 栈溢出和内存泄漏区别？  
  栈溢出是栈空间耗尽；泄漏是堆内存未释放。

---

## C++ 程序内存分区有哪些？
**速答：**  
常见有代码区、全局/静态区、常量区、堆区、栈区。  
局部变量通常在栈，`new` 出来的对象在堆，全局/静态变量在静态存储区。  
要区分“指针变量本身的位置”和“它指向对象的位置”。

**追问速答：**

- 字符串字面量在哪里？  
  通常在只读常量区/只读数据段。
- `static` 局部变量在哪里？  
  静态存储区。
- `new` 对象和指针变量分别在哪里？  
  对象在堆；指针变量取决于定义位置。

---

## new[]/delete[] 和 new/delete 为什么不能混用？
**速答：**  
`new[]` 构造数组多个元素，必须用 `delete[]` 逐个析构并释放。  
`new/delete`、`new[]/delete[]` 必须配对。  
混用是未定义行为，即使内置类型数组也不行。

**追问速答：**
- 内置类型数组也要 `delete[]` 吗？  
  要，释放形式必须匹配。
- `malloc` 能用 `delete` 吗？  
  不能。
- `delete nullptr` 安全吗？  
  安全，没有效果。

---

## placement new 是什么？
**速答：**  
placement new 在已有内存上构造对象，不负责分配内存。  
常用于内存池、对象池、STL 容器底层。  
对象不用时要手动调用析构函数。

**追问速答：**
- placement new 后为什么手动析构？  
  它只构造对象，不管理释放。
- 和普通 new 区别？  
  普通 new 分配内存并构造；placement new 只构造。
- 和 STL allocator 关系？  
  容器常先分配原始内存，再用 placement new 构造元素。

---

## 内存泄漏怎么排查？
**速答：**  
内存泄漏是堆内存申请后没有释放。  
常见原因：裸 `new/delete`、异常路径漏释放、容器保存裸指针、`shared_ptr` 循环引用。  
排查用 RAII、智能指针、日志、ASan、Valgrind、代码审查。

**追问速答：**
- double free 是什么？  
  同一块内存释放两次。
- use after free 是什么？  
  释放后继续访问旧地址。
- ASan 能查什么？  
  越界、UAF、重复释放、缓冲区溢出等。

---

# C++11/现代C++高频题

## 什么是左值、右值、右值引用？移动语义解决了什么问题？
**速答：**  
左值有稳定身份、可取地址；右值多是临时对象、即将销毁。  
右值引用 `T&&` 让程序能识别可被“搬走”的对象。  
移动语义通过转移资源代替深拷贝，减少临时对象和容器搬迁成本。

**追问速答：**
- 移动后的对象还能用吗？  
  能，但只保证有效，值未指定。
- 哪些类型移动收益大？  
  持有堆内存、句柄、缓冲区的重对象。
- 为什么移动语义对容器重要？  
  扩容、插入、返回对象时可减少深拷贝。

---

## std::move 和 std::forward 有什么区别？
**速答：**  
`std::move` 无条件把表达式转成右值，表示“可以搬走”。  
`std::forward` 用在模板转发引用中，保留实参原本的左值/右值属性。  
二者都只是类型转换工具，真正移动取决于后续是否调用移动构造/赋值。

**追问速答：**
- `std::move` 会真的搬数据吗？  
  不会，只是转换成右值。
- 什么是完美转发？  
  在模板包装层中原样保留参数值类别和 const 属性。
- `T&&` 一定是右值引用吗？  
  不一定，模板推导中可能是转发引用。

---

## 为什么很多类的移动构造函数建议写 noexcept？
**速答：**  
标准库容器扩容搬移元素时更愿意使用不抛异常的移动构造。  
如果移动构造不是 `noexcept`，`vector` 可能为保证异常安全而退回复制。  
资源转移类如果移动不会失败，应标记 `noexcept`。

**追问速答：**

- `vector` 为什么关心移动是否抛异常？  
  扩容要迁移元素，需要保证异常安全。
- 所有移动构造都必须 `noexcept` 吗？  
  不是，但不会失败就建议写。
- 不写 `noexcept` 后果？  
  容器可能不用移动而用拷贝，性能下降。

---

## auto 和 decltype 有什么区别？
**速答：**  
`auto` 根据初始化表达式推导变量类型，通常会忽略顶层 const 和引用。  
`decltype` 根据表达式形式推导，能保留引用和 const。  
`decltype((x))` 通常得到引用类型，`decltype(x)` 得到声明类型。

**追问速答：**
- `decltype((x))` 和 `decltype(x)` 一样吗？  
  不一样，前者通常是 `T&`。
- `auto&` 和 `const auto&` 区别？  
  前者可改，后者只读且能绑定临时对象。
- 什么时候用 `decltype(auto)`？  
  需要保留返回表达式引用/const 属性时。

---

## 什么是 RVO 和 NRVO？
**速答：**  
RVO/NRVO 是编译器对返回对象的拷贝省略优化。  
返回临时对象或具名局部对象时，编译器可能直接在调用者位置构造。  
`return std::move(local)` 不一定更好，可能阻止 NRVO。

**追问速答：**
- 返回局部对象一定拷贝吗？  
  不一定，可能 NRVO。
- `return std::move(local)` 一定好吗？  
  不一定，可能破坏 NRVO。
- C++17 有什么变化？  
  某些临时对象返回场景强制拷贝省略。

---

## lambda 表达式有什么价值？在 STL 里常怎么用？
**速答：**  
lambda 可以就地定义短小可调用对象，适合比较器、过滤条件、回调逻辑。  
它减少样板代码，逻辑就在使用点，也能捕获局部上下文。  
异步/延迟执行时要小心引用捕获悬空。

**追问速答：**
- 值捕获和引用捕获怎么选？  
  异步或延迟执行更偏值捕获；同步短生命周期可引用捕获。
- lambda 本质是什么？  
  编译器生成的匿名函数对象。
- 为什么适合 STL 算法？  
  算法常需要短小比较/过滤逻辑，lambda 最直接。

---

## lambda 捕获值和捕获引用有什么区别？
**速答：**  
值捕获把变量拷贝进 lambda，更安全但可能有拷贝成本。  
引用捕获不拷贝，但 lambda 延迟执行时变量可能已销毁。  
捕获 `this` 要保证对象执行时仍然存活。

**追问速答：**

- `[=]` 和 `[&]` 风险？  
  `[=]` 可能多拷贝，`[&]` 易悬空引用。
- 捕获 `this` 要注意什么？  
  对象析构后再执行会悬空。
- lambda 本质是什么？  
  匿名仿函数对象，捕获变量是成员。

---

## std::function 和函数指针有什么区别？
**速答：**  
函数指针只能保存普通函数地址。  
`std::function` 能包装函数、lambda、仿函数、`bind` 结果，更灵活。  
代价是类型擦除、间接调用和可能的动态分配。

**追问速答：**
- `std::bind` 是什么？  
  绑定函数和部分参数，生成新可调用对象。
- lambda 和仿函数关系？  
  lambda 本质是匿名仿函数。
- `std::function` 有性能成本吗？  
  有，性能敏感路径要谨慎。

---

## 模板的本质是什么？为什么说 STL 是模板库？
**速答：**  
模板是编译期代码生成机制，不是运行期泛型。  
编译器根据具体类型实例化出对应代码。  
STL 的容器、算法、迭代器都依赖模板实现多类型复用和较低运行时开销。

**追问速答：**
- 模板是编译期还是运行期机制？  
  主要是编译期。
- STL 为什么依赖模板？  
  为了支持多类型、多容器复用且减少运行时开销。
- 模板代价是什么？  
  报错复杂、编译慢、可能代码膨胀。

---

## 模板特化和偏特化是什么？
**速答：**  
全特化是为某个确定类型提供专门实现，如 `A<int>`。  
偏特化是为一类类型模式提供实现，如 `A<T*>`。  
类模板支持全特化和偏特化，函数模板不支持偏特化，常用重载替代。

**追问速答：**

- 全特化和偏特化区别？  
  全特化针对具体类型，偏特化针对类型模式。
- 函数模板能偏特化吗？  
  不能。
- 特化有什么用？  
  给特殊类型提供更合适或更高效实现。

---

## typename 和 class 在模板参数里有什么区别？
**速答：**  
模板参数列表中 `typename` 和 `class` 基本等价，都表示类型参数。  
但在模板内部，`typename` 可用于说明依赖名字是类型。  
如 `typename T::value_type`，否则编译器不知道它是类型还是成员。

**追问速答：**
- 哪个更推荐？  
  类型参数两者都行，现代常用 `typename`。
- 什么是依赖类型？  
  依赖模板参数才能确定的类型。
- 为什么必须写 `typename`？  
  告诉编译器依赖名字是类型。

---

# STL容器与工程选型

## vector、list、deque 分别适合什么场景？后端开发里怎么选？
**速答：**  
默认优先 `vector`：连续内存、缓存友好、随机访问快。  
`list` 适合已知迭代器位置下的频繁中间插删，但空间开销和缓存表现差。  
`deque` 适合首尾频繁插删且仍需要随机访问。

**追问速答：**
- 为什么 `vector` 常比 `list` 快？  
  连续内存缓存友好。
- `list` 的 O(1) 插删前提？  
  已经拿到位置迭代器。
- `deque` 和 `vector` 最大区别？  
  `deque` 分段存储，首尾插删更强。

---

## vector 的底层扩容机制是什么？reserve 和 resize 有什么区别？
**速答：**  
`vector` 空间不足会申请更大连续内存，把旧元素搬过去，再释放旧空间。  
扩容会导致原迭代器、指针、引用失效。  
`reserve` 只改容量不改元素数；`resize` 改 size，可能构造或析构元素。

**追问速答：**
- 扩容后为什么迭代器失效？  
  元素搬到新内存，旧地址失效。
- `reserve` 会创建元素吗？  
  不会。
- `resize` 变大时发生什么？  
  新增元素并默认/值初始化。

---

## map 和 unordered_map 有什么区别？线上如何选？
**速答：**  
`map` 通常红黑树实现，key 有序，查找/插入/删除 O(log n)，支持范围查询。  
`unordered_map` 哈希表实现，无序，平均 O(1)，最坏 O(n)，有 rehash 成本。  
需要有序、范围、上下界选 `map`；高频点查且哈希分布好选 `unordered_map`。

**追问速答：**
- `unordered_map` 为什么退化？  
  哈希冲突严重，桶内查找接近线性。
- rehash 是什么？  
  扩桶并重新分布元素。
- 什么时候 `map` 更合适？  
  有序遍历、范围查找、上下界查询。

---

## set/map 的底层为什么常说是红黑树？红黑树相对普通 BST 的价值是什么？
**速答：**  
红黑树是近似平衡二叉搜索树，能把查找、插入、删除稳定控制在 O(log n)。  
普通 BST 可能因有序插入退化成链表，变 O(n)。  
红黑树不追求绝对平衡，但插入删除调整成本相对可控。

**追问速答：**

- 红黑树严格平衡吗？  
  不是，是近似平衡。
- 普通 BST 为什么退化？  
  有序插入会一边倒。
- 红黑树为什么适合关联容器？  
  动态维护有序性且复杂度稳定。

---

## 什么是迭代器失效？不同容器下有哪些典型场景？
**速答：**  
迭代器失效是容器结构变化后，原迭代器/指针/引用不能再安全访问。  
`vector` 扩容会使全部失效，`erase` 使删除点及后面失效。  
`list/map` 节点型容器更稳定，删除通常只影响被删元素；`unordered_map` rehash 会整体失效。

**追问速答：**
- `vector` 扩容为什么全部失效？  
  元素整体搬迁到新内存。
- `map` 插入为什么通常不失效？  
  节点结构，不整体搬迁。
- `unordered_map` 最危险场景？  
  触发 rehash。

---

## 一边遍历一边删除元素，正确写法是什么？
**速答：**  
不能 `erase(it)` 后继续用旧 it。  
正确写法是删除时 `it = c.erase(it)`，不删除时 `++it`。  
`erase` 返回下一个有效迭代器，是为了安全继续遍历。

**追问速答：**
- 为什么 `erase` 返回迭代器？  
  原迭代器可能失效。
- `remove` 和 `erase` 一回事吗？  
  不是，remove 只重排，erase 才真正删。
- `list` 删除能随便写吗？  
  不建议，仍用返回值最稳。

---

## STL 中 remove/erase 惯用法是什么？为什么很多人会写错？
**速答：**  
`std::remove` 不真正删除元素，只把保留元素前移并返回新逻辑末尾。  
真正删除要再调用容器 `erase`。  
常见写法：`v.erase(remove(v.begin(), v.end(), x), v.end())`。

**追问速答：**
- 为什么 `remove` 不直接删？  
  它是通用算法，只操作迭代器区间，不拥有容器缩容能力。
- `list` 也要这样吗？  
  不一定，`list` 有成员 `remove`。
- 最容易犯什么错？  
  只写 `remove`，忘了 `erase`。

---

## 迭代器和指针一样吗？为什么 STL 需要迭代器这一层抽象？
**速答：**  
不完全一样。指针是地址，迭代器是统一访问容器元素的位置抽象。  
`vector` 迭代器可能像指针，但 `list/map/unordered_map` 不能只用原始指针统一表示。  
STL 用迭代器解耦容器和算法。

**追问速答：**

- 为什么 `sort` 不能直接排 `list`？  
  `sort` 要随机访问迭代器，`list` 只有双向迭代器。
- `vector` 迭代器为什么像指针？  
  底层连续内存。
- STL 为什么算法和容器分离？  
  让同一算法复用于不同容器。

---

## push_back 和 emplace_back 有什么区别？
**速答：**  
`push_back` 把已有对象拷贝/移动进容器。  
`emplace_back` 用参数在容器内部原地构造对象。  
`emplace_back` 可能减少临时对象，但不一定永远更快。

**追问速答：**

- `emplace_back` 一定更快吗？  
  不一定。
- 哪些场景优势明显？  
  构造成本高、参数多、对象不可拷贝、希望原地构造。
- 会触发扩容吗？  
  会，容量不足仍扩容。

---

## vector 的 size 和 capacity 有什么区别？
**速答：**  
`size` 是当前元素个数，`capacity` 是已分配空间最多能容纳的元素数。  
`reserve` 改 capacity 不改 size；`resize` 改 size。  
`clear` 通常只清空元素，不释放 capacity。

**追问速答：**
- `clear` 会释放 capacity 吗？  
  通常不会。
- 如何释放多余容量？  
  `vector<T>().swap(v)` 或尝试 `shrink_to_fit()`。
- `shrink_to_fit` 一定释放吗？  
  不一定，只是请求。

---

## vector 里存指针要注意什么？
**速答：**  
`vector<T*>` 只管理指针值，不管理指针指向的对象。  
删除元素或 `clear` 不会自动 `delete` 对象，容易泄漏。  
更推荐 `vector<unique_ptr<T>>` 或按所有权需求用 `shared_ptr`。

**追问速答：**

- `vector<T*>` 删除元素会自动 delete 吗？  
  不会。
- `vector<unique_ptr<T>>` 能拷贝吗？  
  不能，只能移动。
- 智能指针进容器注意什么？  
  `unique_ptr` 移动，`shared_ptr` 注意循环引用。

---

## string 的底层是什么？SSO 是什么？
**速答：**  
`std::string` 管理字符缓冲区，自动处理内存和长度。  
SSO 是小字符串优化，短字符串直接存在 string 对象内部，避免堆分配。  
SSO 是实现优化，不同标准库细节不同。

**追问速答：**
- `string` 和 `char*` 区别？  
  string 自动管理内存，char* 只是指针。
- `c_str()` 什么时候失效？  
  string 修改、扩容、析构后可能失效。
- string 是否以 `\0` 结尾？  
  `c_str()` 返回结果以 `\0` 结尾，string 中间也可含 `\0`。

---

## set 和 unordered_set 有什么区别？
**速答：**  
`set` 通常红黑树实现，有序，查找/插入/删除 O(log n)。  
`unordered_set` 哈希表实现，无序，平均 O(1)，最坏 O(n)。  
需要顺序/范围操作选 set，只做点查可选 unordered_set。

**追问速答：**

- 为什么 set 不能直接改元素值？  
  值决定排序位置，改了会破坏树结构。
- unordered_set 何时 rehash？  
  负载因子超过阈值时。
- 自定义类型需要什么？  
  hash 函数和相等比较。

---

## priority_queue 底层是什么？
**速答：**  
`priority_queue` 是容器适配器，底层通常是 `vector + heap`。  
默认大根堆，取堆顶 O(1)，插入/删除堆顶 O(log n)。  
堆只保证堆顶最优，不保证整体有序遍历。

**追问速答：**

- 怎么改小根堆？  
  `priority_queue<int, vector<int>, greater<int>>`。
- 堆和 BST 区别？  
  堆保证父子优先级，BST 保证左小右大。
- 能有序遍历吗？  
  不能直接，只能不断 pop。

---

## stack 和 queue 为什么叫容器适配器？
**速答：**  
它们不是独立底层容器，而是基于已有容器封装受限接口。  
`stack` 提供后进先出，`queue` 提供先进先出。  
默认底层通常是 `deque`。

**追问速答：**
- stack 默认底层是什么？  
  通常 `deque`。
- stack 能用 vector 实现吗？  
  能，因为只需尾部操作。
- queue 为什么不用 vector 默认实现？  
  头删效率低。

---

## sort 和 stable_sort 有什么区别？
**速答：**  
`sort` 不保证相等元素原相对顺序；`stable_sort` 保证稳定性。  
`sort` 通常更快；`stable_sort` 可能需要更多额外空间。  
`std::sort` 要求随机访问迭代器。

**追问速答：**

- `sort` 要求什么迭代器？  
  随机访问迭代器。
- 为什么 list 不能直接用 `std::sort`？  
  list 只有双向迭代器，有自己的 `list::sort()`。
- 比较函数要满足什么？  
  严格弱序。

---

# 类型转换补充

## C++ 四种 cast 有什么区别？
**速答：**  
`static_cast` 做常规类型转换；`dynamic_cast` 做多态类型运行时检查转换。  
`const_cast` 去掉/添加 const；`reinterpret_cast` 底层二进制重解释，风险最高。  
C++ 风格转换比 C 风格强转语义更明确。

**追问速答：**

- `dynamic_cast` 依赖什么？  
  RTTI，通常要求基类有虚函数。
- 转换失败返回什么？  
  指针失败返回 nullptr，引用失败抛 `bad_cast`。
- 为什么不推荐 C 风格强转？  
  语义混杂，不利于审查。

---

## 隐式类型转换有哪些坑？
**速答：**  
隐式转换可能来自单参数构造、转换运算符、算术提升、指针转 bool 等。  
它会让代码能编译但语义不符合预期。  
用 `explicit`、花括号初始化和清晰类型设计减少风险。

**追问速答：**
- `explicit` 为什么重要？  
  阻止构造函数参与隐式转换。
- `operator bool` 风险？  
  可能发生意外条件/算术转换，建议 `explicit operator bool()`。
- `{}` 初始化好处？  
  更严格，禁止窄化转换。

---

# 多线程与并发补充

## 进程和线程有什么区别？
**速答：**  
进程是资源分配基本单位，线程是 CPU 调度基本单位。  
同一进程内线程共享地址空间、堆、全局变量、文件描述符等。  
每个线程有自己的栈、寄存器上下文、程序计数器和线程局部存储。

**追问速答：**
- 线程共享哪些资源？  
  地址空间、堆、全局/静态变量、文件描述符。
- 线程私有哪些资源？  
  栈、寄存器、PC、TLS。
- 哪个切换开销大？  
  通常进程切换更大。

---

## 多线程中堆变量是否共享？
**速答：**  
同一进程的线程共享地址空间，所以堆区共享。  
但具体堆对象是否被共享，取决于多个线程是否拿到同一个地址。  
多个线程同时读写同一对象必须同步，否则数据竞争。

**追问速答：**

- 栈变量是否共享？  
  默认每线程独立，但地址传出去也可能被访问。
- 全局变量是否共享？  
  共享。
- `thread_local` 是什么？  
  每个线程独立一份变量。

---

## 什么是数据竞争？
**速答：**  
多个线程访问同一内存位置，其中至少一个写，并且没有同步，就是数据竞争。  
C++ 中数据竞争是未定义行为。  
用 mutex、atomic、线程局部存储或避免共享可变状态解决。

**追问速答：**

- 数据竞争和竞态条件一样吗？  
  不完全一样，竞态条件更广义。
- 如何避免？  
  锁、原子变量、线程局部、减少共享。
- 只读共享安全吗？  
  通常安全。

---

## mutex、lock_guard、unique_lock 有什么区别？
**速答：**  
`mutex` 是锁本体；`lock_guard` 是简单 RAII 锁，构造加锁析构解锁。  
`unique_lock` 更灵活，支持延迟加锁、手动解锁、移动所有权。  
简单临界区用 `lock_guard`，条件变量等复杂场景用 `unique_lock`。

**追问速答：**

- 为什么 RAII 管理锁？  
  防止异常/提前返回忘记解锁。
- `unique_lock` 为什么配合条件变量？  
  wait 需要临时释放并重新获得锁。
- 手动 lock/unlock 风险？  
  漏解锁导致死锁。

---

## 条件变量 condition_variable 怎么用？
**速答：**  
条件变量用于线程等待某个条件成立。  
等待线程持有 `unique_lock`，`wait` 时释放锁阻塞，被唤醒后重新加锁并检查条件。  
必须用谓词或 while 防止虚假唤醒。

**追问速答：**

- 为什么 wait 配合锁？  
  条件检查和修改要互斥保护。
- 什么是虚假唤醒？  
  条件不满足也可能返回。
- `notify_one` 和 `notify_all` 区别？  
  一个唤醒一个，另一个唤醒全部。

---

## 什么是死锁？如何避免？
**速答：**  
死锁是多个线程互相等待对方持有的资源，导致都无法继续。  
四个必要条件：互斥、占有且等待、不可抢占、循环等待。  
避免方式：固定加锁顺序、缩短持锁时间、使用 `std::lock`、避免持锁调用外部逻辑。

**追问速答：**

- 死锁四条件？  
  互斥、占有且等待、不可抢占、循环等待。
- 两把锁如何安全加？  
  固定顺序或 `std::lock`。
- `std::lock` 有什么用？  
  同时锁多把锁，降低死锁风险。

---

## atomic 和 mutex 有什么区别？
**速答：**  
`atomic` 适合单个变量的简单原子操作。  
`mutex` 适合复杂临界区和多个变量一致性。  
`atomic` 不能完全替代锁。

**追问速答：**
- `atomic<int>++` 线程安全吗？  
  是。
- 多变量一致性用 atomic 够吗？  
  不一定，通常需要锁。
- CAS 是什么？  
  Compare-And-Swap，无锁编程基础。

---

## shared_ptr 是线程安全的吗？
**速答：**  
`shared_ptr` 的引用计数操作通常线程安全，但被管理对象不自动线程安全。  
不同 `shared_ptr` 对象并发拷贝/析构没问题；多个线程读写同一个 `shared_ptr` 变量本身需要同步。  
通过 `shared_ptr` 并发访问对象时，仍要保护对象状态。

**追问速答：**

- 引用计数安全和对象安全区别？  
  计数安全只保证生命周期管理，对象数据仍需同步。
- 多线程同时 reset 同一个 shared_ptr 变量安全吗？  
  不安全。
- `weak_ptr::lock` 为什么安全？  
  成功后生成 shared_ptr，访问期间对象不会释放。

---

# 编译链接补充

## C++ 从源码到可执行文件经历哪些阶段？
**速答：**  
经历预处理、编译、汇编、链接。  
预处理展开宏和头文件；编译生成汇编；汇编生成目标文件；链接解析符号生成可执行文件。  
编译错误多在单文件阶段，链接错误多是找不到定义或重复定义。

**追问速答：**

- 头文件重复包含怎么解决？  
  include guard 或 `#pragma once`。
- 静态链接和动态链接区别？  
  静态合进可执行文件，动态运行时加载。
- 符号表是什么？  
  记录函数/全局变量等符号名和地址信息。

---

## 静态库和动态库有什么区别？
**速答：**  
静态库链接时复制进可执行文件，运行时不依赖库文件，但体积更大。  
动态库运行时加载，多个程序可共享，便于升级，但部署要保证库存在且版本匹配。  
Linux 静态库常见 `.a`，动态库 `.so`。

**追问速答：**

- Linux 后缀？  
  静态库 `.a`，动态库 `.so`。
- 动态库找不到怎么办？  
  检查库路径、`LD_LIBRARY_PATH`、rpath、缓存和版本。
- 动态链接好处？  
  共享库代码、节省空间、便于升级。

---

## 头文件里为什么不要定义普通全局变量？
**速答：**  
头文件会被多个 `.cpp` 包含，普通全局变量定义会在多个翻译单元中重复出现。  
链接时可能重复定义。  
正确做法是头文件 `extern` 声明，一个 `.cpp` 中定义；C++17 可用 `inline variable`。

**追问速答：**

- 声明和定义区别？  
  声明告诉符号存在，定义分配存储或提供实现。
- `#pragma once` 能解决全局变量重复定义吗？  
  不能，只防止同一翻译单元重复包含。
- `inline variable` 是什么？  
  C++17 允许变量定义放头文件并由链接器合并。

---

## extern "C" 的作用是什么？
**速答：**  
`extern "C"` 告诉 C++ 编译器按 C 方式处理函数名，避免 C++ 名字改编。  
常用于 C++ 调 C 库，或 C 调 C++ 暴露的普通函数接口。  
C++ 类、重载、异常通常不能直接作为 C 接口暴露。

**追问速答：**

- `extern "C"` 能修饰类吗？  
  不能按 C 接口导出 C++ 类。
- C++ 为什么需要名字改编？  
  支持重载、命名空间、成员函数，需要符号含类型信息。
- C 调 C++ 注意什么？  
  暴露普通函数接口，避免直接暴露类和异常。

---

# 异常与错误处理补充

## C++ 异常机制是什么？
**速答：**  
异常通过 `throw` 抛出，沿调用栈查找匹配 `catch`。  
查找过程中会析构已构造局部对象，这叫栈展开。  
RAII 能保证异常路径资源释放；析构函数不建议抛异常。

**追问速答：**
- 析构函数能抛异常吗？  
  不建议，可能导致 `terminate`。
- 什么是栈展开？  
  异常向上传播并析构沿途局部对象。
- RAII 为什么异常安全？  
  异常时局部对象析构自动清理资源。

---

## noexcept 有什么作用？
**速答：**  
`noexcept` 表示函数承诺不抛异常，违反会调用 `std::terminate`。  
它影响编译器优化和标准库选择，比如 `vector` 扩容是否使用移动构造。  
移动操作不会失败时应尽量标记 `noexcept`。

**追问速答：**
- `noexcept` 只是说明文档吗？  
  不是，会影响优化和容器行为。
- 析构函数默认 `noexcept` 吗？  
  通常是。
- `noexcept` 函数真抛异常会怎样？  
  调用 `std::terminate`。

---

# 哈希与算法复杂度补充

## 哈希冲突有哪些解决方式？
**速答：**  
常见有链地址法和开放寻址法。  
链地址法是在桶里挂链表/节点；开放寻址是在数组中继续探测空位。  
哈希性能依赖哈希函数、负载因子和冲突处理。

**追问速答：**

- 两者优缺点？  
  链地址删除简单但有节点开销；开放寻址缓存友好但删除复杂。
- 负载因子过高怎样？  
  冲突增加，效率下降。
- 为什么 rehash？  
  扩桶、降低负载因子、减少冲突。

## unordered_map 为什么平均 O(1)，最坏 O(n)？
**速答：**  
哈希表通过哈希函数把 key 映射到桶，理想情况下直接定位，平均 O(1)。  
如果大量 key 落到同一桶，桶内查找退化为线性，最坏 O(n)。  
所以 O(1) 是平均复杂度，不是绝对保证。

**追问速答：**
- 什么是哈希冲突？  
  不同 key 映射到同一桶。
- 什么是负载因子？  
  元素数量 / 桶数量。
- 线上为什么不能只看平均复杂度？  
  数据分布、哈希函数、攻击输入都可能导致退化。

## 红黑树和 AVL 树有什么区别？
**速答：**  
AVL 更严格平衡，查找路径更短；红黑树近似平衡，插入删除调整成本通常更低。  
STL `map/set` 常用红黑树，因为查找、插入、删除都要兼顾。  
数据库索引常用 B+ 树，因为外存场景要减少 IO。

**追问速答：**
- 红黑树查找为什么 O(log n)？  
  颜色规则限制树高。
- 红黑树严格平衡吗？  
  不是，近似平衡。
- 为什么数据库用 B+ 树？  
  一个节点存多个 key，树高低，磁盘 IO 少。

## 快排平均 O(nlogn)，最坏 O(n²) 的原因是什么？
**速答：**  
分区均匀时递归深度 O(log n)，每层处理 O(n)，平均 O(nlogn)。  
pivot 每次极端时递归深度退化为 O(n)，总复杂度 O(n²)。  
可用随机 pivot、三数取中、三路快排优化。

**追问速答：**
- 如何优化快排？  
  随机 pivot、三数取中、三路快排、小数组插入排序。
- 快排稳定吗？  
  通常不稳定。
- 快排和归并怎么选？  
  快排原地缓存友好；归并稳定且最坏 O(nlogn)，但要额外空间。
