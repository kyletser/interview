# Redis 基础与数据结构

## Redis 为什么这么快？

**面经热度：极高频**

**15秒简答：**  
Redis 快主要靠四点：内存访问、核心命令执行路径短、命令执行线程模型简单、数据结构针对高频场景做了优化。

再往工程上说，很多瓶颈其实不在 Redis 本身，而在网络 RTT、慢命令、BigKey、热 Key 和持久化策略。

**3分钟详答：**  

第一，Redis 是内存数据库。数据主要在内存里，少了磁盘随机 I/O，这个是最基础的前提。  
第二，Redis 的大部分核心命令时间复杂度很低，比如 `GET/SET` 平均是 O(1)，很多集合操作也有专门的数据结构支撑。  
第三，Redis 的命令执行模型很轻。传统多线程程序会有大量锁竞争、上下文切换、共享状态同步的问题，而 Redis 的核心命令执行路径很短，单线程主循环让实现和性能都更稳定。  
第四，Redis 的对象编码和底层结构做了很多工程优化。比如**小对象会采用更省内存的编码**；字符串、哈希、集合、有序集合都不是“只用一种结构硬扛”，而是根据数据规模做不同编码切换。  
第五，网络层还能用 pipeline 降低往返次数。如果业务一条请求里发很多条命令，不做 pipeline，真正慢的往往是 RTT，不一定是 Redis 算命令慢。  

“Redis 快是建立在命令不要太重、Key 不要太大、网络 RTT 可控、持久化和复制策略合理的前提下。像 `KEYS *`、超大集合、频繁 fork 持久化，都可能把 Redis 拖慢。”  
这句很像真实后端工程师的回答。

**可能追问：**  
- 追问：Redis 既然快，是不是所有场景都适合放 Redis？  
  十五秒简答：不是。Redis 适合**高频读写、低延迟、可接受内存成本的场景**；**不适合超大冷数据、强事务复杂查询和对成本特别敏感**的全量存储。

- 追问：Redis 的性能瓶颈常见在哪？  
  十五秒简答：常见在**网络 RTT、慢命令**、BigKey、HotKey、AOF fsync、主从复制积压、fork 持久化带来的阻塞，而不只是 CPU。

- 追问：为什么说 pipeline 也会影响“快不快”？  
  十五秒简答：因为很多时候 Redis 处理命令很快，慢的是一来一回的网络往返；pipeline 能把多条命令合并发送，明显降低 RTT 成本。

## Redis 是单线程吗？

**面经热度：极高频**

**15秒简答：**  
经典回答是：Redis 的**命令执行主路径长期以来主要是单线程，**但不是“整个 Redis 完全没有线程”。

后台持久化、部分 I/O、复制等都可能用到其他线程。面试里最好说“命令执行单线程为主”。

**3分钟详答：**  

为什么要这么设计？  
因为 Redis 的主要目标是高吞吐、低延迟、简单稳定。单线程执行命令可以省掉大量锁，避免多线程共享数据结构时的并发复杂度。对于很多场景来说，内存访问本来就很快，单线程已经够用了。  

但 Redis 不是“纯单线程世界”。  
像 RDB/AOF 相关的后台工作、主从复制的一些流程、以及新版本中的 I/O 线程，都可能由其他线程承担。

注意这里的重点是：**多线程更多是在做 I/O 和后台任务优化，不等于把所有命令逻辑都改成并行执行。**

面试时最好补一层理解：  
Redis 不是靠“单线程”才快，而是靠“内存 + 高效数据结构 + 简单执行路径 + 少锁开销”才快。单线程只是它在工程上做出的一个权衡。

**可能追问：**  

- 追问：单线程的好处是什么？  
  十五秒简答：**实现简单、没有大量锁竞争、上下文切换少、性能更稳定，也更容易保证命令执行顺序**。

- 追问：单线程的坏处是什么？  
  十五秒简答：单个实例没法自然吃满多核，重命令、BigKey、阻塞操作更容易拖慢整个实例。

- 追问：Redis 6 以后不是有多线程了吗？  
  十五秒简答：有 I/O 线程等优化，但面试里仍然应说“命令执行主路径主要是单线程”，不要把 Redis 说成完全的多线程数据库。

## Redis 常见数据类型有哪些？各自适合什么场景？

**面经热度：极高频**

**15秒简答：**  
常见有 **String、Hash、List、Set、ZSet**，扩展里常见 Bitmap、HyperLogLog、Geo、Stream。

面试里别只背定义，最好顺手说场景：缓存对象、计数器、消息队列、去重、排行榜、UV 统计、地理位置、消息流。

**3分钟详答：**  
这题表面是“背类型”，其实面试官在看你有没有项目感。

`String`：最常用。适合缓存对象、计数器、分布式锁 token、session、简单配置。很多业务 80% 的 Redis Key 都是 String。  
`Hash`：适合存对象的多个字段，比如用户信息、商品信息。好处是字段级操作方便，不一定每次都整对象覆盖。  
`List`：本质是有序序列，常用于简单消息队列、评论流、最近访问记录。  
`Set`：无序去重集合，适合标签、共同好友、黑名单、签到去重、点赞去重。  
`ZSet`：带 score 的有序集合，特别适合排行榜、延迟队列、滑动窗口限流、按时间排序的 Feed。  

扩展结构里：  
`Bitmap` 适合布尔位状态，例如签到、在线状态、活跃标记；  
`HyperLogLog` 适合基数统计，例如 UV，大幅节省空间；  
`Geo` 适合附近的人、门店距离；  
`Stream` 适合更像消息流的场景，支持消费组。  

真正好的回答要带一句“怎么选型”：  
- 只要一个值：String  
- 想按字段操作对象：Hash  
- 想保留顺序：List  
- 想去重：Set  
- 想排序 + 范围查询：ZSet  

**可能追问：**  
- 追问：为什么缓存用户对象常用 Hash，而不是把整个 JSON 放 String？  
  十五秒简答：Hash 方便按字段读写，局部更新更自然；但如果对象整体读写更多、跨语言序列化统一，String + JSON 也很常见。

- 追问：排行榜为什么常用 ZSet？  
  十五秒简答：因为 ZSet 自带 score 排序，支持按分数和排名查询，天然适合排行榜、TopN 和时间窗口。

- 追问：Set 和 ZSet 怎么选？  
  十五秒简答：只需要去重用 Set；既要去重又要排序、排名、范围查，就用 ZSet。

## Redis 的 ZSet 为什么常被问？底层实现是什么？

**面经热度：高频**

**15秒简答：**  
ZSet 常被问，因为它很像“既能哈希查，又能排序查”的复合结构。

面试里通常答：底层会结合字典和跳表来兼顾按 member 查分数和按 score 排序查询；小对象场景还会用更紧凑的编码。

**3分钟详答：**  
ZSet 是 Redis 特别有代表性的结构，因为它很适合真实业务：排行榜、限流、延迟任务、热度榜都离不开它。  

如果只用哈希表，你可以很快找到某个 member 的 score，但做范围排序会很难。  
如果只用平衡树或跳表，按 score 排序很方便，但按 member 精确查找和更新又不够快。  
所以 ZSet 的设计思路是：  

- 用字典维护 member -> score 的映射，方便 O(1) 平均复杂度查 member  
- 用跳表按 score 维护有序结构，方便做排名、范围查询、TopN  

为什么 Redis 常说跳表而不是红黑树？  
从工程上看，**跳表实现简单、范围查询友好、插入删除平均复杂度也很稳定**，特别适合 Redis 这种追求实现简洁和性能均衡的系统。  

不过你要注意版本细节：  
很多面试题里会说“ZSet 底层是跳表 + 哈希”，这是大方向；但小数据场景下 Redis 还会用更紧凑的编码，不是一上来就大结构。  
所以最稳的表达是：  
“ZSet 在常规编码下会结合哈希和跳表；小对象场景会用更省空间的紧凑编码，数据规模上来再升级。”

**可能追问：**  

- 追问：跳表查找复杂度是多少？  
  十五秒简答：平均 O(logN)，最坏 O(N)，但在工程上表现稳定，且实现比平衡树更简单。

- 追问：为什么不直接用红黑树？  
  十五秒简答：跳表实现更简单，范围查询和分层前进很自然，在 Redis 这种场景下是很合适的工程权衡。

- 追问：为什么还要配一个哈希表？  
  十五秒简答：因为只靠跳表按 member 精确查找不够高效，配哈希表后可以兼顾“按 member 查”和“按 score 排序”。

## Bitmap、HyperLogLog、Geo、Stream 什么时候会用到？

**面经热度：中高频**

**15秒简答：**  
Bitmap 适合二值状态统计，HyperLogLog 适合 UV/去重计数，Geo 适合地理位置检索，Stream 适合可持久化的消息流。面试里关键不是背名字，而是说出“为什么它比普通结构更省”。

**3分钟详答：**  
这类题常出现在项目追问里，面试官想看你是否真正理解 Redis 不只是五种基础类型。  

`Bitmap`：底层本质上还是位操作，适合 0/1 状态，比如用户某天是否签到、某设备是否在线。优势是空间极省，一位就能表示一个状态。  
`HyperLogLog`：适合做“近似去重计数”，典型是 UV。它不适合拿来做精确去重，但非常省空间。面试时可以说“为了统计量级，不是为了拿到具体元素”。  
`Geo`：适合附近门店、附近骑手、附近用户之类的地理位置场景，本质上还是对经纬度做编码和范围检索。  
`Stream`：Redis 5.0 引入，更像一个带持久化能力和消费组机制的消息流。它比 Pub/Sub 更适合“消息不能随便丢”的场景，但如果业务对消息可靠性、堆积、回溯要求更强，通常还是会考虑专业 MQ。  

这题的高分点在于你能明确边界：  
- Bitmap/HLL 是“统计工具”  
- Geo 是“位置检索工具”  
- Stream 是“轻量消息流工具”  
不是所有消息队列场景、所有统计场景都应该用 Redis 硬做。

**可能追问：**  
- 追问：UV 统计为什么会想到 HyperLogLog？  
  十五秒简答：因为它做基数估计特别省内存，适合大规模近似去重计数，典型就是 UV。

- 追问：Bitmap 适合做在线人数吗？  
  十五秒简答：适合做“状态位”记录，但前提是 ID 可映射且稠密；如果 ID 很散，直接 bitmap 可能会浪费空间。

- 追问：Stream 和 Pub/Sub 最大区别是什么？  
  十五秒简答：Pub/Sub 更像实时广播，订阅者断线就可能丢消息；Stream 更像可积压、可消费组处理的消息流。

# 缓存设计与性能优化

## 缓存穿透、缓存击穿、缓存雪崩分别是什么？怎么解决？

**面经热度：极高频**

**15秒简答：**  
穿透是查不存在的数据，缓存和数据库都没有；

击穿是某个热点 Key 失效，大量请求同时打到数据库；

雪崩是大量 Key 同时失效或 Redis 故障。

对应常见解法：空值缓存/布隆过滤器、互斥锁或 singleflight、过期时间打散 + 限流熔断 + 多级缓存。

**3分钟详答：**  
这是 Redis 面试的经典老三样，但很多人会把三个概念混掉。面试里最好一口气区分清楚：

**缓存穿透**：  
请求的数据本来就不存在，缓存查不到，数据库也查不到，于是每次请求都打到数据库。  
解决思路：  

- 缓存空值，给一个较短 TTL  
- 布隆过滤器做前置拦截  
- 参数校验和非法请求拦截  

**缓存击穿**：  
某个非常热点的 Key 在失效瞬间，大量并发请求同时回源数据库。问题不在“数据不存在”，而在“热点数据失效”。  
解决思路：  

- 互斥锁 / singleflight，让同一时刻只有一个线程去回源  
- 热点 Key 逻辑过期 + 异步刷新  
- 对超热点 Key 做永不过期 + 后台更新  

**缓存雪崩**：  
大量 Key 在同一时间失效，或者 Redis 整体挂了，导致流量整体压到下游。  
解决思路：  

- TTL 加随机值，错开失效时间  
- 分层缓存，本地缓存 + Redis  
- 限流、降级、熔断  
- 缓存预热和高可用部署  

这题如果你能再补一句，会更像工程思维：  
“这三类问题本质上分别对应不存在数据、单热点失效、整体缓存能力下降。解决方案也应该分别针对‘非法流量、并发回源、整体保护下游’。”  

**可能追问：**  
- 追问：空值缓存有什么副作用？  
  十五秒简答：会占一点额外内存，而且如果数据库后来补了数据，在空值 TTL 结束前会有短暂不一致。

- 追问：布隆过滤器有问题吗？  
  十五秒简答：它有误判，可能把不存在判断成存在，但不会把存在判断成不存在，适合做前置筛选而不是最终真相。

- 追问：热点 Key 一定要加分布式锁吗？  
  十五秒简答：不一定。单机内 singleflight、本地互斥、逻辑过期都可以；是否用分布式锁取决于部署方式和并发规模。

## 怎么保证 Redis 和数据库的一致性？

**面经热度：极高频**

**15秒简答：**  
高频标准回答是“更新数据库，再删除缓存”，也就是 cache aside。因为双写很容易乱序。更进一步的工程补充是：删除失败要重试，必要时用延迟双删、MQ、binlog 订阅做最终一致性。

**3分钟详答：**  
这是最能看出后端工程意识的 Redis 题。很多同学会背“先删缓存再更新数据库”或“先更新数据库再删缓存”，但说不出为什么。

最常见推荐策略是：**先更新数据库，再删除缓存**。  
原因是数据库一般被当作最终真相。你先把 DB 改掉，再把旧缓存删掉，后续读取会重新回源并写入新值。  

为什么不推荐“更新数据库 + 更新缓存”双写？  
因为双写有两个问题：  
- 两次操作不是原子，容易乱序  
- 缓存更新可能失败，或者并发覆盖成旧值  

为什么“先删缓存，再更新数据库”也有问题？  
因为在删缓存和更新数据库之间，可能有并发读请求进来，把旧数据从数据库读出来重新写进缓存，造成脏缓存。  

所以实际工程里常见的是：  
- 先更新数据库  
- 再删除缓存  
- 如果删缓存失败，引入重试机制  
- 对一致性要求更高的场景，用 MQ、异步补偿、订阅 binlog 刷缓存  
- 有些场景会加“延迟双删”，即删缓存 -> 更新 DB -> 延迟再删一次缓存，用来降低并发读写窗口的问题  

面试里真正加分的是你能说边界：  
Redis 和 DB 很难做到严格强一致，大多数互联网业务做的是**最终一致性**。如果面试官问“强一致怎么做”，你就可以说这通常不靠 Redis 缓存层自己解决，而要靠事务边界、串行化、库存扣减模型或更严格的架构来保证。

**可能追问：**  
- 追问：为什么推荐“更新 DB，再删缓存”？  
  十五秒简答：因为 DB 是最终真相，删缓存比改缓存更简单、更少出错，后续读请求会自然回源构建新缓存。

- 追问：删除缓存失败怎么办？  
  十五秒简答：要有补偿机制，比如重试、异步消息、订阅 binlog 或定时扫描修复，否则一致性只是口头上的。

- 追问：延迟双删是银弹吗？  
  十五秒简答：不是。它只是降低并发窗口造成旧缓存回填的概率，不能从理论上彻底消灭所有竞态。

## HotKey 和 BigKey 分别是什么？怎么处理？

**面经热度：高频**

**15秒简答：**  
HotKey 是访问特别集中，单 Key 扛高并发；

BigKey 是单 Key 体量特别大，读写、删除、迁移都重。HotKey 重点是分流和本地化，BigKey 重点是拆分和避免重操作。

**3分钟详答：**  
这题很工程化，面试官经常会结合“线上 Redis 突然打满”来问。

**HotKey**：  
某个 Key 被极高频访问，比如首页配置、爆款商品库存、热门帖子详情。问题在于请求集中到单个 Key，会导致单实例压力陡增，严重时会把网络和 CPU 打满。  
常见处理：  
- 本地缓存 / 多级缓存  
- 热点 Key 永不过期，后台异步刷新  
- 把读请求打散到副本或前置层  
- 对热点接口限流  
- 对热点数据提前预热  

**BigKey**：  
指一个 Key 里的 value 特别大，比如一个 List 里几十万条元素、一个 Hash 里塞太多字段、一个 String 很大。问题在于读取、删除、迁移、复制时都很重，还容易阻塞事件循环。  
常见处理：  
- 设计上拆 Key，而不是一个 Key 塞全量  
- 避免一次性 `SMEMBERS`、`LRANGE 0 -1` 这种全量拉取  
- 用渐进式删除、异步删除、分批删除  
- 对集合做分片存储  
- 提前用工具识别大 Key，而不是出事故后再看  

这题的高分点在于你能说出二者本质区别：  
HotKey 是“访问集中”，BigKey 是“单 Key 太重”；一个 Key 可以同时又热又大，那是最危险的。

**可能追问：**  
- 追问：BigKey 为什么危险？  
  十五秒简答：因为读写、删除、复制都可能很重，容易阻塞主线程，还会放大网络、持久化和主从同步成本。

- 追问：HotKey 为什么加本地缓存？  
  十五秒简答：因为目标是把大量重复读挡在应用侧，减少所有请求都穿到 Redis 同一个 Key 上。

- 追问：删除 BigKey 为什么要小心？  
  十五秒简答：因为一次性删除可能触发很重的内存释放和阻塞，线上更适合渐进式或异步删除。

## 为什么线上不建议用 `KEYS`？那应该怎么遍历 Key？

**15秒简答：**  
`KEYS` 会扫描整个 keyspace，复杂度 O(N)，大库上容易阻塞 Redis。线上通常用 `SCAN` 增量遍历，或者提前维护索引集合，别把 Redis 当关系型数据库那样到处模糊查。

**3分钟详答：**  
这题看起来小，但特别能体现你是否有“线上意识”。

`KEYS pattern` 的问题是它会做全量扫描。Redis 是单线程命令执行模型，扫描期间会占用主线程，数据量一大就很危险。开发环境、小数据调试可以偶尔用，生产环境不能当常规业务命令。  

更合适的是 `SCAN`。  
`SCAN` 是增量遍历，不会像 `KEYS` 那样一次性把整个库堵住。它的特点是：  
- 分批扫描  
- 可以带游标逐步遍历  
- 更适合线上  
但它也不是“绝对精确的一次性快照”。在遍历过程中如果数据在变化，可能会有重复返回，所以业务代码要能容忍或去重。  

更进一步，真正工程化的做法不是“临时扫描 Redis”，而是：  
- 设计好 Key 命名空间  
- 提前维护集合索引  
- 业务查询时知道自己该查哪些 Key  
Redis 适合按 Key 访问，不适合拿它当全表扫描型数据库。

**可能追问：**  
- 追问：`SCAN` 有什么坑？  
  十五秒简答：它是增量遍历，不保证遍历过程中的强一致快照，数据变化时可能重复返回，所以业务上要能容忍重复。

- 追问：为什么说 Redis 不适合模糊查询？  
  十五秒简答：因为 Redis 核心设计就是按 Key 高效访问，不是为复杂条件检索和全量扫描优化的。

- 追问：除了 `SCAN` 还有什么更好的办法？  
  十五秒简答：提前维护索引集合或业务分桶，尽量让查询从一开始就能定位范围，而不是临时全库扫。

## Pipeline 是什么？什么时候用？有什么代价？

**面经热度：高频**

**15秒简答：**  
Pipeline 是把多条命令批量发给 Redis，不必每条都等响应后再发下一条，主要是降低网络往返 RTT。适合批量读写，但不是事务，不保证语义上的原子性。

**3分钟详答：**  
这题很容易被误答成“pipeline 就是批量执行”。更准确地说，它是**批量发送命令，减少 request-response 往返次数**。  

Redis 单条命令处理很快，但如果应用层一次要发 100 条命令，而每条都要等响应回来再发下一条，真正浪费的是网络往返时间。Pipeline 的价值就是把这 100 条先一口气发过去，再统一读回结果。  

适用场景：  
- 批量写缓存  
- 批量读取多个 Key  
- 初始化数据、预热缓存  
- 批量统计场景  

但注意几个边界：  
- Pipeline 不是事务，不保证多条命令在语义上“要么都成功要么都失败”  
- Pipeline 太大时，会占用更多客户端和服务端缓冲区内存  
- 单次塞入过多命令，可能反而让单次响应变重，影响延迟抖动  
- 如果业务依赖上一条命令结果决定下一条，pipeline 就不适合  

这题如果你补一句“Redis 的瓶颈常常是 RTT 而不是命令本身”，会很加分。

**可能追问：**  
- 追问：Pipeline 和事务有什么区别？  
  十五秒简答：Pipeline 解决的是网络往返效率，事务解决的是命令顺序执行和隔离；两者目标完全不同。

- 追问：Pipeline 一定越大越好吗？  
  十五秒简答：不是。太大可能占内存、放大响应时延，通常要按业务请求规模控制批次。

- 追问：批量查多个 Key，用什么命令还是 pipeline？  
  十五秒简答：固定多 Key 可以先考虑 `MGET/MSET`；如果命令不统一或数量动态，pipeline 更灵活。

# 持久化、过期与事务脚本

## Redis 如何持久化？RDB、AOF、混合持久化怎么选？

**面经热度：极高频**

**15秒简答：**  
RDB 是定时快照，恢复快、文件紧凑，但可能丢最近一段数据；AOF 是追加写命令，数据更完整，但文件更大、恢复更慢；混合持久化想同时兼顾恢复效率和数据完整性。纯缓存场景甚至可以不持久化。

**3分钟详答：**  
Redis 默认是内存数据库，但并不代表它不能落盘。面试里要先把三种思路讲清楚：

**RDB**：  
某个时刻把内存数据做成快照写到磁盘。  
优点：文件紧凑、适合备份、恢复通常更快。  
缺点：两次快照之间如果实例宕机，最近修改可能丢。  

**AOF**：  
把写命令按追加日志方式记录下来，重启时重放命令恢复数据。  
优点：理论上数据更完整，可通过 fsync 策略在性能和安全之间权衡。  
缺点：文件可能更大，恢复慢于 RDB。  

**混合持久化**：  
可以理解为把 RDB 的恢复效率和 AOF 的增量记录结合起来，是更偏工程化的折中方案。  

怎么选？  
- 纯缓存：可以不持久化，或者只做轻量持久化  
- 更重视恢复速度：偏向 RDB  
- 更重视数据完整性：偏向 AOF  
- 线上常见是 AOF + 合理 fsync，或者混合持久化  

面试里别把 Redis 说成“持久化了就和 MySQL 一样安全”。  
Redis 的持久化是为了提升恢复能力，不代表它天然适合承担强事务、强持久化的主存储职责。

**可能追问：**  
- 追问：RDB 和 AOF 谁恢复更快？  
  十五秒简答：通常 RDB 更快，因为直接把快照载入；AOF 需要重放写命令。

- 追问：AOF 为什么更安全？  
  十五秒简答：因为它记录的是写操作日志，理论上能保留更接近故障前的数据，具体还要看 fsync 策略。

- 追问：什么场景可以不持久化？  
  十五秒简答：纯缓存场景可以不持久化，因为 Redis 中的数据只是数据库副本，丢了能重新构建。

## Redis 的过期删除策略和内存淘汰策略是什么？

**面经热度：高频**

**15秒简答：**  
过期删除解决“Key 到期了怎么删”，常见是惰性删除 + 定期/主动抽样删除；

内存淘汰解决“内存满了删谁”，典型策略有 LRU、LFU、随机、只淘汰带 TTL 的 Key 或淘汰全部 Key。

**3分钟详答：**  
这题两个概念很容易混：  
- 过期删除：Key 设置了 TTL，到时间后如何被清理  
- 内存淘汰：内存到上限了，为了继续写入，应该淘汰哪些 Key  

**过期删除**：  
Redis 不是给每个 Key 都起一个定时器，那样成本太高。更常见的做法是两类结合：  
- 惰性删除：访问某个 Key 时发现它过期了，就顺手删掉  
- 主动删除：后台定期抽样检查一些带 TTL 的 Key，把过期的清掉  

这样兼顾成本和效果。  

**内存淘汰**：  
当设置了 `maxmemory` 后，Redis 需要决定删谁。常见策略包括：  
- 不淘汰，直接报错  
- 只从设置了过期时间的 Key 里淘汰  
- 从所有 Key 里淘汰  
- 淘汰依据可以是 LRU、LFU 或随机  

怎么理解？  
如果 Redis 就是纯缓存，那通常会更愿意设置淘汰策略；  
如果 Redis 还承担重要状态存储，就要非常谨慎，不能随便让系统在内存满时“自动删”。  

面试时加一句特别好：  
“过期删除关注的是时间维度，内存淘汰关注的是容量维度，它们解决的不是同一个问题。”

**可能追问：**  
- 追问：惰性删除有什么问题？  
  十五秒简答：如果一个过期 Key 一直不被访问，就可能长期占内存，所以还需要主动抽样删除配合。

- 追问：LRU 和 LFU 怎么选？  
  十五秒简答：数据访问有明显长期热度差异时 LFU 更合适；如果更看重最近访问特征，LRU 更直观。

- 追问：为什么很多缓存 Key 要带 TTL？  
  十五秒简答：为了控制脏数据生命周期、避免缓存永久堆积，同时也给淘汰和更新提供边界。

## Redis 事务是怎么实现的？有原子性和回滚吗？

**面经热度：高频**

**15秒简答：**  
Redis 事务基于 `MULTI/EXEC/WATCH`。它能保证事务内命令串行执行、不会被别的客户端插到中间，但它不是关系型数据库那种完整 ACID 事务，也不支持传统意义上的回滚。

**3分钟详答：**  
这题的难点在于很多同学会把 Redis 事务和 MySQL 事务混为一谈。

Redis 的事务核心命令有：  
- `MULTI`：开启事务  
- 后续命令先入队，不立即执行  
- `EXEC`：一次性按顺序执行  
- `DISCARD`：放弃事务  
- `WATCH`：乐观锁，监视某些 Key 是否被改动  

Redis 事务能保证什么？  
最重要的是：**事务中的命令会按顺序串行执行，不会被其他客户端的命令插进来。**  
这就是它的隔离性来源。  

但它不像关系型数据库那样支持复杂回滚。  
如果 `EXEC` 之前连接断了，那整个事务不会执行；  
如果 `EXEC` 之后某条命令运行时报错，不会自动回滚之前已经执行成功的命令。  

所以面试里稳妥的表达是：  
“Redis 事务更像批量顺序执行 + 可选乐观锁，而不是完整的关系型 ACID 事务模型。”  

如果面试官再追问“那为什么还要用事务”，你可以说：  
它适合多条命令需要打包顺序执行的场景，或者配合 `WATCH` 做 CAS 风格更新。再复杂一点的原子逻辑，很多时候会改用 Lua 脚本。

**可能追问：**  
- 追问：Redis 事务支持回滚吗？  
  十五秒简答：不支持传统意义上的回滚；执行中某条命令报错，不会撤销之前已成功执行的命令。

- 追问：`WATCH` 是干什么的？  
  十五秒简答：它相当于乐观锁，监视某些 Key；如果在事务提交前这些 Key 被别人改了，事务可以放弃执行。

- 追问：为什么说 Redis 事务不像 MySQL 事务？  
  十五秒简答：因为它没有 MVCC、隔离级别、undo log 那套完整机制，更像“排队顺序执行 + 乐观锁”。

## 为什么很多 Redis 场景会用 Lua 脚本？它和事务有什么区别？

**面经热度：高频**

**15秒简答：**  
Lua 脚本常用来把“读 + 判断 + 写”放在 Redis 端一次原子执行，避免多次网络往返和竞态。它和事务的区别是：脚本能把条件判断逻辑一起放进去，很多原子场景比 `MULTI/EXEC` 更自然。

**3分钟详答：**  
Lua 脚本是 Redis 面试里很加分的点，因为它经常出现在分布式锁、库存扣减、幂等控制这些场景里。

为什么要用 Lua？  
因为很多业务操作不是简单“连发几条命令”，而是：  

1. 先读某个值  
2. 判断是否满足条件  
3. 满足就更新，不满足就返回失败  

如果你在客户端分三步做，就会有两个问题：  
- 网络往返多  
- 中间可能被其他请求插队，导致竞态条件  

Lua 的优势就是：  
把逻辑放到 Redis 服务器端执行，整个脚本在执行期间是原子的。这样能同时解决性能和一致性问题。  

典型例子：  
- 释放分布式锁时，先判断 value 是否等于自己的 token，再删除  
- 扣库存时，先判断库存是否足够，再扣减  
- 限流时，读计数、判断阈值、设置过期，一次完成  

Lua 和事务的区别：  
- 事务更像“把多条命令排队顺序执行”  
- Lua 可以把判断逻辑和多命令操作封装成一个原子单元  
所以很多“读改写”的原子场景，Lua 更好用。  

**可能追问：**  
- 追问：Lua 为什么适合释放分布式锁？  
  十五秒简答：因为要把“先校验锁 owner，再删除锁”做成一个原子动作，避免误删别人的锁。

- 追问：Lua 有什么风险？  
  十五秒简答：脚本执行期间会阻塞 Redis 主线程，脚本太重、跑太久，会影响整个实例延迟。

- 追问：Lua 和 pipeline 谁更快？  
  十五秒简答：解决的问题不同。pipeline 主要减少 RTT，Lua 主要解决原子逻辑和竞态；有时 Lua 也顺带减少网络往返。

# 高可用与分布式

## Redis 主从复制是怎么工作的？

**面经热度：高频**

**15秒简答：**  
Redis 主从复制本质是主节点把数据变更同步给从节点。正常情况下从节点持续接收命令流；断线重连后尽量做部分重同步，不行再全量同步。默认是异步复制，所以读写分离下要接受短暂不一致。

**3分钟详答：**  
主从复制是 Redis 高可用的基础。没有复制，就谈不上副本、读写分离和故障切换。  

基本思路是：  
- 主节点负责写  
- 从节点复制主节点数据  
- 读请求可以分担到从节点  

复制分两种主要情况：

**正常连接状态**：  
主节点持续把写命令流发给从节点，从节点按顺序应用。  

**断线重连状态**：  
如果中间断开了，从节点重连后会尽量做**部分重同步**，只补丢失的那一段；  
如果条件不满足，就做**全量同步**，主节点生成快照给从节点，再继续同步增量命令。  

这里的关键点是：  
Redis 默认复制是**异步**的。也就是说主节点处理完写请求后，不会每次都等所有从节点确认再返回。所以读写分离场景下，从节点读到旧数据是可能的。  

面试里加一句会很像工程经验：  
“主从复制提高的是可用性和读扩展能力，不是强一致能力。对一致性敏感的读，还是要谨慎从从节点读取。”

**可能追问：**  
- 追问：主从复制能保证强一致吗？  
  十五秒简答：不能。默认是异步复制，所以从节点可能短暂落后于主节点。

- 追问：什么是全量同步？  
  十五秒简答：就是主节点把当前数据快照发给从节点，再继续同步增量命令，成本比部分重同步高很多。

- 追问：读写分离有什么坑？  
  十五秒简答：最大的问题是复制延迟导致读到旧数据，所以强一致读不能盲目打到从节点。

## 哨兵和集群有什么区别？

**面经热度：高频**

**15秒简答：**  
Sentinel 主要解决非分片 Redis 的高可用和自动故障转移；

集群 同时解决分片扩容和高可用。前者更像“主从 + 哨兵”，后者是“分布式分片集群”。

**3分钟详答：**  
这题经常和“主从复制”连着问。你可以这样理解：

**Sentinel**：  
适合单主多从架构。它本身不存业务数据，主要负责：  

- 监控主从节点  
- 发现主节点故障  
- 自动选举新主  
- 通知客户端新的主节点是谁  

所以 Sentinel 解决的是“**高可用**”，不是“水平分片”。  

**集群**：  
Redis 集群 既考虑高可用，也考虑数据分片。它把整个 keyspace 划分成多个槽位，分布到多个主节点上，每个主节点再有从节点做副本和故障转移。  
所以 集群 解决的是“**高可用 + 横向扩展**”。  

怎么选？  
- 数据量不大，只想做高可用：主从 + Sentinel  
- 数据量和并发都上来，需要分片扩容：集群  

这题的高分点是补一句使用代价：  
集群 架构更复杂，多 Key 操作、事务、Lua 脚本等在跨分片时会有约束，不是“上了集群就更先进”这么简单。

**可能追问：**  
- 追问：Sentinel 会存业务数据吗？  
  十五秒简答：不会。Sentinel 是监控和故障转移组件，不是业务数据节点。

- 追问：集群 主要多解决了什么问题？  
  十五秒简答：除了高可用，它还解决了数据分片和横向扩容问题。

- 追问：为什么说 集群 更复杂？  
  十五秒简答：因为它引入了分片、槽位迁移、跨槽限制、客户端路由等问题，使用和运维成本都更高。

## Redis 集群是怎么分片和保证高可用的？有什么限制？

**面经热度：高频**

**15秒简答：**  
Redis 集群 通过 16384 个 hash slot 做数据分片，Key 经过哈希映射到槽，再由各主节点负责部分槽位。

每个主节点可以挂从节点，主挂了由从提升。限制主要在跨槽多 Key 操作、事务、Lua 和复杂批量场景。

**3分钟详答：**  
集群 是 Redis 横向扩展的核心方案。面试时最好讲三层：

**第一层：怎么分片**  
Redis 集群 不是简单按节点数取模，而是把整个空间划成 16384 个槽位。Key 先映射到某个 slot，slot 再归属某个主节点。这样做的好处是迁移时搬的是槽，不是粗暴整库重分片。  

**第二层：怎么高可用**  
每个主节点通常会有从节点做副本。主节点故障后，集群通过节点间通信和投票，把某个从节点提升为新主，继续接管对应槽位。  

**第三层：有什么限制**  
这才是面试的重点。集群 不是“原来命令都原样可用”：  
- 多 Key 操作要求相关 Key 落到同一个槽  
- 事务和 Lua 涉及多 Key 时也会受跨槽限制  
- 客户端需要处理重定向和槽位路由  
- 分片带来扩展能力，但也带来更多一致性和运维复杂度  

所以你可以总结成一句：  
“集群 用槽位解决分片，用主从副本解决高可用，但代价是跨槽操作能力下降，业务设计要更注意 Key 规划。”

**可能追问：**  
- 追问：为什么是 16384 个槽？  
  十五秒简答：你可以理解为 Redis 选了一个足够细、又不会让元数据过重的槽位数量，方便迁移和路由；面试里不用死磕它的数学来源。

- 追问：什么是跨槽问题？  
  十五秒简答：就是多个 Key 不在同一个 hash slot，很多多 Key 命令或原子操作就没法直接执行。

- 追问：怎么让多个相关 Key 落到同一槽？  
  十五秒简答：可以用 hash tag，让一组 Key 的路由部分保持一致，方便多 Key 操作。

## Redis 分布式锁怎么实现？有哪些坑？

**面经热度：极高频**

**15秒简答：**  
基础实现是 `SET lock_key unique_value NX PX ttl`，表示不存在才加锁并设置过期时间。释放锁不能直接 `DEL`，要用 Lua 先校验 value 再删，避免误删。常见坑有锁过期、业务执行超时、误删他人锁、主从切换一致性问题。

**3分钟详答：**  
这题非常高频，而且面试官通常会一路深挖。回答最好分三步。

**第一步：最基础实现**  
加锁要满足两个条件：  

- 只有不存在时才能设置成功  
- 锁必须有过期时间，防止死锁  

所以经典写法是：  
`SET key value NX PX 30000`  
其中 value 要是唯一标识，比如请求 ID、线程 ID 组合。  

**第二步：正确释放锁**  
很多人会说“加锁用 setnx，解锁用 del”，这其实有坑。  
因为如果线程 A 的锁过期了，线程 B 拿到了新锁，这时 A 再 `DEL`，就会把 B 的锁删掉。  
正确做法是：**用 Lua 脚本把“比较 value 是否是自己 + 删除”做成原子操作。**

**第三步：常见坑**  
- 业务执行时间超过锁 TTL，锁自动过期  
- 释放锁时误删别人的锁  
- 客户端宕机、网络抖动导致续期失败  
- 主从切换时，锁写入还没同步完成，可能带来可见性问题  
- 可重入、公平性、看门狗续期这些问题，手写通常都很麻烦  

所以实际工程里，很多团队会说：  
“面试里我知道底层原理，但生产上通常更愿意用成熟组件，比如 Redisson，而不是手写不完整版本。”

再进一步，如果面试官追问“Redis 分布式锁是不是绝对可靠”，你可以回答：  
它适合很多互联网业务做互斥控制，但如果是强一致、强安全、跨机房极端可靠要求，不能把 Redis 锁神化。

**可能追问：**  
- 追问：为什么释放锁不能直接 `DEL`？  
  十五秒简答：因为锁可能已经过期并被别人重新拿到，直接 `DEL` 会误删别人的锁，所以要先校验 value 再删除。

- 追问：锁过期了业务还没执行完怎么办？  
  十五秒简答：要么把 TTL 评估得足够保守，要么做续期机制；工程里这也是很多人用 Redisson 的原因之一。

- 追问：单机 Redis 锁和 Redlock 怎么看？  
  十五秒简答：面试里知道 Redlock 思路即可，但生产是否采用要看一致性要求和架构，不要把它说成绝对银弹。

# 进阶应用与工程问题

## Redis 能做消息队列吗？List、Pub/Sub、Stream 怎么选？

**面经热度：高频**

**15秒简答：**  
能做，但要分场景。List 适合简单队列，Pub/Sub 适合实时广播但可能丢消息，Stream 更适合可积压、可消费组处理的消息流。

真要做高可靠 MQ，通常还是优先考虑 Kafka、RocketMQ 这类专业中间件。

**3分钟详答：**  
这题的核心不是“能不能”，而是“边界在哪”。

**List**：  
最简单，`LPUSH/RPOP` 或阻塞式命令就能做一个队列。适合轻量级、简单生产消费。  
缺点是消费确认、重试、积压管理这些能力比较弱。  

**Pub/Sub**：  
更像广播。发布了就推给当前在线订阅者，适合聊天室、实时通知、配置广播。  
缺点是如果订阅者不在线或处理失败，消息通常就没了。它更偏“实时通知”，不偏“可靠消息”。  

**Stream**：  
Redis 5.0 引入，更像日志流，支持消费组、待处理消息、重复认领等能力。相比 Pub/Sub，它更适合对消息可靠性和堆积处理有要求的场景。  

但你一定要补一句边界：  
Redis 能做轻量消息队列，但如果业务对吞吐、堆积、顺序性、回溯、死信、投递保障要求很高，还是更推荐专业 MQ。  

面试里最稳的一句话是：  
“Redis 做 MQ 可以，但我会看场景选：实时广播选 Pub/Sub，轻量队列选 List，更完整一点的消息流选 Stream；高可靠核心链路优先专业 MQ。”

**可能追问：**  
- 追问：Pub/Sub 最大问题是什么？  
  十五秒简答：更偏实时广播，订阅者断线或处理失败时消息可能丢，不适合强可靠消息链路。

- 追问：List 做队列有什么问题？  
  十五秒简答：简单好用，但确认、重试、积压管理能力有限，业务复杂后容易自己补很多轮子。

- 追问：Stream 为什么更适合消息流？  
  十五秒简答：因为它支持消费组、待处理消息管理和更像日志的消费模型，比 Pub/Sub 更适合可靠消费场景。

## 怎么排查 Redis 变慢、阻塞、内存暴涨这类线上问题？

**面经热度：中高频**

**15秒简答：**  
先分是命令慢、网络慢、实例阻塞还是内存问题。常见排查顺序：看慢日志、看大 Key/热 Key、看命令统计和延迟、看 CPU/内存/连接数、看持久化和主从复制状态，再回到具体业务命令。

**3分钟详答：**  
这题很像真实工作，不是背概念，而是看你会不会定位问题。一个比较好的回答框架是“先分层，再定位”。

**第一层：先判断是不是 Redis 自己慢**  
不要一上来就怪 Redis。先看调用链耗时，分清楚是应用线程卡住、网络 RTT 高、还是 Redis 命令真的执行慢。  

**第二层：看命令和 Key**  
- 有没有慢命令，比如全量遍历、大范围聚合  
- 有没有 `KEYS`、大 `LRANGE`、大 `SMEMBERS` 这种危险操作  
- 有没有 BigKey、HotKey  
这些通常是最常见根因。  

**第三层：看实例状态**  
- CPU 是否打满  
- 内存是否接近上限  
- 连接数是否异常  
- 持久化是否在高频 fsync 或 fork  
- 主从复制是否积压、网络是否抖动  

**第四层：看工具和指标**  
常用会提到：  
- `SLOWLOG` 看慢命令  
- `INFO` 看内存、复制、命令统计、持久化信息  
- `LATENCY` 相关命令看延迟事件  
- `redis-cli --bigkeys / --memkeys` 查大 Key  
- 监控平台看 QPS、命中率、内存碎片率、主从延迟  

最后落到处理：  
- 慢命令优化成小批量  
- BigKey 拆分  
- HotKey 做本地缓存或分流  
- 调整持久化策略  
- 做容量扩容或分片  

这题如果你答出“先判断是网络还是命令还是资源，再针对性看工具”，面试官会觉得你有线上意识。

**可能追问：**  
- 追问：线上突然延迟抖动，第一反应看什么？  
  十五秒简答：先看调用链分层耗时，再看 Redis 慢日志、CPU、内存、连接数和是否有持久化/fork/大命令。

- 追问：怎么查 BigKey？  
  十五秒简答：可以用 `redis-cli --bigkeys`、`--memkeys`，也结合业务命令和内存使用情况定位。

- 追问：为什么 BigKey 会导致慢？  
  十五秒简答：因为读写、删除、复制、网络传输都变重，单线程实例更容易被拖住。

# 一页速记

## 如果面试只给你 30 秒，你怎么概括 Redis？

**面经热度：压轴总结**

**15秒简答：**  
Redis 是一个以内存为主的数据结构型 Key-Value 系统，擅长高并发、低延迟读写，常用于缓存、计数、排行榜、分布式锁、限流和轻量消息流。面试重点通常围绕五个面：为什么快、数据结构、缓存问题、持久化高可用、分布式锁与一致性。

**3分钟详答：**  
如果面试官最后让你“整体说说 Redis”，你可以按下面这个框架说：

第一，Redis 本质上不是简单的“Map 存字符串”，而是提供多种数据结构，面向高频业务场景做了优化。  
第二，它快的原因不只是“在内存里”，还包括数据结构高效、执行路径短、命令模型简单、能通过 pipeline 降低 RTT。  
第三，在业务里最常见的用途是做缓存，但真正难点不是 `get/set`，而是缓存穿透、击穿、雪崩、以及与数据库一致性。  
第四，如果 Redis 承担的职责更重，就要考虑持久化、主从复制、Sentinel、集群 这些高可用能力。  
第五，Redis 很适合做一些轻量分布式能力，比如分布式锁、限流、排行榜、滑动窗口，但也要清楚它的边界，不要把它当万能中间件。  

最后一句非常重要：  
“Redis 的面试不是背命令，而是看你是否知道它适合什么、不适合什么，以及线上真正会出什么问题。”

**可能追问：**  
- 追问：Redis 最容易被面试官深挖哪几块？  
  十五秒简答：最常被深挖的是缓存一致性、分布式锁、过期与淘汰、持久化、主从复制和 集群 限制。

- 追问：Redis 最能体现工程水平的题是哪类？  
  十五秒简答：我觉得是缓存一致性、热点/大 Key 处理、分布式锁边界，以及“线上 Redis 变慢怎么排查”这几类。
