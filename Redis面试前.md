## 1. 介绍 Redis 常见的数据结构及其应用场景

- **String**: 底层数据结构是SDS（简单动态字符串），用作缓存、计数器、分布式锁等  
- **Set**: 底层数据结构是哈希表或整数集合，可以利用集合的交集和并集操作，计算出用户与其他用户的共同兴趣点/去重用户的访问记录/抽奖  
- **List**: 底层数据结构是双向链表或压缩列表，可以从两端插入或弹出元素。用作消息队列、历史记录等  
- **Hash**: 底层数据结构是哈希表或压缩列表，是一个包含字段和对应值的字典。用于缓存对象等  
- **ZSet**: 底层数据结构是跳表或压缩列表，每个存储元素相当于有两个值组成的，一个是有序集合的元素值，一个是排序值。可以用于排行榜/实时统计  

## 补充：

- **HyperLogLog**: 概率数据结构，用于估计集合的基数。例如，通过 PFADD 将访问页面的每个用户 ID 添加到 HyperLogLog 中，然后通过 PFCOUNT 获取 UV 值。空间恒定为 12KB，无论你插入 100 个还是 1 亿个元素。

## 2. Redis为什么快

- Redis的大部分操作都在内存中完成，并且采用了高效的数据结构
- Redis采用单线程模型可以避免了多线程之间的竞争，省去了多线程切换带来的时间和性能上的开销，而且也不会导致死锁问题
- Redis采用了I/O多路复用机制处理大量的客户端Socket请求，IO多路复用机制是指一个线程处理多个IO流

---

## 3. 介绍Redis的单线程模型

Redis的所有的客户端请求和操作都由一个主线程来处理。

### 实现原理

1. Redis主线程通过epoll或kqueue（根据操作系统不同选择最佳的I/O多路复用方式）监听多个客户端的请求事件。
2. 当有新的请求到来时，主线程会将其加入事件循环队列，按顺序依次处理这些请求。
3. 每个请求的处理时间非常短，通常是O(1)或O(log N)，因此即便单线程，Redis依然能够在高并发场景下处理数万级的QPS。

### 优缺点

**优点：**  
- 避免了多线程之间的竞争，省去了多线程切换带来的时间和性能上的开销  
- 缺点：在单线程模型下，长时间的阻塞操作会成为Redis的性能瓶颈，因此需要小心大键操作  

### 拓展

实际上，Redis 4.0后除了主线程外，它也有后台线程在处理一些较为缓慢的操作，例如清理脏数据、无连接的释放、大Key的删除等等。

并且Redis在6.0版本后引入了多线程I/O，专门用来加速网络数据的读取和发送。

## 4. 介绍布隆过滤器

布隆过滤器（Bloom Filter）是一种空间效率极高的 **概率型数据结构**，用于**快速判断一个元素是否在一个集合中**。  
存在误判问题：**可能会误认为某个元素在集合中，但实际上它并不在集合中。**

### 实现原理

- **使用多个哈希函数和一个固定大小的位数组**来实现  
- 当向布隆过滤器添加一个元素时，布隆过滤器会使用 k 个不同的哈希函数，将该元素的值分别映射到位数组中的 k 个不同位置，并将这些位置的值设置为 1  
- 当你查询一个元素是否存在于集合中时，布隆过滤器会对该元素使用相同的 k 个哈希函数，检查这些位置的值  
  - 如果所有这些位置的值都是 1，那么元素 **可能存在**（这是一个肯定的答案）。但是，由于哈希碰撞的可能性，这也意味着可能会产生误判  
  - 如果其中某个位置的值是 0，那么元素 **一定不存在**  
  - 布隆过滤器的误判率和哈希函数个数 k、位数组大小 m、插入的元素总量 n 有关  

#### 优点：

- **空间效率高**  
- **查询速度快**：布隆过滤器的查询时间是常数时间复杂度 O(k)，其中 k 是哈希函数的数量  

#### 缺点：

- **误判**：布隆过滤器无法避免误判，即它可能错误地判断一个元素存在  
- **不能删除元素**：标准的布隆过滤器不支持删除操作，因为它无法准确地知道哪个元素设置了位数组中的某个位置  

### 布隆过滤器不允许删除，那怎么解决？

- **法 1**：通过两个布隆过滤器来增强删除操作的支持：其中一个用于存储普通元素，另一个用于存储删除元素的标记（可能有一定误差）  
- **法 2**：使用 **布谷鸟过滤器（Cuckoo Filter）**，它支持删除操作，每个元素映射到两个候选位置，删除时清除其中一个位置的指纹即可。  
- **法 3**：定期重建布隆过滤器

## 5. Redis 使用的过期删除策略

Redis 是可以对 key 设置过期时间的。Redis 使用的过期删除策略是 **定期删除+惰性删除**

- **定期删除**：Redis 每隔一定时间（默认 100 毫秒），随机抽样一部分设置了过期时间的键，检查是否过期，并删除已过期的键  
- **惰性删除**：客户端访问键时，检查该键的过期时间，如果键已过期，则立即删除

## 6. Redis 内存淘汰算法

Redis 内存淘汰算法是在内存达到上限时触发的机制，通过删除部分键来释放内存，防止出现 **OOM (Out of Memory)** 错误。

Redis 的淘汰策略可以分为几大类：

- **noeviction**：不淘汰任何 key，但内存满了写入会抛出错误  
- **allkeys-***：全键淘汰，进一步可以分为 `random/lru/lfu`  
- **volatile-***：过期键淘汰，进一步可以分为 `random/lru/lfu`  

### 实现原理

1. **LRU**：Redis 使用了近似 **LRU** 算法，随机采样检查一小部分键的访问时间，从中选择最近最少使用的键进行淘汰。默认采样数量为 5 个键  

2. **LFU**：在 Redis 的 LFU 策略中，每个键都会维护一个 **访问频次计数器（counter）**，用于记录该键被访问的频率。当内存达到上限时，Redis 会根据访问频次优先淘汰访问频次最低的键。Redis 使用一个 **8-bit 的计数器（值范围为 0~255）** 来记录每个键的访问频次，配合时间戳机制，使计数器更符合实际业务需求：
   - 访问频次越高，计数器增长的概率越小，防止频次爆炸  
   - 随着时间的推移，计数器会自动衰减，减少长期未被访问过的计数值  

总结：LRU 实现简单，对短期数据变化敏感；LFU 能更精准淘汰长期低频数据。

## 7. Redis 如何实现分布式锁

使用 Redis 提供的 `SET key value NX EX seconds` 命令来创建锁：

- **NX** 确保只有在锁不存在时才能成功创建锁，防止其他客户端覆盖锁  
- **EX** 设置锁的过期时间，防止锁永久被占用，导致死锁问题  
- **key value** 保证锁能够区分来自不同客户端的加锁操作  

释放锁时：

- 使用 Lua 脚本来原子性确保只有持有锁的客户端才能释放锁，以防误删其他客户端的锁

## 8. Redis 持久化

当 Redis 重启后，内存中的数据就会丢失，为了保证内存中的数据不会丢失，Redis 实现了数据持久化的机制，这个机制会把数据存储到磁盘，这样在 Redis 重启就能够从磁盘中恢复原有的数据。

Redis 提供两种主要的持久化方式：**RDB（Redis Database）快照和 AOF（Append-Only File）日志**（在新版本中也可以结合使用）。

### AOF

- 每次执行写操作，Redis 主线程会将该命令序列化并追加到 AOF 文件的末尾（appendonly.aof）  
- 刷盘策略默认为 everysec  
- 随着命令不断追加，AOF 文件会变得越来越大，影响恢复速度和磁盘空间。Redis 支持定期 AOF 重写，重写过程在后台子进程完成，不阻塞主线程  

#### 优点：

- AOF 可以做到数据丢失最小化，尤其在 appendfsync everysec 模式下，最多丢失 1 秒的数据  

#### 缺点：

- AOF 文件通常比 RDB 文件大得多，耗时较长  
- 在高并发场景下，AOF 会对磁盘 I/O 产生更大的压力  
- 如果单条写操作过长（如存储大量或复杂数据结构），在刷盘时可能导致主线程短暂阻塞

---

### RDB

- 使用 save（当前进程）或 bgsave（子进程）
- Redis 主线程通过 fork 创建一个子进程，子进程共享主线程的内存数据
- 子进程将内存数据序列化成二进制格式，写入 dump.rdb 文件
- 写入完成后，新的 RDB 文件会替换旧文件，确保数据的一致性

#### 优点：

- 使用 RDB 恢复速度快
- 文件体积更小
- 快照生成由于进程完成，主线程几乎不受影响

#### 缺点：

- RDB 是定期生成快照的持久化方式，如果 Redis 崩溃或异常关闭，可能丢失最近一次快照后的数据（几秒到几分钟不等）
- 生成 RDB 快照时会 fork 一个子进程，内存和 CPU 负载会增加

### 混合持久化

混合持久化工作在 AOF 日志重写过程：
- Redis 首先将当前内存中的数据状态以 RDB 格式写入 AOF 文件
- 在写入 RDB 格式数据之后，Redis 会继续将后续的增量写操作以 AOF 格式追加到文件

#### 优点：文件体积小、恢复速度快、数据丢失少  
#### 缺点：兼容性问题

### 追问 1：RDB 为什么是创建子进程（fork()）而不是子线程？

- 为了实现 **内存隔离**，使用子进程能够保证持久化操作（如 RDB 和 AOF 重写）不会干扰主进程处理客户端请求

### 追问 2：RDB 进程和主进程之间具体是如何共享数据的？

- 主进程复制页表给子进程，因此子进程可以直接访问主进程的内存空间
- 在子进程复制页表时，如果主进程需要修改数据，则使用 **写时复制（CopyOnWrite）** 技术。具体来说，需要写时会创建一个新的数据块并拷贝一份内存（直到发生修改）。这使得子进程可以非常高效地读取和操作 Redis 内存中的数据，而不会影响主进程的数据

## 9. 如何避免缓存击穿

缓存击穿（Cache Breakdown）指的是当某个高并发访问的热点数据在缓存中失效时，大量请求同时访问数据库，导致数据库短时间内承受巨大的压力，可能会引发系统崩溃或严重性能下降。

### 解决方案

- **互斥锁**：可以使用分布式锁（如 Redis 分布式锁）让只有一个请求能够访问数据库，其他请求等待锁释放。
- **主动更新缓存**：在缓存即将失效前，提前重新加载数据，避免大量请求在缓存过期时同时访问数据库。

---

## 10. 如何避免缓存穿透

缓存穿透是指当请求的数据既不在缓存中，也不在数据库中时，直接到达数据库查询，绕过了缓存系统，从而对数据库造成不必要的压力和负担。缓存穿透的影响：
- **增加数据库的负担**：大量无效的请求会让数据库承担不必要的查询负担，尤其在高并发的情况下，数据库压力会迅速增大。
- **影响缓存效果**：无效请求直接查询数据库，绕过缓存机制，降低了缓存的效率。

### 解决方案

- **使用布隆过滤器**：布隆过滤器能够在请求到达缓存之前，通过快速判断请求数据是否存在来避免不必要的数据库查询。
- **限制非法请求**：在 API 入口处判断请求参数是否合理，如果是恶意请求就直接返回错误，避免进一步访问缓存和数据库。
- **设置空值或者默认值**：在缓存中设置一个空值或者默认值（对大量恶意请求会占用很多内存）。

## 11. 如何避免缓存雪崩

缓存雪崩指的是大量缓存数据在同一时间过期或失效，导致大量请求直接打到数据库，瞬间增加数据库压力。

### 产生原因

- **缓存集中过期**：如果大量缓存的数据设置了相同的过期时间，会在某一时刻同时失效  
- **Redis 宕机或重启**：Redis 服务不可用，所有请求直接打到数据库  

### 解决方案

- 设置缓存时，避免相同的过期时间，给每个缓存数据增加一个 **随机的过期时间偏移量**  
- 设置多级缓存架构，当 Redis 失效时，可以短时间内从本地缓存获取部分数据  
- 使用限流和降级机制，限制请求速率，防止数据库被击垮  

### 追问1：有哪些降级和限流机制

- **降级**：暂时关闭一些非核心服务  
- **限流**：滑动窗口、令牌桶、漏斗算法

## 12. 介绍 Redis 主从复制原理

Redis 主从复制是指将一个 Redis 服务器的数据（主节点）复制到其他 Redis 服务器（从节点）。从节点可以处理读请求，实现读写分离和热备份。

### 实现原理

- **全量复制**：从节点发送 `PSYNC ? -1` 请求，主节点执行 `bgsave` 生成 RDB 快照，并缓存期间的新写命令。从节点清空旧数据，加载 RDB，然后执行缓存的命令，完成同步。
- **增量复制**：主从断开重连后，主节点根据从节点上报的复制偏移量（`offset`）和积压缓冲区（`repl_backlog`）中的命令，仅发送缺失的写命令。
- **复制偏移量 + 主节点运行ID**：用于断点续传。

### 追问1：主从复制可能造成数据丢失吗？

- 异步复制下，主节点宕机会导致未发送给从节点的数据丢失。可配合 **半同步复制（如腾讯云Redis）** 或 **哨兵** 的配置减少丢失概率。

### 追问2：主从复制是否阻塞主线程？

- 不阻塞。`bgsave` 和命令传播都由子进程或异步线程完成。

## 13. 哨兵机制是如何实现高可用的？

哨兵（Sentinel）是 Redis 的高可用解决方案，用于监控主从节点，并在主节点故障时自动进行故障转移。

### 实现原理

1. **监控**：哨兵定期向所有节点发送 `PING`，检测健康状态。
2. **主观下线**：单个哨兵认为主节点不可达。
3. **客观下线**：多数哨兵（通过 `quorum` 配置）同意主观下线，触发选主。
4. **选举领头哨兵**：使用 Raft 类似算法从哨兵中选出一个 Leader 执行故障转移。
5. **故障转移**：领头哨兵从从节点中选举新主节点（优先级、复制偏移量、运行ID等），并通知其他从节点复制新主，最后更新客户端配置。

### 追问1：哨兵集群自身如何保持高可用？

- 多个哨兵实例组成集群，使用订阅发布机制协同工作，单个哨兵挂掉不影响整体。

### 追问2：故障转移期间能否继续写入？

- 不能。原主节点下线后，新主节点选举完成前，集群不可写入。一般耗时几十秒。

## 14. Redis Cluster 集群的数据分片和通信机制

Redis Cluster 提供分布式存储能力，自动将数据分散到多个节点。

### 实现原理

- **哈希槽**：整个键空间被划分为 **16384 个哈希槽**，每个节点负责一部分槽。键的槽位计算公式：`CRC16(key) % 16384`。
- **节点通信**：使用 **Gossip 协议** 交换槽信息、节点状态等。每个节点维护集群元数据。
- **重定向**：客户端请求一个键，若节点不负责该键，返回 `MOVED` 或 `ASK`，引导客户端访问正确节点。

### 追问1：集群扩容时如何迁移槽？

- 使用 `CLUSTER SETSLOT` 和 `CLUSTER GETKEYSINSLOT`，逐个迁移键。迁移过程中，访问旧节点的客户端可能收到 `ASK` 重定向。

### 追问2：集群模式下支持多键操作吗？

- 仅当所有键属于同一个哈希槽时才支持（可通过 `hash tag` 强制分配到一个槽）。

## 15. Redis 事务是如何实现的？它真的原子吗？

Redis 事务允许一次性执行多个命令，通过 `MULTI`、`EXEC`、`WATCH`、`DISCARD` 实现。

### 实现原理

- `MULTI` 开启事务，后续命令放入命令队列。
- `EXEC` 顺序执行队列中的所有命令。
- `WATCH` 实现**乐观锁**：监视一个或多个键，若在执行 `EXEC` 前被修改，则事务放弃（返回 `nil`）。

### 是否原子？

- **无回滚机制**：如果事务中某条命令执行失败（如语法错误），其他命令仍会执行（除非是服务器错误如内存不足）。因此 **不保证传统 ACID 原子性**，但保证了 **隔离性**（单线程顺序执行）。

### 追问1：事务中可以用 Lua 脚本代替吗？

- 可以。Lua 脚本默认原子执行，且更灵活，通常推荐使用 Lua 脚本代替事务。
