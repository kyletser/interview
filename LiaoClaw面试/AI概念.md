> 这些概念其实分别对应 AI Agent 开发中的不同层面。
> Vibe Coding 更偏开发方式，强调用 AI 提升实现效率；
> Spec Coding 更偏需求约束，强调先把规格说清楚再实现；
> Harness Engineering 更偏测试验证，强调建立可重复评估的运行框架；
> ReAct 是 Agent 的决策模式，让模型边思考边调用工具；
> MCP 是工具接入协议，用来把外部能力标准化接进 Agent；
> Skills 则是把特定任务场景沉淀成可复用的能力模板。
> 这些东西合起来，才构成一个比较完整的 Agent 工程体系。

------

# Vibe Coding

> 我理解的 Vibe Coding，不是把开发完全交给 AI，而是把 AI 当成高效协作工具。开发者仍然负责需求拆解、架构设计、关键逻辑和质量把关，AI 主要帮助做样板代码生成、重复代码补全、局部重构、测试样例生成和问题排查。
> 它的价值在于能明显提升迭代速度，但风险在于如果缺少人工控制，代码结构、边界处理和可维护性容易失控。

------

# Spec Coding

> Spec Coding 的核心是把需求从“模糊描述”变成“可执行规格”。
> 它会提前定义清楚功能目标、输入输出、边界条件、异常处理、性能要求和验收标准。这样无论是人工开发还是 AI 辅助开发，都能减少理解偏差和返工。
> 对 Agent 项目来说，Spec Coding 很重要，因为 Agent 系统链路长、模块多，如果规格不清楚，很容易出现局部能跑、整体不稳的问题。

------

# Harness Engineering

> Harness Engineering 更关注验证过程的工程化。
> 在 AI 系统里，很多优化如果只靠主观感觉判断，很难知道到底有没有变好。所以需要准备固定测试集、统一执行环境、自动记录指标和日志，用来比较不同模型、不同 prompt、不同工具策略下的效果。
> 它解决的是系统“不可测、不可比、不可回归”的问题。

# ReAct

> ReAct 可以理解成 Reasoning + Acting。
> 模型先根据当前任务做一步推理，决定下一步该调用什么工具；拿到工具返回结果后，再继续推理，再决定后续动作。
> 它的价值在于，不是让模型一次性凭空生成最终答案，而是让它结合外部工具、环境反馈和中间结果，逐步完成复杂任务。
> 对 Agent 来说，这种模式非常关键，因为很多任务不是纯文本生成，而是“理解问题—查信息—执行操作—根据结果再调整”。

------

# MCP

> MCP 的核心价值是解耦。
> 如果每接一个工具都要单独写一套绑定逻辑，Agent 系统会越来越难扩展。
>
> MCP 通过统一协议，把工具能力抽象成标准接口，让 Agent 可以用更一致的方式发现、调用和管理外部能力。
> 这样做的好处是，工具接入更规范，扩展新能力更方便，也更适合多工具协同。

------

# Skills

> Skills 本质上是在做经验复用。
> 因为很多任务场景是重复出现的，比如代码审查、写测试、性能优化、报错排查、需求转实现。
> 如果每次都从零开始让模型自由发挥，效果不稳定，而且成本也高。
> 所以可以把这些高频场景沉淀成模板，里面包含任务目标、执行步骤、约束条件、输出格式，甚至可以自动注入系统 prompt 或绑定对应工具。
> 这样模型在遇到相似任务时，行为会更稳定，结果也更可控。



> 我理解这几个概念其实分别对应 AI Agent 开发里的不同层面。
> Vibe Coding 是开发方式，强调 AI 辅助提升编码和迭代效率；
> Spec Coding 是需求约束方式，强调先把输入输出、边界条件和验收标准说清楚；
> Harness Engineering 是验证方法，强调用固定任务和统一指标去评估系统是否真的变好；
> ReAct 是 Agent 的核心运行模式，让模型可以边推理边调用工具；
> MCP 是工具接入规范，用来解耦 Agent 和外部能力；
> Skills 是把高频任务场景沉淀成可复用模板，提升稳定性。
> 在我的 LiaoClaw 项目里，这几个思路其实都有体现：开发上用了 Vibe Coding 辅助实现，任务链路上用了 ReAct，工具扩展上支持 MCP，场景能力上做了 Skills，同时我也比较重视规格定义和验证闭环，而不是只追求把功能堆出来。

## 你觉得这几个里面哪个最重要？

> 如果是从 Agent 真正可用的角度看，我觉得 ReAct 和工具体系最关键，因为 Agent 的核心不是会说，而是会做。
> 但如果从工程落地角度看，Spec 和 Harness 也很重要，因为没有清晰约束和验证框架，系统很容易越做越乱。

## 你觉得 Vibe Coding 最大的问题是什么？

> 最大的问题是容易让项目表面可跑，但内部结构失控，所以必须有人主导架构和边界控制。

## Skills 和普通 prompt 有什么区别？

> 普通 prompt 更像一次性指令，Skills 更像场景化能力封装，强调可复用、可约束、可组合。

## MCP 和直接写工具调用有什么区别？

> 直接写工具调用更快，但耦合高；MCP 更适合做标准化扩展，后期维护和增加工具更方便。
