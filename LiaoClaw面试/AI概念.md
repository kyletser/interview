> 这些概念其实分别对应 AI Agent 开发中的不同层面。
> Vibe Coding 更偏开发方式，强调用 AI 提升实现效率；
> Spec Coding 更偏需求约束，强调先把规格说清楚再实现；
> Harness Engineering 更偏测试验证，强调建立可重复评估的运行框架；
> ReAct 是 Agent 的决策模式，让模型边思考边调用工具；
> MCP 是工具接入协议，用来把外部能力标准化接进 Agent；
> Skills 则是把特定任务场景沉淀成可复用的能力模板。
> 这些东西合起来，才构成一个比较完整的 Agent 工程体系。

------

# 二、Vibe Coding

## 1. 一句话解释

> Vibe Coding 是一种 AI 辅助开发方式，强调开发者用自然语言和 AI 协作，快速完成代码实现和迭代。

## 2. 面试版回答

> 我理解的 Vibe Coding，不是把开发完全交给 AI，而是把 AI 当成高效协作工具。开发者仍然负责需求拆解、架构设计、关键逻辑和质量把关，AI 主要帮助做样板代码生成、重复代码补全、局部重构、测试样例生成和问题排查。
> 它的价值在于能明显提升迭代速度，但风险在于如果缺少人工控制，代码结构、边界处理和可维护性容易失控。

## 3. 结合你的项目怎么说

> 在 LiaoClaw 里，我采用的是 Vibe Coding 辅助开发方式。比如一些基础工具封装、样板逻辑、测试样例，我会借助 AI 提升效率；但系统架构、Agent 主循环、上下文管理、工具调度和飞书接入这些关键模块，还是我自己主导设计和验收。

------

# 三、Spec Coding

## 1. 一句话解释

> Spec Coding 就是先写清楚规格说明，再按规格实现。

## 2. 面试版回答

> Spec Coding 的核心是把需求从“模糊描述”变成“可执行规格”。
> 它会提前定义清楚功能目标、输入输出、边界条件、异常处理、性能要求和验收标准。这样无论是人工开发还是 AI 辅助开发，都能减少理解偏差和返工。
> 对 Agent 项目来说，Spec Coding 很重要，因为 Agent 系统链路长、模块多，如果规格不清楚，很容易出现局部能跑、整体不稳的问题。

## 3. 结合你的项目怎么说

> 比如在 LiaoClaw 里做危险命令拦截时，我不会只说“拦截危险命令”，而是会先明确输入是什么、哪些命令属于高风险、返回结果结构是什么、是否需要二次确认、误判怎么处理，这种先定义规则再实现的方式，其实就是 Spec Coding。

------

# 四、Harness Engineering

## 1. 一句话解释

> Harness Engineering 就是给模型或 Agent 搭建一套可重复、可验证、可回归的测试评估框架。

## 2. 面试版回答

> Harness Engineering 更关注验证过程的工程化。
> 在 AI 系统里，很多优化如果只靠主观感觉判断，很难知道到底有没有变好。所以需要准备固定测试集、统一执行环境、自动记录指标和日志，用来比较不同模型、不同 prompt、不同工具策略下的效果。
> 它解决的是系统“不可测、不可比、不可回归”的问题。

## 3. 结合你的项目怎么说

> 在 LiaoClaw 里，如果我优化上下文压缩或者工具调用策略，我不能只看某一次回答好不好，而是应该用一批固定任务，比如代码阅读、命令执行、错误恢复这些场景，去统计成功率、耗时、工具调用次数、token 开销。这种验证框架思路就是 Harness Engineering。

------

# 五、ReAct

## 1. 一句话解释

> ReAct 是一种让大模型在“思考”和“行动”之间循环的 Agent 工作模式。

## 2. 面试版回答

> ReAct 可以理解成 Reasoning + Acting。
> 模型先根据当前任务做一步推理，决定下一步该调用什么工具；拿到工具返回结果后，再继续推理，再决定后续动作。
> 它的价值在于，不是让模型一次性凭空生成最终答案，而是让它结合外部工具、环境反馈和中间结果，逐步完成复杂任务。
> 对 Agent 来说，这种模式非常关键，因为很多任务不是纯文本生成，而是“理解问题—查信息—执行操作—根据结果再调整”。

## 3. 结合你的项目怎么说

> 我的 LiaoClaw 里 Agent 主循环就是按 ReAct 思路做的。用户发来任务后，系统不会直接硬答，而是先判断要不要读文件、搜代码、执行命令或者调用其他工具，再根据工具结果继续决策，直到任务结束。

## 4. 面试时再说得通俗一点

> 通俗说，ReAct 就是让 Agent 像人一样，一边想一边做，而不是只坐在那里想。

------

# 六、MCP

## 1. 一句话解释

> MCP 可以理解为把外部工具和能力标准化接入 Agent 的协议或接口规范。

## 2. 面试版回答

> MCP 的核心价值是解耦。
> 如果每接一个工具都要单独写一套绑定逻辑，Agent 系统会越来越难扩展。MCP 通过统一协议，把工具能力抽象成标准接口，让 Agent 可以用更一致的方式发现、调用和管理外部能力。
> 这样做的好处是，工具接入更规范，扩展新能力更方便，也更适合多工具协同。

## 3. 结合你的项目怎么说

> 在 LiaoClaw 里，除了内置工具，我还支持 MCP 扩展。我的理解是，内置工具解决高频核心场景，MCP 让系统具备可扩展能力，这样后面想接新的开发工具、外部服务或者平台能力时，不需要把 Agent 内核大改。

## 4. 面试时更稳一点的说法

> 我会把 MCP 说成“标准化工具接入层”，这样不容易说得太悬，也比较工程化。

------

# 七、Skills

## 1. 一句话解释

> Skills 是把常见任务场景沉淀成可复用的能力模板。

## 2. 面试版回答

> Skills 本质上是在做经验复用。
> 因为很多任务场景是重复出现的，比如代码审查、写测试、性能优化、报错排查、需求转实现。
> 如果每次都从零开始让模型自由发挥，效果不稳定，而且成本也高。
> 所以可以把这些高频场景沉淀成模板，里面包含任务目标、执行步骤、约束条件、输出格式，甚至可以自动注入系统 prompt 或绑定对应工具。
> 这样模型在遇到相似任务时，行为会更稳定，结果也更可控。

## 3. 结合你的项目怎么说

> LiaoClaw 里的 Skills 就是针对不同编程场景做的能力模板，比如代码审查、写测试、性能优化这些任务，不是每次裸调用模型，而是根据场景自动注入相应提示词和执行规范，提高稳定性和可复用性。

------

# 八、这几个词之间的关系

这个部分特别适合面试官追问“它们之间有什么联系”时回答。

你可以这样说：

> 这几个概念其实是从不同维度描述 AI Agent 开发。
> Vibe Coding 解决的是“怎么高效开发”的问题；
> Spec Coding 解决的是“怎么把需求讲清楚”的问题；
> Harness Engineering 解决的是“怎么验证优化是否有效”的问题；
> ReAct 解决的是“Agent 怎么完成复杂任务”的问题；
> MCP 解决的是“工具怎么标准化接入”的问题；
> Skills 解决的是“高频任务经验怎么复用”的问题。
> 它们不是并列竞争关系，而是可以组合在一起，形成一套完整的 Agent 工程方法。

# 九、最适合你项目的整合版回答

这段你可以直接背，适合面试官问你“你对这些概念怎么理解”。

> 我理解这几个概念其实分别对应 AI Agent 开发里的不同层面。
> Vibe Coding 是开发方式，强调 AI 辅助提升编码和迭代效率；
> Spec Coding 是需求约束方式，强调先把输入输出、边界条件和验收标准说清楚；
> Harness Engineering 是验证方法，强调用固定任务和统一指标去评估系统是否真的变好；
> ReAct 是 Agent 的核心运行模式，让模型可以边推理边调用工具；

> 在我的 LiaoClaw 项目里，这几个思路其实都有体现：开发上用了 Vibe Coding 辅助实现，任务链路上用了 ReAct，工具扩展上支持 MCP，场景能力上做了 Skills，同时我也比较重视规格定义和验证闭环，而不是只追求把功能堆出来。

# 十一、面试官可能继续追问的点

## 1. 你觉得这几个里面哪个最重要？

你可以答：

> 如果是从 Agent 真正可用的角度看，我觉得 ReAct 和工具体系最关键，因为 Agent 的核心不是会说，而是会做。
> 但如果从工程落地角度看，Spec 和 Harness 也很重要，因为没有清晰约束和验证框架，系统很容易越做越乱。

## 2. 你觉得 Vibe Coding 最大的问题是什么？

> 最大的问题是容易让项目表面可跑，但内部结构失控，所以必须有人主导架构和边界控制。

## 3. Skills 和普通 prompt 有什么区别？

> 普通 prompt 更像一次性指令，Skills 更像场景化能力封装，强调可复用、可约束、可组合。

## 4. MCP 和直接写工具调用有什么区别？

> 直接写工具调用更快，但耦合高；MCP 更适合做标准化扩展，后期维护和增加工具更方便。

# 十二、给你一个最像面试现场的话术

> 我不会把这些词当成单独的概念去背，而是会把它们放到一个完整 Agent 工程里理解。
> 比如开发时我会用 Vibe Coding 提高效率，但为了避免失控，需要用 Spec Coding 把需求和约束写清楚；
> Agent 执行任务时，用 ReAct 组织推理和工具调用；
> 外部能力接入用 MCP 做标准化；
> 常见任务场景再沉淀成 Skills；
> 最后为了验证系统是否真的优化了，还要靠 Harness Engineering 去做回归和对比测试。
> 我觉得这样理解才更像真正做过项目，而不是只记住几个术语。
