# AI / Agent 面试宝典

## 一、Agent 基础与架构

### 1. 模型和 Agent 的区别是什么？

模型本质上是一个推理引擎，它负责**根据输入生成下一个最合理的输出**；

而 Agent 是围绕模型构建的一套执行系统，除了模型本身，还包括**工具调用、上下文管理、记忆、规划、执行、反馈和安全控制**。
**模型负责想，Agent 负责做**。
一个纯模型只能回答问题，一个 Agent 才能完成多步任务，比如先查文档、再调用接口、再汇总结果、最后给用户回复。

------

### 2. Agent 的核心组件有哪些？

我通常把 Agent 拆成七个核心组件：**模型、指令层、上下文层、规划器、工具层、记忆层、评测与观测层**。
模型负责推理，指令层定义角色和边界，上下文层提供任务所需信息，规划器决定是一步做还是分阶段做，工具层负责和外部系统交互，记忆层解决跨轮和跨会话问题，评测与观测层用来做质量保障、回放和定位问题。

如果是生产级 Agent，我还会补充两层：**权限控制**和**降级兜底**，因为实际线上问题往往不是“答不出来”，而是“答错了还去执行了”。

------

### 3. ReAct 和 Plan-and-Execute 的区别是什么？分别适合什么场景？

ReAct 的核心思想是“边想边做”，模型每一步根据当前观察结果决定下一步动作，所以它适合不确定性强、环境反馈快的任务，比如搜索、调试、排障、网页操作。
Plan-and-Execute 则是“先想清楚，再开始做”，先产出整体计划，再按阶段执行，适合目标明确、链路长、需要稳定性的任务，比如代码重构、批量文档生成、复杂工作流编排。
我的判断标准是：**如果任务过程中信息会不断变化，就偏 ReAct；如果任务结构比较稳定，就偏 Plan-and-Execute。**

------

### 4. 如果一个任务既需要 Plan-and-Execute，又需要 ReAct，怎么设计？

这种场景我会做成**分层闭环**：外层用 Plan-and-Execute 做阶段规划，内层每个阶段用 ReAct 处理执行中的动态反馈。
比如“排查线上问题”这个任务，外层先拆成日志定位、根因分析、修复方案、验证回归四步；但在“日志定位”这一步里，Agent 还是要根据查询结果不断调整搜索关键词和分析方向。
也就是说，**计划决定大的路线，ReAct 负责局部自适应**。这样既保证全局稳定，又保留局部灵活性。

------

### 5. Multi-Agent 是什么？子 Agent 会继承父 Agent 的全部上下文吗？

Multi-Agent 是把不同任务拆给不同角色的 Agent 去并行或协作完成，比如检索 Agent、规划 Agent、代码 Agent、评审 Agent。
在工程上我不建议子 Agent 继承父 Agent 的全部上下文，因为这样会导致上下文膨胀、噪声增加、成本上升，而且容易把无关历史也带进去。
更好的做法是给子 Agent 传**任务目标、必要约束、输入材料和期望输出格式**，让它在隔离上下文里完成任务，再返回摘要。

------

### 6. 为什么不一定直接用现成框架做编排？

现成框架很适合快速验证，但到了生产环境，很多团队会做轻量自研编排，原因主要有四个：**可控性、可观测性、性能成本、和业务适配性**。
现成框架往往抽象层太厚，调试和定位问题不够直观；而且很多业务需要自定义状态机、重试策略、权限模型和审计链路，这些在框架里不一定自然。
所以我的看法不是“框架不好”，而是**原型期优先用框架提速，规模化后按瓶颈做定制化收敛**。

------

## 二、Prompt / Context / Harness

### 7. Prompt 是怎么设计的？一个好的 System Prompt 应该长什么样？



我设计 Prompt 的原则是：**角色明确、目标明确、输入输出明确、边界明确、失败策略明确**。
一个好的 System Prompt 不应该只是“你是一个很聪明的助手”，而应该告诉模型：你是谁、你能做什么、你不能做什么、优先级顺序是什么、输出格式是什么、证据不足时该怎么做。
比如我会写清楚：“先判断是否需要检索；有证据就引用；没证据要明确不确定；禁止编造接口和字段；最终输出 JSON/Markdown 的固定结构。”
本质上，System Prompt 不是“文学创作”，而是**行为约束和决策规范**。

------

### 8. Prompt Engineering、Context Engineering、Harness Engineering 的区别是什么？

**Prompt Engineering** 解决的是“**怎么说**”；
**Context Engineering** 解决的是“**给模型什么信息**”；
**Harness Engineering** 解决的是“**模型在什么运行框架里工作**”。

举例来说，同一个代码 Agent，Prompt 是你告诉它“先读后改、改完跑测试”；Context 是 repo 结构、接口文档、编码规范、历史决策；Harness 则是权限控制、工具接入、超时重试、评测回放、人工确认这些机制。
很多人只盯 Prompt，但真正决定上限的，往往是 **Context 和 Harness**。

------

### 9. 在工程类任务中，什么 Context 最重要？

最重要的不是“越多越好”，而是**最相关、最稳定、最影响决策的信息**。
对工程任务来说，我最优先给四类上下文：**项目结构和模块边界、编码规范和约束、接口/数据结构定义、验证方式和上线规则。**

------

### 10. 上下文过长怎么优化？Summary 怎么做？

我通常做**分层摘要**。
第一层是长期稳定信息，比如项目规范、接口约束、常见坑；第二层是任务状态，比如已经查过什么、排除了什么、下一步要做什么；第三层是证据指针，比如关键文件路径、日志链接、文档锚点。
这样摘要不是把所有内容压成一段自然语言，而是把它变成“事实层 + 状态层 + 指针层”。
这样模型下轮既能快速恢复任务状态，也能顺着指针回到原始证据，避免只剩一句抽象总结。

------

### 11. 怎么缓解压缩导致的失真？

我会尽量用**抽取式压缩优先于生成式压缩**。
也就是说，先保留原文中的关键句、关键字段、关键路径，再做轻量总结，而不是直接让模型“自由概括”。
另外，摘要里要保留证据引用和来源路径，重要结论要能回跳原文验证。
我的经验是，**压缩不是替代证据，而是帮助重新定位证据**。一旦把摘要当成事实源，就很容易失真。

### 12. 大模型幻觉是什么导致的？

幻觉本质上是**模型在信息不足、约束不清或者检索失败**时，仍然**被迫生成一个“看起来合理”的答案**。
常见来源有五类：训练知识和现实脱节、用户问题歧义、检索召回不准、上下文不完整、工具返回错误但模型没识别。
所以幻觉不是只靠一句“不要编造”就能解决，它通常是**检索、上下文、工具链和输出约束共同失效**的结果。

------

### 13. 除了 Prompt，还有什么更好的抗幻觉方案？

第一层是**检索增强**，让模型基于证据说话；

第二层是**结构化输出和 schema 校验**，限制它乱填字段；

第三层是**工具验证**，例如让它先查数据库、查文档、跑测试，再下结论；

第四层是**不确定性治理**，证据不足就输出“不确定”或转人工。
所以我的思路是：**少依赖“提醒模型诚实”，多依赖“让系统具备核验能力”。**

------

### 14. RAG 的混合检索怎么理解？

混合检索一般是 **BM25/关键词检索 + 向量检索 + rerank 重排**。
关键词检索擅长找精确实体，比如订单号、错误码、类名、字段名；向量检索擅长找语义相近但措辞不同的内容；重排负责把真正相关的结果排到前面。
所以混合检索不是“堆技术”，而是因为真实问题既有精确匹配需求，也有语义召回需求。
像 OpenClaw 当前官方文档里对 memory search 的描述，也是 hybrid search：同时结合向量相似度和关键词匹配。 ([OpenClaw](https://docs.openclaw.ai/concepts/memory))

------

### 15. RAG 文档分片策略怎么做？

分片不是按固定字数硬切，我更倾向于**按语义结构切分**。
文档类内容按标题、段落、章节切；代码类内容按函数、类、模块边界切；表格要尽量保留表头和行上下文；FAQ 要让“问题+答案”在同一个 chunk 里。
一般还会保留少量 overlap，并把标题、路径、时间、来源这些 metadata 一起带上。
原则很简单：**让一个 chunk 尽量形成最小可理解单元**。

------

### 16. 如果召回一个片段不足以回答，怎么优化？

我会做三种增强。
第一种是**邻接扩展**，把前后 chunk 一起拿回来；第二种是**父子检索**，先召回细粒度 chunk，再回到上层章节或整页文档；第三种是**多跳检索**，允许当前片段继续指向其他相关文档。
如果文档间有组织关系，我会加引用图谱、目录树或者 hyperlink 跳转，让检索从“找一句话”升级成“找一组相关证据”。
也就是说，复杂知识库里不能只做单段召回，要做**关系化召回**。

------

### 17. RAG 和微调的区别、优缺点？

RAG 适合解决**知识更新快、需要可追溯证据**的问题，比如企业知识库、客服问答、代码检索。
微调更适合解决**输出风格、任务习惯、格式遵循、领域动作模式**的问题，比如固定写作风格、固定 JSON 结构、固定流程执行。
简单说，**RAG 更像“给模型查资料”，微调更像“改模型习惯”**。
如果问题核心是“它不知道最新知识”，优先 RAG；如果问题核心是“它知道但总是做得不稳定”，再考虑微调。

------

### 18. 了解过 Agentic Search 吗？

Agentic Search 和普通检索的区别在于，它不是“查一次就答”，而是**会先搜、再判断是否足够、再改写查询、再继续搜，最后整合证据**。
所以它更像一个有目标的检索闭环，而不是单次召回。
这种方式特别适合开放问题、资料分散、线索不完整的场景，比如竞品研究、复杂排障、跨文档问答。
代价是链路更长、成本更高，所以要配好停止条件和证据阈值。

------

## 四、MCP、Skill、A2A、记忆

### 19. MCP 是什么？和 Function Calling 的区别是什么？

我会这样答：**Function Calling 是模型层面的工具调用能力，MCP 是应用层的标准化连接协议。**
Function Calling 解决的是“模型如何按结构化参数发起调用”；MCP 解决的是“外部数据源和工具如何用统一协议被 AI 应用发现、描述和调用”。
当前 MCP 官方规范里，除了 tools，还包括 resources、prompts 等能力，底层采用 JSON-RPC，并区分 host、client、server 角色，所以它比单纯 function calling 更像一层标准化集成协议。

------

### 20. 本地工具、MCP Tools、Skill 三者的区别是什么？

本地工具是运行环境自带的能力，比如读文件、grep、执行命令；

MCP Tools 是通过协议接入的外部系统能力，比如数据库、工单系统、内部服务；

Skill 则不是“连接器”，而是**把知识、流程和操作手册打包成可复用工作流**。
所以我会把它们理解成三层：
**本地工具 = 基础执行能力**
**MCP = 外部系统接入能力**
**Skill = 业务方法论和最佳实践封装**。

------

### 21. 为什么已经有了 MCP，还要做 Skill？有了 Skill 后 MCP 还有必要吗？

有必要，而且两者最好配合使用。
MCP 解决的是“连上系统”，但**连上系统不等于会高质量地使用系统**；

Skill 解决的是“**如何稳定地按最佳流程使用这些能力**”。

Anthropic 当前官方文档里明确把 **Skill + MCP** 作为典型组合模式：MCP 负责连接数据库或服务，Skill 负责告诉模型表结构、查询模式和执行流程。与此同时，Skill 内容按需加载，长参考材料在未触发时几乎不占上下文。
所以更准确的说法不是“Skill 替代 MCP”，而是**Skill 补上了工作流和领域知识这一层**。

------

### 22. MCP 和 Skill 的 token 消耗怎么看？

原则上，**常驻上下文的东西最贵，按需加载的东西更省**。
以 Claude Code 当前文档为例：CLAUDE.md 是每轮都在的；Skill 默认只在会话开始加载描述，真正内容在触发时再加载；

MCP server 也是启动时先暴露工具名，完整 schema 按需加载；

subagent 则在独立上下文里运行，不直接挤占主会话。文档里还明确提到可以通过 `/mcp` 查看每个 server 的 token 成本。
所以面试里可以总结成一句：**不是看“有没有接入”，而是看“什么内容什么时候进入上下文”。**

------

### 23. A2A 是什么？

A2A 可以理解为 **agent 和 agent 之间的标准化通信协议**。
如果 MCP 更偏“Agent 如何访问工具和数据”，那 A2A 更偏“Agent 如何发现另一个 Agent、知道它擅长什么、再把任务交给它”。
Google 当前公开资料里，A2A 的关键设计之一是 Agent Card，通常通过 `/.well-known/agent-card.json` 暴露能力描述和入口地址。
所以我会说，**MCP 更像 tool integration，A2A 更像 agent integration**。

------

### 24. Claude Code 的记忆架构是什么？上下文等于记忆吗？

上下文不等于记忆。
**上下文是当前会话里模型能看到的内容**；**记忆则是被显式持久化、并能在未来会话重新取回的信息。**

按 Claude Code 当前官方文档，项目级 auto memory 存在 `~/.claude/projects/<project>/memory/` 下，其中 `MEMORY.md` 是索引，启动时只会加载前 200 行或前 25KB，详细 topic 文件按需读取；而 CLAUDE.md 是另一类指令文件，不等同于自动记忆。
所以更准确地说，**上下文是当前工作集，记忆是持久化知识库**。

------

### 25. 为什么 Claude Code 这类工具常用 grep / code search，而不是先上 RAG？

因为代码检索和通用语义检索不是一个问题。
在代码场景里，类名、函数名、字段名、路径、调用链这些信息通常要求**精确命中、可审计、可回跳**，所以 grep、symbol search、LSP 导航往往更直接、更稳定。
Claude Code 官方文档里本身就有 Grep 这类内建工具，也支持通过语言服务器做定义跳转、引用查找、类型信息读取；这类手段在代码库里通常比先做向量化 RAG 更可控。
**代码库优先确定性检索，知识库再考虑语义检索**。

------

### 26. OpenClaw 的记忆机制怎么讲？

按 OpenClaw 当前官方文档，它的记忆是**明文 Markdown 文件落盘**，不是“模型脑子里有隐藏状态”。
核心有三类文件：`MEMORY.md` 负责长期记忆，`memory/YYYY-MM-DD.md` 负责日常运行笔记，`DREAMS.md` 是实验性的 consolidation 结果。它还提供 `memory_search` 和 `memory_get` 两个工具，检索上支持 hybrid search；另外在 compaction 前会触发 automatic memory flush，尽量先把重要上下文写入记忆，避免被压缩丢失。
所以这套设计的核心思路不是“让模型无限记住”，而是**把记忆外部化、结构化、可审计化**。 ([OpenClaw](https://docs.openclaw.ai/concepts/memory))

------

## 五、AI Coding 与工程落地

### 27. 你平时怎么用 AI Coding？有没有方法论？

我一般遵循 **Explore → Plan → Implement → Verify** 四步。

先让它读代码和文档，不允许上来就改；

然后让它给出方案和影响范围；

接着只改最小闭环；最后必须跑测试、lint、类型检查和关键路径验证。
也就是说，我不会把 AI 当“自动写代码机器”，而是当成**会提速的初级到中级工程伙伴**，但最终质量责任还是我来兜底。

------

### 28. 如果用 AI coding 重做一个已有业务项目，怎么给 Agent 部署上下文和约束？

我会先准备一套“最小但关键”的工程上下文包：项目结构、核心业务流程、关键实体关系、接口契约、编码规范、测试命令、上线约束。
然后在执行上做三层约束：第一层要求它先读后改；第二层要求所有改动给出影响面说明；第三层要求所有改动必须有验证动作，比如补测试或跑回归。
这样做的目标不是让 AI 更聪明，而是让它**更不容易乱写**。

------

### 29. 大模型从 0 到 1 很会写，但修改已有项目怎么做？比如把异步下单改成同步下单

我不会直接让它“把异步改同步”，而是先让它做**边界识别**。
先找调用链、队列消费者、幂等策略、超时处理、失败补偿、数据库事务边界，再画出当前时序图；

之后再设计同步方案，看是否需要拆成“同步确认 + 异步补偿”的混合模式。
最后一定要补契约测试和回归测试，因为旧系统改动的风险不在“功能写不出来”，而在“兼容性被破坏了”。

------

### 30. AI 能保证不破坏原有逻辑吗？

AI 可以提高开发效率，但不能替代验证机制。
所以真正靠谱的做法不是问“它会不会出错”，而是建立一套**让错误能被快速发现和拦住的工程机制**，比如测试、静态检查、review、灰度发布、监控告警和快速回滚。
面试里我会明确表达：**AI 是提效工具，不是正确性的担保人。**

------

### 31. 公司内部组件很私有，AI 没见过，这种情况下怎么辅助开发？

这种场景我不会期待模型“提前知道”，而是给它喂接口定义、示例代码、调用文档、常见错误和 CLI 帮助信息。
本质上要把私有组件从“模型没见过的黑盒”变成“当前上下文里可学习的白盒”。

Claude Code 官方最佳实践里也明确提到，Claude 可以通过 CLI 帮助信息去学习它原本不熟悉的工具。
所以我的方法是：**别赌它先验知道，给它足够可推理的局部真相。**

------

### 32. 用 AI 编程出问题了，怎么排查？线上出了 AI 没见过的问题怎么办？

我的原则是，**排查流程不能 agent 化得失去工程纪律**。
先止血，控制影响面；

再看日志、指标、链路追踪和最近变更；

然后做最小复现，区分是代码问题、配置问题、依赖问题还是环境问题；

最后再让 AI 帮我做假设生成、日志归类和可能修复路径对比。
也就是说，AI 很适合辅助“缩小搜索空间”，但不能代替事故指挥和根因判定。

------

### 33. Agent 调工具失败了，一般怎么处理？你做过哪些 self-refine / 自我修正？

我通常把工具失败分成三类：调用失败、返回异常、结果不满足约束。
对应的修正策略是：先重试和参数修复；

再做 schema 校验（数据格式合法性检查）和字段补全判断；最后如果还是不可靠，

就切换工具、降级回答或转人工。

如果是 self-refine，我会让 Agent 先输出“失败原因 + 当前假设 + 下一步修正动作”，而不是盲目重试。
好的自我修正不是多试几次，而是**知道为什么错、改哪一层**。

------

### 34. 上线前怎么保证质量？只是靠 code review 吗？

不够。
我会把质量保障分成四层：**静态质量、功能质量、场景质量、线上质量**。
静态质量包括 lint、类型检查、扫描；

功能质量是单测、集成测试、E2E；

场景质量是典型任务集和回归集；

线上质量则是灰度、监控、告警和回滚。
如果是 Agent，还要有**评测集和离线回放**，不然你根本不知道升级模型、升级工具、升级 prompt 后能力有没有退化。

------

### 36. 在 AI 时代，程序员的核心壁垒是什么？

我觉得不是“会不会写代码”，而是**能不能定义问题、拆解系统、约束质量、处理复杂权衡并对结果负责**。

AI 会把编码门槛往下拉，但不会替你做业务抽象、边界判断、架构权衡、事故处置和组织协同。

所以真正值钱的能力会更偏向：**问题建模、工程判断、领域理解、质量兜底和持续迭代能力。**

------

## 六、系统设计场景题

### 37. 设计一个智能客服 Agent，怎么做？怎么防幻觉？

我会把客服 Agent 设计成“五层架构”：
第一层是**意图识别与路由**，先判断是 FAQ（Frequently Asked Questions）、订单查询、售后、投诉还是转人工；
第二层是**知识与工具层**，FAQ 走知识库检索，订单类问题走业务系统工具查询；
第三层是**回答生成层**，要求必须基于检索证据或工具结果生成；
第四层是**风险控制层**，涉及退款、赔付、政策承诺的回答必须有规则校验和高风险兜底；
第五层是**运营评测层**，持续看命中率、转人工率、满意度、幻觉率。

防幻觉的关键不是“提示它别编”，而是让它遵循一句话：**能查就查，查不到就说不知道，敏感动作必须走系统校验。**

------

### 38. 电商大促流量大、有尖刺，Agent 服务怎么扛？

大促场景我会先做**分层降级**。
高频简单问题走缓存和静态 FAQ；

中等复杂问题走小模型 + 检索；

高风险、长链路问题才走重型 Agent。

系统层面要做限流、熔断、队列削峰、热点缓存、异步化和多级超时控制，尤其不能让所有请求都穿透到大模型和后端核心服务。
本质上，大促不是考“Agent 多聪明”，而是考**系统能不能把贵的链路留给最需要的人**。

------

### 39. 设计一个微信小程序排查问题的 Agent，你会怎么做？

我会按“发布链路 + 运行链路 + 观测链路”来设计。
发布链路看版本、灰度、配置变更、依赖升级；

运行链路看前端报错、接口请求、鉴权、缓存、CDN、环境差异；

观测链路看日志、埋点、告警和用户反馈。
Agent 具体能力上，我会给它接入版本信息、发布记录、日志平台、接口网关、配置中心、监控面板和知识库。
回答上先做故障分类，再给排查路径，最后给“最可能根因 + 验证动作 + 回滚建议”。这样它不是只会“猜问题”，而是能给**有证据的排障闭环**。

------

## 七、模型基础题

### 40. CNN 和 Transformer 的区别是什么？

CNN 更擅长利用局部感受野和参数共享去提取局部模式，所以在图像早期任务里非常强；Transformer 通过 self-attention 直接建模任意位置之间的关系，更适合处理长距离依赖和全局上下文。
如果一句话概括，CNN 偏“局部模式提取”，Transformer 偏“全局关系建模”。
现在很多多模态模型仍然会结合两者思想，只是主流通用大模型大多以 Transformer 为核心。

------

### 41. 为什么位置编码能够体现句子中分词的关系？

因为 self-attention 本身只看 token 内容，不天然知道顺序。
位置编码把“第几个 token”这种位置信息注入表示空间后，模型在计算注意力时，就能同时利用“词本身是什么”和“它在什么位置”。
所以位置编码不是直接“存语法关系”，而是给模型提供一个可以学习相对和绝对顺序关系的坐标系。

------

### 42. 为什么说模型有上下文限制？为什么不把上下文窗口做得无限大？

因为上下文越大，计算、显存、时延和成本都会明显上升，而且噪声也会增加。
不是窗口越大效果就一定越好，很多时候上下文太长反而会让模型注意力分散。
从工程视角看，最重要的不是“能塞多少”，而是**能不能把最相关的信息在最合适的时候塞进去**。

Claude Code 官方最佳实践也明确提到，随着上下文填满，性能会下降，管理上下文是最重要的资源管理之一。

------

### 43. 模型的参数量是什么时候定的？为什么有这么多不同参数量的模型？

参数量在模型架构设计阶段基本就定下来了，后续训练是在这个参数空间里优化权重。
之所以会有不同参数量，是因为实际场景需要在**效果、成本、时延、吞吐和部署条件**之间做平衡。
大模型通常能力更强，但更贵更慢；小模型更便宜、更容易部署，也更适合高并发和边缘场景。

### 44. 手撕一个 Skill 系统，怎么讲设计？

我会先说目标：实现 **Skill 的注册、发现、调用** 三个能力。
目录扫描层遍历 `.agents/skills/*`，读取每个 skill 下的 `yaml` 配置和 `scripts/*.js`；注册层做 schema 校验，把 name、description、tags、入口脚本、参数定义、权限需求写入 registry；发现层支持按名称、标签、描述做匹配；调用层负责参数校验、权限检查、脚本执行、超时控制、结果回传。
再往上一层，我会补充生产能力：日志、监控、失败重试、并发控制、缓存和审计。
这样答会显得你不是只会“写个 demo”，而是知道一个可用系统至少要有哪些工程骨架。

**我对 Agent 的理解不是“让模型多想一点”，而是通过上下文、工具、记忆、评测和系统设计，把不稳定的模型能力变成相对稳定的工程能力。”**
