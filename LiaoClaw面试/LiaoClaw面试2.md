**你先一句话介绍一下 LiaoClaw。**
LiaoClaw 是我仿照pi-mono做的一套可运行编程 Agent 系统，

把统一模型接口、Agent 主循环、工具调用、会话持久化、上下文压缩，以及 Web/CLI/飞书三种入口打通了。

它本质上更像一个 Agent 框架，再往上叠了编程助手这个具体应用。



**你为什么把它拆成四层架构？**

我拆成四层是为了把“**模型调用”“Agent 编排”“具体业务能力”“外部接入渠道**”解耦。

底层统一 LLM 接口只负责屏蔽不同厂商协议差异；

中间的 agent_core 只负责主循环和工具协议；

coding_agent 负责会话、工具、提示词和扩展；

最上层 IM 桥接只负责飞书消息接入和回复。这样每层都能独立演进，不会因为接一个新平台就改乱核心逻辑。



**统一模型接口你具体是怎么做的？**
我的 AI 统一接口层采用**注册中心设计**，将不同厂商、不同协议的流式调用实现统一注册到注册表。

运行时根据厂商 provider 和模型 model_id 查找对应模型，再分发到对应协议实现执行调用。

新增模型或新协议时只需新增配置并注册，无需修改 Agent 业务主逻辑，符合开闭原则，实现底层模型与上层 Agent 编排解耦。



**Agent 主循环是怎么工作的？**（思考—用工具—再思考）
本质上就是一个可执行的 ReAct loop。

用户消息进来之后，**先进入上下文，再调用 LLM**；

如果 **LLM 回复里带工具调用，我就执行工具**，**把工具结果再拼回上下文，继续下一轮**；

**直到模型不再请求工具，循环结束**。这个设计的关键是让模型不是一次性回答，而是能“思考—用工具—再思考”。



**你为什么不做成“一问一答”，而要做多轮工具循环？**
因为真实编程任务通常不是一次推理能完成的。

比如先找文件、再读文件、再 grep、再编辑、最后总结，这天然就是多步链路。

做成循环之后，它可以基于真实执行结果继续决策，正确率和可解释性都会更好。



**你的 Web、CLI、飞书三入口，为什么能共用一套内核？**

因为我**把“交互入口”和“Agent 能力”**分开了。

无论是浏览器、命令行还是飞书，最终都只是把用户输入送到同一个 AgentSession，再**订阅同一套事件流去展示结果**，所以入口不同，但**底层 Agent、工具、会话、上下文压缩全是复用的**。这样调试和上线是一条链路。 



**工具系统是怎么设计的**

**把每个工具抽象成统一协议：名字、描述、JSON Schema 参数定义、执行函数。**

模型看到的是标准化工具描述，运行时 Agent 根据 ToolCall 找到对应工具执行，再把 ToolResultMessage（工具结果消息） 塞回上下文。这样工具层和模型层是松耦合的，工具既可以是同步函数，也可以是异步函数。



**你现在内置了哪些工具？为什么先做这几类？**
我现在内置的是 **ls、read、write、edit、grep、find、bash** 这 7 类，基本覆盖编程助手最核心的“看目录、读文件、搜内容、改文件、执行命令”闭环。先做这几类，是因为它们组合起来已经能支撑大部分代码理解、重构、排错和小规模生成场景。

**ls**：列出目录内容，查看某个目录下有哪些文件和子目录。

**read**：读取文本文件内容。

**write**：写入文件内容。

**edit**：对文件做精确文本替换，适合小范围修改已有代码。

**grep**：按正则在文件内容里搜索，快速定位关键字、函数名或报错信息。

**find**：按路径模式查找文件或目录，适合找某类文件。

**bash**：执行 shell 命令，适合跑脚本、编译、测试或查看命令行结果。



**你为什么同时支持串行和并行工具执行？**
因为不同工具调用关系不一样。像**同时读取多个文件、同时搜索多个位置，这种天然适合并行；**但像**先创建目录再写文件**，这种**有依赖的步骤就更适合串行**。所以我把它做成可配置策略，而不是写死一种模式。



**并行执行最大的风险是什么？你怎么控制？**
最大的风险是**顺序依赖和共享状态冲突，**所以我不会默认把“有先后关系”的操作交给并行。

并行更多用于只读、互不依赖的任务；

如果是修改型操作，我会切回串行，或者通过工具语义和业务层控制来避免冲突。项目里也专门有并行工具执行的测试去验证行为。 



**上下文太长了你怎么处理？**
我不是简单截断，而是做了溢出检测加压缩。

会话在 prompt 前会先检查消息数和估算 token，超过阈值就把较早的历史拿去做摘要，把摘要作为一条上下文总结消息，再保留最近若干条消息继续对话。这样能保住关键事实和近期交互，不会像硬裁剪那样丢掉重要上下文。



**你为什么要用 LLM 摘要，而不是固定规则压缩？**

编程对话里真正重要的通常不是字数，而是**决策、路径、约束、代码片段和用户偏好**，所以我优先让 LLM 生成摘要；

如果失败，再退回 fallback summary。这样准确性和稳定性两边都能兼顾。



**你怎么保证工具执行的安全性？**

第一层是 workspace 边界校验，所有路径都必须落在工作区内，不能逃逸；

第二层是 bash 风险拦截，像 rm -rf、format、shutdown 这类高风险命令会被直接阻断。 



**MCP 在你项目里是什么定位？**
MCP 在我这里不是重写一套工具系统，而是把外部 MCP server 代理成内部可调用的 AgentTool。这样 Agent 对 MCP 工具和本地内置工具的使用方式是统一的，扩能力时不需要动核心循环，只是多接一个 server 配置。 



**你为什么还做了 extensions 和 skills，不直接把能力都写死？**
因为编程 Agent 的变化点太多了，写死会很快失控。extensions 更偏代码级扩展，可以注册工具、hook、命令；skills 更偏提示词和任务能力装配，负责补充 prompt guideline、附加系统提示和命令。我的目标是让核心稳定、变化外置。  



**系统提示词你是怎么组织的？**
会将**自定义角色 Prompt、当前启用工具描述、工具片段、全局约束规范、全局记忆以及扩展 Skills** 内容动态合并拼接，生成最终系统提示词。

实现同一 Agent 内核在不同工作区、不同项目下拥有独立风格、行为约束与专业能力。



**飞书接入里，为什么要做“频道队列”？**
因为同一个频道里的消息必须串行处理。AgentSession 同一时刻只能跑一个 prompt，如果用户在群里连续发三条消息，不排队就会出现并发 prompt 冲突，甚至把上下文打乱。所以按频道维度建队列，保证同一条会话链路上的消息按顺序消费。



**飞书里为什么还要做去重和过期过滤？**
 答：这是 IM 场景里很现实的问题。Webhook 或长连接有可能重复推送消息，服务重启后也可能收到延迟旧消息，如果不做去重和 stale event 过滤，用户会看到重复回复。所以我会基于 message_id 去重，并过滤掉过期消息。



**飞书流式“打字机效果”你是怎么做的？**
我不是等最终答案一次性回，而是先发一个占位卡片“思考中”，然后订阅 Agent 的 message_update 事件，定期把当前累积文本 PATCH 到飞书消息里。这样用户能感知到过程，而且我还做了更新限频，避免过于频繁触发平台限流。 



**你说的记忆系统，和普通会话历史有什么区别？**
普通会话历史是一次会话内部的上下文，**记忆是跨会话保留的长期信息**。我这里分了全局记忆和频道级记忆，前者放通用偏好和项目约束，后者放某个群或线程的局部背景，再把它们合并注入系统提示词。这样即使会话轮换了，Agent 仍然能带着长期偏好工作。



**你觉得这个项目目前最大的不足是什么？**
目前它更强在单工作区和个人效率场景，还不是大规模生产级平台。

比如**权限体系、更加细粒度的审计、任务级调度、跨会话共享缓存、分布式执行**这些我还没完全做；

另外**上下文压缩虽然可用，但质量仍然依赖模型本身。**这个项目的定位是“可运行、可扩展的编程 Agent 雏形”，不是把所有企业级能力一次做完。



**如果再给你一段时间，你下一步最想做什么？**
我会优先做三件事。

第一，补更完整的工具权限模型，把不同入口和不同工具的安全等级做细；

第二，把任务拆解、结果缓存和异步任务队列做起来，提升复杂任务吞吐；



**你是怎么处理不同模型对消息格式兼容性问题的？**
我专门做了一层 `convert_to_llm`。它不是把 Agent 内部消息原样丢给模型，而是先做标准化转换。

第一，过滤不适合直接发给模型的消息；

第二，超长 `ToolResult` 做截断，防止上下文被工具输出撑爆；

第三，`thinking` 块根据目标模型能力决定是保留、转成文本还是移除。这样切 provider 的时候，上层 Agent 不用跟着改。



**工具结果特别长的时候，你为什么不直接全量保留？**
因为编程 Agent 最大的问题之一就是上下文污染。比如一次 `read` 或 `bash` 返回非常长的内容，如果不做裁剪，很容易把真正重要的历史对话和约束挤掉。

所以我在转换层对 `ToolResult` 做了最大长度控制，**优先保留“足够让下一轮推理继续”的信息，而不是盲目全塞进去。**我的目标是**让上下文对模型更有信息密度**，而不是更长。



**你的 system prompt 是固定写死的吗？为什么这样设计？**

我的系统提示词**并非固定写死**，而是采用**动态模块化构建**。运行时会自动合并**当前可用工具、工具调用规范、全局行为约束、长期记忆、当前工作目录环境以及业务层定制要求**，拼接生成最终系统提示词。

实现同一 Agent 内核在**不同工作区、不同入口、不同技能配置下**都能被稳定约束，避免使用单一通用提示词导致行为松散、不可控的问题。



**你怎么让模型“少猜、多用工具”？**
我主要用了**两层约束**。第一层是 system prompt 里显式规定：查目录优先 `ls/find`，查内容优先 `read/grep`，不要用 bash 代替常规读写工具；修改前先读上下文，再 edit/write。

第二层是运行时工具边界，比如危险命令拦截、workspace 路径约束。不是只靠 prompt 说服模型，而是 prompt 和执行层同时收口。 

**1. workspace 路径约束**
所有文件类工具在真正执行前，都会先把用户传入路径和 `workspace_dir` 拼接后做 `resolve`，再检查目标路径是不是仍然位于工作区内部。
如果路径逃出了工作区边界，就直接报错 `Path escapes workspace boundary`，这样就避免了访问工作区外文件。

2**. 危险命令拦截**
`bash` 工具执行前，会先检查命令里是否包含高风险模式，比如 `rm -rf`、`format`、`shutdown`、`reboot` 之类。
如果命中危险命令，并且没有显式允许，就直接拦截，不会执行。
另外还支持 allowlist 和 blocklist 规则，方便做更细的命令控制



**如果模型调用失败了，你的系统怎么恢复？**

我实现了**智能条件重试机制**。

仅当模型返回停止原因为报错`error`或中断`aborted`这类临时异常时才触发重试；

对于密钥无效、认证失败、未授权等**永久配置错误不进行重试**。同时重试采用**指数退避策略**，避免高频请求压垮接口。

最终实现偶发网络抖动自动恢复，与永久性配置错误精准区分，提升 Agent 稳定性。



**你怎么观测一次 Agent 执行到底花了多少 token、多少成本？**
 答：我在 `AgentSession` 里保留了最近一次 usage 和整个会话的累计 usage。最近一次可以看 input/output/total tokens 和 cost，累计统计可以看整段会话总共花了多少。这个设计主要是为了后续做成本观测、提示用户开销，以及做不同 prompt 和不同工具策略的效果对比。



**你为什么还做了 runner / RPC 模式？这不是多此一举吗？**
 答：不是多此一举，我把它看成“统一能力出口”。CLI 的 print 模式适合单次问答，interactive 适合本地调试，而 RPC/jsonl 模式适合被外部进程接管，比如 IDE、插件或者其他服务侧进程。也就是说，同一个 AgentSession，不只是人能在终端里用，也能作为一个轻量协议服务被程序集成。这个设计让系统更像平台能力，而不只是命令行脚本。



**你这个“会话分叉”是做什么的？**
它本质上是**把对话树结构化**。比如一段上下文已经走到某个节点，我想基于那个节点尝试两个不同方案，就可以 fork 一条新分支，而不是把原会话覆盖掉。对编程场景来说，这很像从同一上下文分出两个解法，便于比较和回退。 

------

## 会话怎么存

我的思路是：**一个会话一个目录**。

第一类是**元信息**，比如这个会话用的什么模型、什么系统提示词、当前走到了哪条分支。
第二类是**当前上下文**，也就是恢复会话时真正要喂给模型的消息序列。
第三类是**运行过程和历史结构**，包括事件日志，还有整棵会话树。

------

## 上下文怎么恢复

恢复的时候，我优先恢复“**当前正在走的那条分支**”，而不是简单恢复最后几条消息。

- 先找到当前分支末尾
- 再把这条分支从头到尾的消息链恢复出来
- 然后交给 Agent 继续往下跑

这样即使服务重启了，也不是丢失上下文重新开始，而是能真正接上之前的工作现场。

------

## 为什么还要做上下文压缩

超出模型上下文窗口，直接用不了；
旧信息太多也会稀释重点，让模型越来越不稳定。

------

## 为什么要做分支

**如果只有线性会话，一旦往下聊，原来的思路就被覆盖了。**
所以我做了分支，让会话像一棵树一样，可以从某个历史节点分出一条新路继续尝试。

这个设计的核心价值是：**支持多方案并行尝试，而不是把所有思路都糊在一条聊天记录里。**

------

## 分支具体怎么理解

每条消息都会挂在上一条消息后面，所以自然形成链；
如果从中间某个节点重新出发，就会形成新的分支。所以我不是把会话当成“消息数组”，而是当成“消息树”。

------

## 为什么还要支持切换

有了分支以后，切换就很自然了。

支持把当前活跃分支切到任意历史节点，切过去以后，系统恢复那条路径上的上下文，再从那里继续工作。

**我不仅支持分叉，也支持回到任意一个历史节点继续往下做。**

------

## 为什么不只保留当前上下文，非要保留整棵树

因为只保留当前上下文，意味着你只能“继续”，不能“回头”，也不能“比较不同方案”。
而编程场景很多时候恰恰需要：

- 回看之前为什么这么改
- 从某一步重新走另一条路
- 比较两个方案的差异

我同时保留：

- 当前可直接恢复的上下文
- 整个会话树结构

前者保证可运行，后者保证可回溯、可分叉、可切换。

------

## 这套会话设计解决了什么问题

**我这套会话设计，主要解决了四个问题：**
第一，服务重启后上下文还能恢复，不会断；
第二，对话过长时不会直接崩，而是能压缩；
第三，复杂任务可以从中间分出多条方案并行尝试；
第四，走错了还能切回历史节点继续，不会一条路走到底。

这就让它更像一个“可持续工作的编程助手”，而不是一次性聊天机器人。
