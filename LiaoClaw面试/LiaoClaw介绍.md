**普通大模型**像一个“会聊天的专家”，
**LiaoClaw**更像一个“会聊天、会动手、还能接入工作群的开发助手”。

它和普通问答型 AI 的差别，在于有一套完整的执行链路：

1. 先接收你的自然语言任务
2. 调用大模型理解任务
3. 判断需不需要用工具
4. 去读文件、查目录、搜索代码、编辑文件、执行 shell 命令
5. 拿到工具结果后再继续推理
6. 最终给出可执行结果或代码修改建议

这个核心循环，在 `agent_loop.py` 里写得很清楚：**用户消息 → LLM → 工具调用 → LLM → … → 结束**，并且支持事件流、工具执行、再推理的标准 agent loop 。



**这是一个用 Python 从零搭建的 AI 编程助手系统，它把不同大模型统一成同一种调用方式，再用 Agent 循环把“思考—调用工具—继续思考”串起来，最后还能接进飞书，让它在群聊里像一个会写代码、会查文件、会执行命令的工程助手一样工作。**



### 它解决了三个现实问题

第一，**不同大模型接口不统一**。
OpenAI、Anthropic、GLM 这些模型厂商的 API、返回格式、流式输出方式都不一样。项目最底层先做了一层统一封装，把这些模型都抽象成同一种调用方式，这样上层不用关心具体厂商差异。这个定位在 README 和 `pyproject.toml` 里都很明确：项目描述就是 unified LLM API + agent core + IM bridge，并且暴露了 CLI、IM、Web 三个入口 。

第二，**大模型本身不会真正操作你的工程目录**。
模型知道“应该怎么改”，但它本身不能读本地文件、改源码、执行命令。所以项目中做了一整套内置工具，包括列目录、读文件、写文件、文本替换、正则搜索、文件查找、运行 bash 命令等，等于给模型装上了“手”和“脚” 。

第三，**真实工作场景需要会话、上下文、长期使用，而不是一次性问答**。
一个能长期使用的编程助手，必须能保存会话、管理上下文、在上下文过长时自动压缩，还要能接入 IM 场景。`agent_session.py` 里就承担了这些能力：会话持久化、消息恢复、上下文溢出检测、自动摘要压缩、失败重试等 。`im/service.py` 则把它接到飞书频道中，支持每个频道维护长期 session、流式更新回复、附带 token/cost 统计、记忆注入等 。

## 这个项目的核心组成

### 1. 统一 LLM 接口层

这一层的作用是：**屏蔽不同模型厂商的差异**。

README 里列出的 `src/ai/` 模块包括模型注册、统一流式入口、Provider 注册、API Key 管理、事件流封装等 。
你可以理解为：不管底层接的是 Claude、OpenAI 兼容接口，还是别的模型，上层 agent 都以同一种方式调用。

这层的意义不是“功能炫”，而是**工程扩展性强**。以后换模型、加模型，都不需要重写上层业务。

------

### 2. Agent 编排内核

这是项目最核心的部分。

`agent_loop.py` 实现的是标准的 agent loop：模型先产出回答，若回答里包含 tool call，就执行工具，再把工具结果喂回模型继续推理，直到没有后续工具调用为止 。

这一层最重要的不是“能调工具”，而是它具备几个工程化特征：

- 有统一事件机制，能发出 `agent_start`、`message_update`、`tool_execution_start`、`tool_execution_end` 等事件
- 支持顺序执行和并行执行工具
- 支持工具调用前后的拦截钩子，便于做权限控制、日志记录、结果后处理
- 支持流式接收模型输出，不是等模型一次吐完才展示

**它像一个“任务指挥器”。模型负责想，工具负责做，指挥器负责在“想”和“做”之间来回协调。**

------

### 3. 编程 Agent 应用层

这一层是把抽象 Agent 变成真正的“编程助手”。

README 里列出的 `src/coding_agent/` 包含工厂、会话管理、内置工具、会话持久化、系统提示词、CLI、运行模式、扩展系统、MCP 工具代理等 。

这里最重要的是两个东西。

#### 一个是内置工具

`builtin_tools.py` 里直接实现了编程助手常见能力：

- `ls` / `list_dir`：列目录
- `read` / `read_file`：读文件
- `write` / `write_file`：写文件
- `edit`：替换文本修改文件
- `grep`：代码搜索
- `find`：查找文件
- `bash`：执行 shell 命令

而且不是裸实现，还做了工程安全处理，比如：

- 路径不能越过工作区边界
- 高危 bash 命令可以拦截
- 文本替换支持唯一匹配、指定第几次匹配、期望匹配次数校验等

这些都说明它不是一个“玩具 demo”，而是在朝可控、可落地的 coding agent 做 。



#### 另一个是会话系统

`agent_session.py` 里做的事情很关键：

- 管理会话目录
- 把 agent 事件和消息写入持久层
- 提供稳定的 prompt / continue 接口
- 上下文过长时做压缩摘要
- 支持自动重试
- 支持会话分叉、切换历史分支

这意味着它不是“问一句答一句”，而是支持持续协作的。尤其是上下文溢出检测与压缩这一点，非常像真实 agent 产品会做的事情：上下文长了就摘要，只保留最近关键消息和浓缩历史 。

**这一层相当于“把聪明的模型，包装成一个能长期陪你一起改项目的开发助手”。**

------

### 4. IM 飞书桥接层

很多 agent 项目停留在命令行或网页 demo，但这个仓库专门做了 `im/` 模块，目标就是把 agent 接到飞书里。README 里列出了 `IMService`、飞书适配器、长连接/Webhook 服务、会话路由、长期记忆等组件 。

`im/service.py` 里可以看到几个非常实际的设计：

- **每个频道维护长期 session**，而不是每条消息重新开一轮会话
- **先发占位消息，再流式更新内容**，更像真实聊天体验
- **会显示 token 和 cost 统计**，便于观察使用成本
- **支持 MEMORY.md 记忆注入**，让机器人记住频道级长期信息
- 支持 `/clear`、`/new`、`/session`、`/help` 等控制命令

这说明项目不是只想做一个“本地跑通”的技术实验，而是在往**团队协作中的智能开发机器人**方向走。

## 这个项目的运行形态

从 `pyproject.toml` 可以直接看出，这个项目至少有三种使用方式：

- `liaoclaw`：命令行入口
- `liaoclaw-im`：IM 机器人入口
- `liaoclaw-web`：Web 控制台入口

README 里也给了对应方式：

- CLI 交互模式
- quickstart 示例脚本
- Web 控制台
- 飞书 IM 模式

所以它并不是单点产品，而是一个**“多入口的 AI agent 系统”**。同一套内核，既能本地命令行跑，也能网页跑，还能进飞书当机器人。

## 它和普通“聊天机器人”的差别

你可以专门这样解释，外行会很容易懂：

普通聊天机器人是：

**你问，它答。**

这个项目是：

**你提需求，它理解需求，然后自己决定要不要看文件、搜代码、改文件、执行命令，最后把处理结果回给你。**

所以它更像：

- 半自动开发助手
- 面向代码工程的智能代理
- 具备工具执行能力的协作机器人

而不是单纯的对话模型壳子。



**LiaoClaw 是一个用 Python 搭建的 AI 编程助手框架。它底层先把不同厂商的大模型接口统一起来，让上层不用关心 Claude、OpenAI 这些 API 的差异；中间用 Agent 循环把“理解任务—调用工具—继续推理”串起来；上层再针对编程场景提供文件读写、代码搜索、文本编辑、命令执行等能力。这样用户只需要用自然语言提要求，系统就可以像一个开发助手一样去分析代码、操作工程、反馈结果。除此之外，它还支持会话持久化、上下文压缩、长期记忆以及飞书接入，所以既能本地当 coding agent 用，也能作为团队群聊里的智能开发机器人使用。**

------

## 这个项目最值得强调的亮点

**第一，它是分层清晰的。**
不是把所有逻辑写在一个脚本里，而是按模型层、Agent 核心层、应用层、IM 集成层拆开。

**第二，它是真正带工具执行的。**
不只是文本问答，而是具备文件系统和命令行操作能力。

**第三，它有会话与持久化能力。**
支持长期会话、上下文压缩、分叉切换，不是一次性 demo。

**第四，它有产品接入能力。**
能接入飞书，不局限于本地实验。

**第五，它兼顾扩展性。**
有扩展系统、MCP 工具代理、多 Provider 统一接口，后续可继续拓展。
