## 你说支持并行执行，并行的到底是什么？

并行的不是整条推理链，而是**同一轮里多个彼此独立的工具调用**。
比如模型一轮里要同时读两个文件、查两个位置，这种可以并行；但如果后一个工具依赖前一个工具结果，比如先搜再改，那就要串行。这个设计的目的不是炫技，而是减少一轮任务里的等待时间。测试里也专门验证过两个 60ms 的工具并行时，总耗时明显小于串行累加。

------

## 你怎么处理流式输出里那种“参数还没传完”的工具调用？

这是比较容易出 bug 的地方。我的处理方式不是等整包，而是按工具调用的序号持续聚合增量片段，把半截参数先暂存起来，边收边尝试解析。

这样模型流式吐参数时，即使 JSON 还没完全闭合，系统也不会直接炸掉，而是等到足够完整再执行。这个能力对流式工具调用很关键。

------

## 你说有 hook，面试官会问：这不是多此一举吗？

不是。
**工具调用前后的扩展点很有必要，**因为很多非功能逻辑都不应该写死在主循环里，**比如审计、拦截、日志、权限判断、结果修饰**。把这些放成前后置 hook，主流程仍然保持干净，而扩展能力会强很多。测试里也覆盖了“前置 hook 可以阻止工具执行”这种情况。

------

## 你简历里写“支持会话分支和切换”，这个很容易被追问，真实做法是什么？

我不是简单把消息 append 到一个列表里，而是把会话按“树”来存。
每条消息都知道它接在谁后面，当前会话则记住自己现在停在哪个叶子节点。这样恢复的时候，不是把历史全读一遍，而是沿着当前分支一路回溯出来。
这个设计的好处是可以从某个历史节点继续开新分支，相当于“在旧上下文基础上继续尝试另一个方案”，而不是把整个会话复制一大份。测试里也专门验证了分叉和切换。

------

## 你说“上下文恢复”和“上下文压缩”，这两个分别是什么意思？

上下文恢复，是指程序重启后还能把这个会话之前的状态接回来，包括消息、事件和一些元信息，不会每次都当成新对话。
上下文压缩，则是当历史太长、快超出模型上下文上限时，不是直接丢掉，而是把较早的内容整理成摘要，保留最近轮次，再继续对话。这样做是为了在成本和连续性之间取平衡。

------

## 如果面试官继续追问：你的上下文压缩是精确 token 计算吗？是 RAG 吗？

都不是。
我这里不是做向量检索，也不是对每个模型都做精确 tokenizer 预算，而是先做较轻量的溢出估算，再在快超限时触发 LLM 摘要压缩。
这个方案的优点是实现简单、跨模型比较好迁移；缺点也很明显，就是精度不如专门 tokenizer 方案高。所以这个点我会明确说成“工程上先跑通的方案”，而不是吹成最优解。

------

## 你工具里最有工程价值的部分是什么？

我觉得不是“工具数量”，而是**闭环能力**。
一个编程助手最核心的是：能看目录、能读文件、能搜索、能写改、能执行命令。只有这几个组合起来，模型才不是只会空谈，而是真的能对代码库做任务。
所以我的工具设计是围绕“感知代码库—定位问题—修改—验证”这个闭环来的，不是为了堆功能点。

------

## 你说做了“工作区沙箱”和“危险命令拦截”，这两个是怎么落地的？

**答法：**
第一层是路径边界控制。用户或者模型给出文件路径后，系统会把它解析成绝对路径，再检查它是不是还在工作区目录里面，防止越界访问。
第二层是命令拦截。对于明显危险的 shell 命令，会在执行前做模式识别和阻断。
但我也会主动承认，这还不是容器级隔离，更多是工程上的第一层保护，所以如果上生产，后面还要补真正的隔离环境和资源配额。这样回答更真实。

------

## 面试官很可能会问：你这个项目的“记忆”到底是不是长期记忆系统？

**答法：**
我不会把它吹成复杂记忆系统。
目前更准确的说法是：我做了**全局和频道级别的长期提示注入**。也就是把全局说明和某个频道自己的记忆内容合并后，附加到系统提示里，让这个频道的长期会话保留一些稳定背景信息。
它有长期效果，但本质上更接近“结构化上下文注入”，而不是向量检索式 memory retrieval。

------

## 飞书消息重复、乱序、过期你怎么处理？

这是 IM 系统的基本功。
我的处理思路是：
先做消息去重，已经处理过的 ID 不再重复消费；
再过滤掉明显过期的事件；
同时按频道维持独立状态，限制队列长度并串行处理，避免同一频道里多条消息互相打架。
这样才能保证长期会话的上下文不容易串掉。

------

## MCP 你到底做到了什么程度？

我做的是把外部 MCP 工具纳入我的工具体系里统一使用。
也就是说，系统既能用自己的内置工具，也能连到外部工具服务，把它们当成同一种调用对象来调度。
所以我做的是“接入和调度层”，不是重新发明 MCP 协议本身。这个说法比较稳，也符合真实实现。

------

## 面试官可能会怀疑：这项目是不是大部分是别人框架，你只是拼起来？

你可以这样答：
这个项目当然会借鉴现有生态，比如模型协议、IM 接入方式、MCP 生态，但我自己做的重点是把这些能力组织成一个能运行、能扩展、能保状态的完整系统。
真正体现我工程能力的，不是某个概念，而是统一抽象、主循环、工具闭环、会话树、上下文压缩、消息去重和长期 session 这些组合起来后的系统行为。
这样回答比硬吹“全原创”更稳。

------

## 这个项目目前最大的短板是什么？

**答法：**
我建议你主动承认三个短板：
第一，长度控制还是估算型方案，不是精确 token 预算；
第二，命令安全目前主要靠规则拦截，不是容器级隔离；
第三，记忆更偏提示注入，还不是检索增强型长期记忆。
这样回答不会减分，反而显得你知道项目边界。

------

## 如果明天让你把它真拿去服务团队，你最先补什么？

**答法：**
我会先补三件事：
一是更严格的安全隔离，包括命令执行环境和资源限制；
二是更稳的上下文预算和观测指标；
三是多用户、多会话下更正式的存储和检索能力。
因为现在它已经能形成闭环了，下一步最缺的不是“再多几个功能”，而是把可靠性、安全性和可运营性补齐。这个回答会显得你不是停留在 demo 心态。

## 最后给你一个适合面试开场的版本

你可以这样讲：

> 这个项目是一个 AI 编程助手系统，我重点不是做一个聊天界面，而是把模型、工具、会话和 IM 场景真正串起来。
> 它能把模型输出转成结构化事件，如果需要就调用工具，再把结果继续回灌给模型，形成读代码、改代码、执行验证的闭环。
> 我还做了长期会话、会话分支、上下文压缩、飞书消息去重和流式更新，以及外部工具接入。
> 我觉得这个项目最有价值的地方不是某个点功能，而是它已经是一个能持续执行任务、并且有状态管理和扩展能力的系统。 ([GitHub](https://github.com/kyletser/LiaoClaw/raw/refs/heads/master/src/agent_core/agent_loop.py))

## 你理解的 Agent 和普通聊天机器人有什么区别？

我理解的核心区别不是“会不会多轮对话”，而是**能不能把任务拆解、调用外部能力、根据外部结果继续决策**。
普通聊天更像一次性生成文本，Agent 则是一个带状态的执行体。它会根据目标不断做“判断—行动—观察—再判断”，直到任务完成。
所以我这个项目里最重要的不是接了模型，而是让模型能调工具、读结果、继续执行，形成闭环。

------

## 你为什么觉得你的项目是 Agent，不是 Workflow？

Workflow 更偏预定义流程，路径基本提前写好；
Agent 更偏开放式决策，下一步做什么由模型结合上下文和工具结果动态决定。
我的项目里，工具不是按固定脚本串起来的，而是模型在每一轮根据任务状态决定要不要调、调哪个、调几个，所以它更接近 Agent。
但我也会承认，它不是完全开放世界 Agent，而是**面向编程任务的工具型 Agent**。

------

## 你这个 Agent 的边界是什么？它擅长什么，不擅长什么？

它擅长的是**本地代码相关任务**，比如读代码、搜代码、修改、执行命令、辅助测试和代码审查。
它不擅长开放域复杂规划，也不适合做高风险自动执行，因为工具集和安全边界都是围绕本地编程场景设计的。
我会主动强调边界，因为 Agent 做得好不好，不是看吹得多大，而是看任务域收得准不准。

------

## 为什么要支持多轮工具调用？

因为很多真实任务不是一次工具调用就能完成。
比如改 bug，通常要先搜位置，再读文件，再改代码，再跑命令验证。
如果只能单轮调用工具，Agent 本质上就退化成“模型 + 单次插件”，做不了复杂任务。
所以多轮工具调用是 Agent 从“会调用工具”升级到“会做任务”的关键。

------

## 你为什么做并行工具调用？什么时候不该并行？

并行适合彼此没有依赖的工具调用，比如同时读多个文件、查多个位置。
这样能减少等待时间，尤其是读类和搜索类工具。
但只要后一步依赖前一步结果，就不能并行，比如先搜索再编辑，这种必须串行。
所以并行不是默认越多越好，而是看任务依赖关系。

------

## 你的工具设计原则是什么？为什么不是做更多工具？

我更看重**闭环能力**，不是工具数量。
对编程 Agent 来说，最关键的是几类能力：看代码、找位置、改代码、执行验证。
只要这几类能组合起来，就能完成大量真实任务；反过来，工具再多但不能闭环，实际价值不高。
所以我的设计是围绕任务链路来的，而不是为了堆功能点。

## 你怎么处理长上下文问题？

我的思路不是硬塞全部历史，而是**在快溢出时做压缩**。
保留最近几轮关键交互，把更早的内容整理成摘要，再继续执行。
这样做的好处是能维持任务连续性，不至于因为上下文太长直接丢失历史。
我会明确说，这是一种工程上可用的压缩策略，不是最精细的 token 最优解。

------

## 你这个“记忆”到底是什么？长期记忆吗？

更准确地说，我做的是**长期上下文注入**：把全局信息和频道/会话级别的信息整理后放进系统上下文，让后续轮次持续受到影响。
它确实能让 Agent“记住”一些稳定背景，但本质更接近结构化上下文管理，而不是完整的 memory retrieval 系统。

------

## 为什么你要做会话分支？

因为真实任务经常要从某个中间状态分叉尝试不同方案。
比如同一个 bug，我可能想保留原有上下文，再试另一种修法；或者同一段代码，在已有分析基础上继续走不同方向。
如果没有分支，就只能复制粘贴大量历史，成本高而且容易乱。
所以会话分支本质上是在提升 Agent 的“探索能力”和“复用历史状态”的能力。

## 你觉得 Agent 最容易失败在哪？

我觉得最容易失败在四类地方：
一是任务理解偏了，第一步就走错方向；
二是工具调用不稳定，比如参数不完整、工具返回异常；
三是上下文过长后遗忘关键约束；
四是模型过度自信，拿不完整的工具结果继续推理。
所以 Agent 不是只看 happy path，关键是怎么处理这些失败链条。

------

## 你怎么降低 Agent 幻觉带来的问题？

我主要靠两类方法：
一类是让它尽量多依赖工具结果而不是空想，比如要求先读代码、先搜索、先执行验证；
另一类是把工具边界和错误明确返回给模型，让它知道哪些事情没成功。
我的理解是，Agent 幻觉不能完全消灭，但可以通过“外部反馈闭环”明显降低。

------

## 如果模型瞎调危险命令怎么办？

这个问题不能只靠 prompt 约束。
我做的是在工具层加硬限制，包括工作区边界和危险命令拦截。
也就是说，就算模型想执行高风险操作，也要先过系统规则这一关。
我也会承认，这只是第一层保护，真正线上化还要补更强的执行隔离。

## 你怎么判断这个 Agent 做得好不好？

**答法：**
我会从三个层次看：
第一是**任务完成率**，能不能真的完成改代码、查问题、生成测试这些任务；
第二是**过程质量**，比如工具调用是否合理、是否有无效循环、是否频繁走错；
第三是**交互体验**，比如响应延迟、流式反馈、失败是否可解释。
我会强调，Agent 的评估不能只看最终文本质量，一定要看执行过程。

------

**答法：**
你要稳一点答。
可以说：我项目里更偏工程验证，已经会看一些局部指标，比如工具执行耗时、任务链路是否跑通、并行和串行的差异，但还没有做成大规模标准化 benchmark 平台。
这种答法比硬吹“我做了很完整的评估体系”安全很多。

------

## 为什么你要把 IM 接入也放进项目里？

**答法：**
因为 IM 场景能逼出很多真实问题。
终端模式更像单机工具，而 IM 场景要求你解决消息去重、长期 session、流式更新、频道隔离这些更工程化的问题。
从面试角度说，这也能证明你不是只会写一个本地 demo，而是考虑过真实协作环境。

**答法：**
我会优先补三件事：
第一，做更系统的任务评估和失败分析；
第二，提升上下文管理精度，而不是只靠较粗的压缩策略；
第三，把工具执行环境做得更安全、更可控。
因为我现在这版已经把主链路打通了，下一步最重要的是让它更稳、更可量化。
