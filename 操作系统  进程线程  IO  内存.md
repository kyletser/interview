# 一、操作系统 / 进程线程 / IO / 内存

## 协程了解吗

> 协程本质上是用户态调度的轻量级执行流，特点是切换成本低、适合高并发 IO 场景。
>
> 它更像在线程内部再切分很多可挂起恢复的小任务，常用于异步编程、网络框架、游戏逻辑等场景。协程擅长提高并发效率，但单线程协程本身并不等于多核并行。

------

## 在应用程序中读文件涉及进程的切换吗

> 一般不涉及“切换到其他进程去读”，但会涉及 **用户态到内核态的切换**；
>
> 如果发生阻塞，还可能导致当前线程被挂起，CPU 去调度别的线程或进程，就会涉及到进程切换。

### 详答

调用 `read()` 读文件时，流程通常是：

1. **应用进程在用户态执行**
2. **调用系统调用 `read`**
3. **CPU 陷入内核态**
4. **如果数据已在页缓存，内核直接拷贝给用户态**
5. **如果不在缓存，发起磁盘 IO，请求设备完成**
6. **当前线程可能睡眠等待**
7. **数据到达后唤醒线程，返回用户态**

所以：

- **不一定切换到别的进程去执行“读文件逻辑”**
- 但**当前线程可能阻塞**，操作系统会调度其他线程 / 进程运行

> 读文件本质上会触发系统调用，涉及用户态和内核态切换；如果 IO 阻塞，当前线程会睡眠，调度器再去运行其他线程或进程，但不是切到“某个专门读文件的用户进程”。

------

## 5. 你觉得为什么需要切换进程

> 因为 CPU 是有限的，而系统里同时有很多任务要运行，所以操作系统需要通过调度在不同进程或线程之间切换，保证系统并发执行、资源公平使用以及响应及时性。
>
> 进程切换本质是为了提高 CPU 利用率和系统响应性。一个进程阻塞或时间片用完后，如果不切走，CPU 就可能空转或被单个任务长期占用，所以操作系统需要不断切换到其他可运行任务。

------

## 6. 为什么不在当前进程内直接读而是要切换到其他进程

> 不是说“必须切到其他进程才能读”，而是文件读取往往依赖磁盘等慢设备，当前线程发起读请求后，如果数据暂时没准备好，就没必要一直占着 CPU 等待。
>
> 操作系统会把当前线程挂起，让 CPU 去执行其他就绪任务，这样整体效率更高。
>
> 因为磁盘 IO 速度远慢于 CPU，当前线程发起读请求后，如果必须原地等待，会浪费 CPU 时间。操作系统会让当前线程阻塞，把 CPU 分给其他可运行任务，从而提高整体吞吐和响应性。

------

## 7. 什么 IO 会阻塞当前的线程

> 当 IO 操作不能立即完成，而程序又采用阻塞方式调用时，就会阻塞当前线程。

常见阻塞场景：

- 阻塞式 `read`
- 阻塞式 `recv`
- 阻塞式 `accept`
- 阻塞式 `connect`
- 管道 / 终端 / socket 等没有数据可读时
- 磁盘文件读写等待底层设备完成时

### 补充

普通磁盘文件在页缓存命中时可能很快返回，不明显阻塞；
网络 socket 更常见阻塞，比如收不到数据时。

> 凡是采用阻塞模式、且数据或资源当前还没准备好的 IO，都可能阻塞当前线程，比如阻塞式 socket 的 `accept/read/recv`，以及等待磁盘、管道、终端等设备数据的场景。

------

## 8. 访问磁盘时是直接访问到硬件还是通过一些系统级别的转换

> 一般不是应用程序直接访问磁盘硬件，而是通过操作系统的文件系统、页缓存、块层、驱动程序等一系列内核模块，最后再由设备控制器和磁盘硬件完成真正的读写。

### 大体路径

用户程序
→ 系统调用
→ VFS（虚拟文件系统）
→ 具体文件系统（ext4/xfs/ntfs 等）
→ 页缓存 / 块层
→ 设备驱动
→ 磁盘控制器
→ 硬件

> 应用程序通常不会直接碰硬件，而是通过系统调用进入内核，由文件系统、缓存层、驱动层逐级处理，最后才由磁盘控制器访问硬件。这么设计是为了统一抽象、隔离硬件差异、提高安全性和缓存命中效率。

------

## 9. 虚拟内存的概念

> 虚拟内存就是操作系统给**每个进程提供的一套独立的**、连续的“假想地址空间”，实际再通过页表映射到物理内存，必要时还能和磁盘交换空间配合使用。

虚拟内存的核心作用：

### ① 地址空间隔离

每个进程看到的地址空间相互独立，提高安全性

### ② 让程序感觉内存连续

程序员不必关心物理内存是否连续

### ③ 按需分配

不是一次性给所有实际物理页，而是访问时再建立映射

### ④ 扩大可用内存“表象”

物理内存不足时，可借助 swap / 页置换机制

> 虚拟内存本质上是“进程看到的是虚拟地址，CPU 和操作系统再通过页表映射到物理地址”。它的价值主要在于进程隔离、简化内存管理、支持按需加载和页置换，从而让系统更安全、灵活且更高效。

------

## 10. 如果把换页的事情交给应用进程来做可以吗

> 一般不适合。理论上某些用户态系统或特殊运行时可以参与内存管理，但普遍场景下，换页更适合由操作系统统一管理。

1. **应用无法全局掌握系统内存状态**
2. **不同进程之间需要统一公平分配物理内存**
3. **页表和异常处理涉及特权操作**
4. **安全性和隔离性要求高**
5. **换页策略需要全系统协调**

> 一般不能把换页完全交给普通应用进程来做，因为页表管理、缺页异常处理和物理内存分配都涉及系统级权限和全局资源协调。操作系统统一管理能保证隔离性、安全性、公平性和整体效率。

------

## 11. 这种事情（内存管理）交给系统做的核心目的是什么

> 核心目的是 **统一管理全局资源，保证安全隔离、提高利用率，并降低应用开发复杂度**。
>
> 把内存管理交给操作系统，核心是为了对全局物理内存做统一控制，同时保证进程隔离和系统安全，并把复杂的页表、换页、缺页处理对应用屏蔽掉，让应用只面向统一的虚拟地址空间编程。

------

## 12. IO 涉及用户态和内核态的切换吗

> 涉及。只要应用程序通过系统调用发起 IO，例如 `read`、`write`、`send`、`recv`，就会从用户态切到内核态，由内核完成资源检查、缓存处理、驱动交互等，再返回用户态。IO 通常会涉及用户态和内核态切换，因为应用层没有权限直接操作设备，必须通过系统调用让内核代为完成。尤其是文件、网络、管道等 IO，本质都要经过内核。

------

## 13. 不停 read 几千个字节，循环 read 是不是涉及很多次用户态和内核态的切换

> 是的，如果你每次都调用一次 `read`，那每次都是一次系统调用，通常都会发生一次用户态到内核态再返回用户态的切换。所以循环 `read` 小块数据会带来较多系统调用开销。
>
> 循环 `read` 小块数据会带来很多次系统调用，因此也会有很多次用户态和内核态切换，这部分开销在高频 IO 场景下会比较明显。

------

## 14. Buffer 是有上限的，如果文件达到很多 MB，那还会有很多次的切换吗

> 通常还是会有很多次，因为应用层缓冲区、页缓存、内核处理窗口都有限，大文件一般不可能一次性全搬完。即使内核层有缓存优化，应用层如果反复调用 `read`，本质上还是会有多次系统调用和多次拷贝过程。
>
> 大文件通常还是分块读取，所以往往会有多次系统调用和多次用户态/内核态切换。
>
> 优化方向一般是**增大单次读写块**、**使用缓冲**、**内存映射**、**异步 IO 或零拷贝机制。**

------

## 15. 除了 IO 多路复用还了解其他解决读效率问题的方法

### ① 批量读写 / 合理缓冲

减少系统调用次数

### ② 异步 IO

发起 IO 后不阻塞当前线程

### ③ 线程池 / 并发处理

让阻塞 IO 不至于拖死主线程

### ④ `mmap`

内存映射文件，减少显式 `read/write` 调用，适合大文件随机访问

### ⑤ 零拷贝

如 `sendfile`、`splice` 等，减少用户态和内核态之间的数据拷贝

### ⑥ 页缓存 / 预读 / 顺序访问优化

利用 OS 页缓存和磁盘预读机制

### ⑦ 协程

把大量 IO 等待任务轻量调度起来

> 除了 IO 多路复用，还可以通过增大缓冲区、减少系统调用次数、使用异步 IO、线程池、协程、`mmap` 和零拷贝等方式提高读效率。
>
> 核心思路要么是**减少切换和拷贝**，要么是**减少等待时 CPU 的浪费**。

------

## 18. 面向连接的“连接”是什么概念

> TCP 所谓“连接”，本质上不是物理上拉了一根线，而是通信双方在内核协议栈中维护的一份会话状态。

这份状态包括：

- 双方 IP 和端口
- 序列号 / 确认号
- 收发缓冲区
- 连接状态（ESTABLISHED、FIN_WAIT 等）
- 定时器
- 拥塞控制 / 流量控制信息

> 面向连接的“连接”本质上是一种由双方协议栈维护的逻辑会话状态，而不是物理连接。建立连接就是双方为后续可靠传输准备好必要的状态信息。

------

## 19. 连接占的是什么资源

> 一个 TCP 连接主要占用的是内核中的连接控制块、socket 结构、发送和接收缓冲区、协议状态、定时器以及文件描述符等资源。

- socket 对象
- TCP 控制块（TCB）
- 收发缓存
- fd 表项
- backlog / 半连接队列资源
- 协议状态机资源

> **TCP 连接占用的核心资源是内核中的 socket 和 TCP 状态结构，以及对应的收发缓冲区、文件描述符和各种协议管理信息，**所以高并发连接场景常常受 fd 数量和内核内存限制。

------

## 20. 连接所占的内存具体指什么

> 主要包括 socket 结构体、TCP 控制块、发送缓冲区、接收缓冲区，以及一些附加的协议状态、队列和定时器数据结构。

1. **socket 本身**
2. **TCB（Transmission Control Block）**
3. **发送缓存**
4. **接收缓存**
5. **重传队列 / 乱序队列**
6. **定时器**
7. **文件描述符关联结构**

> 连接占用的内存不仅是“一个 fd”那么简单，更多是内核里维护这个 TCP 会话所需的数据结构和收发缓冲区。大量空闲长连接的主要消耗之一就是这些内核内存。
