# 后端架构总览
聊天TCP服务器的代码位于`ChatServer.cpp`文件中：该服务器会启动一个基于 **Boost.Asio** 的TCP监听器，同时搭配一个用于跨服务器通知的gRPC服务器（`ChatServiceImpl`）；服务器通过Redis统计登录人数，并能响应系统信号实现优雅停机。
网关服务器的HTTP服务逻辑在`GateServer.cpp`中实现，其中`HttpConnection`类负责将GET/POST请求路由至`LogicSystem.cpp`中的处理函数，支持注册、密码重置、登录、验证码申请等功能。
状态/代理服务的代码在`StatusServiceImpl.cpp`中，该服务对外暴露gRPC接口，功能是从Redis中读取各聊天节点的登录人数，筛选出负载最低的节点，并生成用户令牌存入Redis。
验证码服务（基于Node.js）的代码在`server.js`中，对外提供`GetVerifyCode`的gRPC接口，可生成4位数字验证码，将验证码存入Redis并设置过期时间（TTL），再通过SMTP协议发送至用户邮箱（邮件发送逻辑在`email.js`中）；Redis的交互逻辑封装在`redis.js`中。
项目中存在另一个聊天服务器副本，代码路径为`ChatServer2`，与主服务器代码完全一致，推测用于多节点部署。

# 核心聊天服务器（TCP）
- **连接处理**：`CServer.cpp`负责监听并接收Socket连接，为每个连接创建对应的`CSession`会话对象。会话会解析消息的4字节消息头（包含**消息ID + 消息长度**），维护待发送消息队列；当连接断开时，会话会自动销毁，并解除用户与会话的绑定关系。
- **消息分发**：`LogicSystem.cpp`启动一个工作线程，维护一张“消息ID-处理回调”的映射表。支持的消息类型可参考`const.h`文件，包括登录、用户搜索、好友添加/好友验证、文本聊天等。
- **登录流程**：验证Redis中存储的用户令牌（键名格式为`utoken_<用户ID>`）→ 从Redis或MySQL中加载用户基础信息 → 获取用户的待处理好友申请列表与好友列表 → 在Redis的`logincount`哈希表中，将当前服务器的登录人数加1 → 绑定用户ID与会话的对应关系 → 记录该用户当前所在的服务器节点（键名格式为`uip_<用户ID>`）。
- **好友添加/验证流程**：通过`MysqlMgr/MysqlDao`将操作意图/状态写入MySQL数据库 → 根据目标用户的`uip_`映射信息判断其所在节点：若目标用户在本地节点，则直接通过会话推送消息；若在其他节点，则通过`ChatGrpcClient`调用目标节点`ChatServiceImpl`提供的gRPC接口完成跨节点通知。
- **文本聊天**：不存储聊天历史记录，仅将JSON格式的消息体转发至目标用户的在线会话；若目标用户在其他节点，则通过gRPC完成跨节点转发。转发成功后，服务器会向发送方回显已发送的消息内容，供客户端确认。
- **其他核心组件**：内存级别的“用户-会话”映射表在`UserMgr.cpp`中维护；Redis令牌/节点地址的操作工具类是`RedisMgr.cpp`；异步工作线程池的实现代码为`AsioIOServicePool.cpp`。

# 网关服务器（HTTP API）
- **接口路由**：提供`/user_register`（注册）、`/reset_pwd`（密码重置）、`/user_login`（登录）、`/get_verifycode`（获取验证码）等接口，同时包含少量测试接口，所有接口的处理逻辑均在`LogicSystem.cpp`中。
- **注册流程**：校验Redis中存储的邮箱验证码（键名格式为`code_<邮箱地址>`）→ 校验两次输入的密码是否一致 → 调用`MysqlDao.cpp`中的事务接口完成用户注册，生成新用户ID并写入用户的头像、昵称等信息。
- **密码重置流程**：重新校验邮箱验证码 → 校验邮箱与用户名是否匹配 → 更新MySQL中存储的用户密码。
- **登录流程**：通过MySQL校验用户邮箱与密码的正确性 → 调用状态服务器（`StatusServer`）的`GetChatServer`接口，获取负载最低的聊天节点信息与用户令牌 → 将聊天节点的地址、端口、用户令牌返回给客户端。
- **HTTP传输**：`HttpConnection.cpp`负责解析GET请求参数、分发请求至对应处理函数、生成JSON格式的响应数据并返回。

# 状态服务器
- **负载均衡**：读取Redis哈希表`logincount`中的数据，筛选出登录人数最少的聊天服务器；候选节点列表从配置文件中读取（代码在`StatusServiceImpl.cpp`）。
- **令牌签发**：生成UUID格式的用户令牌，存入Redis（键名格式为`utoken_<用户ID>`），并将令牌与选中的聊天节点地址、端口一并返回（接口为`GetChatServer`）。用户登录时，聊天服务器会校验客户端传入的令牌与Redis中的存储是否一致。

# 验证码服务（Node.js）
- `server.js`：实现gRPC的`GetVerifyCode`接口，先检查Redis中是否已存在该邮箱的有效验证码；若不存在，则生成新验证码并存入Redis并设置过期时间，再通过SMTP协议发送至用户邮箱。
- 配置文件为`config.json`；Redis交互工具类是`redis.js`；邮件发送工具类是`email.js`；Protobuf协议定义文件（`message.proto`）由C++服务与Node.js服务共享。

# 数据存储与通信协议
- **协议定义**：通过`message.proto`文件定义共享的Protobuf协议，包含状态查询、验证码申请、好友事件、文本聊天等场景的RPC接口与数据结构。
- **持久化数据**：MySQL中包含`user`（用户表）、`friend`（好友关系表）、`friend_apply`（好友申请表）、`user_id`（自增ID表）；Redis中存储的键值包括：用户令牌（`utoken_*`）、用户-服务器节点映射（`uip_*`）、节点登录人数（`logincount`）、验证码（`code_*`）、用户/好友信息缓存（`ubaseinfo_*`、`nameinfo_*`）。

# 典型端到端流程
1.  **注册流程**：客户端 → 网关服务器`/user_register`接口 → 校验Redis中的邮箱验证码 → 调用MySQL事务接口写入新用户数据 → 返回用户ID给客户端。
2.  **登录流程**：客户端 → 网关服务器`/user_login`接口 → MySQL校验邮箱密码 → 状态服务器筛选负载最低的聊天节点并生成令牌 → 客户端连接该聊天节点的TCP端口，发送登录消息（消息ID 1005），携带用户ID与令牌 → 聊天服务器校验Redis令牌有效性，加载用户资料与好友列表，绑定用户与会话。
3.  **消息/好友操作流程**：客户端通过TCP连接向所在的聊天服务器发送聊天/好友相关指令；若目标用户在其他节点，当前服务器会通过gRPC将指令转发至目标节点的`ChatService`服务，由目标节点推送消息至目标用户的会话。

# 补充说明
- `ChatServer2`是一个独立的部署单元；状态服务器的负载均衡功能，依赖配置文件中注册的多个聊天节点。
- 系统不支持消息持久化与离线消息队列，仅能实现在线消息的实时投递。
- 身份认证完全依赖Redis令牌，令牌由状态服务器在用户登录选节点时生成。
