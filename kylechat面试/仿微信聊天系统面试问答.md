# 仿微信聊天系统面试问答（去重排序版）


## 一、项目整体、架构与部署


### 1. 请你整体介绍一下这个仿微信聊天系统



项目定位：仿微信 IM → 再说客户端 Qt → 再说后端四服务 → 再说 HTTP 登录、TCP 聊天、gRPC 内部通信 → 最后说 Redis/MySQL/连接池亮点。

**回答：**
这个项目是一个基于 C++ 的仿微信分布式即时通讯系统。客户端使用 Qt 实现，支持注册、登录、好友搜索、好友添加和文字聊天。

后端拆分为四大服务，网关对外提供 HTTP 接口，验证服务 负责邮箱验证码，状态服务 负责选择聊天服务器和生成 token，聊天服务负责 TCP 长连接、用户会话和消息转发。

------


### 2. 为什么要把系统拆成 四大服务？



职责不同 → Gate 入口、Verify 验证码、Status 选服务器、Chat 长连接 → 独立部署、扩展 → 故障隔离。

**回答：**
因为这几个模块的职责和流量特点不一样。GateServer 负责统一入口和 HTTP 请求；VerifyServer 只负责验证码；StatusServer 负责选择合适的 ChatServer 和生成 token；ChatServer 负责 TCP 长连接、用户会话和聊天业务。
比如聊天连接压力大时，可以单独扩容 ChatServer；验证码服务出问题，也不会影响已经在线用户继续聊天。

**追问：不拆分，全部写在一个服务里行不行？**
功能上可以，但代码会高度耦合，后期扩容和维护都困难。尤其 IM 系统里登录、验证码和长连接聊天的流量模型完全不同，拆开更合理。

------


### 3. 这个项目最有亮点的地方是什么？



不是单机 demo → 分布式 IM 主链路 → Redis 用户位置 + gRPC 跨服转发 → Asio 异步长连接 → 消息队列解耦。

**回答：**
我觉得最大的亮点是它不是单机聊天 demo，而是实现了分布式 IM 的核心主链路。

用户可能连接在不同 ChatServer 上，系统通过 Redis 记录用户位置，再通过 gRPC 做跨服消息转发。

另一个亮点是网络层和业务层解耦。ChatServer 使用 Asio 处理 TCP 长连接，IO 线程读完消息后投递到消息队列，由业务线程统一处理。

**追问：相比普通聊天 demo，它强在哪里？**
普通 demo 往往只有一个服务器收发消息，而这个项目考虑了服务拆分、跨服务器转发、负载均衡和连接池。

从 CRUD 到通信系统 → 理解长连接、协议、异步 IO、服务拆分、缓存、数据库 → 工程上考虑异常和扩展。

**回答：**

以前可能更多是理解接口和数据库，但这个项目让我真正接触到长连接、消息协议、异步 IO、服务拆分、Redis 状态管理、MySQL 持久化和跨服务器通信。

同时我也认识到工程项目不只是功能能跑，还要考虑异常、断线、连接泄露、消息丢失、扩容和服务故障。

**追问：这个项目对你找后端有什么帮助？**
它能证明我不仅会写接口，还理解网络通信、并发模型和分布式系统的基本设计。

重点是自己理解架构和链路 → 能部署调试、压测、排查问题 → 能说不足和优化。

对我来说，重点不是“代码是不是从零闭门造车”，而是我是否真正理解系统设计，能不能定位问题，能不能说明不足和优化方向。


### 6. 为什么选择项目中的这些网络库和技术？



Asio 处理 TCP 长连接 → Beast 处理 HTTP → gRPC 处理内部 RPC → Redis 存状态缓存 → MySQL 存持久化 → Qt 做客户端。

**回答：**
ChatServer 需要处理大量 TCP 长连接，所以用 Boost.Asio 来实现异步 IO 和高并发连接管理。

GateServer 需要提供 HTTP 接口，所以用 Boost.Beast 处理 HTTP 请求。服务之间需要结构化通信，所以用 gRPC 和 Protobuf。

Redis 用来存 token、验证码、用户在线位置和热点用户信息，因为这些数据读写频繁，适合缓存和临时状态。

MySQL 用来存用户信息、好友关系和好友申请，因为这些数据需要持久化和关系查询。客户端用 Qt，是因为 Qt 适合做桌面 GUI，也提供网络模块和信号槽机制。

所以整体不是随便堆技术，而是 HTTP、TCP、RPC、缓存、数据库分别对应不同问题。

**追问：如果让你重新选技术，你会怎么改？**
如果做 Web 端，我可能会增加 WebSocket 网关；如果做生产级分布式部署，会引入服务注册中心、消息队列、异步日志和监控系统。

------


### 7. 为什么需要网关服务？没有 GateServer 行不行？



没有也能跑 → 但客户端要知道所有服务 → 不安全、不好扩展 → （GateServer 作为统一入口，可以屏蔽后端服务细节。）

**回答：**
没有 GateServer 功能上也可以跑，比如客户端直接请求 VerifyServer、StatusServer 或 ChatServer。但这样会带来很多问题。客户端需要知道多个服务地址，服务地址一变客户端也要改；内部服务直接暴露给客户端，安全性差；鉴权、限流、日志、路由也不好统一管理。
GateServer 作为统一入口，可以屏蔽后端服务细节。客户端只需要请求 GateServer，GateServer 再去调用内部服务。这样后端服务可以独立扩容、迁移，客户端感知不到。
所以 GateServer 的价值不只是转发请求，还包括统一入口、协议管理、安全控制、路由分发和后续扩展。

**追问：GateServer 会不会成为瓶颈？**
会，所以生产环境中 GateServer 也应该多实例部署，前面再加负载均衡，比如 Nginx、LVS 或云负载均衡。

------


### 8. 为什么要将登录和聊天分开不同的服务器？



登录短连接低频认证 → 聊天长连接高频状态 → 流量模型不同 → 独立扩展 → 故障隔离。

**回答：**
登录和聊天的业务特点完全不同。登录是短连接、低频、认证类请求，主要涉及账号密码校验、token 生成和服务器分配。聊天是长连接、高频、状态类业务，需要维护大量 TCP 连接、在线状态和实时消息推送。
如果把两者放在一起，聊天连接压力会影响登录接口，登录逻辑也会让 ChatServer 变得臃肿。拆开以后，登录服务可以专注认证和路由，ChatServer 专注长连接和消息转发。
这样既能独立扩容，也能隔离故障。比如 ChatServer 连接压力大时，扩容 ChatServer 即可，不一定要扩容登录服务。

**追问：登录完成后 GateServer 还参与聊天吗？**
一般不参与。客户端拿到 ChatServer 地址和 token 后，直接和 ChatServer 建立 TCP 长连接进行聊天。

------


### 9. 简历里写多服务器支持 1W 到 2W 活跃用户，为什么多服务器不是单服务器的正比扩容？



先承认理论上可以横向扩展 → 但不是简单正比 → 公共瓶颈：Gate、Status、Redis、MySQL、带宽 → 跨服通信有额外开销 → 压测环境和真实业务不同。

**回答：**
理论上 ChatServer 是可以横向扩展的，增加服务器确实可以提升连接承载能力。

但它不是一个严格的正比关系，不是说单服务器 8000 连接，两台就一定稳定支持 16000，三台就一定 24000。

原因是系统里除了 ChatServer，还有一些公共依赖，比如 GateServer、StatusServer、Redis、MySQL、网络带宽和机器资源。

多台 ChatServer 同时运行时，Redis 要维护更多在线状态，MySQL 要处理更多用户数据和消息数据，跨服务器消息还会产生 gRPC 调用开销。

所以我在简历里写的是一个相对保守的范围。多服务器部署确实能提升整体承载能力，但最终上限取决于整个系统链路，而不是只看 ChatServer 数量。

**追问：那你怎么让它更接近线性扩容？**
可以从几个方面做：**GateServer 多实例，StatusServer 多实例，Redis 做集群或哨兵，MySQL 做读写分离或分库分表，跨服消息引入 MQ，减少 ChatServer 两两直连压力。**这样才能让扩容效果更接近线性。

------


### 10. 不同聊天服务器是物理上的分离，还是逻辑上的分离？



（同机多端口模拟）不同 ChatServer 实例 → 本地测试可同机器不同端口 → 生产可不同机器不同 IP → 对客户端来说由 Status 分配。

**回答：**
从架构上看，不同聊天服务器是不同的 ChatServer 实例。

部署方式上可以灵活选择。本地测试时，为了方便，通常会在同一台机器上启动多个 ChatServer，用不同端口模拟多服务器部署。

如果是生产环境，更合理的是把不同 ChatServer 部署在不同机器或不同容器中，它们可以有不同的内网 IP 和端口。

客户端不需要关心这些服务器怎么部署，它只需要先登录 GateServer，然后由 StatusServer 返回一个合适的 ChatServer 地址。

所以可以总结为：逻辑上是不同服务实例，物理上既可以同机多端口模拟，也可以多机器分布式部署。

**追问：面试官问你本地测试是哪种？**
可以说本地测试一般是同一台机器上多个 ChatServer 实例，通过不同端口模拟分布式部署；真实上线时可以放到不同机器或容器里。

------


### 11. 项目中用的是私域 IP 还是公网 IP？如何获得公网 IP？



本地测试：127.0.0.1/内网 IP → 真实部署：云服务器公网 IP/EIP → 外部只暴露 GateServer → 内部服务走内网。

**回答：**
本地开发和测试时，一般使用 127.0.0.1 或局域网内网 IP，比如多个服务跑在同一台机器不同端口上。
如果真实部署到线上，就需要云服务器的公网 IP，或者云厂商提供的弹性公网 IP。客户端访问系统时，通常只需要访问对外暴露的 GateServer。GateServer、StatusServer、VerifyServer、ChatServer、Redis、MySQL 之间尽量走内网通信，这样更安全，也能减少公网暴露面。
所以我的项目测试阶段主要是本地或内网模拟，真正上线时应该通过云服务器、公网 IP、域名和负载均衡来部署。

**追问：为什么不把所有服务都暴露公网？**
安全风险大，也不好管理。一般只暴露网关或负载均衡入口，内部微服务和数据库走内网。

## 二、登录、认证、网关与状态服务


### 12. 用户登录的完整流程是什么？



两阶段：HTTP 登录校验，状态服务分配 token 和 ChatServer 地址 → TCP 连接 ChatServer → Redis 校验 token → 加载用户/好友 → 绑定 session 和 uid。

**回答：**
登录分为两阶段。第一阶段是 HTTP 登录，客户端把邮箱和密码发给 GateServer，GateServer 校验 MySQL 中的用户信息。

校验成功后，GateServer 调用 StatusServer，StatusServer 选择一台负载较低的 ChatServer，生成 token 存入 Redis，然后返回 host、port、uid 和 token。

第二阶段是 TCP 登录，客户端拿到 ChatServer 地址后建立 TCP 长连接，把 uid 和 token 发给 ChatServer。

ChatServer 从 Redis 中校验 token，成功后加载用户信息、好友列表和好友申请列表，并把 uid 和 session 绑定，同时记录用户所在服务器。

**追问：为什么不直接让 GateServer 完成登录后就聊天？**
因为 GateServer 适合处理短连接 HTTP 请求，而聊天需要长期 TCP 连接和实时推送，所以登录认证和聊天连接要分开。

------


### 13. GateServer 是怎么实现的？



Boost.Beast HTTP → async_read → 解析 GET/POST → 路由表分发 → 登录/注册/验证码 → 内部调 gRPC。

**回答：**
GateServer 是系统的 HTTP 网关，用 **Boost.Beast 处理 HTTP 请求。**

它异步读取请求，根据 method 和 URL 分发到对应 handler。

比如获取验证码、注册、登录、重置密码都是 POST 接口。

GateServer 不处理 TCP 长连接聊天，而是负责账号类入口逻辑，并通过 gRPC 调用 VerifyServer 或 StatusServer。

**追问：GateServer 完全不处理业务吗？**
不能这么说。它不处理聊天长连接业务，但会处理登录注册这类入口业务和协议转换。

------


### 14. 登录用的是 HTTP 的哪个方法？为什么用 POST 不用 GET？



登录用 POST → 账号密码放 body → GET 参数暴露在 URL、日志、历史记录 → POST 语义是提交数据 → 更适合登录。

**回答：**
登录接口应该使用 POST。因为登录需要提交邮箱、密码这类敏感信息，POST 可以把数据放在 request body 中。

如果用 GET，参数会直接出现在 URL 中，可能被浏览器历史、服务器日志、代理日志记录下来，安全性更差。

而且从语义上看，GET 更适合查询资源，POST 更适合提交数据或触发状态变化。

所以登录、注册、重置密码、验证码校验这类接口更适合使用 POST。

**追问：POST 就绝对安全吗？**
不是。POST 只是比 GET 更适合传敏感数据，但真正安全还需要 HTTPS，密码也要做哈希存储，服务端还要防止暴力破解和重放攻击。

------


### 15. VerifyServer 验证码流程是怎样的？



客户端请求验证码 → Gate 调 Verify → 生成验证码 → Redis 设置过期 → 邮件发送 → 注册时比对。

**回答：**
客户端请求验证码时，先请求 GateServer。GateServer 通过 gRPC 调用 VerifyServer。

VerifyServer 生成验证码，写入 Redis，并设置过期时间，然后发送邮件给用户。

注册或重置密码时，GateServer 会根据邮箱从 Redis 中取验证码并比对，验证码正确才继续后续操作。

**追问：验证码为什么放 Redis？**
验证码是临时数据，需要过期时间，读写频繁，不需要长期持久化，所以 Redis 很适合。

------


### 16. Token 为什么放 Redis？ChatServer 如何验证 token？



Status 生成 token → Redis 共享给 ChatServer → 客户端 TCP 登录带 token → ChatServer 查 Redis 比对 → 跨服务认证。

**回答：**
token 由 StatusServer 在登录阶段生成，并存入 Redis。

客户端连接 ChatServer 时，会带上 uid 和 token。

ChatServer 根据 uid 拼出 token key，从 Redis 中取出 token 并进行比对。

这样 GateServer、StatusServer 和 ChatServer 不需要共享内存，只要都能访问 Redis，就可以完成跨服务认证。

**追问：token 有什么安全改进？**
可以设置过期时间、刷新机制、绑定设备信息，也可以改成 JWT 或增加签名校验。

------


### 17. StatusServer 怎么做负载均衡？



Redis 记录每台 ChatServer 在线数 → Status 遍历服务器 → 选连接数最少 → 返回 host、port、token。

**回答：**
StatusServer 使用最小连接数策略。每台 ChatServer 的在线人数记录在 Redis 中。用户登录时，StatusServer 遍历配置中的 ChatServer 列表，读取每台服务器的连接数，选择连接数最少的一台返回给客户端。
相比轮询，最小连接数更能反映当前负载，因为不同用户在线时长不一样。

**追问：这个负载均衡有什么不足？**
依赖 Redis 中的计数，如果断线清理不及时，计数可能不准。

另外 StatusServer 本身也需要做多实例和健康检查。

------


### 18. 用户在线状态如何管理？



两层：本地 UserMgr 存 uid-session → Redis 存 uid-server → 登录写入 → 断线清理。

**回答：**
在线状态分两层。本地层用 UserMgr 维护 uid 到 session 的映射，用于当前 ChatServer 快速找到用户连接。

全局层用 Redis 维护 uid 到 ChatServer 名称的映射，用于跨服务器查找用户位置。

用户登录时写入这两层状态，用户断线时需要清理本地 session，并删除或更新 Redis 中的在线位置。

**追问：为什么要两层状态？**
本地内存查询快，但只能查当前服务器；Redis 稍慢，但能跨服务器共享用户在线位置。

------


### 19. 如果引入服务注册发现，你会怎么做？



现在配置写死 → 引入 Zookeeper→ ChatServer 注册和心跳 → Status 动态获取健康节点。

**回答：**
当前 ChatServer 列表更多依赖配置文件。后续可以引入注册中心，比如 Zookeeper。

ChatServer 启动时向注册中心注册自己的地址和状态，并定期发送心跳。

StatusServer 从注册中心获取可用 ChatServer 列表，而不是写死配置。

某台 ChatServer 下线或心跳超时后，StatusServer 不再把新用户分配过去。

**追问：注册中心解决什么问题？**
解决服务地址写死、服务上下线无法感知、健康检查不足和扩容不方便的问题。

## 三、客户端通信、TCP 协议与网络模型


### 20. 为什么使用 Qt 做客户端？Qt 里信号槽有什么作用？



Qt 适合桌面客户端 → GUI + Qt Network → 信号槽解耦 UI 和网络 → 收消息通知界面更新。

**回答：**
Qt 适合做桌面客户端，既有 GUI 能力，也有 Qt Network 网络模块。

项目中 Qt 负责登录注册界面、好友列表、聊天窗口、消息气泡，以及 HTTP/TCP 通信。
信号槽可以解耦 UI 和网络逻辑。比如网络层收到服务器消息后，通过信号通知聊天界面更新；

用户点击发送按钮后，也可以通过信号触发消息封装和发送。

**追问：信号槽和普通回调有什么区别？**
信号槽更适合 Qt 对象体系，支持对象自动断开、跨线程队列连接和类型检查，比普通回调更解耦。

------


### 21. 为什么登录用 HTTP，而聊天用 TCP 长连接？为什么不用**websocket**？



登录低频一次性，用 HTTP 简单 → 聊天高频实时，需要服务端主动推送 → TCP 长连接复用，延迟低，自定义TCP更轻量，方便自己自定义消息协议。

**回答：**
登录、注册、验证码属于低频的一次性请求，用 HTTP 更简单，也方便和客户端交互。

聊天是高频实时场景，需要服务端主动推送消息。如果用 HTTP，服务端很难主动给客户端推消息，只能轮询（**客户端不断发起请求**，询问服务端是否有新的消息。），效率低、延迟高。

TCP 长连接建立后可以一直复用，服务端可以随时通过这条连接把好友消息、好友申请、认证结果推送给客户端。

WebSocket 的优势是**浏览器支持好**，但它本质上也是基于 TCP 的应用层协议。我的客户端可以直接使用 TCP Socket，所以直接 TCP 更轻量，也更方便我自己设计 `msgid + length + body` 的消息协议，解决粘包拆包和消息分发问题。

当然，如果后续要支持 Web 端，我会在接入层增加 WebSocket 网关，把 WebSocket 消息转换成内部统一协议，再转发给 ChatServer。这样 TCP 和 WebSocket 可以共存。

**追问：HTTP keep-alive 不是也能复用连接吗？**
HTTP keep-alive 只是减少重复建连，不代表服务端可以自由主动推送。IM 场景更适合 TCP 长连接或 WebSocket。

------


### 22. 客户端和服务端只有一个 TCP 连接吗？没有其他连接吗？



聊天阶段客户端到 ChatServer 一条 TCP 长连接 → 登录注册验证码走 HTTP → 服务间还有 gRPC → 后端访问 Redis/MySQL 连接池。

**回答：**
不能简单说只有一个连接。**对于一个客户端来说，聊天阶段会和某一台 ChatServer 保持一条 TCP 长连接，这条连接用于实时消息收发和服务端推送。**

但登录、注册、验证码这类功能不是走这条 TCP 长连接，而是通过 HTTP 请求 GateServer。

后端服务之间还有 gRPC 连接，比如 GateServer 调 StatusServer，ChatServer 跨服转发消息。后端访问 Redis 和 MySQL 时，也会通过连接池维护多个连接。

所以从客户端视角看，聊天主要是一条 TCP 长连接；从整个系统视角看，还有 HTTP、gRPC、Redis、MySQL 多类连接。

**追问：为什么聊天阶段一条 TCP 连接就够？**
TCP 是全双工的，一条连接可以同时收发数据。普通 IM 文本消息没有必要为每个好友建立单独连接。

### 23. 客户端是用什么接口进行消息发送的？



HTTP 用 QNetworkAccessManager → 聊天 TCP 用 QTcpSocket → 先组包头和 JSON body → socket.write 发送 → readyRead 接收。

**回答：**
客户端中登录、注册、验证码这类 HTTP 请求可以使用 Qt 的 QNetworkAccessManager 发送。
聊天消息则通过 QTcpSocket 和 ChatServer 建立 TCP 长连接。

发送消息时，客户端先把业务数据封装成 JSON，然后加上自定义包头，包括 msgid 和 body 长度，最后调用 QTcpSocket 的 write 接口发送。

接收消息时，客户端监听 readyRead 信号，从 socket 中读取数据，同样按照包头和 body 的格式解析。

**追问：为什么不用 HTTP 接口发送聊天消息？**
HTTP 不适合服务端主动推送，聊天消息更需要长连接和实时性，所以用 TCP 更合适。

### 24. TCP 消息协议是怎么设计的？怎么解决粘包拆包？body部分是什么内容？



自定义协议 → 4 字节头：2 字节 msgid + 2 字节 len → 先读头再读 body → 按长度解决粘包拆包。

**回答：**
项目使用自定义 TCP 协议，格式是固定 4 字节包头加 JSON 数据体。前 2 字节是 msgid，用来区分登录、搜索用户、添加好友、聊天等业务类型；

后 2 字节是 body 长度；后面是 JSON 数据。

接收时不会把一次 read 当作完整消息，而是先读取固定长度包头，解析出 msgid 和 body 长度，再按长度读取完整 body。这样即使 TCP 出现粘包或拆包，也能根据长度字段还原完整消息。

body 里面携带具体的业务数据，JSON 字符串。包头里的 msgid 只负责标识消息类型；len 负责标识 body 长度，用来解决粘包拆包。



真正的业务字段都放在 body 中。比如**登录消息的 body 里有 uid 和 token**；添加好友消息里有 uid、touid、applyname、bakname；

文本聊天消息里有 fromuid、touid 和 text_array，text_array 里包含每条消息的 msgid 和 content。

服务端先根据包头 msgid 找到对应 handler，然后在 handler 里解析 JSON body，取出业务字段继续处理。

**追问：为什么 body 用 JSON，不全用二进制？**
JSON 可读性好，调试方便，适合项目开发和展示。包头用二进制保证基础协议紧凑，body 用 JSON 保证扩展方便。生产环境可以换成 Protobuf。

------


### 25. 客户端怎么区分不同类型的消息？



包头 msgid 区分业务 → 登录/搜索/加好友/聊天不同 ID → body JSON 放字段 → 新增类型加 msgid 和 handler。

**回答：**
**客户端和服务端约定统一的 msgid**。

比如登录、搜索用户、添加好友、认证好友、文本聊天都有不同的请求和响应 ID。

发送时，客户端把 msgid 写入 TCP 包头，body 中放 JSON 业务字段。服务端收到后先解析 msgid，再分发到对应 handler。新增一种消息类型时，只需要新增 msgid、定义 JSON 字段，并在服务端注册处理函数。

**追问：如果是图片、文件怎么办？**
可以在 body 中增加 type 或 msgFlag 字段区分内容类型，但大文件本身不建议直接放进聊天 TCP 通道。

### 26. 为什么不用 UDP？



聊天消息不能丢 → TCP 可靠有序自带重传 → UDP 适合音视频但要自己处理可靠性。

**回答：**
普通聊天消息对可靠性要求比较高，不能随便丢，也需要保证顺序。TCP 自带可靠传输、顺序保证、重传和拥塞控制，更适合文字聊天。
UDP 虽然延迟低，但不保证到达，也不保证顺序。如果用 UDP 做聊天，就要自己实现确认、重传和乱序处理，复杂度会高很多。

**追问：那音视频为什么常用 UDP？**
音视频更看重实时性，少量丢包可以接受，所以更适合 UDP 或基于 UDP 的实时传输协议。文字消息更重视可靠性。

### 27. ChatServer 的网络模型是怎样的？



Asio 异步 IO → 多 io_context 线程池 → 每个连接一个 CSession → 读完消息投 LogicSystem 队列 → 业务线程分发 。

**回答：**
ChatServer 使用 Boost.Asio 实现异步 TCP 服务。

网络层维护多个 io_context，每个 io_context 在一个线程中运行，新连接通过轮询方式分配到不同 io_context。

每个客户端连接对应一个 CSession（**服务端对一个客户端连接的封装类**），负责异步读取包头、包体和发送响应。

完整消息读取后，不直接在 IO 线程处理业务，而是投递到 LogicSystem 消息队列，由业务线程根据 msgid 分发到不同处理函数。

**追问：这种模型有什么好处？**
IO 线程只负责网络读写，不会被业务阻塞；业务逻辑集中处理，也能减少共享数据竞争。

------


### 28. 为什么要用异步 IO，不直接一个连接一个线程？



一个连接一个线程资源浪费 → 长连接大部分时间空闲 → 异步事件驱动少量线程管理大量连接 → 降低上下文切换。

**回答：**
IM 系统有大量长连接，但**很多连接大部分时间是空闲**的。

如果一个连接一个线程，会创建大量线程，占用很多内存，还会产生频繁上下文切换。
**异步 IO 是事件驱动**的，线程不用阻塞等待某个连接，而是等 socket 可读可写时再触发回调。这样少量线程就可以管理大量连接，更适合长连接场景。

**追问：异步 IO 有什么缺点？**
**代码复杂度更高，需要处理回调、对象生命周期和并发问题**，所以项目里通过 shared_ptr 和消息队列降低复杂度。

------


### 29. io_context是什么， 线程池是怎么设计的？



多个 io_context → 每个一个线程 run → 新连接轮询分配 → work 保活 → stop 时统一退出。

**回答：**

`io_context` 是 Boost.Asio 的事件循环对象，可以理解为异步 IO 的调度中心。

客户端连接、异步读、异步写这些操作都会注册到 `io_context` 上，线程调用 `io_context.run()` 后，它就会不断等待 IO 事件完成，并执行对应的回调函数。

项目封装了 AsioIOServicePool，内部维护多个 io_context，每个 io_context 由一个独立线程执行 run。新连接到来时，通过 Round-Robin 方式选择一个 io_context，把连接分配给它。
这样可以利用多核 CPU，避免所有连接都集中在一个事件循环上。项目中还通过 work 对象保持 io_context 不会因为暂时没有任务而退出，停止时统一 stop 并 join 线程。

**追问：为什么不用一个 io_context 多线程 run？**
也可以，但多个 io_context 可以减少同一个事件队列上的竞争，每个连接绑定一个 io_context，结构更清晰。

------


### 30. io_context （asio网络库的事件循环对象）在单线程和多线程下分别是怎么工作的？



单线程：一个 io_context 一个 run，回调串行 → 多线程：多个线程 run 同一个或多个 io_context → 项目是多个 io_context 轮询分配连接。

**回答：**
单线程情况下，一个 io_context 由一个线程执行 run，所有异步事件的回调都在这个线程中串行执行。

优点是简单，不需要太多锁；缺点是只能利用一个 CPU 核心，高并发时容易成为瓶颈。
多线程情况下有两种常见方式。一种是多个线程同时 run 同一个 io_context，这样多个回调可以被不同线程执行，但共享事件队列可能有竞争。另一种是创建多个 io_context，每个 io_context 绑定一个线程，新连接按轮询或 hash 分配到不同 io_context。
我的项目采用的是多个 io_context 的思路，新连接轮询分配。这样可以利用多核，也能让连接和事件循环的关系更清晰。

**追问：多线程 io_context 会不会有并发问题？**
会。只要多个线程可能访问共享数据，就需要加锁或通过消息队列串行化处理。项目里 IO 层和业务层通过消息队列解耦，可以减少共享数据冲突。

------


### 31. Asio 网络库底层原理是什么？



io_context 是事件循环 → async_read/write 注册事件 → 操作系统监听可读可写 → Linux epoll、Windows IOCP → 就绪后执行回调。

**回答：**
Asio 的核心是异步事件驱动。io_context 可以理解为事件循环，程序调用 async_read、async_write、async_accept 时，不会阻塞当前线程，而是把异步操作注册到 io_context 中。
底层会依赖操作系统的 IO 多路复用机制，比如 Linux 下常见的是 epoll，Windows 下是 IOCP。当 socket 可读、可写或有新连接到来时，io_context 会取出对应事件，并执行之前绑定的回调函数。
项目中多个 io_context 跑在多个线程上，就可以让多个线程并发处理不同连接上的 IO 事件，从而提高长连接处理能力。

**追问：Asio 是不是一定等于 epoll？**
不是。Asio 是跨平台异步 IO 库，Linux 下可以基于 epoll，Windows 下可以基于 IOCP。它屏蔽了不同平台底层差异。

------


### 32. 为什么业务逻辑不直接放在 IO 线程里处理？



IO 线程要轻 → 查库/Redis 会阻塞 → 投消息队列 → 业务线程处理 → 解耦、减少锁。

**回答：**
IO 线程应该尽量只做网络读写。如果在 IO 回调里直接处理业务、查 MySQL 或访问 Redis，一旦某个操作变慢，就会影响同一个 IO 线程上的其他连接。
所以项目中 IO 线程读取完整消息后，只负责投递到 LogicSystem 消息队列。业务线程从队列取消息，根据 msgid 调用对应 handler。这样网络层和业务层解耦，整体更稳定。

**追问：业务线程查数据库也可能阻塞，怎么办？**
当前项目业务量较轻，可以接受。后续可以把数据库访问异步化，或者放入独立业务线程池。

------


### 33. 单业务线程会不会成为瓶颈？



当前业务轻，单线程保证简单正确 → 高并发会瓶颈 → 后续按 uid 分片、多 worker、数据库异步化。

**回答：**
会有成为瓶颈的可能。当前项目中业务相对轻量，单业务线程可以减少并发问题，保证逻辑简单和数据一致。
如果消息量继续增加，就可以升级为多业务线程。比如**按 uid hash 分片，不同用户的消息进入不同队列**；

或者**把数据库查询、跨服务器 RPC 这些耗时操作放到独立线程池中**。

**追问：为什么不一开始就做多业务线程？**
多线程业务会引入更多锁、顺序和一致性问题。项目早期先保证主链路正确，再逐步优化更稳妥。

### 34. CSession 的作用是什么？



一个客户端连接的抽象 → 管 socket、session_id、uid、读缓冲、发送队列 → 异步读写 → 投递业务队列。

**回答：**
**CSession 可以理解为一个客户端 TCP 连接的抽象。**

它内部管理 socket、session_id、用户 uid、读缓冲区和发送队列。

连接建立后，CSession 负责异步读取包头和包体，收到完整消息后投递到 LogicSystem。发送消息时，也通过 CSession 的发送队列进行异步写。

**追问：为什么 CSession 要用 shared_ptr？**
异步回调可能在函数返回后才执行，如果对象提前析构就会出错。shared_ptr 可以保证回调执行期间 session 仍然存活。

------


### 35. 发送消息时如何避免并发写乱序？



同一 socket 不能多个 write 乱写 → 每个 session 一个发送队列 → 一次只 async_write 队首 → 写完再发下一条。

**回答：**
同一个 TCP 连接上不能随便多个 async_write 并发写，否则可能出现数据交错。

**项目里为每个 session 维护发送队列。**

发送时先把消息放入队列，如果当前没有正在写，就启动 async_write。写完成后弹出队首，再继续发送下一条。这样同一连接上的消息是串行写出的，顺序更稳定。

**追问：发送队列满了怎么办？**
可以丢弃、限流或断开慢连接。项目里设置最大队列长度，是为了避免单个连接堆积太多消息占用内存。

------


### 36. 断线清理怎么做？



read/write error → Close socket → ClearSession → 删除 session map → 删除 UserMgr 绑定 → 清 Redis 在线状态。

**回答：**
当 async_read 或 async_write 出现错误，比如客户端断开连接，服务端会关闭 socket，并调用 ClearSession 清理连接。

清理时需要从 session map 删除连接，从 UserMgr 删除 uid 到 session 的映射，并清理 Redis 中用户在线位置，同时更新服务器登录计数。

这个流程很重要，否则会出现用户已经离线，但 Redis 仍然显示在线的脏状态。

**追问：如果服务进程直接崩了，还能清理吗？**
进程崩溃时本地清理逻辑可能执行不了，所以生产环境需要心跳、过期 key 或服务健康检测来兜底。

------


### 37. gRPC 调用是同步还是异步？有没有考虑超时？



当前偏同步调用，简单清晰 → 风险是阻塞业务线程 → 后续加 deadline、重试、异步 gRPC 或 RPC 线程池。

**回答：**
当前项目里的 gRPC 调用更偏同步调用，优点是实现简单、调用链清晰。缺点是如果对端服务慢或不可用，可能阻塞当前业务线程。
更完善的方案是给 gRPC ClientContext 设置 deadline，超时后返回错误，并做有限次数重试。高并发场景下，也可以改成异步 gRPC，或者把 RPC 调用放到独立线程池。

**追问：同步调用一定不好吗？**
不一定。项目规模较小时同步调用简单可靠。只有当调用耗时明显影响吞吐时，才需要异步化。

------


### 38. gRPC 如何实现断线重连？



Channel 底层有连接管理 → 调用层设置 deadline → 失败按状态码处理 → 重试 + 指数退避 → 长期失败摘除节点。

**回答：**
gRPC 的 Channel 底层本身有一定的连接管理和重连能力，但在业务层不能完全依赖底层自动恢复。

更稳妥的做法是：每次 RPC 调用设置 deadline，避免一直阻塞；如果调用失败，根据错误码判断是超时、不可用还是连接断开；然后做有限次数重试。

重试时不能无脑立即重试，否则对端服务恢复前会造成大量请求打过去。一般会使用指数退避，比如第一次 100ms，第二次 200ms，第三次 500ms。

如果某个 ChatServer 或服务长期不可用，应该从可用节点中临时摘除，后续通过健康检查恢复。

**追问：项目里现在做到这么完整了吗？**
当前项目更偏学习版，gRPC 调用主要保证主流程打通。生产级可以继续补充 deadline、重试、健康检查和节点摘除机制。

------


### 39. 如果用户登录后长时间没有请求，需要重新登录吗？心跳机制怎么设计？



空闲不等于断开 → 定期心跳保活 → 服务端记录最后活跃时间 → 超时关闭 → 连接还活着不用重登，断了才重连认证。

**回答：**
用户登录后长时间没有发消息，不代表一定要重新登录。只要 TCP 长连接还活着，用户后续发消息可以继续使用这条连接。
但实际网络中，连接可能因为 NAT、路由器、防火墙或服务端超时被断开，所以需要心跳机制。

客户端可以定期向服务端发送心跳包，服务端收到后更新用户最后活跃时间。如果服务端长时间收不到心跳，就认为连接失效，关闭 socket，并清理用户在线状态。
如果连接还活着，不需要重新登录；如果连接已经断开，客户端需要重新连接 ChatServer，并携带 token 重新认证。

**追问：心跳间隔怎么设置？**
不能太短，否则浪费网络和服务器资源；也不能太长，否则断线发现太慢。可以根据业务要求设置，比如 30 秒或 60 秒一次，服务端允许连续几次心跳丢失后再断开。

### 40. 项目中有没有遇到死锁或内存 crash？怎么避免？



死锁：减少多锁嵌套 → 锁粒度小 → 不持锁做 IO/DB → RAII 管锁 → 内存：shared_ptr 生命周期、session 清理、连接归还。

**回答：**
项目中没有刻意设计复杂的多锁嵌套，主要共享数据是 session map（存储当前在线用户会话的映射表）、发送队列、消息队列和连接池。

为了避免死锁，我会控制锁粒度，只在访问共享容器时加锁，**不在持锁期间做网络 IO、数据库查询或 RPC 调用**。

加锁也尽量使用 lock_guard 或 unique_lock，让锁在作用域结束时自动释放。

内存 crash 主要关注异步回调中的对象生命周期。因为 async_read 和 async_write 的回调可能在函数返回后才执行，所以 CSession 需要通过 shared_ptr 保证回调期间对象还存活。
另外，连接池也要注意连接归还，避免连接泄露。session 断开时要清理 socket、UserMgr 映射和 Redis 在线状态。

**追问：如果真的出现死锁，你怎么排查？**
先看线程堆栈，确认哪些线程卡在哪些锁上；再检查是否有多把锁嵌套、加锁顺序不一致、持锁期间调用外部接口等问题。

------


### 41. 项目是否关注过内存泄漏？如何检查内存泄漏？



关注点：会话连接的生命周期、连接池的归还、new/delete、shared_ptr 循环引用 → 工具：valgrind、VS 诊断 → 日志观察析构。

**回答：**
这个项目中需要重点关注几类内存问题。第一是 session 生命周期，因为异步回调可能延迟执行，如果对象提前释放会 崩溃；第二是连接池中的连接是否都能归还；第三是是否存在 new 后没有 delete 的情况；第四是 shared_ptr 是否形成循环引用，导致对象无法析构。
检查方法上，Linux 下可以用 valgrind ，Windows 下可以用 Visual Studio 的诊断工具。

项目调试时也可以在关键类析构函数中打印日志，比如 CSession 析构、连接池析构等，观察断线后对象是否正常释放。

另外，RAII 是避免泄漏的重要思想。比如锁用 lock_guard，连接归还用 Defer 或封装对象，内存尽量用智能指针管理。

**追问：shared_ptr 一定安全吗？**
不是。shared_ptr 可以解决对象提前释放问题，但如果两个对象互相持有 shared_ptr，可能形成循环引用。需要时可以用 weak_ptr 打破循环。


## 四、gRPC、跨服通信与分布式扩展


### 42. gRPC 在项目中解决了什么问题？



服务间通信 → Gate 调 Verify/Status → ChatServer 跨服转发 → Protobuf 接口清晰 → 内部 RPC 合适。

**回答：**
**gRPC 主要解决后端服务之间的通信问题。**

比如 GateServer 获取验证码时调用 VerifyServer，登录时调用 StatusServer 获取 ChatServer 地址，两个 ChatServer 之间通过 gRPC 转发跨服务器消息。
相比自己再写一套内部 TCP 协议，gRPC 有清晰的 proto 接口定义，使用 Protobuf 序列化，适合内部服务调用。

**追问：为什么客户端和 ChatServer 不直接用 gRPC？**
聊天需要长连接和服务端主动推送，自定义 TCP 协议更灵活。gRPC 更适合后端服务间 RPC 调用。

------


### 43. gRPC 是怎么来的，为什么叫 gRPC？



 RPC 概念 → Google 内部 Stubby → 开源成 gRPC → g 早期可理解为 Google → 本质是高性能 RPC 框架 → 项目里用于服务间通信。

**回答：**
**RPC 的意思是远程过程调用，也就是像调用本地函数一样调用远程服务。**

gRPC 最早可以理解为 Google 开源的一套高性能 RPC 框架，它的前身和 Google 内部使用的 Stubby 系统有关。后来 Google 把类似的思想开源出来，形成了 gRPC。
至于为什么叫 gRPC，早期很多人会把 g 理解为 Google，因为它确实是 Google 推出的。后来官方在不同版本中给 g 赋予过不同含义，但在面试里不需要纠结名字，重点是说清楚它本质上是一个基于 Protobuf 和 HTTP/2 的高性能 RPC 框架。
在我的项目中，gRPC 主要用于服务间通信，比如 GateServer 调用 StatusServer 获取聊天服务器地址，GateServer 调用 VerifyServer 获取验证码，ChatServer 之间进行跨服消息转发。

**追问：gRPC 和普通 HTTP 接口有什么区别？**
普通 HTTP 接口通常是 REST 风格，传 JSON，适合对外接口；gRPC 更像函数调用，接口通过 proto 文件定义，序列化用 Protobuf，通信效率更高，更适合后端内部服务调用。

------


### 44. 两个用户在不同 ChatServer 上，消息怎么转发？



登录写 Redis 用户位置 → A 发给 B 先查 uip_B → 同服本地 session 推送 → 跨服 gRPC 转发。

**回答：**
用户登录成功后，ChatServer 会把用户所在服务器写入 Redis，比如 `uip_uid -> chatserver1`。

同时本机 UserMgr 会保存 `uid -> session`。

A 给 B 发消息时，A 所在的 ChatServer 先查 Redis，判断 B 当前在哪台服务器。

如果 B 在本机，就直接通过 UserMgr 找到 session 推送；

如果 B 在其他 ChatServer，就通过 gRPC 调用目标服务器的通知接口，再由目标服务器推送给 B。

**追问：为什么用户位置不只存在本地内存？**
本地内存只能查当前服务器的用户。分布式部署后，用户可能在其他 ChatServer，所以需要 Redis 存全局在线位置。

------


### 45. 如果有成千上万台 ChatServer，还能两两 gRPC 通信吗？



两两 gRPC 会 O(n²) 连接爆炸 → 引入 MQ/路由层 → ChatServer 投递到队列 → 目标服务器订阅。

**回答：**
不能简单两两直连。ChatServer 数量很大时，如果每台都维护到其他所有服务器的 gRPC 连接，连接数量会接近 O(n²)，资源开销和管理复杂度都会很高。

更合理的方案是引入消息队列或路由层。

**ChatServer 把跨服消息投递到 MQ，目标服务器订阅自己的队列或 topic。这样服务器之间不用全互连，也更容易削峰和扩展。**

**追问：MQ 会不会增加延迟？**
会有一点，但换来了削峰、解耦、可靠投递和扩展性。对于普通文本消息，这个延迟通常可以接受。

------


## 五、Redis、MySQL、缓存与连接池


### 46. Redis 在项目里主要存什么？



验证码 → token → 用户位置 → 登录计数 → 用户基础信息缓存 → 临时状态和热点数据。

**回答：**
Redis 主要存五类数据。

第一是验证码，设置过期时间；

第二是登录 token，用于 ChatServer 校验身份；

第三是用户位置，用于跨服务器消息转发；

第四是登录计数，用于负载均衡；

第五是用户基础信息缓存，用于减少 MySQL 查询。
Redis 适合存临时状态、高频访问数据和跨服务共享状态。

Redis 用来缓存 **高频访问的用户信息**（如在线状态、token 和会话信息等），这些信息变化较慢但访问频繁。将这些信息存储在 Redis 中，能够 **提高查询速度**、**减轻数据库负载**，并能 **支持高并发的访问**。Redis 的 **TTL 机制** 还能确保缓存数据的时效性和一致性。

**追问：Redis 挂了怎么办？**
核心数据以 MySQL 为准。Redis 挂了会影响 token、在线状态和缓存，所以生产环境需要 Redis 高可用，比如 Sentinel 或 Cluster。

------


### 47. MySQL 主要存什么？为什么不全放 Redis？



用户信息 → 好友关系 → 好友申请 → 需要持久化和关系查询 → Redis 只做缓存/状态。

**回答：**
MySQL 主要存用户账号信息、好友关系、好友申请等需要长期保存的数据。这些数据需要持久化，也可能涉及关系查询，更适合放在 MySQL 中。
Redis 虽然快，但内存成本高，也不适合复杂关系查询，更适合作为缓存和临时状态存储。所以项目采用 MySQL 存核心数据，Redis 存热点缓存和在线状态。

**追问：好友关系为什么适合 MySQL？**
好友关系需要长期保存，还涉及双向关系、申请状态、备注等字段，用 MySQL 建表和加索引更清晰。

------


### 48. Redis 和 MySQL 如何保证一致性？



MySQL 为准 → Redis 是缓存/状态 → 查询先缓存后数据库 → 写操作先写库再删/更新缓存 → 最终一致。

**回答：**
项目中核心数据以 MySQL 为准，Redis 主要做缓存和状态共享。

查询时先查 Redis，没命中再查 MySQL，并把结果回填 Redis。

写操作更推荐先写 MySQL，成功后删除或更新 Redis，避免缓存长期脏数据。

严格强一致比较难，一般采用最终一致性。

**追问：为什么不是先删缓存再写数据库？**
先删缓存再写数据库，中间如果有读请求，可能读到旧数据并重新写入缓存。更常见做法是先写库，再删缓存，必要时延迟双删。

------


### 49. 如何防止缓存穿透、击穿、雪崩？



穿透：空值/布隆过滤器 → 击穿：热点 key 锁/不过期 → 雪崩：随机过期、预热、限流。

**回答：**
缓存穿透是查询不存在的数据，可以用空值缓存或布隆过滤器。

缓存击穿是热点 key 过期导致大量请求打到数据库，可以用互斥锁或热点 key 不过期。

缓存雪崩是大量 key 同时过期，可以给过期时间加随机值，并做缓存预热和限流。

**追问：验证码也会有雪崩问题吗？**
验证码 key 分散且生命周期短，通常不是主要雪崩来源。用户信息、热点关系链更需要关注。

------


### 50. 连接池是怎么设计的？为什么要做连接池？



预创建连接 → 队列保存 → 获取加锁等待 → 用完归还 → 减少创建开销 → 控制资源上限。

**回答：**
连接池的核心是预先创建一定数量的连接，放入队列中。

请求来了就从池子里取连接，用完再归还，而不是每次都重新创建和销毁连接。
连接池内部通过 mutex 保证队列线程安全，通过 condition_variable 在无可用连接时等待。

这样可以减少连接创建开销，也能控制 MySQL、Redis、gRPC 的最大并发连接数。

**追问：连接池数量怎么设置？**
不是越大越好。太少会排队，太大会增加数据库压力和锁竞争。需要根据 QPS、平均响应时间、数据库最大连接数和压测结果调整。

------


### 51. 如何防止连接泄露？



问题是中途 return 忘归还 → 用 RAII/Defer → 析构自动 returnConnection → 再加健康检查。

**回答：**
连接泄露常见原因是函数中途 return 或异常退出，导致连接没有归还到连接池。

项目里可以用 RAII 思想，比如 Defer 对象，在作用域结束时自动执行归还连接。
这样无论函数正常结束还是提前返回，连接都会被归还，比靠人工记住 returnConnection 更安全。

**追问：除了防泄露，还要注意什么？**
还要做连接健康检查，比如 Redis PING、MySQL SELECT 1。失效连接要剔除并重建。

------


### 52. 连接池大小怎么确定？



不是越大越好 → 太小排队，太大压垮 DB → 根据 QPS、响应时间、最大连接数、压测调优。

**回答：**
连接池大小不能随便设置，也不是越大越好。太小会导致请求等待，太大会增加数据库或 Redis 压力，也会增加锁竞争。
一般可以根据 QPS、平均响应时间、数据库最大连接数和机器资源进行估算，然后通过压测动态调整。项目里本地测试用较小连接池，是因为测试规模有限。

**追问：连接池满了怎么办？**
可以等待、超时返回、限流或降级，不能无限创建连接，否则可能把数据库打爆。

------


### 53. 高并发场景下单个连接池连接数不足怎么办？



连接池太小会排队 → 等待超时 → 不能无限加 → 调整池大小、限流、缓存、异步化、读写分离。

**回答：**
如果连接池连接数不足，请求会在获取连接时排队，严重时会超时。

最直接的做法是**根据压测结果适当增大连接池大小**，但连接数不是越大越好。

如果无限增大连接池，可能会把 MySQL 或 Redis 压垮，因为数据库本身能承载的并发连接有限。

更合理的做法是综合优化：第一，合理设置连接池上限和等待超时时间；第二，对非核心请求做限流或降级；第三，增加 Redis 缓存减少 MySQL 查询；第四，把慢查询优化掉；第五，在更高规模下做 MySQL 读写分离或分库分表。

所以连接池不足不是只靠加连接数解决，而是要结合后端资源整体承载能力。

**追问：连接池满了请求应该一直等吗？**
不应该。应该设置超时时间，**超过时间就快速失败或返回友好错误**，否则线程会被大量阻塞，导致系统雪崩。

------


### 54. 数据库中用户密码应该如何加密存储？



不能明文存 → 盐值 + 哈希 → bcrypt/PBKDF2/Argon2 → 登录时同样算法比对 → 项目学习版可承认后续优化。

**回答：**
用户密码不能明文存储。比较合理的做法是使用盐值加哈希算法。

注册时，为每个用户生成一个随机 salt，把密码和 salt 一起做哈希，然后把 salt 和哈希结果存入数据库。登录时，再用用户输入的密码和数据库中的 salt 重新计算哈希值，和数据库保存的哈希结果进行比对。

如果面试官问到我的项目，我会坦诚说当前学习版更关注 IM 主链路，密码安全可以进一步优化为盐值加安全哈希，而不是简单保存明文或普通哈希。

**追问：为什么不直接用 MD5？**
MD5 速度太快，而且容易被彩虹表和暴力破解攻击。密码存储不追求快，而是希望攻击者破解成本高，所以更推荐 bcrypt、PBKDF2 或 Argon2。

------


### 55. 如何做验证码防刷和登录限流？



Redis 计数 → 按 IP/邮箱限流 → 60 秒冷却 → 登录错误次数限制 → 防暴力破解和邮件轰炸。

**回答：**
验证码接口容易被刷，需要按 IP 和邮箱做限流。

可以在 Redis 中记录单位时间内的请求次数，比如同一邮箱 60 秒只能发一次，同一 IP 每分钟最多请求 N 次。

登录也需要限制密码错误次数，超过次数后短暂冻结或要求验证码。这样可以防止暴力破解和邮件轰炸。

**追问：限流放 GateServer 还是 VerifyServer？**
最好 **GateServer 入口先限流**，VerifyServer 内部也可以做二次保护。

------


## 六、好友、消息、离线与多端


### 56. 添加好友流程是怎么实现的？



A 搜索 B → 写申请表 → 查 Redis 找 B 位置 → 在线同服直接推 → 跨服 gRPC 推 → B 同意后双向写好友表。

**回答：**
用户 A 搜索到用户 B 后，发送添加好友申请。

ChatServer 先把申请写入 MySQL，然后查询 Redis 判断 B 是否在线、在哪台 ChatServer。
如果 B 在当前服务器，就通过本地 session 直接推送好友申请；

如果 B 在其他服务器，就通过 gRPC 转发给目标 ChatServer，再由目标服务器推送给 B。
B 同意后，服务端更新申请状态，并写入双向好友关系，再通知 A 认证成功。

**追问：为什么要先写数据库再推送？**
先落库可以保证状态可恢复。即使推送失败，用户下次登录也能从数据库看到申请记录。

------


### 57. 主动添加方的消息推送怎么做？如果被添加方同意时，主动添加方不在线怎么办？



B 同意 A → 先更新数据库好友关系 → 查 A 在线位置 → A 在线立即推 → A 不在线存未读事件 → A 下次登录拉取。

**回答：**
当用户 B 同意用户 A 的好友申请时，服务端首先要更新数据库，比如更新申请状态，并写入好友关系表。然后服务端根据 A 的 uid 查询 Redis，判断 A 是否在线。
如果 A 在线，就判断 A 在哪台 ChatServer。如果 A 在本机，直接通过本地 session 推送认证成功通知；如果 A 在其他 ChatServer，就通过 gRPC 转发给对应服务器。
如果 A 不在线，就不能只依赖实时推送，而应该把这个认证结果作为未读事件或离线通知存入数据库。A 下次登录时，ChatServer 从数据库中拉取未读好友通知，再推送给 A。

**追问：为什么必须先写数据库？**
因为推送可能失败。如果先推送但没落库，用户断线后通知可能丢失。先写数据库可以保证状态可恢复。

------


### 58. 同一账号在 A 设备添加好友，之后在 B 设备登录时，如何同步好友信息？



好友关系最终存 MySQL → A 添加成功后写库 → B 登录时拉好友列表 → 多端在线时推送所有设备 → 已读/同步状态另存。

**回答：**
好友关系必须以服务端数据库为准。A 设备添加好友成功后，服务端会把好友关系写入 MySQL。

之后同一账号在 B 设备登录时，ChatServer 会从 MySQL 查询该用户的好友列表，并返回给 B 设备。

如果支持多端同时在线，还可以在好友关系变化时，主动向该账号的所有在线设备推送同步通知。

比如 A 设备添加好友成功后，PC 端或 B 设备也能马上更新好友列表。
如果 B 设备当时不在线，也没关系。因为好友关系已经写入数据库，B 下次登录时重新拉取即可。

**追问：如果多个设备好友列表不一致怎么办？**
以服务端 MySQL 为最终数据源。客户端本地缓存只是加速显示，登录或收到同步通知时应以服务端数据刷新。

------


### 59. 如果用户不在线，好友申请或消息怎么办？



当前主链路在线推送 → 离线要落 MySQL → 标记未读/未送达 → 登录拉取 → ACK 后改状态。

**回答：**
当前项目主要实现在线推送主链路。

如果用户在线，就通过 Redis 找到用户位置并推送。
如果用户不在线，完整方案应该把好友申请或聊天消息存入 MySQL，状态标记为未读或未送达。

用户下次登录时，ChatServer 查询未读数据并推送给客户端，客户端 ACK 后再更新状态。

------


### 60. 消息可靠性怎么保证？



TCP 只保证传输层 → 业务层要消息落库 → msgid 幂等 → 推送后等 ACK → 失败重试/离线补偿。

**回答：**
TCP 只能保证传输层可靠，但业务层还需要 ACK 和落库。完整设计是客户端发送消息时带唯一 msgid，服务端收到后先落库，状态设为未送达，然后推送给接收方。
接收方收到后返回 ACK，服务端再把状态改为已送达或已读。如果客户端重试，服务端根据 msgid 做幂等，避免重复消息。

**追问：ACK 丢了怎么办？**
服务端没有收到 ACK，就认为消息未送达，后续可以重试。客户端收到重复消息时用 msgid 去重。

------


### 61. 聊天过程中的数据如何产生、如何传递、如何存储？



用户输入 → 客户端封装 JSON → 加 TCP 包头 → ChatServer 解析 → 查 Redis 找目标 → 本地/跨服推送 → 落库 → ACK 更新状态。

**回答：**
用户在客户端输入消息后，客户端会把消息内容、发送者 uid、接收者 uid、msgid 等字段封装成 JSON。

然后再加上自定义 TCP 包头，包头中包含 msgid 和 body 长度。

ChatServer 收到数据后，先解析包头，再解析 JSON，得到发送方和接收方。

然后通过 Redis 查询接收方当前在哪台 ChatServer。

如果接收方在本机，就直接通过本地 session 推送；如果在其他 ChatServer，就通过 gRPC 转发给目标服务器。

如果要做可靠消息，还应该在服务端收到消息后先落库，记录消息状态。接收方收到后返回 ACK，服务端再更新消息状态为已送达或已读。

**追问：当前项目有没有完整落库？**
当前项目重点在在线消息转发主链路，完整的消息落库、ACK 和离线补偿可以作为后续增强。

------


### 62. 如果离线消息过大，需要等所有数据发送完之后再删除吗？发送过程中服务断掉怎么办？



不要一次性全删 → 逐条发送 → 逐条 ACK → ACK 后更新状态 → 中途断开下次继续 → msgid 去重。

**回答：**
不能等所有离线消息一次性发送完之后再统一删除，也不能刚开始发送就删除。

更合理的做法是逐条或分批发送，客户端每收到一条或一批消息后返回 ACK，服务端收到 ACK 后再把对应消息状态改为已送达或已读。

如果发送过程中服务断掉，未 ACK 的消息仍然保留在数据库中。用户下次登录时，服务端继续发送未确认的消息。

为了避免重复消息，**消息需要有唯一 msgid，客户端收到消息后可以根据 msgid 去重**。这样即使服务端重发，也不会在客户端显示重复消息。

**追问：离线消息特别多怎么办？**
可以分页拉取或分批推送，不能一次性全部发给客户端。比如每次拉取 50 条或 100 条，根据游标继续拉取。

------


### 63. 如果同一账号多端登录，怎么设计？



uid 不再只对应一个 session → uid 对多个设备 session → 消息广播到多端 → 已读状态同步 → 可选踢旧设备。

**回答：**
当前项目更偏单端登录。

如果支持多端登录，UserMgr（用户会话管理中心） 中不能只存一个 uid 到 session，而应该存 uid 到多个 session 的映射，并区分设备 ID。

**消息到达时，要推送给该用户所有在线设备。已读状态也要同步，比如手机读了，PC 端也更新。也可以做互斥登录策略，比如新设备登录后踢掉旧设备，这取决于产品需求。**

**追问：多端登录时 Redis 怎么存？**
可以存 `uid -> deviceId -> serverName/sessionInfo`，而不是简单的 `uid -> serverName`。

------


### 64. 聊天消息表怎么设计？索引怎么建？



message 表：from、to、content、msg_id、status、time → msg_id 唯一防重复 → from_to_time 联合索引查聊天记录。

**回答：**
消息表可以包含 id、from_uid、to_uid、msg_id、msg_type、content、status、created_at 等字段。

索引上，常用的是 `(from_uid, to_uid, created_at)` 或者会话维度的 `conversation_id + created_at`，用于查询两个人的聊天记录。

msg_id 可以加唯一索引，用于防止客户端重试导致重复消息。

**追问：为什么要 msg_id 唯一索引？**
客户端重试时可能重复发送同一条消息，msg_id 唯一索引可以保证幂等。

### 65. 如果要支持图片、文件、语音，怎么设计？



不走聊天 TCP 大包 → 单独文件服务/对象存储 → 先上传拿 file_id/url → IM 只传元数据 → 分片/断点续传。

**回答：**
不建议把图片、文件、语音直接塞进聊天 TCP 通道，因为**大文件会阻塞普通文字消息。**更合理的方案是单独做文件服务。
客户端先把文件上传到文件服务器或对象存储，拿到 file_id 和 URL。然后 IM 消息只发送 file_id、URL、文件类型、大小和缩略图等元信息。接收方收到消息后再按需下载。大文件可以支持分片上传和断点续传。

**追问：为什么不直接用聊天连接传文件？**
大文件会占用聊天通道，影响普通消息实时性。文件传输和 IM 通知应该解耦。

------


### 66. 如何存储大量聊天消息，包括视频、音频和大文件？



文字消息存数据库 → 大文件不进 MySQL → 对象存储/文件服务 → MySQL 存 file_id、URL、类型、大小、状态 → 分片上传/CDN。

**回答：**
文字消息可以存入 MySQL 的消息表中，字段包括 from_uid、to_uid、msg_id、content、status、created_at 等。

但图片、语音、视频、大文件不应该直接存进 MySQL，更不应该直接放到聊天 TCP 通道里。

更合理的方案是把这些大文件上传到文件服务器或对象存储，比如 OSS、COS、S3 这类系统。MySQL 中只保存 file_id、URL、文件类型、大小、缩略图、发送者、接收者和消息状态。

对于大文件，可以支持分片上传、断点续传、秒传和 CDN 加速。客户端收到消息后，根据 URL 或 file_id 去下载文件。这样数据库不会被大字段拖垮，聊天服务也不会被大流量文件传输阻塞。

**追问：语音消息怎么存？**
语音本质上也是文件，可以上传到文件服务，消息表中保存语音文件地址、时长、大小和格式。

------


### 67. 如果需要添加视频聊天功能，你觉得应该怎么设计？视频需要通过服务器转发吗？



（视频流会占用大量带宽和 CPU）视频不走普通聊天 TCP → 用 WebRTC/RTP/UDP → IM 服务器只做信令 → 优先 P2P → 不通时 TURN/SFU 中继。

**回答：**
视频聊天不应该直接走当前 IM 的 TCP 聊天通道。

文字消息要求可靠，而视频更重视实时性，少量丢包可以接受，所以更适合使用 UDP 或基于 UDP 的 WebRTC、RTP 方案。

我的设计是：现有 ChatServer 负责信令，比如发起通话、接听、拒绝、挂断、交换 SDP 和 ICE 候选信息。真正的视频流尽量让客户端之间 P2P 传输。
如果 P2P 不通，比如 NAT 穿透失败，就需要 TURN 服务器中继；如果是多人视频会议，可以考虑 SFU，由服务器转发媒体流，但不是让普通 ChatServer 直接转发大视频流。

**追问：为什么不能直接让 ChatServer 转发视频流？**
ChatServer 的核心是文字消息和用户状态，不适合承载高带宽视频流。视频流会占用大量带宽和 CPU，影响普通消息实时性。应该把信令服务和媒体服务拆开。

## 七、高并发、可靠性、优化与排查


### 68. 高并发连接的瓶颈在哪里？



不只连接数 → fd 限制 → 内存 socket/session → CPU 解析业务 → Redis/MySQL/连接池 → 网络带宽。

**回答：**
高并发瓶颈不只是 TCP 连接数。

第一是文件描述符限制，Linux 默认 fd 数不够；

第二是内存，每个连接都有 socket 缓冲区和 session 对象；

第三是 CPU，消息解析、JSON 序列化和业务处理都消耗 CPU；

第四是外部资源，比如 Redis、MySQL、gRPC 连接池和网络带宽。


**追问：1 万连接和 1 万 QPS 一样吗？**
不一样。1 万连接表示在线连接数，1 万 QPS 表示每秒请求数。IM 系统中很多连接可能是空闲的。

### 69. 你简历里写 1w 在线连接，面试官问你怎么测的，怎么答？



本地双 ChatServer → 模拟大量客户端 TCP 连接 → 看连接成功率、延迟、CPU、内存 → 强调不是线上生产能力。

**回答：**
我会说明这是本地双 ChatServer 实例下的连接测试，主要验证长连接保持能力和基础消息收发延迟。

测试时模拟大量客户端建立 TCP 连接，完成登录或保持连接，并按一定频率发送消息，观察连接成功率、平均延迟、CPU、内存和服务稳定性。
真实生产还要考虑网络、日志、数据库、Redis、限流和监控。

**追问：你这个 1w 在线有什么边界？**
它主要验证连接保持和基础通信，不代表复杂业务场景下仍然有同样性能。

### 70. 如果 ChatServer 挂了，用户怎么办？



心跳发现断开 → 客户端重连 → 重新找可用 ChatServer → Redis token 继续认证 → 消息靠离线补偿。

**回答：**
当前项目**可以通过客户端重连恢复**，但还不是完全无感切换。

完整方案是客户端通过心跳发现 TCP 断开后，重新请求可用 ChatServer，并用原 token 重新认证。

因为 token 存在 Redis 中，新 ChatServer 可以继续校验身份。为了保证消息不丢，还需要消息落库和离线补偿。

**追问：能做到完全无损吗？**
需要额外设计，包括消息落库、ACK、服务健康检查和客户端重连机制。如果消息只在内存里，还没落库，服务器挂掉就可能丢。

### 71. 项目有什么不足？你会怎么优化？



可靠消息不完整 → 服务发现不足 → 单点问题 → 业务线程可扩展 → 优先补 ACK/离线消息。

**回答：**
第一，可靠消息链路还可以补充消息落库、ACK、重试和离线消息。

第二，服务发现目前不够完善，可以引入 Zookeeper。

第三，Redis、MySQL、StatusServer 都需要高可用。

第四，业务线程可以从单线程升级为多 worker，按 uid 分片处理。
如果让我优先优化，我会先补消息可靠性，因为 IM 系统最重要的是消息不能丢。

**追问：最想先优化哪一个？**
我会先做消息表、ACK 和离线消息，因为这直接影响用户体验和系统可靠性。

### 72. 项目中哪些地方需要加锁？如何控制锁粒度？



session map、UserMgr、发送队列、消息队列、连接池 → 锁粒度小 → 不持锁做网络/数据库 → RAII 防忘 unlock。

**回答：**
需要加锁的地方主要是共享数据结构**，比如 session map、UserMgr、发送队列、消息队列、MySQL/Redis/gRPC 连接池。**

控制锁粒度的原则是**只在访问共享容器时加锁**，业务处理尽量不要长时间持锁。项目中通过消息队列把 IO 和业务解耦，也减少了业务层共享数据的复杂度。

**追问：怎么避免死锁？**
减少多把锁嵌套，固定加锁顺序，使用 RAII 自动释放锁，避免持锁期间做网络或数据库调用。

### 73. 项目中遇到过什么问题？怎么排查？



先看日志 → 打印 msgid/len → 查 Redis 用户位置 → 模拟断线重连 → 看 session 是否清理 → 用断点定位。

**回答：**
比较典型的问题是消息收不完整或粘包。排查时先打印包头 msgid 和 len，再确认是否按固定长度读取。

另一个问题是断线后状态没有清理，导致 Redis 中还认为用户在线。这个要从 read/write error 回调开始查，看是否调用 Close 和 ClearSession。

排查这类问题时，我主要通过日志、断点、Redis key 检查和模拟多客户端测试来定位。

**追问：怎么证明问题解决了？**
用复现场景反复测试，比如频繁断开重连、快速发送多条消息、跨服务器发送消息，观察日志和 Redis 状态是否符合预期。
