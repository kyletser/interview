## 池化技术

---

### 面试官：说一下你项目中的连接池是怎么实现的？

**回答**：

好的面试官，我在项目中实现了三种连接池，我来分别介绍一下：

---

#### 第一个：MySQL连接池

**背景**：
我在MysqlDao中使用了连接池。在没有连接池的情况下，每次执行SQL都需要：TCP三次握手建立连接 → MySQL认证 → 执行SQL → 四次挥手关闭连接。一次简单的查询可能需要30-50ms，高并发时MySQL连接数会瞬间耗尽。

**实现方式**：
我在MysqlDao.h中定义了MySqlPool类，核心是用了三个东西：

1. **std::queue**：用队列存储预创建的连接
2. **std::mutex**：保证多线程访问队列时的线程安全
3. **std::condition_variable**：当队列为空时，让线程阻塞等待

```cpp
// 核心代码
class MySqlPool {
    std::queue<std::unique_ptr<SqlConnection>> pool_;  // 连接队列
    
    // 获取连接
    std::unique_ptr<SqlConnection> getConnection() {
        std::unique_lock<std::mutex> lock(mutex_);
        // 队列为空则阻塞，等有连接归还时唤醒
        cond_.wait(lock, [this] { return !pool_.empty(); });
        auto con = std::move(pool_.front());
        pool_.pop();
        return con;
    }
    
    // 归还连接
    void returnConnection(std::unique_ptr<SqlConnection> con) {
        pool_.push(std::move(con));
        cond_.notify_one();  // 唤醒等待的线程
    }
};
```

**额外功能**：
我还实现了一个心跳检测线程，每60秒遍历所有连接，执行`SELECT 1`检测是否存活。如果连接失效，就重建一个。这是为了防止MySQL服务器主动断开长时间空闲的连接。

**效果**：
原来每次SQL需要30-50ms，用了连接池后只需要1-2ms，性能提升20-30倍。

#### 第二个：gRPC连接池

**背景**：
我们系统是分布式架构，ChatServer之间通过gRPC通信。当用户A在ChatServer1，用户B在ChatServer2时，消息需要通过gRPC转发。

如果每次转发都创建新的gRPC通道，成本非常高：需要TCP三次握手 + TLS握手 + HTTP/2协商，大概需要30-40ms。这在高频消息场景是不可接受的。

**实现方式**：
我参考MySQL连接池，实现了ChatConPool：

```cpp
class ChatConPool {
    std::queue<std::unique_ptr<ChatService::Stub>> connections_;
    
    // 预创建多个Stub
    ChatConPool(size_t poolSize, std::string host, std::string port) {
        for (size_t i = 0; i < poolSize_; ++i) {
            // 创建gRPC通道（成本很高，但只创建一次）
            std::shared_ptr<Channel> channel = 
                grpc::CreateChannel(host + ":" + port, 
                    grpc::InsecureChannelCredentials());
            // 创建Stub
            connections_.push(ChatService::NewStub(channel));
        }
    }
    
    std::unique_ptr<ChatService::Stub> getConnection() {
        // 从池中获取，复用Stub
    }
    
    void returnConnection(std::unique_ptr<ChatService::Stub> stub) {
        // 归还到池中
    }
};
```

**效果**：
gRPC调用从30-40ms降低到1ms以内，性能提升30倍以上。

#### 第三个：IO线程池

**背景**：
ChatServer需要处理大量TCP连接，包括登录、聊天、添加好友等。如果用单线程处理所有连接，CPU利用率很低，无法充分利用多核CPU。

**实现方式**：
我使用Boost.Asio实现了多IO线程池：

```cpp
class AsioIOServicePool {
    std::vector<boost::asio::io_context> _ioServices;
    std::vector<std::thread> _threads;
    
    AsioIOServicePool(std::size_t size) {
        // 创建size个io_context（size = CPU核心数）
        for (size_t i = 0; i < size; ++i) {
            // 创建work防止io_context退出
            _works[i] = std::unique_ptr<work>(new work(_ioServices[i]));
        }
        
        // 每个io_context启动一个线程运行run()
        for (size_t i = 0; i < _ioServices.size(); ++i) {
            _threads.emplace_back([this, i]() {
                _ioServices[i].run();  // 事件循环
            });
        }
    }
    
    // Round-Robin分配
    boost::asio::io_context& GetIOService() {
        auto& service = _ioServices[_nextIOService++];
        if (_nextIOService == _ioServices.size()) {
            _nextIOService = 0;
        }
        return service;
    }
};
```

**工作原理**：

1. 创建多个io_context，每个启动一个线程运行run()事件循环
2. 新TCP连接到来时，通过Round-Robin分配到不同的io_context
3. 每个io_context独立处理自己分配的连接读写事件
4. 充分利用多核CPU，4核CPU可以同时处理4个连接

**关键点**：
我用了`io_context::work`这个对象，它的作用是防止io_context在没有任务时退出。如果没有work，当没有任何事件时，run()会直接返回，线程就退出了。加上work后，即使没有事件，线程也会阻塞等待。

### 面试官：为什么连接池要用条件变量？不用可以吗？

**回答**：

可以用，但不用会有问题。

**不用条件变量的方案**：

```cpp
// 方案1：直接返回，空了报错
std::unique_ptr<SqlConnection> getConnection() {
    if (pool_.empty()) {
        // 等待一会儿再试
        std::this_thread::sleep_for(std::chrono::milliseconds(10));
        return getConnection();  // 递归重试
    }
    // ...
}
```

**问题**：这种方式叫轮询，CPU空转浪费资源，而且响应不及时。

**用条件变量的方案**：

```cpp
// 方案2：条件变量
std::unique_lock<std::mutex> lock(mutex_);
cond_.wait(lock, [this] { return !pool_.empty(); });
// 队列有连接时才会继续执行
```

**优点**：
1. 不消耗CPU，线程阻塞等待
2. 有连接时立刻唤醒，响应及时
3. 不会产生竞态条件

条件变量是线程间同步的标准方式，比轮询高效很多。

### 面试官：连接池的大小怎么确定？

**回答**：

我项目中是这样确定的：

**MySQL连接池**：配置为10个
- 根据预估的并发量，如果QPS是1000，每个请求1ms，10个连接理论上能处理10000 QPS
- 太大浪费内存，太小不够用
- 一般观察MySQL的max_connections，如果最大2000，连接池10-20比较合适

**gRPC连接池**：配置为5个

- 因为跨服务器消息不是特别频繁
- 每个Stub占用资源比MySQL连接少，可以稍微多一点

**IO线程池**：默认使用`std::thread::hardware_concurrency()`
- 自动获取CPU核心数
- 4核CPU就创建4个线程

### 面试官：连接池的线程安全是怎么保证的？

**回答**：

主要靠两点：

1. **std::mutex**：每次访问队列都要加锁
2. **std::condition_variable**：配合mutex使用

```cpp
// 获取连接
std::unique_ptr<SqlConnection> getConnection() {
    std::unique_lock<std::mutex> lock(mutex_);  // 加锁
    cond_.wait(lock, [this] { return !pool_.empty(); });  // 释放锁等待
    // 被唤醒后，锁重新加身，继续执行
    auto con = std::move(pool_.front());
    pool_.pop();
    return con;  // 锁自动释放
}
```

这是C++11标准的线程安全实现，wait函数会自动释放锁，被唤醒后重新获取锁。

### 面试官：如果所有连接都被占用了怎么办？

**回答**：

有两种处理方式：

1. **等待**（我的实现）：
   - 线程阻塞在条件变量上
   - 其他线程用完归还后会唤醒
   - 适合请求量波动不大的场景

2. **超时返回错误**：
   - 等待一定时间后返回nullptr
   - 上层业务自行处理（比如返回"系统繁忙"）
   - 适合对响应时间敏感的场景

我的实现是第一种，因为聊天系统对实时性要求高，宁可等待也不返回错误。
