## 1、怎样做用户登录，为什么用异步不用同步，怎样设计多线程模式，连接池的实现原理（如何设计连接数）

**问题**：怎样做用户登录，为什么用异步不用同步，怎样设计多线程模式，连接池的实现原理（如何设计连接数）

**详细回答**：

**一、用户登录流程**

##### 第一阶段：获取聊天服务器地址

1. 客户端向GateServer提交邮箱密码；
2. GateServer校验密码后，向StatusServer请求聊天服务器信息；
3. StatusServer选负载最低的ChatServer，生成Token并存储，返回给GateServer；
4. GateServer将uid、Token及聊天服务器地址返回客户端。

##### 第二阶段：连接聊天服务器并完成登录

1. 客户端通过TCP连接ChatServer，发送含uid和Token的登录包；
2. ChatServer校验Token，通过后加载用户及好友相关数据；
3. 绑定会话、记录登录信息并更新计数；
4. 返回用户数据，完成登录。

**二、为什么用异步不用同步**

同步IO模式下，一个线程同时只能处理一个客户端请求。而且线程切换需要CPU保存上下文，频繁切换性能很差。

同步模式下线程大部分时间阻塞在网络读写上，CPU利用率很低。

异步 IO：单线程可处理大量连接，非阻塞，依靠 epoll/IOCP 事件驱动，CPU 利用率高，资源消耗小。

**三、多线程模型设计**

项目采用**三级线程模型。**

第一级是IO线程池。io_context，数量默认等于CPU核心数。每个io_context跑一个独立线程。新连接轮询分配到不同io_context。这样做的好处是避免单io_context的锁竞争，充分利用多核CPU。

第二级是gRPC服务线程。ChatServer启动独立的gRPC服务（50055或50056端口），调用server->Wait()阻塞等待。这个线程专门处理其他服务器通过gRPC发来的通知，比如跨服务器的消息转发。

第三级是业务逻辑线程。只有一条线程处理业务逻辑。IO线程收到网络数据后，调用PostMsgToQue方法投到消息队列。业务线程从队列取消息，根据消息ID找到对应处理函数执行。这样做的好处是业务逻辑完全串行，不需要加锁，天然保证数据一致性。

**四、连接池实现原理**

连接池的核心是预创建加复用加线程安全。以RedisConPool为例。

构造函数里，根据poolSize参数（比如5个）循环创建Redis连接。每个连接用redisConnect创建，然后用AUTH命令认证。成功后的连接放入queue队列。同时启动一个检测线程，每60秒遍历所有连接发PING命令，检测到连接断开就重新创建。

获取连接用getConnection方法。用unique_lock加mutex锁，然后cond_.wait等待直到队列不为空或已停止。从队列取出一个连接返回。

归还连接用returnConnection方法。用lock_guard加锁，把连接放回队列，调用cond_.notify_one唤醒等待的线程。

连接数设置为5-10个。经验公式是CPU核心数乘以2。但实际要根据业务来定。如果连接数太多，会增加锁竞争开销；如果太少，高并发时线程会等待。

## 2、gRPC怎么来的，为什么叫gRPC

**问题**：gRPC怎么来的，为什么叫gRPC

**详细回答**：

gRPC是Google开源的RPC框架。Google内部长期以来使用一个叫Stubby的内部RPC系统来连接数据中心内的数千种服务。Stubby在Google使用了十多年，有很好的性能和数据一致性，但和外部系统不兼容，也不够通用。

gRPC用HTTP/2传输，支持多路复用，一个TCP连接上可以并行多个请求和响应。用Protocol Buffers（Protobuf）做序列化，比JSON体积小、解析快。支持四种模式：普通调用、服务端流、客户端流、双向流。

## 3 多服务器和单服务器相比应该是一个正比关系的扩容，这里为什么不是正比关系

**详细回答**：

不是正比是因为系统里有多个瓶颈。

第一是GateServer。所有HTTP请求都要经过GateServer，它只有一台，处理能力有限。如果1万个用户同时登录，GateServer会被压垮。

第二是StatusServer。每次登录都要调用StatusServer获取聊天服务器地址，高并发下压力大。

第三是Redis和MySQL。所有服务共享这些资源，连接数有限制。比如Redis连接池只有5个，如果请求太多会排队等待。

第四是网络带宽。服务器之间的gRPC通信、访问数据库都要占用带宽。

所以虽然ChatServer可以线性扩容（多加几台就能多承载用户），但Gateway和Status是单点，会成为瓶颈。要真正支持更多用户，需要给这些服务也做高可用集群。

## 4、怎么封装的http和tcp

**问题**：怎么封装的http和tcp

**详细回答**：

**HTTP封装（GateServer）**

GateServer用**Boost.Beast封装HTTP**。核心类是HttpConnection，每个TCP连接对应一个HttpConnection对象。

CServer调用async_accept等待新连接。连接来时创建HttpConnection对象，调用Start方法。Start方法里用http::async_read异步读取HTTP请求。回调函数里调用HandleReq处理请求。

HandleReq方法先获取请求的method。如果是GET，调用PreParseGetParam解析URL参数，然后调用LogicSystem的HandleGet方法。如果是POST，直接调用LogicSystem的HandlePost方法。

处理完后调用WriteResponse发送响应，用http::async_write异步写出。

LogicSystem维护两个map：_get_handlers和_post_handlers。注册路由用RegPost或RegGet方法，传入URL路径和处理lambda。HandlePost根据path从map里找对应handler，找不到返回404。

**TCP封装（ChatServer）**

ChatServer用**Boost.Asio封装TCP**。核心类是CSession。

CServer同样调用async_accept等待新连接。连接来时创建CSession对象，调用Start方法。Start方法调用AsyncReadHead读取头部。

AsyncReadHead读取4个字节：前2字节是消息ID，后2字节是数据长度。网络传输用大端序，所以用network_to_host_short转换。解析出msgid和msglen后，调用AsyncReadBody读取实际数据。

AsyncReadBody读取指定长度的数据，然后调用LogicSystem的PostMsgToQue投到消息队列。

发送用Send方法。把数据放入发送队列（防粘包），然后调用boost::async_write异步写出。回调函数HandleWrite里，如果队列还有数据就继续发送。

消息格式：2字节msgid + 2字节长度 + JSON数据。msgid用short表示，len用short表示，都是网络字节序。

## 5、对并发和异步的思考（异步时如何处理共享数据，引导思考如何平衡并发和异步）

**问题**：对并发和异步的思考（异步时如何处理共享数据，引导思考如何平衡并发和异步）

异步编程确实带来了数据共享的问题。我有三种处理方式。

**第一种：消息队列解耦**

这是最核心的方式。IO线程只负责网络读写，读完数据后投到队列。业务逻辑由单独的工作线程处理，完全串行。这样做的好处是业务逻辑不需要任何锁，因为只有一条线程在处理。

代码示例：IO线程调用PostMsgToQue把LogicNode投到_msg_que。业务线程DealMsg用while循环从队列取消息，取的时候需要加锁，但取出后的处理完全不需要锁。

```cpp
void LogicSystem::DealMsg() {
    for(;;) {
        std::unique_lock<std::mutex> unique_lk(_mutex);
        while(_msg_que.empty() && !_b_stop) {
            _consume.wait(unique_lk);
        }
        auto msg_node = _msg_que.front();
        // 这里取出来后，后续处理不需要加锁
        call_back_iter->second(msg_node->_session, msg_node->_recvnode->_msg_id, 
            std::string(msg_node->_recvnode->_data, msg_node->_recvnode->_cur_len));
        _msg_que.pop();
    }
}
```

**第二种：连接池用锁**

Redis和MySQL连接必须共享，因为连接数量有限。这种情况必须加锁，但用条件变量避免忙等。获取连接时如果队列为空，线程在cond_上wait而不是自旋。归还时调用notify_one唤醒。

**第三种：避免共享**

设计时尽量避免共享。比如每个用户绑定到固定的session，不存在多线程同时访问同一用户数据的情况。

**平衡点**：

- 网络IO层：完全异步，事件驱动
- 业务逻辑层：单线程串行，避免锁
- 数据访问层：连接池加锁，这是唯一需要锁的地方

## 6、项目背景，项目中微服务的拆分，职责边界如何划分

**问题**：项目背景，项目中微服务的拆分，职责边界如何划分

**微服务拆分**

项目拆成4种服务：

GateServer：**网关服务，监听8080端口。负责接收客户端HTTP请求，做路由分发，调用其他服务，返回结果。不处理具体业务逻辑。**

VarifyServer：**验证码服务，Node.js实现，监听50051端口。负责生成验证码、存Redis、发邮件。技术栈不同是因为验证码逻辑简单，用Node.js开发快。**

StatusServer：**状态服务，监听50052端口。负责两件事：负载均衡（选择聊天服务器）和Token管理（生成和验证Token）。**

ChatServer：**聊天服务，监听TCP端口8090/8091和gRPC端口50055/50056。负责处理登录、搜索用户、添加好友、聊天等所有业务逻辑。**

**职责边界**

**每个服务独立部署、独立扩展。服务间通过gRPC通信，不直接操作对方数据库。GateServer不存储业务数据，只做路由。ChatServer不暴露给公网，通过GateServer访问。**

## 7、聊天服务器如何实现负载均衡，当有大量请求到来时，如何实现连接的均匀分布

负载均衡在StatusServer实现，**用最小连接数算法。**

StatusServer启动时从配置文件读取所有ChatServer列表，存到_servers这个map里。同时从Redis读取各服务器的登录人数，存到logincount这个Hash。

当GateServer调用GetChatServer时，StatusServer遍历_servers里的每台服务器，从Redis查该服务器的登录数量（用HGET logincount服务器名）。比较找出连接数最少的服务器，返回给客户端。

用户登录成功后，ChatServer会更新Redis的登录计数（用HINCRBY）。这样StatusServer始终知道各服务器的负载情况。

**均匀分布**

当大量请求到来时，因为每台ChatServer的登录人数不同，新用户会被分配到人数最少的那台。这样能保证负载均匀。

## 8连接数的瓶颈在哪儿

**瓶颈分析**

主要有三个瓶颈。

第一是内存。每个TCP连接在Linux里约占用10KB内存（主要是socket缓冲区和数据结构）。3万连接约需300MB，10万连接约需1GB。这是可以接受的。

第二是文件描述符。Linux默认ulimit -n是1024，必须改大才能支持更多连接。生产环境需要配置/etc/security/limits.conf。

第三是CPU。如果业务逻辑复杂，大量消息处理会占用CPU。但如果只是简单的收发，CPU不是瓶颈。

## 9、数据库的连接池是怎么设置的，当有大量请求到来时，如何处理连接超时（连接泄露）问题

**问题**：数据库的连接池是怎么设置的，当有大量请求到来时，如何处理连接超时（连接泄露）问题

**详细回答**：

**连接池设置**

Redis连接池设5个连接。MySQL连接池也是5个。这个数字怎么来的？根据经验，5-10个性价比最高。太多会增加锁竞争，太少会不够用。

**防止连接泄露**

我用RAII思想封装了Defer类。原理是C++中对象离开作用域时一定会调用析构函数，不管函数是正常返回还是抛异常。

```cpp
class Defer {
public:
    Defer(std::function<void()> func) : func_(func) {}
    ~Defer() { func_(); }
private:
    std::function<void()> func_;
};
```

使用方式：

```cpp
bool MysqlDao::CheckPwd(const std::string& name, const std::string& pwd, UserInfo& userInfo) {
    auto con = pool_->getConnection();
    Defer defer([this, &con]() {
        pool_->returnConnection(std::move(con));
    });
    
    // 业务逻辑，可能中途return
    std::unique_ptr<sql::PreparedStatement> pstmt(con->_con->prepareStatement("SELECT * FROM user WHERE name = ?"));
    // ...
    
    // 函数结束，defer自动析构，连接自动归还
    return true;
}
```

这样即使中间return或抛异常，连接一定会归还。

**超时处理**

连接池还有后台检测线程。每60秒遍历所有连接，发SELECT 1探测是否活着。如果连接已断开，重建新连接。

## 10、MySQL连接池的设计要点

**问题**：MySQL连接池的设计要点

**详细回答**：

有四个核心要点。

**第一是预创建**。构造函数里一次性创建poolSize个连接，放到队列里。这样请求来的时候不需要等待创建连接，直接从队列取就能用。

**第二是线程安全**。获取和归还连接必须加锁，因为多个线程会同时访问队列。用std::mutex加std::condition_variable。获取时如果队列空就wait，而不是自旋忙等（自旋会占用CPU）。

**第三是连接复用**。用完不是关闭，而是归还到队列。下次有请求来直接复用，不需要重新创建连接。

**第四是健康检测**。后台启动检测线程，每60秒遍历连接发SELECT 1探测。如果连接已断开，关闭旧连接，创建新连接补充。

## 11、gRPC在项目中解决的什么问题

**问题**：gRPC在项目中解决的什么问题

**详细回答**：

**解决跨服务器通信问题。**

项目有两台ChatServer（ChatServer1和ChatServer2）。用户A可能连着ChatServer1，用户B可能连着ChatServer2。当A发消息给B时，ChatServer1怎么知道B在哪台服务器？怎么把消息推过去？

答案是Redis存用户位置。A登录时，ChatServer1把"uip_A"存到Redis，value是"chatserver1"。B登录时，ChatServer2把"uip_B"存到Redis，value是"chatserver2"。

A发消息给B时，ChatServer1先查Redis，得到"uip_B"="chatserver2"，知道B在ChatServer2上。然后ChatServer1用gRPC调用ChatServer2的NotifyTextChatMsg接口，把消息内容传过去。ChatServer2收到后，找到B的session，把消息推过去。

gRPC的优势是：基于HTTP/2支持多路复用，Protobuf序列化比JSON快很多，跨语言支持好。

## 12、redis缓存的应用场景，如何保证一致性

**问题**：redis缓存的应用场景，如何保证一致性

**详细回答**：

**应用场景**

项目里Redis用在五个地方。

第一是验证码存储。key是"code_"加邮箱，value是4位验证码，过期时间600秒。注册或重置密码时用来验证。

第二是用户Token。key是"utoken_"加uid，value是UUID生成的Token。登录验证时用来确认身份。

第三是用户位置。key是"uip_"加uid，value是服务器名（chatserver1或chatserver2）。跨服务器推送时用来找目标用户在哪台服务器。

第四是登录计数。key是"logincount"，是Hash类型，field是服务器名，value是登录人数。负载均衡时用来选服务器。

第五是用户信息缓存。key是"ubaseinfo_"加uid，value是用户信息的JSON。查询时优先查缓存，缓存没有查MySQL并回填。

**一致性保证**

Redis只做缓存，数据以MySQL为准。写操作时先写MySQL，成功后再更新或删除Redis。查操作时先查Redis，Redis没有查MySQL并回填。

**如果Redis和MySQL数据不一致，以MySQL为准。因为MySQL是持久化存储，Redis可能因为重启丢失数据。**

## 13、讲项目做得好的一个点

**问题**：讲项目做得好的一个点

**详细回答**：

我觉得最好的亮点是**异步IO加消息队列解耦的架构**。

用Boost.Asio实现高性能网络IO，单机能抗3-5万连接。IO线程只管读写，读完后投到消息队列。业务逻辑由单独的工作线程处理，完全串行，不需要加锁。

这样做的好处很多：第一性能高，异步IO不阻塞；第二代码简单，业务逻辑没有并发问题，不需要处理锁；第三容易维护，逻辑清晰。

具体实现：Logic_msg_que队列和System有一个一条工作线程。IO线程收到消息后调用PostMsgToQue投到队列。工作线程用while循环从队列取消息，根据msgid找到对应处理函数执行。

```cpp
void LogicSystem::PostMsgToQue(shared_ptr<LogicNode> msg) {
    std::unique_lock<std::mutex> unique_lk(_mutex);
    _msg_que.push(msg);
    if(_msg_que.size() == 1) {
        unique_lk.unlock();
        _consume.notify_one();
    }
}
```

## 14、大量用户连接时，负载的处理和断连的处理

**问题**：大量用户连接时，负载的处理和断连的处理

**负载处理**

大量连接时，StatusServer的最小连接数算法保证负载均衡。每台ChatServer的登录人数存在Redis里，登录时ChatServer调用HINCRBY增加计数。用户越多，新登录的用户越可能被分到人数少的服务器。

如果某台服务器特别忙，可以通过**增加ChatServer实例来横向扩展。**

**断连处理**

分客户端检测和服务端处理。

客户端检测：启动定时器，每30秒发一次心跳。如果60秒没收到服务器响应，认为断连，尝试重连。

服务端处理：CSession的async_read回调里检测error。如果有错误（比如connection reset），调用Close关闭连接，然后调用Server的ClearSession清理。

清理操作包括：从UserMgr的map里删除uid和session的绑定；从sessions的map里删除session_id；删除Redis里的uip_key。

## 15、客户端请求到达服务端后的通信链路

**问题**：客户端请求到达服务端后的通信链路

**详细回答**：

以用户登录为例。

**第一步：HTTP请求到GateServer**

客户端发起POST请求，目标URL是http://gate:8080/user_login，请求体是JSON：{"email":"xxx","passwd":"xxx"}。

Linux内核通过epoll通知GateServer的IO线程有数据可读。IO线程调用http::async_read读取数据。解析HTTP头和body，调用LogicSystem的HandlePost方法。

**第二步：LogicSystem处理**

HandlePost根据path（/user_login）找到注册的处理lambda。处理lambda解析JSON，取出email和password。调用MysqlMgr的CheckPwd方法验证密码。

**第三步：调用StatusServer**

密码验证通过后，LogicSystem调用StatusGrpcClient的GetChatServer方法。这个方法通过gRPC Stub调用StatusServer的GetChatServer方法，传入uid。

StatusServer返回聊天服务器地址（host、port）和Token。

**第四步：返回响应**

GateServer把StatusServer返回的信息打包成JSON，通过HttpConnection的WriteResponse方法发送HTTP响应。

**第五步：TCP连接ChatServer**

客户端收到响应后解析得到聊天服务器地址。创建TCP连接到chatserver:8090，发送登录包（自定义二进制格式）。

ChatServer的CSession收到连接，async_accept回调创建新session。session读取登录包，调用LogicSystem验证Token。验证通过后绑定uid和session，返回成功。

## 16、数据库中用户密码的加密存储

**为什么不能明文**

如果数据库泄露，所有用户密码都暴露。攻击者可以用这些密码去撞其他网站（很多人密码通用）。

登录时：用户输入密码后，服务器用存着的盐重新计算MD5，和数据库比对。

这样即使数据库泄露，攻击者也难以恢复原始密码（因为加了随机盐，无法用彩虹表）。

## 17、高并发场景下单个连接数不足的问题具体讲讲

**问题现象**

高并发下，如果MySQL连接池只有5个，但有100个请求同时来，那95个请求要等待。等待时间过长会导致超时。

**解决思路**

第一是增加连接池数量。如果CPU够用，可以加到10-20个。但不是越多越好，太多了会增加锁竞争。

第二是水平扩展。多加ChatServer实例，分散请求。请求少了，每台服务器分到的就少。

第三是异步非阻塞。确保每个连接尽可能快地处理完释放，不要在连接上做耗时操作。比如不要在查询数据库时同步等待结果，可以改成异步。

第四是读写分离。把查询和写入分开，读请求走从库。

## 18、为什么要用线程池

**问题**：为什么要用线程池

**第一是避免频繁创建销毁开销**

创建一个线程需要分配栈空间、调用内核函数创建线程控制块，成本大约是几毫秒。如果每个请求都创建线程，1万个请求就要1万次创建销毁，消耗巨大。

线程池预创建一批线程，用完了归还而不是销毁。下次有任务直接从池子里拿，零创建成本。

**第二是控制资源消耗**

如果不限制线程数量，用户请求一多，服务器会创建几千个线程。每个线程1MB栈空间，1000个线程就要1GB内存。线程切换CPU消耗也大。线程池限制同时运行的线程数，保护系统。

**第三是任务缓冲**

请求来的速度和处理速度可能不一样。如果处理速度慢，请求在队列里排队。线程池提供一个缓冲区域，不丢失请求。

## 20、客户端和服务端网络通信的过程，包括具体的协议

**问题**：客户端和服务端网络通信的过程，包括具体的协议

**详细回答**：

**协议选择**

GateServer和客户端之间用HTTP（短连接）。ChatServer和客户端之间用TCP（长连接）。ChatServer之间用gRPC。

**HTTP协议**

标准HTTP/1.1，请求行、请求头、请求体。请求体是JSON格式。比如登录：POST /user_login HTTP/1.1，body是{"email":"xxx","passwd":"xxx"}。

**TCP自定义二进制协议**

格式是4字节头部加变长body：

```
┌──────────┬──────────┬────────────┐
│  MSGID   │   LEN    │   DATA     │
│  2字节   │  2字节   │   N字节    │
└──────────┴──────────┴────────────┘
```

MSGID：消息ID，比如1005是登录，1017是聊天。2字节short，网络字节序（用host_to_network_short转换）。

LEN：数据长度。2字节short，网络字节序。

DATA：JSON格式的业务数据。

**示例**

客户端发送登录包：假设uid=1001，token="abc-def-ghi"。

JSON数据是{"uid":1001,"token":"abc-def-ghi"}，假设长度是25字节。

实际发送：05 E9（1005的16进制，2字节） 00 19（25的16进制） {"uid":1001,"token":"abc-def-ghi"}

服务端解析：先读2字节转成short得到msgid=1005。再读2字节得到len=25。然后读25字节，用Jsoncpp解析出业务数据。

## 26、为什么选择封装tcp（为什么不用udp、为什么不直接用库里的tcp）

**问题**：为什么选择封装tcp（为什么不用udp、为什么不直接用库里的tcp）

**详细回答**：

**为什么不用UDP**

UDP是无连接、不可靠的协议。发出去的消息不保证对方能收到，也没有重传机制。聊天消息不能丢，丢了用户会察觉。

UDP适合的场景：实时音视频（可以容忍丢帧）、DNS查询、广播等。聊天是可靠消息传输，必须用TCP。

**为什么封装TCP**

Boost.Asio只提供底层API，比如async_read_some、async_write等。直接用的话，每个连接要自己处理：协议解析（粘包拆包）、心跳检测、断线重连、会话管理。

封装后形成统一的业务层协议：自定义二进制格式（2字节msgid + 2字节len + JSON），统一的消息处理流程。代码更清晰，也方便替换底层实现。

## 28、如果要进行文件传输，应该怎么做

文件传输不能走IM的TCP通道，原因是：大文件占用通道影响即时消息；断点续传、进度展示需要独立逻辑。

正确方案是分离式文件服务：

第一步是客户端上传文件到文件服务器。文件服务器返回file_id和file_url。

第二步是发送消息。IM消息体只带file_id，接收方根据file_id从文件服务器下载。

第三步是大文件处理。分片上传（每次64KB），支持断点续传。图片压缩（WebP），视频用H.264编码。

文件服务器可以用：阿里云OSS（稳定贵）、FastDFS（开源自建）、MinIO（兼容S3）。

## 29、redis缓存怎么用的，为什么不直接用MySQL

**问题**：redis缓存怎么用的，为什么不直接用MySQL

**详细回答**：

Redis在项目里用在五个地方：验证码（code_邮箱）、Token（utoken_uid）、用户位置（uip_uid）、登录计数（logincount）、用户信息缓存（ubaseinfo_uid）。

为什么不全用MySQL：性能差距大。MySQL每秒几千次查询，Redis能上百万次。热点数据（用户信息）80%请求集中在20%用户，缓存能大幅减轻DB压力。临时数据（验证码、Token）不需要持久化，Redis更合适。

为什么不全用Redis：不可靠，Redis挂了可能丢数据。不支持复杂查询（好友关系需要联表）。内存比磁盘贵10倍。

正确做法是缓存加数据库组合：先查Redis，命中直接返回；未命中查MySQL并回填Redis。写操作先写MySQL再删Redis。

## 30、项目中遇到的困难，如何解决

**问题**：项目中遇到的困难，如何解决

**详细回答**：

困难一：跨服务器消息推送失败。

现象是A在Server1，B在Server2，A发消息给B但B没收到。排查发现gRPC调用超时没处理。解决：添加超时处理和错误日志，RPC失败返回错误码让客户端重试。

困难三：高并发下消息乱序。

现象是快速发多条消息，收到的顺序不对。原因是多线程处理速度不同。解决：在消息体加序号，接收端按序号排序后处理。

## 32、用asio主要实现了哪些功能

**问题**：用asio主要实现了哪些功能

**详细回答**：

Boost.Asio实现了四个核心功能。

1. **TCP服务器**：使用acceptor异步接受新连接，监听8090端口
2. **异步读写**：使用async_read_some和async_write实现全异步IO，不需要阻塞等待。读取时先读取4字节包头，再读取数据体，解决TCP粘包问题。
3. **IO线程池**：创建多个io_context，每个启动一个线程运行run()事件循环。新连接通过Round-Robin分配到不同io_context，充分利用多核CPU。
4. **信号处理**：使用signal_set监听SIGINT/SIGTERM，实现优雅退出。

核心组件是io_context，充当事件循环。调用async_xxx提交任务，底层用epoll（Linux）或IOCP（Windows）监听，事件到了触发回调。

## 33、为什么使用多线程模型，如何处理多个请求

> "我在项目中使用了两种线程模型：
>
> 1. **IO线程池**：创建多个io_context，每个启动一个线程处理TCP连接。连接通过Round-Robin分配到不同io_context，这样可以充分利用多核CPU。Boost.Asio底层会用IOCP（Windows）高效处理海量连接。
> 2. **单线程业务处理**：LogicSystem用一个独立线程从消息队列取消息并处理业务。这样设计的好处是避免竞态条件，不需要对共享数据加锁，简化了并发编程。
>
> 整个流程是：IO线程接收数据 → 投送到消息队列 → 业务线程取出处理 → IO线程响应。整个系统是事件驱动的，没有阻塞IO，效率很高。"

**问：为什么业务处理不放在IO线程中？**

> "如果每个IO线程都处理业务，会有两个问题：1. 多个线程访问同一数据需要加锁，增加复杂度。2. 业务处理可能阻塞（比如查数据库），影响IO效率。分离后，IO线程只管读写，业务线程专注处理逻辑，互不干扰。"

**问：单线程处理业务会不会成为瓶颈？**

> "目前项目规模不大，单线程足够。如果高并发，可以改成多线程队列，或者用线程池。最简单的方式是创建多个LogicSystem，每个处理一部分消息ID。"

## 35、gRPC如何实现断线重连

**问题**：gRPC如何实现断线重连

**详细回答**：

项目目前没实现gRPC自动重连，这是不足。正确做法：

方案一是**gRPC底层配置** 在创建Channel时配置重连参数，设置初始**重连间隔、最大间隔、指数退避倍数、可重试的错误码**。这样底层会自动处理断线重试。

方案二是**手动**封装一个调用失败重试的模板方法，调用失败时判断错误码，如果是UNAVAILABLE等临时错误，就重试3次，每次等待时间指数增长。

方案三是**心跳检测** 启动后台线程，每10秒对每个连接发Ping，超时则重建Channel。这样可以主动发现断连，不需要等到发消息时才知道。

## 36、两个客户端之间如何实现聊天功能

**问题**：两个客户端之间如何实现聊天功能

**详细回答**：

A发消息给B的完整流程：

第一步，A的客户端通过TCP连接到ChatServer1，发送聊天消息。消息体包含：fromuid（A的uid）、touid（B的uid）、text_array（消息内容）。

第二步，ChatServer1收到消息，调用LogicSystem的DealChatTextMsg处理。

第三步，LogicSystem查Redis，用key"uip_"+B的uid得到B在哪台服务器。如果B在ChatServer1，直接推送。如果B在ChatServer2，用ChatGrpcClient调用gRPC转发。

第四步，ChatServer2收到gRPC调用，通过UserMgr找到B的session，调用session->Send推送消息。

核心是Redis存储用户位置：登录时存"uip_uid"="服务器名"，发送时查这个key就知道目标用户在哪。

## 37、服务器如何确定用户连接到哪个聊天服务器（用户状态管理）

**问题**：服务器如何确定用户连接到哪个聊天服务器（用户状态管理）

**详细回答**：

两层存储。

第一层是内存。UserMgr用unordered_map<int, shared_ptr<CSession>>存uid到session的映射。查询O(1)，速度快。缺点是只存当前服务器的。

第二层是Redis。key是"uip_"加uid，value是服务器名。登录时Set：RedisMgr::GetInstance()->Set("uip_1001", "chatserver1")。查询时Get：RedisMgr::GetInstance()->Get("uip_1001", value)。

查找用户位置：先从UserMgr查（本地服务器），查不到从Redis查（跨服务器）。

断开时清理：删除UserMgr里的映射，删除Redis的key。

## 38、如何实现发送验证码功能

**问题**：如何实现发送验证码功能

**详细回答**：

验证码流程五步走：

第一步，客户端POST /get_varifycode到GateServer，带email。

第二步，GateServer调用VerifyGrpcClient的GetVarifyCode方法，发gRPC请求到VarifyServer。

第三步，VarifyServer收到请求，用uuid生成4位随机码。

## 40、为什么需要网关服务，没有行不行

**问题**：为什么需要网关服务，没有行不行

**详细回答**：

没有网关也能跑，但有四个问题。

第一，客户端需要知道所有服务地址。登录连GateServer，聊天连ChatServer，没网关的话客户端要配多个地址，维护困难。

第二，安全没法保证。每个服务都暴露公网，容易被攻击。有网关统一鉴权。

第三，协议不统一。内部用gRPC，对外用HTTP，网关做协议转换。

第四，负载均衡难做。没有网关，客户端直连ChatServer，没法做负载均衡。

有网关的好处：统一入口；鉴权限流；协议转换；隐藏内部服务。

## 41、以什么标准进行服务拆分呢

**问题**：以什么标准进行服务拆分呢

**详细回答**：

服务拆分原则是单一职责、独立部署、边界清晰。

单一职责：每个服务只做一件事。GateServer只做路由，VarifyServer只管验证码，StatusServer只管状态和负载均衡，ChatServer只管聊天。

独立部署：每个服务可以单独启动、升级、回滚。不相互影响，一台挂了其他还能跑。

边界清晰：服务间通过API通信，不直接读写对方数据库。依赖明确，接口稳定。

按业务能力拆分：登录注册是一类（GateServer+VarifyServer），状态管理是一类（StatusServer），聊天是一类（ChatServer）。

## 42、如何实现断线重连

分客户端和服务端。

客户端重连：检测到断开后，尝试重连。设置重试间隔（比如3秒），最多重试N次。重连成功后，用缓存的token重新登录。

```cpp
void Reconnect() {
    for(int i=0; i<maxRetry; i++) {
        if(Connect(serverIP, serverPort) == SUCCESS) {
            // 重连成功，重新登录
            SendLoginReq(token);
            return;
        }
        Sleep(3000);  // 3秒后重试
    }
}
```

服务端清理：检测到client close后，从UserMgr删除uid映射，删除Redis的uip_key，更新登录计数。

## 44、tcp长连接的作用

第一是减少握手延迟。TCP三次握手需要30-100ms。长连接建立后一直保持，下次发消息直接用，不需要重新握手。对于频繁通信的场景，节省很多时间。

第二是服务器推送。聊天需要实时收到对方消息。短连接只能是客户端主动发请求，服务器没法主动推。长连接建立后，服务器可以随时通过这条连接发数据给客户端。

## 45、如何解决大量tcp连接的性能问题，如果用户登录后长时间没有请求（心跳机制），如果又突然要发消息了，需要重新登陆吗

大量连接性能问题：用异步IO加事件驱动（epoll），单线程处理万级连接。瓶颈在内存和文件描述符。

心跳机制：客户端每30秒发一次ping。服务端用steady_timer设置60秒超时。收到数据重置定时器。60秒没收到数据就关闭连接。

不需要重新登录：心跳只是检测连接是否活着，Token还在Redis里有效。断线重连后，用原Token重新认证即可恢复会话。不需要用户重新输入密码。

## 47、asio网络库底层原理

"**Asio底层原理**：

- Linux用epoll，Windows用IOCP
- epoll是事件驱动，注册socket的读写事件，当事件就绪时回调执行
- IOCP是完成端口，异步操作完成后通知回调
- io_context是调度器，管理所有异步操作和回调
- async_accept/async_read等都是提交任务，立即返回，等事件到了再执行回调

## 50、消息发送的协议是自定义的还是用了什么

**问题**：消息发送的协议是自定义的还是用了什么

**详细回答**：

TCP是自定义二进制协议。

格式：2字节msgid + 2字节len + JSON数据。msgid和len都是short类型，网络字节序（用host_to_network_short转换）。

为什么自定义：比JSON更紧凑（2+2=4字节头部vsJSON {"msgid":1001}），解析更快（memcpy vs JSON解析），适合高频聊天。

gRPC用Protocol Buffers。定义在message.proto里，编译生成C++代码。序列化后体积比JSON小5-10倍，解析速度更快。

## 51、为什么要将登录和聊天分开不同的服务器

1. **职责分离**：登录和聊天的业务逻辑完全不同
   - 登录：一次性HTTP请求，验证账号密码，返回服务器地址
   - 聊天：长连接TCP，高频消息收发，Session管理

   - ChatServer挂了，可以登录但不能聊天

这也是微服务架构的核心理念：单一职责、独立部署、按需扩展。"

## 57、登录用的是http的哪个方法，为什么用post，不用get

用POST。

GET和POST区别：GET参数在URL里，POST在body里。GET URL有长度限制（通常2KB），POST无限制。GET适合查询，POST适合提交。

用POST原因：密码在body里，不在URL里，相对安全（URL会记日志、存浏览器历史）。密码可能很长，GET放不下。语义上，登录是提交数据，不是查询。

## 58、项目中MySQL主要存储哪些数据，redis主要存什么，为什么这么设计？

"项目中MySQL主要存三类数据：

1. **用户信息**：uid、name、email、pwd等，注册后长期保存
2. **好友关系**：friend表，双向存储
3. **好友申请**：apply_friend表，存储申请记录

Redis主要存五类数据：

1. **Token**：utoken_用户ID，登录验证用，有过期时间
2. **用户位置**：uip_用户ID，存储在哪台ChatServer，转发消息用
3. **用户缓存**：ubaseinfo_用户ID，JSON格式，登录时快速返回
4. **在线人数**：logincount的Hash，负载均衡用
5. **验证码**：code_邮箱，5分钟有效期

**为什么这样设计**：

- MySQL是持久化存储，保证数据不丢，适合用户信息、好友关系等长期数据
- Redis是内存数据库，读写快，适合频繁变化的数据（Token、用户位置）和需要TTL的数据（验证码）
- 用户信息用MySQL+Redis缓存，既保证持久化又提升查询性能
- 好友关系用MySQL是因为需要复杂关联查询，Redis不适合"

## 59、聊天过程中的数据如何产生，如何传递，如何存储

数据产生：用户打开聊天窗口，输入文字或选择图片，点击发送。客户端把数据封装成JSON（包含fromuid、touid、text_array等）。

数据传递（TCP）：客户端用TCP socket发送。数据格式是自定义二进制：2字节msgid（1017是聊天） + 2字节len + JSON数据。发送用async_write。

服务端接收：CSession的async_read读数据。先读4字节头，再读body。解析msgid和len，读取body。用Json解析业务数据。

业务处理：LogicSystem收到消息，投到队列。业务线程处理：查Redis找目标用户位置；本地推送或gRPC转发。

数据存储：正确做法是收到消息立即存MySQL，标记status=0（未送达）。对方收到后ACK，status=1（已送达）。对方上线时拉取离线消息。

## 60、存储聊天数据的表如何设计，如何建立索引

**问题**：存储聊天数据的表如何设计，如何建立索引

**详细回答**：

表设计：

```sql
CREATE TABLE message (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    from_uid INT UNSIGNED NOT NULL,
    to_uid INT UNSIGNED NOT NULL,
    type TINYINT UNSIGNED DEFAULT 1 COMMENT '1:text 2:image 3:voice 4:video',
    content TEXT COMMENT '文字内容',
    file_id VARCHAR(64) COMMENT '文件ID',
    file_url VARCHAR(500) COMMENT '文件URL',
    msg_id VARCHAR(64) COMMENT '客户端消息ID，去重',
    status TINYINT DEFAULT 0 COMMENT '0:已发送 1:已送达 2:已读',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    INDEX idx_from_uid (from_uid),
    INDEX idx_to_uid (to_uid),
    INDEX idx_created_at (created_at),
    INDEX idx_from_to (from_uid, to_uid, created_at),
    UNIQUE KEY uk_msg_id (msg_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

索引设计：idx_from_uid按发送者查询；idx_to_uid按接收者查询；idx_created_at按时间排序；idx_from_to两人聊天记录（最常用）；uk_msg_id消息去重（重发时避免重复存储）。

分页优化：不要用OFFSET，用主键游标。比如查最新20条：WHERE to_uid=? AND id < last_id ORDER BY id DESC LIMIT 20。

## 61、如果离线消息过大，需要等所有数据发送完之后再删除吗，如果发送过程中服务断掉，那下一次要重复发送吗

正确做法是边发边删，不用等全部发完。

流程：用户上线，查MySQL未送达消息（status=0）。逐条发送，收到客户端ACK后更新status=1。如果没ACK，status保持0。

不需要等所有发完再删：发一条ACK一条，标记一条。效率更高，体验更好（用户立刻能看到消息）。



断线重发问题：消息存MySQL，status=0表示未送达。用户重连上线，再拉取status=0的消息重发。

可靠性保证：发送方存消息到MySQL再发。接收方收到ACK才标记已送达。ACK丢失不怕，下次重发。

## 63、客户端和服务端的消息是如何封装和解析的，数据包有哪些字段

项目使用自定义二进制协议，格式如下：

```
┌────────────┬────────────┬──────────────────┐
│   MSGID    │    LEN     │      DATA        │
│   2字节    │   2字节    │     N字节        │
└────────────┴────────────┴──────────────────┘
```

字段解释：MSGID是消息ID，2字节short类型，用于标识消息类型（如1005是登录，1017是聊天）。LEN是数据长度，2字节short类型，表示后续JSON数据有多少字节。DATA是实际的业务数据，JSON格式。



"客户端和服务端使用一致的4字节定长包头协议：

- 前2字节是消息ID（MSG_ID）
- 后2字节是数据长度（DATA_LEN）
- 后面是JSON格式的数据

**发送时**：客户端用QDataStream把ID和长度写入bytearray，加上JSON数据，用QTcpSocket::write发送

**接收时**：先读取4字节头，解析出ID和长度，再读取对应长度的数据，用QJsonDocument解析JSON

这样设计解决了TCP粘包问题：每次都先读固定长度的头，根据长度再读数据，保证完整解析。"

```cpp
enum MSG_IDS {
    MSG_CHAT_LOGIN = 1005,        // 用户登录
    MSG_CHAT_LOGIN_RSP = 1006,    // 登录响应
    ID_SEARCH_USER_REQ = 1007,     // 搜索用户
    ID_SEARCH_USER_RSP = 1008,    // 搜索响应
    ID_ADD_FRIEND_REQ = 1009,     // 添加好友请求
    ID_ADD_FRIEND_RSP = 1010,     // 添加好友响应
    ID_NOTIFY_ADD_FRIEND_REQ = 1011, // 通知添加好友
    ID_AUTH_FRIEND_REQ = 1013,    // 认证好友请求
    ID_AUTH_FRIEND_RSP = 1014,    // 认证好友响应
    ID_NOTIFY_AUTH_FRIEND_REQ = 1015, // 通知认证好友
    ID_TEXT_CHAT_MSG_REQ = 1017,  // 文字聊天消息请求
    ID_TEXT_CHAT_MSG_RSP = 1018,  // 文字聊天消息响应
    ID_NOTIFY_TEXT_CHAT_MSG_REQ = 1019, // 通知文字聊天消息
};
```

## 65、QT的信号和槽是什么机制，和回调函数对比说明优点，qt对象树和内存管理怎么做的，qt的信号和事件是什么，区别。

"Qt的信号槽机制是观察者模式的实现。

**信号槽优点**：

1. 类型安全：编译期检查参数匹配
2. 解耦：发送者和接收者不需要知道彼此
3. 动态连接：可以运行时连接/断开
4. 多对多：一个信号可以连多个槽，一个槽可以监听多个信号

**对象树管理**：

- Qt通过父子关系形成对象树
- 父对象销毁时，子对象自动销毁
- 不需要手动delete，简化内存管理

**信号和事件的区别**：

- 信号是对象主动发出的，用connect连接槽处理
- 事件是系统触发的，重写event函数处理
- 信号用于自定义行为，事件用于系统交互（如鼠标、键盘）
- 信号优先级更高，直接调用；事件走事件队列"

**"MFC和Qt的区别：**

- MFC是Windows专用，Qt跨平台
- MFC用消息映射表，Qt用信号槽
- MFC需要手动管理内存，Qt有对象树自动管理

**信号槽底层原理**：

- 编译时moc生成元对象信息（信号索引、槽函数索引）
- connect时把信号和槽的信息存到ConnectionList
- emit信号时调用QMetaObject::activate，遍历所有连接
- 通过方法索引找到槽函数并调用

**交互触发流程**：

## 66、gRPC通信用什么协议，protobuf的作用

"gRPC在项目中用的是HTTP/2协议 + Protobuf序列化。

- 头部压缩：省带宽

- 体积小60-80%（二进制vs文本）

## 67、项目中有没有遇到死锁的情况，有没有遇到内存的crush等情况

**死锁情况**

没遇到过死锁。因为项目里业务逻辑是单线程处理的，通过消息队列解耦，不需要加锁。锁只用在连接池的地方（Redis、MySQL），但那些锁的粒度很小，而且用了std::lock_guard或std::unique_lock，作用域清晰，不容易死锁。

**内存问题**

遇到过两个问题。

问题一是MySQL连接泄漏。某个函数提前return导致连接没归还。解决：用Defer类封装归还逻辑，析构时自动执行。

问题二是测试时频繁断连重连导致内存增长。查了一下是CSession对象没有及时释放。解决：用了stdstd::shared_ptr和::enable_shared_from_this，通过智能指针自动管理生命周期。

**其他崩溃**

曾经遇到过段错误（SIGSEGV），原因是访问了空指针。后来加了断言和防御性编程，检查指针有效性。

## 70、具体的添加好友的业务是怎么实现的

1. **申请**：用户A点击添加用户B，发送ID_ADD_FRIEND_REQ请求
2. **处理**：ChatServer收到请求后：
   - 把申请写入apply_friend表

   - 通知用户A"你已成为好友"

这样就完成了从申请到同意的完整流程。"

## 71、主动添加方的消息推送是怎么做的（因为被添加方同意时，主动添加方可能不在线）

离线消息存储：

第一步，收到认证请求时，先判断目标用户是否在线。

第二步，如果在线，立即推送；如果不在线，把消息存入MySQL的离线消息表。

第三步，用户下次登录时，先查询离线消息表，拉取所有未读消息，发送给用户。用户收到并ACK后，删除离线消息或标记为已读。

## 72、如果同一账号可以多端登录，如何解决消息同步问题

"同一账号多端登录的消息同步，有几种方案：

1. **消息广播**：发送消息时，查询UserMgr中该用户的所有Session，遍历发送。这样手机、电脑、平板都能同时收到。
2. **消息存储+拉取**：消息存入数据库，每个设备登录时从数据库拉取未读消息。离线期间的消息也不会丢失。
3. **已读同步**：消息增加已读状态，用消息ID+设备ID标记。任一设备已读后，同步到其他设备。

## 73、同一账号，在a设备上时添加的好友，之后在b设备上登陆时，如何同步这个好友信息

1. **存储在MySQL**：好友关系存在friend表中，是双向的（A的好友列表里 有B，B的好友列表里也有A）
2. **登录时同步**：用户登录时，ChatServer会从MySQL查询该用户的好友列表，返回给客户端
3. **具体流程**：
   - A设备登录 → 查询friend表 → 返回好友列表

   - B设备登录 → 查询friend表 → 包含C → 返回给B
4. **实时性**：好友是持久化存储在数据库的，登录时从数据库读取，所以不同设备登录都能看到同样的好友列表。

**改进方案**：

- 当前是登录时同步，好友变更不会实时推送到其他设备
- 如果需要实时同步，可以：A添加好友 → 推送给A在线的所有设备 + 存入离线消息，B登录时拉取"

## 74、什么是asio，具体可以实现哪些功能

"Boost.Asio是C++的异步I/O库，项目中主要用它实现了三个功能：

1. **TCP服务器**：用tcp::acceptor监听端口，异步接受连接
2. **异步读写**：用socket.async_read_some()和async_write()实现全异步IO，不需要阻塞等待数据
3. **IO线程池**：创建多个io_context，每个启动一个线程运行run()，处理海量连接

**为什么用Asio**：

- 底层用epoll（Linux）或IOCP（Windows），性能高
- 支持百万级并发连接
- 代码风格优雅，用回调而不是回调地狱
- C++标准库有对应版本（std::asio）

## 75、两个chatsever间怎么通信（答：gRPC），那如果有成千上万台服务器呢

**大规模服务器的问题**

如果服务器数量达到成千上万台，gRPC直连模式会有问题：每台服务器都要维护到其他所有服务器的连接，连接数爆炸（O(n^2)）；服务地址hardcode，无法动态感知服务上下线；没有负载均衡和故障转移。

**解决方案**

引入中间件：

消息队列。用RabbitMQ或Kafka做消息转发。服务器只和消息队列连，消息发给队列，由队列转发到目标。优点是解耦、削峰、持久化。

## 76、如果需要添加一个视频聊天功能，你觉得应该怎么设计，视频需要通过服务器转发吗

视频聊天不能走IM的TCP通道，因为视频流太大，会阻塞正常消息。

## 78、聊天项目中，客户端给服务器发消息，有tcp连接过程吗，还是直接用http发送

"在项目中，客户端和服务器的通信分两种：

1. **HTTP短连接**：用于登录、注册、获取验证码等一次性操作。客户端发请求到GateServer，服务器响应后连接关闭。这些操作不频繁，用HTTP简单直接。
2. **TCP长连接**：用于聊天。客户端登录成功后，拿到ChatServer的地址，建立TCP长连接，后续所有聊天消息都通过这个连接收发。TCP连接建立后一直保持，服务器可以随时推送消息给客户端，不需要轮询。

**为什么分开**：

- HTTP适合轻量级的一次性操作
- TCP适合高频、实时的消息通信
- 登录只需要一次，聊天可能每分钟发多条，用TCP更高效

**为什么用TCP不用HTTP**

HTTP是短连接。每次请求完就关闭，下一次发消息要重新建立连接。三次握手开销大，频繁通信效率低。HTTP无法实现服务器主动推送（只能轮询或长轮询）。

TCP是长连接。建立一次，长期使用。服务器可以主动推送消息给客户端。

## 79、聊天服务器如何验证客户端的token

"Token验证流程是这样的：

1. **Token生成**：用户登录时，StatusServer生成一个UUID作为Token，存入Redis（key是utoken_uid），并返回给客户端。
2. **客户端保存**：客户端拿到Token后，保存到本地，后续连接ChatServer时带上这个Token。
3. **Token验证**：ChatServer收到客户端的登录请求后，从Redis中读取utoken_uid的value，和客户端传来的token对比：
   - 如果Redis中没有这个key，返回uid无效
   - 如果Token不匹配，返回Token无效
   - 验证通过后，绑定Session和用户ID
4. **后续请求**：用户后续发消息时，已经绑定了uid，不需要再验证Token。

目前的设计比较简单，生产环境可以改进：

- Token加过期时间（比如24小时）
- 定期刷新Token
- 或者用JWT等更完善的认证方案"

## 81、介绍聊天项目，gRPC是怎么用的，同步调用还是异步调用，有考虑过超时的问题吗

**项目介绍**

这是分布式即时通讯系统，四大微服务：GateServer（HTTP网关）、VarifyServer（验证码）、StatusServer（状态和负载均衡）、ChatServer集群。核心功能：用户登录、验证码、添加好友、聊天。

**gRPC使用方式**

gRPC用来做ChatServer之间的跨服务器通信。比如A在ChatServer1，B在ChatServer2，A发消息给B时，ChatServer1用gRPC调用ChatServer2的NotifyTextChatMsg接口转发消息。

**同步**

- 调用后阻塞等待响应
- 简单，但可能阻塞业务线程

这样写的问题是：如果对方服务器响应慢，会阻塞当前线程。虽然LogicSystem是单线程，但消息处理会被拖慢。

**超时考虑**

gRPC默认有超时设置。如果对端服务挂了，调用会超时返回。超时会返回错误码，调用方做容错处理。

## 82、网络模型如何工作的，需要处理什么样的并发场景

"项目主要处理以下并发场景：

1. **大量客户端连接**：每个TCP连接分配到不同的io_context，独立处理，互不阻塞。
2. **并发收发消息**：读和写都是异步的，同一连接可以同时读和写。不同连接的消息由不同io_context并行处理。
3. **跨服务器通信**：多个ChatServer之间通过gRPC通信，用连接池管理Stub，获取和归还需要加锁。
4. **数据库并发访问**：多个IO线程可能同时需要数据库连接，通过连接池+条件变量管理，只允许10个线程同时使用。
5. **消息队列生产消费**：IO线程投消息，LogicSystem线程消费，用mutex+condition_variable同步。

**设计原则**：

- IO线程只管读写，不做业务
- 业务逻辑单线程处理，避免数据竞争
- 共享资源用连接池+锁保护

这样设计保证了高并发下的稳定性和性能。"

## 83、io_context在单线程和多线程下分别是怎么工作的

"io_context是Boost.Asio的核心，可以理解为一个事件循环调度器。

它内部有一个任务队列，异步操作完成后会把回调放到队列里，然后io_context执行这些回调。

**多线程模式**（ChatServer）：

- 创建多个io_context，每个启动一个线程运行run()
- 新连接通过Round-Robin分配到不同io_context
- 每个io_context独立处理自己的连接，互不干扰
- 充分利用多核CPU

**单线程模式**（GateServer）：

- 只有一个io_context，一个线程
- 所有连接都在这个线程处理
- 简单，但无法利用多核

**底层原理**：

- Linux用epoll_wait等待IO事件
- Windows用IOCP（完成端口）
- 当socket可读/可写时，触发回调

这样设计的好处是：开发者不需要关注线程，只需要关注业务逻辑，异步操作由io_context调度。"

## 84、你的项目有没有数据共享情况的处理，是怎么控制并发中的锁的

1. **消息队列**：IO线程投递消息到LogicSystem线程，使用mutex+condition_variable同步，保证队列操作的原子性。
2. **用户Session管理**（每个TCP连接对应一个Session）：UserMgr中维护uid到Session的映射，多个IO线程可能同时读写，用mutex保护。
3. **连接池**：MySQL、Redis、gRPC连接池在获取和归还连接时加锁，保证队列的线程安全。
4. **发送队列**：CSession的发送队列可能被IO线程和回调线程同时访问，用mutex保护。

**为什么锁不多**：

- LogicSystem是单线程，业务处理不需要加锁

- 每个服务器独立管理自己的Session，不跨服务器竞争

- 合理设计后，大部分数据都是单线程访问，只有共享资源才加锁 

  

**经验**：

- 尽量减少锁的粒度
- 用RAII（lock_guard）自动释放锁
- 考虑用无锁队列或原子操作提升性能"

**避免锁的思路**

最好的办法是避免共享。LogicSystem的消息队列把IO和业务分离，业务线程完全串行，不需要锁。UserMgr虽然是共享的，但每个用户绑定到固定session，处理时也是局部数据。
