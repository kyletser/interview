## 1. InnoDB 和 MyISAM 有什么区别？为什么现在默认讲 InnoDB？

**主答：**
最核心的区别是：**InnoDB 支持事务、行级锁、MVCC 和崩溃恢复**，

而 MyISAM 不支持事务，锁粒度也更粗。

现代业务更看重并发控制和数据可靠性，所以面试里默认重点讲 InnoDB。

**追问 1：InnoDB 为什么更适合高并发 ？**
因为它有**行级锁**和**一致性非锁定读**，多个会话可以同时读写同一张表，而不是像粗粒度锁那样更容易互相阻塞。

**追问 2：是不是所有表都无脑选 InnoDB？**
面试里别说绝对。标准回答是：**绝大多数业务核心表优先 InnoDB**；只有非常特殊、非常明确的场景，才会考虑别的引擎。

------

## 2. 为什么 MySQL 常见索引是 B+ 树，而不是红黑树或 Hash？

**主答：**
因为 InnoDB 普通索引本质是 **B-tree 家族结构**，它天然支持**等值查找、范围查找、排序扫描**；

而 Hash 索引只擅长等值匹配，不适合范围查询。

至于红黑树，它是二叉树，放到数据库页 I/O 场景里层高更高，不如高扇出的 B-tree 适合磁盘页访问。

**追问 1：Hash 为什么不适合数据库主索引？**
因为 Hash 只能高效处理等值比较，**不能高效支持范围查询`**。

**追问 2：B-tree 在 MySQL 里能做什么？**
它可以快速处理 `=`、`>、<、BETWEEN、IN` 等条件；而且按索引顺序扫描时，对排序和范围查询都很友好。

------

## 3. 什么是聚簇索引、二级索引、回表？

InnoDB 表数据是按**聚簇索引**组织的，通常就是主键索引；

**二级索引**里保存的是“索引列值 + 主键值”。如果先从二级索引查到主键，再去聚簇索引把整行取出来，这一步就叫**回表**。

**追问 1：为什么主键建议短？**
因为 **InnoDB 会把主键列复制到每个二级索引项里**。主键越长，所有二级索引都会变胖，空间和 I/O 成本都会上升。

**追问 2：如果没有显式主键怎么办？**
InnoDB 会自己选一个合适的键；如果没有可用的唯一非空键，还会生成隐藏列。面试里最稳的说法是：**自己显式定义主键，别把选择权留给引擎。**

------

## 4. 联合索引为什么讲最左前缀原则？

因为多列索引的键值是按列顺序拼出来的，优化器可以利用它的**左侧连续前缀**。比如 `(a,b,c)`，能稳定利用 `(a)`、`(a,b)`、`(a,b,c)`，但跳过 `a` 直接用 `b,c`，通常就不能高效定位。

**追问 1：`(a,b,c)`，`where a=1 and c=3` 能用多少？**
最稳的回答是：**通常先稳定利用到 `a`**；`c` 不一定还能继续缩小扫描范围，但后续有时还能参与过滤、覆盖或优化器别的策略。

**追问 2：联合索引字段顺序怎么定？**
先看高频 SQL。一般优先考虑：**高频过滤列、经常一起出现的列、排序/分组列**；不是看哪个字段“重要”，而是看查询怎么写。

------

## 5. 什么是覆盖索引？为什么它很重要？

覆盖索引就是：**查询需要的列都能直接从同一个索引里拿到**，不需要再回表。`EXPLAIN` 常见信号是 `Using index`。它的重要性在于通常能减少一次到聚簇索引取行数据的访问。

**追问 1：`select \*` 为什么经常不利于覆盖索引？**
因为你把很多不在索引里的列也一起取了，优化器就很难只靠索引完成查询。

**追问 2：覆盖索引是不是一定比回表快？**
多数场景更优，但别说绝对。更稳的说法是：**通常更省 I/O，尤其在随机访问明显时更有价值。**

------

## 6. 常见索引“失效”场景有哪些？

高频场景包括：不满足最左前缀、前导通配符 `LIKE '%xx'`、字符集或类型不一致。

**追问 1：`like 'abc%'` 和 `like '%abc'` 有什么区别？**
前者通常还能按 B-tree 前缀走索引；后者因为开头没有确定前缀，普通 B-tree 很难高效定位。

**追问 2：列上包函数怎么办？**
稳妥答法是：**普通索引列被函数包裹时，往往要改写 SQL；如果业务长期按表达式查，可以显式建 functional index。**这是 MySQL 8.4 可支持的方向。

------

## 7. 你会怎么设计索引？主键怎么设计？

设计索引一定是**先看查询，再看字段**。如果经常按多列组合查询，就优先考虑**联合索引**；索引不是越多越好，因为它会拖慢写入。

主键则要**短、稳定、尽量少改动**。

**追问 1：为什么很多时候推荐自增整型主键？**
因为 InnoDB 按主键组织数据，自增整型一般**短、稳定、好管理**；如果没有天然合适的业务键，也可以单独造一个自增主键。

**追问 2：什么时候不建议建索引？**
**低频查询列、写特别频繁的列、表很小、重复索引、冗余索引，都不建议乱建。**官方也明确提醒：**索引提升检索，但会拖慢插入和更新。**

------

## 8. EXPLAIN 你重点看什么？（语句前加上explain）

我最先看 `type`、`key`、`rows`、`filtered`、`Extra`。其中：`type` 看访问方式，`key` 看实际用了哪个索引，`rows` 看预估扫描行数，`Extra` 看有没有 `Using index`、`Using filesort`、`Using temporary` 这些危险信号。

**追问 1：`rows` 有多重要？**
非常重要。它表示 MySQL **估计要检查多少行**。就算“走了索引”，如果 `rows` 很大，SQL 依然可能很慢。

**追问 2：`Extra` 里最该警惕什么？**
先盯 `Using filesort` 和 `Using temporary`。官方文档也明确说了，这两个值通常值得重点关注。

------

## 9. 线上慢 SQL 你怎么排查？（语句前加上explain）

先定位具体 SQL，再用 `EXPLAIN` 看执行计划，判断是**没索引、索引顺序不对、回表多、排序分组代价高还是锁等待**。

如果优化器误选了表扫，还要考虑更新统计信息，比如 `ANALYZE TABLE`，必要时再用索引提示。

**追问 1：走了索引为什么还慢？**
因为“走索引”不等于“扫描少”。可能是**扫描行数仍大、回表很多、排序没走索引、或者锁冲突严重**。

**追问 2：什么是 索引下推，为什么能优化慢 SQL？**
索引下推 是索引下推。没有 索引下推 时，引擎按索引定位后全部回表，由 Server 层过滤条件；开启 索引下推 后，索引里自带的字段条件会下推到存储引擎提前过滤，不用回表再判断，减少无效回表 IO，从而优化慢 SQL。

------

## 10. 什么是事务？ACID 分别是什么？

事务就是一组**要么都成功、要么都失败**的操作。

ACID 分别是：**原子性、 一致性、隔离性、持久性**。

InnoDB 的事务、提交/回滚、崩溃恢复和隔离机制，都是围绕这四点展开的。

**追问 1：原子性主要靠什么？**
面试里答：**靠事务和回滚机制**，底层和 undo 相关。

**追问 2：持久性主要靠什么？**
答：**提交后的修改不能因为宕机丢掉**，底层与 redo log 和 crash recovery 相关。

------

## 11. MySQL 的隔离级别有哪些？默认是什么？读已提交 和 可重复读 差别是什么？

**主答：**
四种隔离级别是：**读未提交、读已提交、可重复读、Serializable**。MySQL 8.4 默认是 **可重复读（Repeatable Read）**。面试里最常被追问的是 读已提交 和 可重复读：读已提交 每次读已提交最新版本；可重复读 在同一事务里普通一致性读通常基于同一个读视图，所以重复读更稳定。

**追问 1：读已提交 下还有 gap lock 吗？**
标准答法：**大幅减少**。在 读已提交 下，锁定读、`UPDATE`、`DELETE` 通常只锁索引记录，不锁 gap；gap locking 主要保留在外键检查和重复键检查等场景。

**追问 2：可重复读 能完全解决幻读吗？**
别答死。更稳的说法是：**普通快照读下，可重复读 通过一致性视图让结果更稳定；而对锁定读/更新删除，InnoDB 通过 next-key locking 防止幻读。**

------

## 12. 什么是 MVCC？它解决了什么问题？

MVCC 是**多版本并发控制**。核心思想是：读操作很多时候不直接和写操作抢当前版本，而是读一个对自己可见的历史版本，从而减少读写冲突，提高并发。

**追问 1： MVCC 依赖什么实现？**
你可以答：**undo log、读视图、以及 InnoDB 记录上的隐藏事务信息**。

undo 用于一致性读，InnoDB 也有隐藏列如 `DB_TRX_ID`、`DB_ROLL_PTR`。

**追问 2： MVCC 的直接收益是什么？**
很多普通 `SELECT` 能做成**一致性非锁定读**，从而减少读写互相阻塞。

------

## 13. 快照读和当前读有什么区别？

快照读读的是**某个时间点的一致性视图**；当前读读的是**最新版本**，并且会参与加锁或锁检查。`SELECT ... FOR UPDATE`、`SELECT ... FOR SHARE`、`UPDATE`、`DELETE` 都按当前读理解。

------

## 14. 行锁、表锁、间隙锁、临键锁分别是什么？为什么会有 gap lock？

行锁锁的是索引记录；表锁锁整张表；**间隙锁**锁的是索引记录之间的空隙；**临键锁（next-key lock）= 记录锁 + gap lock**。它们存在的核心原因，是在范围读写里防止别人插入“新满足条件的行”。

**追问 1：InnoDB 的“行锁”真的是锁一行吗？**
别这么说。更准确的说法是：**它本质上是基于索引项加锁**。如果没走到合适索引，影响范围会扩大。

**追问 2：意向锁是什么？**
它是表级元信息锁，用来表示“我接下来要在这张表的某些行上加什么类型的锁”，方便多粒度锁共存。

------

## 15. 死锁是怎么产生的？MySQL 怎么处理？

死锁就是多个事务互相等待对方持有的锁，形成循环等待。InnoDB 默认会做**死锁检测**，发现后主动回滚其中一个事务来打破僵局。

**追问 1：死锁和锁等待超时一样吗？**
不一样。死锁是**循环等待**；锁等待超时不一定成环，只是等太久。

**追问 2：怎么减少死锁？**
统一加锁顺序、尽量走索引缩小锁范围、缩短事务时间、拆小事务，并对死锁错误做重试。官方也明确建议：**死锁发生后应重试整个事务。**

------

## 16. redo log、undo log、binlog 分别是什么？

**主答：**
三者分工不同：**redo log** 用于崩溃恢复和持久性；**undo log** 用于回滚和一致性读；**binlog** 是 Server 层日志，记录数据库变更事件，用于复制和恢复。

**追问 1： undo log 为什么和 MVCC 有关？**
因为一致性读要能构造旧版本，update undo log 会被一致性读使用**，直到没有事务再需要它。

**追问 2： binlog 为什么不等于 redo log？**
因为它们层次和用途不同。redo 偏 InnoDB 存储引擎恢复；binlog 偏 MySQL Server 层的**复制、归档、增量恢复**。

------

## 17. 为什么事务提交要讲两阶段提交？

**主答：**
因为事务提交时既涉及 **InnoDB 数据/日志**，又涉及 **binlog**。如果只成功写了一边，宕机后就会出现“数据状态”和“复制/恢复日志状态”不一致。两阶段提交的目标就是让这两边在提交点上保持一致。

**追问 1：它和 XA 的 2PC 是一回事吗？**
思想相似，都是先 prepare 再 commit；但面试里你要强调：这里重点是 MySQL 内部 InnoDB 和 binlog 的一致性问题。

**追问 2：那配置上还要注意什么？**
二阶段提交解决流程一致性，但为了把日志真正刷到磁盘，还要关注例如 `sync_binlog=1` 这类刷盘策略。

------

## 18. MySQL 为什么能做崩溃恢复？

**主答：**
因为 InnoDB 有 redo log。数据页落盘和事务提交不是完全同步的，崩溃后可以根据 redo 把应该生效的修改重放出来；而不该留下来的未完成修改，再靠事务回滚相关机制处理。

**追问 1：为什么不是直接每次都把数据页立刻刷盘？**
因为那样随机 I/O 太重。日志先行的方式更高效，同时还能保留恢复能力。

**追问 2： redo log 能关吗？**
默认是开的，而且官方明确警告：**生产环境不要关闭 redo logging**，否则异常宕机会有数据丢失和实例损坏风险。

------

## 19. MySQL 主从复制的大致流程是什么？

**主答：**
最简流程：**源库写 binlog；副本拉取 binlog；先写到本地 relay log；再由 applier 线程回放到副本数据。**副本按自己的节奏追主库，所以复制天然可能有延迟。

**追问 1：为什么会有主从延迟？**
因为副本是**拉取后再异步回放**，它可以按自己的节奏处理，不必和源库完全同步。

**追问 2：那读写分离有什么风险？**
最常见就是：**刚写主库，立刻读副本，可能读不到最新数据。**本质就是复制延迟。

------

## 22. 并发事务下会出现哪些问题？

常见并发异常包括：**脏读、不可重复读、幻读、丢失更新**。
脏读是读到了别人还没提交的数据；不可重复读是同一事务中两次读取同一行，结果不一样；幻读是同样条件两次查询，结果集的行数变了；丢失更新是两个事务同时修改同一条数据，后提交的把前一个结果覆盖了。
MySQL / InnoDB 主要通过**隔离级别、多版本并发控制、锁机制、临键锁**来处理这些问题。

**追问 1：丢失更新？**
可以说：它虽然不是最常单独列出来的那三个“读异常”，但在工程里非常重要。
处理方式一般有两类：
一种是**悲观锁**，比如 `select ... for update`；
另一种是**乐观锁**，比如版本号、条件更新。
前者适合冲突比较明显的场景，后者适合读多写少场景。

**追问 2：幻读为什么和不可重复读不一样？**
不可重复读强调的是：**同一行的数据值变了**。
幻读强调的是：**满足条件的行集合变了**，通常是别的事务插入了新行。
比如你第一次查有 10 条，第二次查变成 11 条，这更像幻读，而不是单行值变化。

------

## 23. 四种隔离级别是怎么实现的？ACID 里一致性怎么说？

更稳的说法是：InnoDB 通过**不同的锁策略**和**一致性读策略**来实现不同的隔离级别。
**读未提交**最宽松；**读已提交**下，每次一致性读都会拿一个新的快照；**可重复读**下，同一事务中的普通一致性读通常会复用第一次读建立的快照；对于锁定读、更新、删除这类操作，InnoDB 还会结合**间隙锁、临键锁**来控制并发；**串行化**则更严格，相当于强制事务按串行方式执行。

**追问 1：读已提交和可重复读最关键的区别是什么？**
在**读已提交**里，同一事务中的每次普通读取，看到的都是当时最新已提交的数据版本。
在**可重复读**里，事务第一次普通读取建立好快照后，后续普通读取通常都基于同一个快照。
所以可重复读里，同一事务内多次读同一条数据，结果更稳定。

**追问 2： ACID 里的“一致性”怎么答？**
不要答成“只靠某一个日志保证”。
更稳的说法是：**一致性是指事务执行前后，数据都满足约束和业务规则。**
数据库里的事务、锁、约束、恢复机制提供基础保证；表结构约束和业务代码逻辑也共同参与保证一致性。
所以一致性不是单靠数据库某一个功能点就能完成的。

------

## 25. 关系型数据库和非关系型数据库？

最稳的答法不是说谁更高级，而是说：**两者数据模型不同，擅长场景不同。**
关系型数据库强调**表结构、约束、连接查询、事务**，更适合强约束、复杂查询、强一致业务；
非关系型数据库强调**模型灵活、扩展方便、按特定访问模式优化**，比如文档、键值、宽列、图。
所以不是谁一定更强，而是要看具体业务需求。

------

## 26. 分表怎么讲？分表、分区、分库分表有什么区别？

面试里一定要分清这三个概念：
**分区** 是 MySQL 原生能力；
**分表 / 垂直拆表** 是应用层表设计；
**分库分表** 是更大规模下的水平拆分方案。
MySQL 原生分区本质上讲的是**水平分区**，也就是把不同的行放到不同分区里。
而你原文里说的垂直切分，是工程上的拆表思路，不是 MySQL 原生分区。

**追问 1：水平切分和垂直切分分别是什么？**
**水平切分**是按**行**拆。
比如按用户 ID 取模、按时间范围拆分，把结构相同的数据分到多个表甚至多个库中。
**垂直切分**是按**列**拆。
比如把大字段、不常用字段单独拆出去，或者把高频访问列和低频访问列拆开。
一个是拆行，一个是拆列，这个一定要区分清楚。

**追问 2：什么时候考虑分库分表？**
当单表数据量太大、单机写入瓶颈明显、热点严重、索引膨胀、备份恢复窗口过长时，才考虑。
不是一开始就上分库分表。
因为分库分表会引入很多复杂度，比如：
**跨分片查询、跨分片事务、主键生成、扩容迁移、数据路由**。
所以一般先靠索引优化、冷热分离、归档、读写分离去顶，实在不够再上分库分表。

1. 为什么用<font style="background-color:yellow;">B+树</font>，不用<font style="background-color:yellow;">B树</font>或者<font style="background-color:yellow;">AVL</font>（优点）/<font style="background-color:yellow;">哈希表</font>

IO次数少，B+树中间节点只存放索引，数据都在叶子节点中，中间节点可以存放更多索引项，索引树更加矮胖

范围查询效率高，B树需要遍历整个树，B+树只需要遍历叶子节点中的链表

查询效率更稳定，每次查询从根节点到叶节点路径长度相同

有大量的冗余节点，插入删除效率更高

哈希索引只支持精确查找，不支持部分和范围查找，无法用于排序和分组，并且遇到大量哈希值相等的情况后查找效率会降低

2. <font style="background-color:yellow;">聚簇</font>和<font style="background-color:yellow;">非聚簇</font>索引的区别

数据的物理排列顺序和索引排列顺序一致，优点是查询速度快。非聚簇索引只记录逻辑顺序，并不改变物理顺序

6. 什么是<font style="background-color:yellow;">事务</font>

是一个操作序列，不可分割的工作单位，以begin transaction开始，以rollback/commit结束

7. Mysql 的<font style="background-color:yellow;">隔离级别</font>，如何实现四个隔离级别的

未提交读：导致脏读，不可重复读，幻读

提交读：只能看见已经提交事务所做的改变，避免脏读

可重复读，避免不可重复读

可串行化：强制事务串行执行，使之不可能相互冲突，解决幻读问题。

8. <font style="background-color:yellow;">并发事务下</font>会发生什么<font style="background-color:yellow;">问题</font>

丢失修改，脏读，不可重复读（一个事务内多次读同一个数据前后不一样），幻读（当同一查询多次执行时，由于其它事务在这个数据范围内执行了插入操作，会导致每次返回不同的结果集）

9. 数据库事务的<font style="background-color:yellow;">原子性</font>如何保证，<font style="background-color:yellow;">持久性</font>和<font style="background-color:yellow;">隔离性</font>呢（<font style="background-color:yellow;">ACID还有个一致性</font>）

用回滚日志实现，反向执行日志中的操作

重做日志

MVCC（多版本并发控制）或锁机制

10. <font style="background-color:yellow;">快照读</font>和<font style="background-color:yellow;">当前读</font>

快照读：在事务开始时确定一个一致的时间点（即快照），然后从该时间点读取数据。是事务开始时的一个静态快照，事务过程中其他事务的修改不会对其产生影响

当前读：在读取数据时，直接读取当前数据的最新版本

12. <font style="background-color:yellow;">关系型数据库</font>和<font style="background-color:yellow;">非关系型数据库</font>的区别

前者高度组织化结构化数据；后者存储的数据结构不固定更加灵活，可以减少一些空间和时间的开销后者更加容易水平扩展前者支持结构化查询语言，支持复杂的查询功能和表关联。后者只能进行简单的查询前者支持事务，具有ACID特性。后者则是BASE，最终一致性

14. <font style="background-color:yellow;">三种日志</font>

<font style="background-color:yellow;">redo log</font>（重做日志）：记录事务提交后的数据修改信息，以便在数据库重启后重做这些操作，保证事务的持久性。

<font style="background-color:yellow;">undo log</font>（回滚日志）：记录数据修改前的状态，当事务回滚时，通过读取 Undo Log 中的记录，撤销事务对数据的修改，保证事务原子性。

<font style="background-color:yellow;">binlog</font>（归档日志）：主要记录用户对数据库操作的sql语句（除了查询语句），用于主从同步和基于时间点的还原，与 Redo log 不同，Binlog 是按顺序记录的，并且在服务器重启后不会自动清除。

16. <font style="background-color:yellow;">分表</font>：

水平切分：将同一个表中的记录拆分到多个结构相同的表中（策略：哈希取模；根据ID范围来分）。当一个表的数据不断增多时，Sharding 是必然的选择，它可以将数据分布到集群的不同节点上，从而缓解单个数据库的压力；垂直切分：将一张表按列切分成多个表。可以将不常用的字段单独放在同一个表中；把大字段独立放入一个表中；或者把经常使用的字段（关系密切的）放在一张表中。垂直切分之后业务更加清晰，系统之间整合或扩展容易，数据维护简单

17. 什么是<font style="background-color:yellow;">MVCC</font>：

多版本并发控制，MVCC在每行记录后面都保存有两个隐藏的列，用来存储创建版本号和删除版本号。当事务要访问数据时，它会看到该数据的一个一致的快照，这个快照是在事务开始之前最新的版本。这样，读操作不会被写操作阻塞。

18. <font style="background-color:yellow;">mysql</font>的<font style="background-color:yellow;">server层</font>

server层负责建立连接、分析和执行sql。主要包括连接器，查询缓存，解析器，预处理器，优化器，执行器等。
