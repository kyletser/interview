## 1. InnoDB 和 MyISAM 有什么区别？为什么现在默认讲 InnoDB？

**主答：**
最核心的区别是：**InnoDB 支持事务、行级锁、MVCC 和崩溃恢复**，

而 MyISAM 不支持事务，锁粒度也更粗。

现代业务更看重并发控制和数据可靠性，所以面试里默认重点讲 InnoDB。

**追问 1：InnoDB 为什么更适合高并发 ？**
因为它有**行级锁**和**一致性非锁定读**，多个会话可以同时读写同一张表，而不是像粗粒度锁那样更容易互相阻塞。

- **一致性锁定读**：就是 `SELECT ... FOR UPDATE` 或 `SELECT ... LOCK IN SHARE MODE`，会加行锁，读的是最新已提交数据，可能阻塞其他事务。

- **当前读**：`UPDATE`、`DELETE`、`SELECT ... FOR UPDATE` 都属当前读，必须读最新的已提交版本，并加锁。

  **一致性锁定读**：特指**查询语句**主动加锁

  **当前读**：指所有**必须读最新数据并加锁**的 SQL

- **一致性非锁定读**：读取数据时，不加锁，直接通过快照（Snapshot）读取某个历史版本的数据，从而保证读到的是一个事务开始时的“一致性”视图，且不会阻塞其他事务的写操作。

------

## 2. 为什么 MySQL 常见索引是 B+ 树，而不是红黑树或 Hash？

**主答：**
因为 InnoDB 普通索引本质是 **B-tree 家族结构**，它天然支持**等值查找、范围查找、排序扫描**；

而 Hash 索引只擅长等值匹配，不适合范围查询。

至于红黑树，它是二叉树，放到数据库页 I/O 场景里层高更高，不如高扇出的 B-tree 适合磁盘页访问。

**追问 1：Hash 为什么不适合数据库主索引？**
因为 Hash 只能高效处理等值比较，**不能高效支持范围查询`**。

**追问 2：B-tree 在 MySQL 里能做什么？**
它可以快速处理 `=`、`>、<、BETWEEN、IN` 等条件；而且按索引顺序扫描时，对排序和范围查询都很友好。

## 3. 什么是聚簇索引、二级索引、回表？

- **聚簇索引**：数据本身按索引顺序存放，叶子节点存的是**完整行数据**。
- **二级索引**：叶子节点存的是**聚簇索引的键值**，不是完整行。
- **回表**：通过二级索引查到聚簇索引键后，再拿这个键去聚簇索引里**查一遍**拿完整行。

---

### 1. 聚簇索引

InnoDB 中，表数据本身就是按一棵 B+ 树组织的，这棵树就是聚簇索引。

- **叶子节点** = 整行数据（所有列都在这）
- 表必须有且只有一个聚簇索引
- 有主键，主键就是聚簇索引；没主键，用第一个唯一非空索引；都没有，InnoDB 自动生成隐藏的 `row_id`

---

### 2. 二级索引

你手动创建的普通索引、唯一索引，**只要不是聚簇索引，都是二级索引。**

- **叶子节点** = 索引列的值 + **主键值**
- 不存完整行数据，**只存主键引用**

---

### 3. 回表（回表查询）

当二级索引不够用时，拿着主键回聚簇索引再查一遍的过程。

1. 在 `name` 二级索引找到 `('张三', id=5)`
2. 拿 `id=5` 去聚簇索引查整行，拿到 `id, name, age`
3. 这个第二步就是**回表**，多一次 B+ 树查找

---

### 回表多了怎么办？覆盖索引

如果查询的列**全在二级索引里**，就不用回表，这叫**覆盖索引**：

`Extra: Using index` 就表示覆盖索引，性能好很多。

### 二级索引叶子为什么存主键而不是数据指针？

因为数据可能因页分裂而移动物理位置，指针会失效。

**存主键值**，数据再移、页再变，主键逻辑定位不变，这是逻辑索引的重大优势。代价就是多一次回表——但这是接受的权衡。

**追问 1：为什么主键建议短？**
因为 **InnoDB 会把主键列复制到每个二级索引项里**。主键越长，所有二级索引都会变胖，空间和 I/O 成本都会上升。

**追问 2：如果没有显式主键怎么办？**
InnoDB 会自己选一个合适的键；如果没有可用的唯一非空键，还会生成隐藏列。面试里最稳的说法是：**自己显式定义主键，别把选择权留给引擎。**

------

## 4. 联合索引为什么讲最左前缀原则？

联合索引的 B+ 树是多级排序结构，只有从最左列开始、不跳过中间列的查询条件，才能利用这个排序做快速定位。

跳过任何一列，后面列的排序就失去全局有序性，索引退化为全索引扫描。

比如 `(a,b,c)`，能稳定利用 `(a)`、`(a,b)`、`(a,b,c)`，但跳过 `a` 直接用 `b,c`，通常就不能高效定位。

**追问 1：`(a,b,c)`，`where a=1 and c=3` 能用多少？**
最稳的回答是：**通常先稳定利用到 `a`**；`c` 不一定还能继续缩小扫描范围，但后续有时还能参与过滤、覆盖或优化器别的策略。

**追问 2：联合索引字段顺序怎么定？**
先看高频 SQL。一般优先考虑：**高频过滤列、经常一起出现的列、排序/分组列**；不是看哪个字段“重要”，而是看查询怎么写。

------

## 5. 什么是覆盖索引？为什么它很重要？

覆盖索引就是：**查询需要的列都能直接从同一个索引里拿到**，不需要再回表。

`EXPLAIN` 常见信号是 `Using index`。它的重要性在于通常能减少一次到聚簇索引取行数据的访问。

**追问 1：`select \*` 为什么经常不利于覆盖索引？**
因为你把很多不在索引里的列也一起取了，优化器就很难只靠索引完成查询。

**追问 2：覆盖索引是不是一定比回表快？**
更稳的说法是：**通常更省 I/O，尤其在随机访问明显时更有价值。**

------

## 6. 常见索引“失效”场景有哪些？

**查询条件无法利用 B+ 树的有序结构**。比如违反最左前缀、索引列上做函数或运算、隐式类型转换、LIKE 百分号在前、不等于操作等。

**① 违反最左前缀原则**
联合索引 `(a, b, c)`，查询跳过 `a` 或中间断列，后续列无法利用索引排序。MySQL 8.0 的索引跳跃扫描只是有限缓解，条件苛刻。

**② 对索引列做函数或运算**
索引存的是原始值，函数/运算后的值与索引无法匹配。

- `WHERE YEAR(create_time) = 2024` → 改成范围查询
- `WHERE age + 1 = 20` → 改成 `WHERE age = 19`
- **隐式类型转换**是常见陷阱：`WHERE phone = 13800000000`，phone 是 VARCHAR，数据库会做隐式转换，等同于在索引列上用了函数。

**③ LIKE 以 `%` 开头**
`LIKE '%xxx'` 前缀未知，无法定位区间；`LIKE 'xxx%'` 可以走索引。

**④ 范围查询右边的列失效**
联合索引中，第一个范围查询后面的列不再保证有序，索引用到范围列就停。
`WHERE a = 1 AND b > 10 AND c = 2`，`c` 无法走索引。

**⑤ `!=`、`NOT IN` 等不等于操作**
排除式条件无法圈定连续区间，通常是全表扫描。但如果排除后剩的数据很少，优化器可能选择走索引（极少见）。

**追问 1：`like 'abc%'` 和 `like '%abc'` 有什么区别？**
前者通常还能按 B-tree 前缀走索引；后者因为开头没有确定前缀，普通 B-tree 很难高效定位。

------

## 7. 你会怎么设计索引？主键怎么设计？

主键优先选有序且短的，自增 BIGINT 是大多数场景的最优解。

索引从慢查询出发，等值在前、范围在后，尽量覆盖来避免回表，利用最左前缀让一个索引服务多个查询，同时在写入频率高的表上控制索引数量。

**追问 1：为什么很多时候推荐自增整型主键？**
因为 InnoDB 按主键组织数据，自增整型一般**短、稳定、好管理**；如果没有天然合适的业务键，也可以单独造一个自增主键。

**追问 2：什么时候不建议建索引？**
**低频查询列、写特别频繁的列、表很小、重复索引、冗余索引，都不建议乱建。**

## 8. EXPLAIN 你重点看什么？（语句前加上explain）

我最先看 `type`、`key`、`rows`、`filtered`、`Extra`。其中：`type` 看访问方式，`key` 看实际用了哪个索引，`rows` 看预估扫描行数，`Extra` 看有没有 `Using index`、`Using filesort`、`Using temporary` 这些危险信号。

**追问 1：`rows` 有多重要？**
非常重要。它表示 MySQL **估计要检查多少行**。就算“走了索引”，如果 `rows` 很大，SQL 依然可能很慢。

**追问 2：`Extra` 里最该警惕什么？**
先盯 `Using filesort` 和 `Using temporary`。官方文档也明确说了，这两个值通常值得重点关注。

------

## 9. 线上慢 SQL 你怎么排查？（语句前加上explain）

通过慢查询日志，先定位具体 SQL，再用 `EXPLAIN` 看执行计划，判断是**没索引、索引顺序不对、回表多、排序分组代价高还是锁等待**。

一是看 **type**，全表扫（ALL）或全索引扫（index）就需要重点检查。

二是看 **key**，确认实际用了哪个索引，和预想是否一致。

三是看 **rows**，估算扫描行数，能判断索引过滤效果好不好。

四是看 **Extra**，特别留意 **Using filesort** 和 **Using temporary**，说明排序或分组没利用上索引。其他像 `Using index` 是好事，代表覆盖索引不回表。”

**追问 1：走了索引为什么还慢？**
因为“走索引”不等于“扫描少”。可能是**扫描行数仍大、回表很多、排序没走索引、或者锁冲突严重**。

**追问 2：什么是 索引下推，为什么能优化慢 SQL？**
索引下推 是索引下推。没有 索引下推 时，引擎按索引定位后全部回表，由 Server 层过滤条件；开启 索引下推 后，索引里自带的字段条件会下推到存储引擎提前过滤，不用回表再判断，减少无效回表 IO，从而优化慢 SQL。

## 10.事务

---

## 一、原子性（Atomicity）

原子性的核心是**不可分割**。事务是数据库状态迁移的最小逻辑单元，它包含的所有操作对外必须表现为一个整体。

### 实现机制：undo log（回滚日志）

InnoDB 在修改任何数据之前，**先把旧值记录到 undo log**。这个过程是：

1. 事务想改某行，先在 undo log 里记下"怎么恢复回去"
2. 然后才真正修改数据页
3. 如果事务失败或被主动回滚，就顺着 undo log 把修改过的行逐一恢复原状

### 关键点

- **undo log 保证的是"回头路"**：记的是怎么撤销，不是正向操作的日志
- **undo log 本身也有 redo log 保护**：写入 undo log 的操作也会记 redo log，所以崩溃不会丢回滚信息
- **MVCC 也依赖 undo log**：旧版本数据通过 undo log 链条串联，实现一致性读

---

## 二、一致性（Consistency）

一致性不是数据库靠某个单一技术实现的，它是**业务层面的约束**，是事务追求的**最终目标**。AID 三种机制共同保障它。

**1. 数据库层面的约束**
- 主键唯一 → 不能插入重复主键
- 外键约束 → 不能引用不存在的行
- NOT NULL → 不能插入空值
- CHECK 约束 → 必须满足自定义规则

违反任何一条，事务直接失败回滚。

**2. 业务层面的逻辑**
- 转账前后，A + B 总金额不变
- 库存不能为负
- 订单状态流转必须合法

这层数据库不知道，但 **AID 机制保证了：事务要么完整执行（业务规则满足的结果对外可见），要么完全撤销（回到执行前的合法状态）。**中间态永远不会暴露出去。

```
原子性 + 隔离性 + 持久性 → 一致性
        ↑                       ↑
    "怎么保证"              "保证什么"
```

- **原子性**：保证不会出现"扣了钱没加钱"，即不出现非法的中间状态
- **隔离性**：保证并发场景下，一个事务不会读到另一个事务的中间态，否则基于中间态做的业务判断可能违反规则
- **持久性**：保证已经达成一致的状态不会因崩溃而丢失

**一致性是用户定义的数据约束和业务规则，原子性、隔离性、持久性则是数据库为保证这些规则不被破坏而提供的机制。**

---

## 三、隔离性（Isolation）

| 问题           | 说明                                 | 例子                                                         |
| -------------- | ------------------------------------ | ------------------------------------------------------------ |
| **脏读**       | 读到别的事务未提交的数据             | A 转钱但未提交，B 读到了扣后金额，A 随后回滚                 |
| **不可重复读** | 同一事务内多次读同一行结果不同       | 事务内前后两次 `SELECT * WHERE id=1`，中间被别人改了且提交了 |
| **幻读**       | 同一事务内多次查，符合条件的行数不同 | 前后两次查 `WHERE age > 18`，中间被人插了一条新数据          |

### 四级隔离级别

| 级别         | 脏读 | 不可重复读 | 幻读          | 实现方式            |
| ------------ | ---- | ---------- | ------------- | ------------------- |
| **读未提交** | ✗    | ✗          | ✗             | 无 MVCC，读最新数据 |
| **读已提交** | ✓    | ✗          | ✗             | 每次读生成新快照    |
| **可重复读** | ✓    | ✓          | ✗（基本解决） | 事务内用同一个快照  |
| **串行化**   | ✓    | ✓          | ✓             | 读写加锁，完全串行  |

### 实现机制：MVCC + 锁

**MVCC（多版本并发控制）**：
- 每行数据有隐藏字段 `trx_id`（最近修改的事务 ID）和 `roll_pointer`（指向 undo log 旧版本）
- 读操作根据事务开始时的快照，顺着版本链找到"那个时间点应该看到的数据"
- 这就是一致性非锁定读——读不加锁，通过版本控制隔离

**锁**：
- 写操作之间需要互斥（两个事务不能同时改同一行）
- 当前读（`SELECT FOR UPDATE`、`UPDATE`、`DELETE`）需要加锁读最新版本
- 间隙锁（Gap Lock）解决幻读：在 RR 级别下，不仅锁住已有的行，还锁住行之间的间隙，阻止插入

### InnoDB 默认 RR（可重复读） 就基本解决了幻读

这得益于**间隙锁 + 快照读**的组合：

- 快照读（普通 SELECT）：MVCC 保证读到的是事务开始时的快照，新插入的行有更新的 `trx_id`，在快照下不可见
- 当前读（`SELECT FOR UPDATE`）：间隙锁阻止了其他事务在查询的范围内插入新行

## 四、持久性（Durability）

事务一旦提交，结果就不能丢失，即使数据库在提交后立即崩溃，重启后数据依然在。

### 实现机制：redo log（重做日志）

InnoDB 的数据先在**缓冲池（内存）**里修改，不是立刻写盘。如果每次修改都直接写磁盘（随机 I/O），性能会非常差。但这样，如果缓冲池的脏页还没来得及刷盘就崩溃了，已提交的事务就会丢数据。

**redo log 怎么解决？**

- **WAL（Write-Ahead Log，预写日志）**：修改数据前，先把"我要做什么"记到 redo log
- **redo log 是顺序写**：文件是循环使用的固定大小，写起来非常快
- **事务提交时强制刷盘**：`COMMIT` 时 redo log 必须持久化到磁盘（`innodb_flush_log_at_trx_commit = 1`）
- **崩溃恢复**：重启时，比对数据页和 redo log，把已提交但未写盘的操作"重放"一遍

```
事务执行 UPDATE SET money = 400：

① 修改缓冲池中 id=1 的数据页，money 改为 400
② 生成一条 redo log：把 id=1 的 money 改为 400
③ COMMIT 时，redo log 刷盘
④ 脏页可能过一段时间才被刷到磁盘

崩溃发生：
→ 重启时检测 redo log，发现已提交但数据页还是旧值的记录
→ 重放 redo log，把数据页恢复成 400
→ 持久性得到保证
```

### 和 undo log 的关系

|          | undo log                 | redo log                 |
| -------- | ------------------------ | ------------------------ |
| 记录内容 | 修改前的旧值（怎么撤销） | 修改后的新值（怎么重做） |
| 用途     | 回滚 + MVCC              | 崩溃恢复                 |
| 保护什么 | 原子性（失败可撤销）     | 持久性（提交不丢失）     |

| 特性       | 核心问题             | 实现机制                             | 记忆点                 |
| ---------- | -------------------- | ------------------------------------ | ---------------------- |
| **原子性** | 事务失败怎么办？     | undo log 回滚                        | 要么全做，要么全撤     |
| **一致性** | 数据规则怎么不破坏？ | AID + 约束共同保障                   | 事务是目标，机制是手段 |
| **隔离性** | 并发事务怎么不互扰？ | MVCC（快照读）+ 锁（当前读）+ 间隙锁 | 四级隔离，默认 RR      |
| **持久性** | 提交后崩溃怎么办？   | redo log + WAL                       | 顺序写日志，崩溃后重放 |

------

这三个日志是 MySQL 最重要的三个日志。面试时先一句话讲清各自定位，再展开机制。

---

## 一句话区分三大日志。

| 日志         | 一句话                  | 保障什么            |
| ------------ | ----------------------- | ------------------- |
| **undo log** | 记录**怎么撤销**        | 原子性 + MVCC       |
| **redo log** | 记录**怎么重放**        | 持久性              |
| **binlog**   | 记录**执行的 SQL 逻辑** | 主从复制 + 数据恢复 |

---

## undo log（回滚日志）

1. **事务失败时回滚，把数据恢复到修改前的样子**
2. **MVCC 读快照时，通过 undo log 链条找到历史版本**

```
UPDATE SET money = 400 WHERE id = 1;

① 写 undo log：money 原值是 500，记下来
② 修改缓冲池数据页：money 改为 400
③ 成功就提交，undo log 未来被清理
  失败就回滚，用 undo log 把 money 改回 500
```

**关键**：事务回滚靠它，一致性非锁定读也靠它（读旧版本）。

---

## redo log（重做日志）

**作用**：崩溃恢复。事务提交了但数据页还没写盘，重启后用 redo log "重放"一遍，保证不丢数据。

```
① 修改缓冲池数据页：money 改为 400
② 写 redo log：记录"把 id=1 的 money 改为 400"
③ COMMIT 时，redo log 刷盘（顺序写，很快）
④ 脏页后续某个时刻刷回磁盘

崩溃发生：
重启 → 检查 redo log → 发现已提交但未写盘的数据 → 重放
```

**关键**：WAL（Write-Ahead Log）机制，先写日志再改数据页，日志是顺序写所以快，用它弥补内存易失和磁盘随机写慢的问题。

---

## binlog（归档日志 / 二进制日志）

1. **主从复制**：从库通过读主库 binlog 同步数据变更
2. **数据恢复**：备份 + binlog 可以恢复到某个时间点

```
① 事务执行、修改数据
② 事务提交时，生成 binlog 记录
③ binlog 刷盘
④ 从库的 I/O 线程拉取 binlog → 从库 SQL 线程回放
```

**关键**：**binlog 是 Server 层的，所有引擎通用；undo/redo 是 InnoDB 引擎层的。**

|              | undo log         | redo log           | binlog                |
| ------------ | ---------------- | ------------------ | --------------------- |
| **层面**     | InnoDB 引擎层    | InnoDB 引擎层      | Server 层             |
| **记录内容** | 旧值（怎么撤销） | 新值（怎么重做）   | SQL 语句或行变更      |
| **用途**     | 回滚 + MVCC      | 崩溃恢复           | 主从复制 + 时间点恢复 |
| **写方式**   | 随机写           | 循环写（固定大小） | 追加写（独立文件）    |
| **保障**     | 原子性           | 持久性             | 数据可复制、可恢复    |

---

## 两阶段提交

redo log 和 binlog 都要在事务提交时写盘，怎么保证两者一致？用**两阶段提交**：

```
① Prepare 阶段：写 redo log，标记事务为 prepare 状态
② Commit 阶段：写 binlog
③ 将 redo log 中的事务标记为 commit 状态
```

这样崩溃恢复时：
- 如果 redo log 是 prepare、binlog 没写完 → 回滚
- 如果 redo log 是 prepare、binlog 写完了 → 提交

保证了两者的一致性，主从数据不会出现偏差。

> undo log 记录旧值，用来回滚和 MVCC，保障原子性。
>
> redo log 记录新值，用来崩溃恢复，保障持久性，它用循环写的方式实现了 WAL 机制。
>
> binlog 是 Server 层的归档日志，记录 SQL 逻辑，用来主从复制和时间点恢复。redo log 和 binlog 通过两阶段提交保证一致。



## 14. 行锁、表锁、间隙锁、临键锁分别是什么？

行锁锁的是索引中的具体记录，表锁锁整张表。

间隙锁锁的是索引记录之间的区间，阻止别的插入，是用来解决幻读的。

临键锁是行锁和间隙锁的组合，是 InnoDB 在 RR 级别下的默认加锁方式，既保护已有行不被修改，也防止间隙里插入新行。

RC 级别下没有间隙锁，所以有幻读，但并发性能更高。

**追问 1：InnoDB 的“行锁”真的是锁一行吗？**
别这么说。更准确的说法是：**它本质上是基于索引项加锁**。如果没走到合适索引，影响范围会扩大。

**追问 2：意向锁是什么？**
它是表级元信息锁，用来表示“我接下来要在这张表的某些行上加什么类型的锁”，方便多粒度锁共存。

## 19. MySQL 主从复制的大致流程是什么？

主库事务提交时写 binlog，从库的 I/O 线程主动连接主库拉取 binlog，主库 Dump 线程推送给从库。

从库 I/O 线程收到后先写入本地的 relay log，再由 SQL 线程从 relay log 读取事件，逐条在从库上回放。

因为是异步的，主库不等待从库确认，所以天然可能有延迟。延迟的大小取决于主库写入量和从库回放速度的差距。并行复制通过多线程回放能显著降低延迟。

**追问 1：为什么会有主从延迟？**
因为副本是**拉取后再异步回放**，它可以按自己的节奏处理，不必和源库完全同步。

**追问 2：那读写分离有什么风险？**
最常见就是：**刚写主库，立刻读副本，可能读不到最新数据。**本质就是复制延迟。

## 22. 并发事务下会出现哪些问题？

**脏读是读到了别人还没提交的数据；**

**不可重复读是同一事务中两次读取同一行，结果不一样；**

**幻读是同样条件两次查询，结果集的行数变了；**

**丢失更新是两个事务同时修改同一条数据，后提交的把前一个结果覆盖了。**
MySQL / InnoDB 主要通过**隔离级别、多版本并发控制、锁机制、临键锁**来处理这些问题。

**追问 1：丢失更新？**
可以说：它虽然不是最常单独列出来的那三个“读异常”，但在工程里非常重要。
处理方式一般有两类：
一种是**悲观锁**，比如 `select ... for update`；
另一种是**乐观锁**，比如版本号、条件更新。
前者适合冲突比较明显的场景，后者适合读多写少场景。

**追问 2：幻读为什么和不可重复读不一样？**
不可重复读强调的是：**同一行的数据值变了**。
幻读强调的是：**满足条件的行集合变了**，通常是别的事务插入了新行。
比如你第一次查有 10 条，第二次查变成 11 条，这更像幻读，而不是单行值变化。

## 26. 分表怎么讲？分表、分区、分库分表有什么区别？

分区是数据库原生能力，逻辑上是一张表，物理上分文件，对应用透明。

分表是在同一个库里把大表拆成多张小表，需要应用层做路由。

分库分表是把表和库都拆到不同机器上，解决单机性能瓶颈，但会引入分布式事务、跨库查询等复杂问题。"

|                | 分区           | 分表           | 分库分表                |
| :------------- | :------------- | :------------- | ----------------------- |
| **粒度**       | 同一张表       | 同一库，多张表 | 多个库，多台机器        |
| **代码改动**   | 不需要         | 需要改路由逻辑 | 需要改路由 + 分布式组件 |
| **解决的问题** | 单表数据多     | 单表行数过多   | 单机扛不住了            |
| **跨分片查询** | 数据库自动处理 | 应用层拼接     | 中间件或应用层拼接      |
| **复杂度**     | 低             | 中             | 高                      |

**追问 1：水平切分和垂直切分分别是什么？**
**水平切分**是按**行**拆。
比如按用户 ID 取模、按时间范围拆分，把结构相同的数据分到多个表甚至多个库中。
**垂直切分**是按**列**拆。
比如把大字段、不常用字段单独拆出去，或者把高频访问列和低频访问列拆开。
一个是拆行，一个是拆列，这个一定要区分清楚。

**追问 2：什么时候考虑分库分表？**
当单表数据量太大、单机写入瓶颈明显、热点严重、索引膨胀、备份恢复窗口过长时，才考虑。
不是一开始就上分库分表。
因为分库分表会引入很多复杂度，比如：
**跨分片查询、跨分片事务、主键生成、扩容迁移、数据路由**。
所以一般先靠索引优化、冷热分离、归档、读写分离去顶，实在不够再上分库分表。



1. 为什么用<font style="background-color:yellow;">B+树</font>，不用<font style="background-color:yellow;">B树</font>或者<font style="background-color:yellow;">AVL</font>（优点）/<font style="background-color:yellow;">哈希表</font>

IO次数少，B+树中间节点只存放索引，数据都在叶子节点中，中间节点可以存放更多索引项，索引树更加矮胖

范围查询效率高，B树需要遍历整个树，B+树只需要遍历叶子节点中的链表

查询效率更稳定，每次查询从根节点到叶节点路径长度相同

有大量的冗余节点，插入删除效率更高

哈希索引只支持精确查找，不支持部分和范围查找，无法用于排序和分组，并且遇到大量哈希值相等的情况后查找效率会降低

2. <font style="background-color:yellow;">聚簇</font>和<font style="background-color:yellow;">非聚簇</font>索引的区别

数据的物理排列顺序和索引排列顺序一致，优点是查询速度快。非聚簇索引只记录逻辑顺序，并不改变物理顺序

6. 什么是<font style="background-color:yellow;">事务</font>

是一个操作序列，不可分割的工作单位，以begin transaction开始，以rollback/commit结束

7. Mysql 的<font style="background-color:yellow;">隔离级别</font>，如何实现四个隔离级别的

未提交读：导致脏读，不可重复读，幻读

提交读：只能看见已经提交事务所做的改变，避免脏读

可重复读，避免不可重复读

可串行化：强制事务串行执行，使之不可能相互冲突，解决幻读问题。

8. <font style="background-color:yellow;">并发事务下</font>会发生什么<font style="background-color:yellow;">问题</font>

丢失修改，脏读，不可重复读（一个事务内多次读同一个数据前后不一样），幻读（当同一查询多次执行时，由于其它事务在这个数据范围内执行了插入操作，会导致每次返回不同的结果集）

9. 数据库事务的<font style="background-color:yellow;">原子性</font>如何保证，<font style="background-color:yellow;">持久性</font>和<font style="background-color:yellow;">隔离性</font>呢（<font style="background-color:yellow;">ACID还有个一致性</font>）

用回滚日志实现，反向执行日志中的操作

重做日志

MVCC（多版本并发控制）或锁机制

10. <font style="background-color:yellow;">快照读</font>和<font style="background-color:yellow;">当前读</font>

快照读：在事务开始时确定一个一致的时间点（即快照），然后从该时间点读取数据。是事务开始时的一个静态快照，事务过程中其他事务的修改不会对其产生影响

当前读：在读取数据时，直接读取当前数据的最新版本

12. <font style="background-color:yellow;">关系型数据库</font>和<font style="background-color:yellow;">非关系型数据库</font>的区别

前者高度组织化结构化数据；后者存储的数据结构不固定更加灵活，可以减少一些空间和时间的开销后者更加容易水平扩展前者支持结构化查询语言，支持复杂的查询功能和表关联。后者只能进行简单的查询前者支持事务，具有ACID特性。后者则是BASE，最终一致性

14. <font style="background-color:yellow;">三种日志</font>

<font style="background-color:yellow;">redo log</font>（重做日志）：记录事务提交后的数据修改信息，以便在数据库重启后重做这些操作，保证事务的持久性。

<font style="background-color:yellow;">undo log</font>（回滚日志）：记录数据修改前的状态，当事务回滚时，通过读取 Undo Log 中的记录，撤销事务对数据的修改，保证事务原子性。

<font style="background-color:yellow;">binlog</font>（归档日志）：主要记录用户对数据库操作的sql语句（除了查询语句），用于主从同步和基于时间点的还原，与 Redo log 不同，Binlog 是按顺序记录的，并且在服务器重启后不会自动清除。

16. <font style="background-color:yellow;">分表</font>：

水平切分：将同一个表中的记录拆分到多个结构相同的表中（策略：哈希取模；根据ID范围来分）。当一个表的数据不断增多时，Sharding 是必然的选择，它可以将数据分布到集群的不同节点上，从而缓解单个数据库的压力；垂直切分：将一张表按列切分成多个表。可以将不常用的字段单独放在同一个表中；把大字段独立放入一个表中；或者把经常使用的字段（关系密切的）放在一张表中。垂直切分之后业务更加清晰，系统之间整合或扩展容易，数据维护简单

17. 什么是<font style="background-color:yellow;">MVCC</font>：

多版本并发控制，MVCC在每行记录后面都保存有两个隐藏的列，用来存储创建版本号和删除版本号。当事务要访问数据时，它会看到该数据的一个一致的快照，这个快照是在事务开始之前最新的版本。这样，读操作不会被写操作阻塞。

18. <font style="background-color:yellow;">mysql</font>的<font style="background-color:yellow;">server层</font>

server层负责建立连接、分析和执行sql。主要包括连接器，查询缓存，解析器，预处理器，优化器，执行器等。
