
# C++语言基础与对象模型

## new/delete 和 malloc/free 有什么区别？
**15秒简答：**  
`new/delete` 是 C++ **运算符**，除了分配/释放内存，还会调用构造/析构函数，支持类型安全和重载；

`malloc/free` 是 C **函数**，只负责裸内存管理，不会调用对象生命周期函数。

写 C++ 业务代码优先 `new/delete`，更推荐进一步用 RAII 和智能指针替代手写释放。

**3分钟详答：**  
`malloc` **做的是“给你一块指定字节数的原始内存”，返回 `void*`**，所以需要手动转换类型，也不会自动初始化对象。`free` 只是把内存还回去，不会调用析构。 
`new` 则是“**先分配内存，再在这块内存上构造对象**”，因此它和类型绑定，会调用构造函数；`delete` 会先调用析构函数，再释放内存。  



面试里最好再补三点：  

1. `new` 失败默认抛 `std::bad_alloc`，而 `malloc` 失败返回 `nullptr`。  
2. `new/delete` 可以被类重载，适合做对象池、定制分配器；`malloc/free` 不能体现这种面向对象扩展能力。  
3. 混用有风险：`malloc` 出来的内存不能用 `delete` 释放，`new` 出来的对象不能用 `free` 释放，否则是未定义行为。  

**如果资源需要生命周期管理，我尽量不直接写裸 `new/delete`，而是交给对象、容器、智能指针管理**。这就自然引到 RAII。

**可能的追问问题：**  

- `new` 失败和 `malloc` 失败的表现有什么不同？  
  **15秒简答：** `new` 默认抛异常 `std::bad_alloc`，`malloc` 返回 `nullptr`，所以错误处理方式不同。  
- 为什么不建议在现代 C++ 里到处手写 `new/delete`？  
  **15秒简答：** 因为容易泄漏、重复释放、异常路径漏释放；现代 C++ 更推荐 RAII、容器和智能指针托管资源。

## 什么是 RAII？为什么 C++ 特别强调它？
**15秒简答：**  
RAII 就是“资源获取即初始化”，**把资源绑定到对象生命周期**：对象**构造时拿资源**，对象**析构时释放资源**。这样即使函数中途返回或抛异常，也不会漏释放。锁、文件句柄、堆内存都适合这样管理。

**3分钟详答：**   
RAII 的思路是：**不要把“释放资源”写成业务逻辑的一部分，而是写进对象析构函数里**。这样资源的释放就和作用域绑定了。  

最经典例子是加锁：  
进入函数时构造 `std::lock_guard<std::mutex>`，它在构造时加锁；离开作用域时自动析构解锁。这样不管是正常 return、异常退出，还是中间分支，锁都能被释放。  
再比如智能指针，本质也是 RAII：对象离开作用域时自动回收所管理的堆内存。  

**可能的追问问题：**  

- `lock_guard` 为什么是 RAII 的典型例子？  
  **15秒简答：** 因为它在构造时加锁，析构时解锁，锁的生命周期和对象作用域绑定。  
- RAII 只能管理内存吗？  
  **15秒简答：** 不只，锁、文件、socket、数据库连接等都可以，只要资源有获取和释放动作。  
- RAII 和异常安全有什么关系？  
  **15秒简答：** 发生异常时栈展开会调用局部对象析构函数，所以 RAII 能自动完成清理。

## 深拷贝、浅拷贝，以及为什么会引出 Rule of Three/Five/Zero？
**15秒简答：** 
浅拷贝只是复制成员值，如果成员里有裸指针，就会导致多个对象指向同一块资源；

深拷贝会复制资源本身。只要类自己管理资源，就要认真处理拷贝、赋值、析构，C++11 以后还要考虑移动，所以有 Rule of Five；更推荐 Rule of Zero，尽量交给标准库成员管理资源。

**3分钟详答：** 
如果一个类里只有普通值类型成员，比如 `int`、`std::string`、`std::vector`，编译器默认生成的拷贝行为通常没问题，因为这些成员自己就会正确拷贝。  
但如果类里自己维护了裸资源，比如 `char* data` 指向堆内存，默认拷贝就只是把这个地址复制一份，这就是浅拷贝。结果两个对象会指向同一块内存：一边修改会互相影响，更严重的是析构时可能 double free。  

所以只要**类自己管理资源，通常就要显式定义：**  
- 析构函数  
- 拷贝构造  
- 拷贝赋值  
这叫 Rule of Three。  

到 C++11 以后，为了支持移动语义，通常还要考虑：  
- 移动构造  
- 移动赋值  
这就是 Rule of Five。  

但真正更现代的做法是 Rule of Zero：**尽量不要自己管理裸资源，而是把资源交给 `std::string`、`std::vector`、`std::unique_ptr` 这类已经实现好语义的成员。**这样很多特殊成员函数都可以交给编译器默认生成。  

- 能不用裸指针就不用。  
- 真的需要自己管理资源时，先想清楚对象是否可拷贝、是否可移动。  
- 如果语义上不允许拷贝，就 `= delete`。  

**可能的追问问题：**  
- 什么情况下会发生 double free？  
  **15秒简答：** 两个对象因浅拷贝共享同一块堆内存，析构时都去释放它，就会重复释放。  
- Rule of Five 比 Rule of Three 多了什么？  
  **15秒简答：** 多了移动构造和移动赋值，因为 C++11 引入了移动语义。  
- 为什么很多时候更推崇 Rule of Zero？  
  **15秒简答：** 因为把资源管理交给标准库成员更安全，能少写特殊成员函数，也更不容易出错。

## 指针和引用有什么区别？什么时候更推荐引用？
**15秒简答：** 
引用是对象别名，定义时必须初始化，语义上通常不表示“空”；

指针本身是独立对象，可以改指向、可以为空。参数传递里，若函数要求“必须有对象”，优先用引用；若要表达“可空”“可重新绑定”或多态间接访问，才更常用指针。

**3分钟详答：**  

- 引用更像“这个参数一定存在，而且我就是在操作原对象”。  
- 指针更像“我手上拿的是一个地址，它可能为空，也可能 later 改成指向别的对象”。  

从语言层面看：  
1. 引用定义时必须绑定到对象，之后不能改绑。  
2. 指针可以先不指向任何对象，也可以多次修改指向。  
3. 使用时引用语法更自然，不需要显式解引用。  
4. 指针可以做地址运算，引用不行。  

**引用适合表达必然存在的别名关系，指针适合表达可空、可变绑定、间接寻址和所有权封装的底层能力。**

**可能的追问问题：**  
- 为什么很多函数参数喜欢写成 `const T&`？  
  **15秒简答：** 既避免拷贝开销，又表达“函数不会修改参数”，适合传大对象。  
- 引用底层是不是指针？  
  **15秒简答：** 实现上通常会借助指针，但语言语义上引用不是对象，不能简单等同。  
- 返回引用要注意什么？  
  **15秒简答：** **不能返回局部变量引用，**否则会悬空；只能返回仍然有效对象的引用。

# 智能指针与资源管理

## unique_ptr、shared_ptr、weak_ptr 有什么区别？（所有权模型）怎么选？
**15秒简答：**  
`unique_ptr` 独占所有权，开销最小，默认首选；

`shared_ptr` 共享所有权，靠引用计数管理；

`weak_ptr` 不拥有对象，只是观察者，用来打破 `shared_ptr` 循环引用。工程上能独占就别共享，避免不必要的计数和生命周期复杂度。

**3分钟详答：**  
这三者的区别不是“功能多少”，而是**所有权模型不同**。  

`unique_ptr`：  

- 一个对象同一时刻只能有一个拥有者。  
- 不能拷贝，只能移动。  
- 几乎没有引用计数开销。  
所以如果资源的生命周期很清晰，比如某个连接对象、上下文对象只归一个模块所有，优先它。  

`shared_ptr`：  
- 多个对象共同拥有同一资源。  
- 内部有控制块，维护引用计数。  
- 最后一个拥有者离开时自动释放资源。  
它适合“资源确实需要被多方共享”的场景，但代价是额外内存和原子计数成本，而且生命周期更难推理。  

`weak_ptr`：  

- 不增加引用计数。  
- 不能直接解引用，要先 `lock()` 成 `shared_ptr` 再访问。  
它主要用于两种场景：  
1. 打破循环引用。  
2. 做缓存/观察者，表达“我想看这个对象，但不拥有它”。  

面试里建议再补两点，让答案更像工程判断：  
1. **默认优先 `unique_ptr`**，只有在确实有共享生命周期需求时才升级成 `shared_ptr`。  
2. `shared_ptr` 不是“更高级”，它常常意味着更高的复杂度和更难排查的生命周期问题。  

**可能的追问问题：**  

- 为什么说默认优先 `unique_ptr`？  
  **15秒简答：** 因为语义最清晰、开销更小、生命周期更容易推理，符合“能独占就不共享”的原则。  
- `weak_ptr` 为什么不能直接访问对象？  
  **15秒简答：** 因为它不拥有对象，目标对象可能已经释放，必须先 `lock()` 检查是否还活着。  
- `shared_ptr` 的额外开销主要来自哪里？  
  **15秒简答：** 来自控制块、引用计数维护，以及多线程场景下可能涉及的原子操作成本。

## shared_ptr 为什么会有循环引用问题？怎么解决？
**15秒简答：**  
因为 `shared_ptr` 会增加强引用计数，**如果两个对象互相持有 `shared_ptr`**，计数永远不归零，析构就不会触发。

解决方法是在其中一边改成 `weak_ptr`，让它只观察不拥有。

**3分钟详答：**    
`shared_ptr` 的释放条件是“最后一个强引用消失”。如果对象 A 里有一个 `shared_ptr<B>`，对象 B 里又有一个 `shared_ptr<A>`，那么即使外部都不再使用它们了，A 和 B 仍然互相把对方的计数托住，于是两个对象都析构不了，这就是循环引用。  

典型场景有：  
- 双向链表节点互指  
- 父子节点互相持有  
- 观察者模式里相互保存对象  

正确做法通常是：  
- **真正拥有资源的一边用 `shared_ptr`**  
- **仅做反向引用、回指针、观察关系的一边用 `weak_ptr`**  

比如树结构里，父节点持有子节点可以是 `shared_ptr`，子节点回指父节点时用 `weak_ptr`。  
这样当外部不再持有父节点时，强引用链断掉，整棵树就能释放。  

再进一步一点，面试里可以补一句：  
循环引用不是 `shared_ptr` 的 bug，而是共享所有权模型天然带来的问题。所以选型时要先想清楚：**这到底是拥有关系，还是仅仅引用关系？**

**可能的追问问题：**  

- `weak_ptr` 怎么访问对象？  
  **15秒简答：** 通过 `lock()` 获取一个临时 `shared_ptr`，拿到后再安全访问。  
- 所有双向关系都该用 `shared_ptr + weak_ptr` 吗？  
  **15秒简答：** 不一定，关键看谁拥有谁；很多“回指”其实不该是拥有关系。  
- 裸指针也能解决循环引用吗？  
  **15秒简答：** 能表达非拥有关系，但要自己保证对象生命周期；`weak_ptr` 更安全也更明确。

# 面向对象、多态与虚函数

## C++ 多态的实现原理是什么？虚函数表是什么？
**15秒简答：**  
运行时多态靠虚函数机制实现。含虚函数的类通常会有一张虚函数表，对象里有隐藏的虚表指针；通过基类指针/引用调用虚函数时，会在运行时根据实际对象类型分派到正确实现。

**3分钟详答：**  
在 C++ 里，如果成员函数不是虚函数，那么调用哪个版本在编译期就确定了；

而虚函数则支持运行时根据对象真实类型做分派。  

常见实现方式是：  
- 每个含虚函数的类维护一张虚函数表（vtable）。  
- 每个对象内部有一个隐藏指针（vptr）指向所属类的虚表。  
- 当你用基类指针/引用调用虚函数时，程序先通过对象里的 vptr 找到虚表，再取出对应槽位的函数地址执行。  

所以多态成立通常需要三个条件：  
1. 有继承关系。  
2. 基类函数声明为 `virtual`。  
3. 通过基类指针或引用调用。  


- 虚函数调用通常不能像普通函数那样轻易内联，因此会有一点间接调用开销。  
- 但不要把这个开销夸大，大多数业务代码的瓶颈不在这里，关键还是语义正确和设计合理。   

**可能的追问问题：**  

- 为什么必须通过基类指针或引用调用才体现多态？  
  **15秒简答：** 因为对象本体调用时静态类型和动态类型一致，只有基类视角才需要运行时分派。  
- 虚函数的开销主要在哪里？  
  **15秒简答：** 主要是间接调用、可能不易内联，以及一定的对象额外布局成本。  
- 纯虚函数是什么？  
  **15秒简答：** 用 `=0` 声明的虚函数，表示基类只定义接口不提供默认实现，含纯虚函数的类通常是抽象类。

## 为什么基类析构函数通常要声明成 virtual？
**15秒简答：**  
因为如果通过基类指针删除派生类对象，基类析构不写成 `virtual` 会导致只调用基类析构，行为未定义；写成虚析构才能保证派生类析构链完整执行。

**3分钟详答：**  
这题几乎是“虚函数”题后的固定追问。  
场景很常见：  
```cpp
Base* p = new Derived();
delete p;
```
如果 `Base` 的析构函数不是虚函数，那么通过 `Base*` 删除 `Derived` 对象时，程序不一定会正确调用 `Derived` 的析构逻辑，这属于未定义行为。结果可能是派生类资源没有释放、逻辑没有执行，甚至内存破坏。  

如果基类析构函数是 `virtual`，那么 delete 时就会根据对象真实类型，先调用派生类析构，再逐层回到基类析构，保证整个销毁链完整。  

面试里常见一个延伸点:
- **构造函数不能是虚函数**，因为对象都还没构造好，多态基础尚未建立。  
- **析构函数可以而且多态基类通常应该是虚函数**，因为销毁时需要根据动态类型执行正确清理。  

工程上可以这样表述更成熟：  
**只要一个类打算被当作多态基类使用，也就是它会被基类指针/引用持有并通过基类接口操作，那么析构函数几乎总该是虚的。**

**可能的追问问题：**  
- 构造函数为什么不能是虚函数？  
  **15秒简答：** 因为构造阶段对象的动态类型还没完整建立，虚表分派基础不成立。  
- 如果类根本不打算被继承，还需要虚析构吗？  
  **15秒简答：** 不需要，虚析构会带来额外开销，只有多态基类通常才需要。  
- 只有析构函数是 virtual，没有别的虚函数，可以吗？  
  **15秒简答：** 可以，这至少保证通过基类指针删除对象时行为正确。

# C++11/现代C++高频题

## 什么是左值、右值、右值引用？移动语义解决了什么问题？
**15秒简答：**  
**左值一般有稳定身份、可取地址；右值更偏临时对象。**

右值引用 `T&&` 让我们能识别“即将销毁的对象”，把内部资源转移给新对象而不是深拷贝，这就是移动语义，核心价值是减少不必要的拷贝开销。

**3分钟详答：**  
这个题是现代 C++ 的核心题。  
在 C++03 时代，很多临时对象虽然马上就销毁了，但传递、返回、容器扩容时仍可能发生昂贵拷贝，尤其对象内部持有堆内存、文件句柄等资源时，深拷贝代价很高。  

C++11 引入右值引用，本质上是给编译器和程序员一个信号：  
**这个对象是临时值，资源可以“搬走”而不是“复制”。**  

例如一个 `std::vector<std::string>` 扩容时，如果元素支持移动，那么旧空间里的字符串可以把内部 buffer 直接转移到新位置，而不一定重新分配和复制字符内容。  
这就是移动语义的价值：  

1. 提升性能  
2. 降低临时对象成本  
3. 让资源独占类型也能参与返回值传递和容器操作  

面试里别把左值右值讲得太学院派，抓住两个关键词就够：  
- **左值：有名字、可持续存在**  
- **右值：临时结果、即将销毁**  

然后补一句：右值引用并不等于“万能高性能”，它只是让类作者可以定义移动构造和移动赋值，从而把资源转移而非复制。  
真正有价值的是：**理解哪些类型适合移动、移动后对象是什么状态、容器为何受益。**

**可能的追问问题：**  
- 移动之后的对象还能用吗？  
  **15秒简答：** 能，但只保证“有效但值未指定”，可以析构、赋新值，不该依赖原内容。  
- 哪些类型从移动语义里收益大？  
  **15秒简答：** 持有堆内存、句柄、缓存等资源的重对象收益大，像 `int` 这种小对象收益很小。  
- 为什么说移动语义对容器很重要？  
  **15秒简答：** 因为容器扩容、插入、返回对象时经常要搬运元素，移动能显著减少深拷贝成本。

## std::move 和 std::forward 有什么区别？
**15秒简答：**  
`std::move` 是无条件把表达式转成右值，表示“可以搬走”；`std::forward` 是有条件转发，通常配合模板里的转发引用 `T&&` 使用，用来保留实参原本的左值/右值属性。简单说：move 用于显式移动，forward 用于完美转发。

**3分钟详答：**  
这是很容易答混的高频题。  
首先两者本质都不是“真的移动对象”，而是**类型转换工具**。真正移不移动，取决于后续有没有匹配到移动构造、移动赋值等重载。  

`std::move` 的语义很直接：  
- 不管传进来原来是左值还是右值，统统转成右值视角。  
- 适用于你已经决定“这个对象后面不再保留原值语义了，可以把资源交出去”。  

`std::forward` 的使用场景更窄但很重要：  
- 通常出现在函数模板里，参数形式是 `T&&`。  
- 因为模板推导会发生引用折叠，函数体里这个参数本身是有名字的，表达式层面又成了左值。  
- 为了把它原始的值类别保留下去传给下游函数，就需要 `std::forward<T>(arg)`。  

一句话区分：  
- `move`：我主动宣布“把它当右值用”。  
- `forward`：我替调用者原样转交，不擅自改变它的左值/右值属性。  

面试里可以加一个非常加分的坑点：  
**在转发函数里误用 `std::move`，会把本该保持为左值的参数强行搬走，这是常见 bug。**

**可能的追问问题：**  
- `std::move` 会真的搬数据吗？  
  **15秒简答：** 不会，它只是转成右值；真正是否发生移动取决于后续调用的重载。  
- 什么是完美转发？  
  **15秒简答：** 在模板包装层里，把参数的值类别和 const 属性尽量原样传给下游函数。  
- 为什么模板参数 `T&&` 不一定是右值引用？  
  **15秒简答：** 在类型推导场景里它可能是转发引用，左值实参会推导成 `T&`，发生引用折叠。

## 为什么很多类的移动构造函数建议写 noexcept？
**15秒简答：**  
因为标准库容器在搬运元素时更愿意使用“不抛异常的移动”；如果移动构造不是 `noexcept`，像 `vector` 在扩容时可能退回复制，以保证异常安全，性能就可能下降。

**3分钟详答：**  
这个题能很好地区分“背过 move”和“真的理解 STL 行为”的人。  
以 `std::vector` 为例：当它扩容时，需要把旧空间里的元素迁移到新空间。理论上移动比拷贝更便宜，但容器还要保证异常安全。  
如果元素类型的移动构造可能抛异常，那么迁移过程中一旦抛了，容器会更难维持强异常安全保证。于是实现往往会优先使用拷贝，或者在某些条件下不得不承担更复杂的异常处理。  

所以对于一个“移动本来就只是转移资源指针/句柄”的类型，通常移动构造和移动赋值都应该是 `noexcept`。这样标准库就更放心地在扩容、重排时使用移动路径。  

你可以这样总结这题：  
- `noexcept` 不只是语法修饰。  
- 它会直接影响标准库容器是否采用更高效的移动路径。  
- 因此在自定义资源管理类里，这是很实用的性能点。  

**如果一个类型的移动操作本质不会失败，就应尽量标成 `noexcept`，否则你可能“写了移动语义，但容器不敢用”。**

**可能的追问问题：**  
- `vector` 为什么关心移动构造会不会抛异常？  
  **15秒简答：** 因为扩容时要迁移元素，容器要维持异常安全，如果移动可能抛异常就更难保证状态一致。  
- 所有移动构造都必须写 `noexcept` 吗？  
  **15秒简答：** 不是必须，但如果语义上确实不会失败，建议写，这样更利于标准库优化。  
- 如果不写 `noexcept`，最常见后果是什么？  
  **15秒简答：** 某些容器操作可能退回复制而不是移动，性能下降。

# STL容器与工程选型

## vector、list、deque 分别适合什么场景？后端开发里怎么选？
**15秒简答：**  
默认首选 `vector`，因为连续内存、缓存友好、随机访问快；

`list` 适合已知迭代器位置下的频繁插删，但额外指针开销大、缓存不友好；

`deque` 适合首尾频繁插删又需要随机访问。工程里不要因为“插删 O(1)”就盲选 `list`，多数业务场景 `vector` 更快。

**3分钟详答：**  

- `vector` 尾插均摊 O(1)  
- `list` 任意位置插删 O(1)  
- `deque` 首尾插删 O(1)  
这些没错，但还不够工程化。  

真正做后端选型时，要关注的不只是渐进复杂度，还包括：  
1. **内存布局**  
   `vector` 连续存储，CPU 缓存友好；`list` 节点分散，遍历时 cache miss 多。  
2. **真实操作模式**  
   `list` 的 O(1) 插删前提通常是“你已经拿到了那个位置的迭代器”。如果还得先查找，整体不一定划算。  
3. **内存开销**  
   `list` 每个节点至少多两个指针，空间成本高。  
4. **接口需求**  
   需要随机访问就别选 `list`。需要稳定引用/迭代器时可以考虑 `list` 或节点型容器。  

- 默认先想 `vector`。  
- 只有当业务真的有大量中间插删、并且能长期持有有效位置时，才考虑 `list`。  
- 需要双端操作又想保留随机访问能力时考虑 `deque`。  

**可能的追问问题：**  
- 为什么很多时候 `vector` 比 `list` 更快？  
  **15秒简答：** 因为连续内存更缓存友好，虽然理论上中间插删慢，但真实 CPU 上遍历和搬移常常更高效。  
- `list` 的 O(1) 插删有什么前提？  
  **15秒简答：** 前提是已经拿到待插删位置的迭代器，否则先查找的成本可能很高。  
- `deque` 和 `vector` 最大区别是什么？  
  **15秒简答：** `deque` 擅长首尾插删且支持随机访问，但内存不要求整体连续，实现通常是分段存储。

## vector 的底层扩容机制是什么？reserve 和 resize 有什么区别？
**15秒简答：** 
`vector` 空间不够时会申请更大的新内存，把旧元素搬过去，再释放旧空间，所以可能导致迭代器、指针、引用失效。`reserve` 只改容量不改元素个数，`resize` 会改变 size，必要时还会默认构造新元素。

**3分钟详答：** 
`vector` 的核心特点是**连续内存**。正因为要求连续，所以当现有容量不够放新元素时，它不能像链表那样随便挂个节点，而是要：  

1. 申请一块更大的连续内存  
2. 把旧元素迁移过去（拷贝或移动）  
3. 释放旧内存  
4. 更新内部指针  

这就解释了三个经典现象：  
- 扩容有成本  
- 扩容后地址可能变  
- 迭代器/引用/指针可能失效  

至于 `reserve` 和 `resize`：  
- `reserve(n)`：提前预留至少 n 的容量，`size` 不变，不会凭空多出元素。适合已知数据规模时减少扩容次数。  
- `resize(n)`：把逻辑元素个数改成 n。若 n 变大，会新增元素并进行默认构造或值初始化；若 n 变小，只是减少有效元素个数。  

后端场景里最好补一句：  
**如果你大致知道元素规模，提前 `reserve` 能显著减少扩容和搬运成本。**  
再加一句更深的：  
如果元素类型的移动构造是 `noexcept`，扩容迁移通常更容易走移动路径。  

**可能的追问问题：**  

- 扩容后为什么迭代器会失效？  
  **15秒简答：** 因为元素可能整体搬到新内存，原先地址不再有效。  
- `reserve` 会创建新元素吗？  
  **15秒简答：** 不会，它只扩容量，不改 `size`。  
- `resize` 把大小变大时会发生什么？  
  **15秒简答：** 会新增元素，内置类型值初始化，自定义类型会调用相应构造函数。

## map 和 unordered_map 有什么区别？线上如何选？
**15秒简答：**  
`map` 通常基于红黑树，键有序，查找/插入/删除是 O(log n)，支持范围查询；`unordered_map` 基于哈希表，平均 O(1)，但无序，最坏可能退化，还会有 rehash 成本。需要顺序、上下界、范围操作选 `map`；只关心高频单点查找、且 key 分布合适时优先 `unordered_map`。

**3分钟详答：**  
这是后端实习岗几乎必问题，常常还会问“工程里怎么选，而不是书上怎么选”。  
先讲底层：  

- `map` 是关联有序容器，通常用红黑树实现。  
- `unordered_map` 是无序关联容器，通常用哈希表实现。  

因此它们差异主要在四个维度：  
1. **顺序性**  
   `map` 天然按 key 有序，支持 `lower_bound`、区间遍历；`unordered_map` 不保证顺序。  
2. **复杂度**  
   `map` 稳定 O(log n)；`unordered_map` 平均 O(1)，但冲突严重时会退化。  
3. **空间与常数项**  
   哈希表常有桶数组、负载因子、rehash，空间和常数可能更大。  
4. **稳定性与场景特性**  
   对范围查询、排序输出、前缀区间等，`map` 更自然。对高频点查、缓存索引等，`unordered_map` 更常用。  

面试里最加分的是别说“unordered_map 一定更快”。因为真实线上选型还要看：  
- key 分布是否均匀  
- 是否需要自定义哈希  
- 是否担心冲突退化  
- 是否会频繁 rehash  
- 是否关心遍历顺序  
- 内存是否紧张  

成熟回答通常是：  
**没有绝对更优，只有更贴近业务访问模式的容器。**

**可能的追问问题：**  
- `unordered_map` 为什么会退化？  
  **15秒简答：** 因为大量哈希冲突会让桶内元素过多，查找接近线性。  
- 什么是 rehash？  
  **15秒简答：** 当负载因子过高时，哈希表会扩桶并重新分布元素，这会带来额外成本。  
- 什么时候 `map` 明显更合适？  
  **15秒简答：** 需要有序遍历、范围查找、上下界查询时，`map` 更适合。

## set/map 的底层为什么常说是红黑树？红黑树相对普通 BST 的价值是什么？
**15秒简答：**  
因为红黑树是一种近似平衡的二叉搜索树，**能把查找、插入、删除稳定控制在 O(log n)。**

相比普通 BST，不容易因为数据有序插入而退化成链表。

**3分钟详答：**  

普通二叉搜索树理论上查找很快，但如果插入序列接近有序，它就可能退化成一条链，复杂度变成 O(n)。这对标准库容器是不够稳定的。  
红黑树通过节点颜色和几条平衡规则，保证树的高度不会太离谱，因此查找、插入、删除都能维持对数级复杂度。  

为什么 STL 喜欢这种结构？因为关联容器需要：  
- 动态插入删除  
- 保持有序  
- 提供稳定复杂度  
红黑树恰好平衡了这三点。  

面试里不需要背所有旋转细节，但最好能讲出它的工程价值：  
- 它不是“绝对最平衡”的树，但调整成本相对可控。  
- 相比 AVL 树，红黑树在插入删除时通常旋转更少，更适合通用容器场景。  

如果你再补一句“STL 标准并不强制 `map` 一定是红黑树，但常见实现通常如此”，会显得更严谨。

**可能的追问问题：**  
- 红黑树为什么不是严格平衡树？  
  **15秒简答：** 它只保证**近似平衡**，通过约束黑高把高度控制在对数级，不追求每层绝对平衡。  
- 为什么普通 BST 会退化？  
  **15秒简答：** 因为若按有序序列插入，节点会一边倒，树高接近 n。  
- 为什么说红黑树适合关联容器？  
  **15秒简答：** 因为它支持动态维护有序性，同时查改删复杂度比较稳定。

# STL迭代器、失效与算法题

## 什么是迭代器失效？不同容器下有哪些典型场景？
**15秒简答：**  
**迭代器失效就是容器结构变化后，原迭代器、指针或引用不再能安全访问原元素。**

`vector` 扩容最典型，会导致全部迭代器失效；`erase` 往往会让删除点及其后的迭代器失效；`list`/`map` 这类节点型容器通常更稳定，删除一般只影响被删元素本身。

**3分钟详答：**  

核心原因是： 
容器修改后，元素位置、内部缓冲区或节点关系发生变化，原来的迭代器不一定还指向有效位置。  

最常见规律：  
- `vector`：  
  - 扩容时几乎所有迭代器、引用、指针都失效。  
  - `erase` 后，被删位置及后面的迭代器失效。  
- `deque`：  
  - 规则比 `vector` 更复杂，中间插删常影响更大。  
- `list`：  
  - 节点独立存储，插入通常不影响其他迭代器；删除只使被删元素迭代器失效。  
- `map/set`：  
  - 插入通常不让已有迭代器失效；删除通常只影响被删元素。  
- `unordered_map`：  
  - 插入如果触发 rehash，迭代器会整体失效。  


1. 了解容器规则，不拍脑袋写。  
2. 对 `vector` 预留容量，减少扩容。  
3. 删除时使用 `erase` 返回的迭代器更新当前位置。  
4. 需要长期持有稳定位置时考虑节点型容器。  

这个题如果答得好，会让面试官觉得你不是只会刷题，而是知道 STL 真正危险在哪。

**可能的追问问题：**  

- 为什么 `vector` 扩容会导致全部迭代器失效？  
  **15秒简答：** 因为元素整体搬到新连续内存，旧地址都失效了。  
- `map` 插入通常为什么不影响已有迭代器？  
  **15秒简答：** 因为它是节点型结构，插入只调整树链接，不会整体搬迁已有节点。  
- `unordered_map` 哪种场景下最要小心迭代器失效？  
  **15秒简答：** 触发 rehash 时，所有迭代器都可能失效。

## 一边遍历一边删除元素，正确写法是什么？
**15秒简答：**  
不能删完还继续用旧迭代器。对 `vector/list/map` 等，通常写成 `it = c.erase(it);`，因为 `erase` 会返回下一个有效迭代器；如果没删当前元素才 `++it`。

**3分钟详答：**  
这个题非常贴近编码实战。  
很多人第一反应是：

```cpp
for (auto it = v.begin(); it != v.end(); ++it) {
    if (need_delete(*it)) v.erase(it);
}
```
这通常有问题，因为 `erase(it)` 后，`it` 本身可能已经失效，循环里的 `++it` 再用它就是未定义行为。  

正确思路是：  
- 删除当前元素后，立刻接住 `erase` 返回的“下一个有效迭代器”。  
- 如果当前不删除，再正常自增。  

典型写法：
```cpp
for (auto it = c.begin(); it != c.end(); ) {
    if (need_delete(*it)) it = c.erase(it);
    else ++it;
}
```

这个模式特别适合面试，因为它体现的是：  
1. 你理解迭代器失效。  
2. 你知道 STL 设计接口的用意。  

再补一句区分会更完整：  
- 对 `vector`，删除后后续元素前移，所以更要依赖返回值。  
- 对 `map/set`，虽然通常只删当前节点，但统一使用返回值是最稳妥的写法。  

**可能的追问问题：**  
- 为什么 `erase` 要返回迭代器？  
  **15秒简答：** 因为删除后原迭代器可能失效，返回值让你能继续安全遍历。  
- `remove` 和 `erase` 是一回事吗？  
  **15秒简答：** 不是，`remove` 主要是算法层“重排保留元素”，真正删容器元素还要结合 `erase`。  
- `list` 删除时是不是就可以随便写？  
  **15秒简答：** 也不建议，虽然它更稳定，但接住 `erase` 返回值仍是最通用安全的写法。

## STL 中 remove/erase 惯用法是什么？为什么很多人会写错？
**15秒简答：**  
`std::remove` 不会真的删容器元素，它只是把“不需要删除的元素”搬到前面，并返回新的逻辑末尾；

真正删除还要调用容器的 `erase`。所以常见写法是 `v.erase(std::remove(v.begin(), v.end(), x), v.end());`。

**3分钟详答：**  
这是一个特别能区分“只会 API”还是“理解算法与容器分工”的题。  
`remove` 是 `<algorithm>` 里的通用算法，它并不知道底层容器怎么真正释放元素，也不会改容器 `size`。它做的是：  

- 遍历区间  
- 把需要保留的元素前移  
- 返回“新逻辑结尾”迭代器  

因此对 `vector/string/deque` 等容器，如果你只调用 `remove`，容器长度其实没变，后面那一段只是变成了“待忽略垃圾区”。  
所以必须再调用容器的 `erase` 把尾部真正删掉，这就是 erase-remove idiom。  

面试里最好补一句：  
为什么标准库要这样设计？因为算法和容器是分离的。`remove` 只依赖迭代器，不依赖具体容器类型，这体现了 STL 的泛型设计思想。  

如果能说到这一步，面试官通常会觉得你对 STL 的理解比较到位。

**可能的追问问题：**  

- 为什么 `remove` 不直接删元素？  
  **15秒简答：** 因为它是通用算法，只操作迭代器区间，不拥有具体容器的缩容能力。  
- `list` 也一定要用 `std::remove + erase` 吗？  
  **15秒简答：** 不一定，`list` 自己有成员函数 `remove`，能直接按节点删除。  
- 这个惯用法最容易犯什么错？  
  **15秒简答：** 只写了 `remove`，没写 `erase`，结果以为删了其实容器大小没变。

## 迭代器和指针一样吗？为什么 STL 需要迭代器这一层抽象？
**15秒简答：**  
不完全一样。指针是地址，迭代器是“**统一访问容器元素的位置抽象**”。**连续容器的迭代器可能像指针，但树、链表、哈希表容器显然不能只靠原始指针统一表示**。STL 用迭代器把算法和容器解耦。

**3分钟详答：**  
如果只有数组，指针足够表示遍历位置；但 STL 里有 `vector`、`list`、`map`、`unordered_map` 等完全不同结构。  
它们底层分别可能是：  

- 连续数组  
- 双向链表  
- 红黑树  
- 哈希桶  
显然没法要求所有容器都暴露原始地址来让算法工作。  

于是 STL 引入迭代器，作为一种统一接口：  
- `*it` 取元素  
- `++it` 前进  
- 某些容器支持 `--it`  
- 某些容器支持随机跳转 `it + n`  

这样算法只依赖“迭代器能力”，比如随机访问迭代器、双向迭代器，而不依赖具体容器类型。  
这就是为什么 `sort` 需要随机访问迭代器，所以可以作用于 `vector` 但不能直接作用于 `list`。  

面试里如果你能顺手讲出“STL 的核心是容器、算法、迭代器三者解耦”，会很加分。

**可能的追问问题：**  
- 为什么 `sort` 不能直接排序 `list`？  
  **15秒简答：** 因为 `sort` 需要随机访问迭代器，而 `list` 只有双向迭代器。  
- `vector` 的迭代器为什么常常看起来像指针？  
  **15秒简答：** 因为它底层是连续内存，实现上迭代器通常可以很接近原生指针。  
- STL 为什么强调“算法和容器分离”？  
  **15秒简答：** 这样同一套算法可以复用到不同容器上，体现泛型编程的价值。

# 函数对象、lambda 与模板意识

## lambda 表达式有什么价值？在 STL 里常怎么用？
**15秒简答：**  
lambda 能就地定义短小可调用对象，特别适合配合 STL 算法写比较器、过滤条件、回调逻辑。

它减少样板代码，也更容易把局部上下文一起捕获进来。

**3分钟详答：**  
lambda 是 C++11 以后非常实用的特性，面试常问“你项目里怎么用过”。  
在 STL 里，很多算法都需要你提供一个“行为”：  

- `sort` 需要比较规则  

- `find_if` 需要判定条件  

- `for_each` 需要处理动作  

  如果每次都单独写一个函数或仿函数类，代码会很重。lambda 则允许你在调用点直接写出这段逻辑。  

它的工程价值在于：  
1. **局部性好**：逻辑就在使用点。  
2. **减少模板样板代码**。  
3. **能捕获上下文**：不像普通函数那样必须全靠参数传。  



**捕获方式要谨慎。**  
尤其在异步或延迟执行场景里，引用捕获很容易把已经失效的局部变量带进去。  

所以更成熟的回答是：  
lambda 很适合 STL 和回调，但要根据生命周期正确选择值捕获还是引用捕获。

**可能的追问问题：**  

- 值捕获和引用捕获怎么选？  
  **15秒简答：** 生命周期短、同步场景里引用捕获可减少拷贝；异步或延迟执行时更要小心，通常值捕获更安全。  
- lambda 本质上是什么？  
  **15秒简答：** **本质上是编译器生成的匿名函数对象，捕获的变量会成为它的成员。**  
- lambda 为什么适合 STL 算法？  
  **15秒简答：** 因为算法常需要短小的比较/过滤逻辑，lambda 写在调用点最直观。

## 模板的本质是什么？为什么说 STL 是模板库？
**15秒简答：** 
模板本质上是“**编译期代码生成机制**”，不是运行时泛型。

**编译器会根据具体类型实例化出对应代码**。

STL 的容器、算法、迭代器大量依赖模板，才能做到一套接口适配多种类型和容器。

**3分钟详答：** 
C++ 模板不是 Java/C# 那种偏运行时的泛型机制，它更像一份“类型相关的代码蓝图”。  
当你写：

```cpp
std::vector<int>
std::vector<std::string>
```
编译器会为不同类型实例化出对应版本。  

这带来两个好处：  
1. **零额外抽象成本**  
   编译后就是具体类型代码，不必为泛型引入统一基类或运行时装箱。  
2. **强大的复用能力**  
   同一个容器模板能支持任意元素类型；同一个算法模板能工作在不同迭代器上。  

这也是 STL 的基础：  
- 容器是模板  
- 算法是模板  
- 迭代器能力也通过模板约束组合起来  
所以 STL 才能做到“同一套 `sort/find/accumulate` 面向很多数据结构工作”。  

模板也会带来编译报错复杂、代码膨胀、调试难度增加等代价，所以工程里既要会用，也要懂得控制复杂度。

**可能的追问问题：**  
- 模板是编译期还是运行期机制？  
  **15秒简答：** 主要是编译期机制，编译器按具体类型实例化代码。  
- 为什么 STL 这么依赖模板？  
  **15秒简答：** 因为它要支持多类型、多容器复用，同时尽量不牺牲运行时性能。  
- 模板的工程代价有哪些？  
  **15秒简答：** 主要是报错长、编译慢、可能造成代码膨胀和阅读复杂度上升。

# 速刷建议（建议背法）

1. **先背“工程判断句”**：  
   比如“默认优先 `vector`”“默认优先 `unique_ptr`”“容器选型看访问模式，不只看复杂度”。这类话最像真实面试回答。

2. **每题先答结论，再答原理，再补场景**：  
   面试不是写教材。先给结论，面试官才愿意继续追问。

3. **尽量把 STL 题和工程风险绑定**：  
   例如：  
   - `vector` → 扩容、迭代器失效  
   - `unordered_map` → rehash、冲突退化  
   - `shared_ptr` → 循环引用、生命周期复杂  

4. **把“背概念”升级成“会选型”**：  
   面试官更喜欢听到：  
   - 为什么业务里优先 `vector`  
   - 为什么默认 `unique_ptr`  
   - 为什么移动构造建议 `noexcept`
