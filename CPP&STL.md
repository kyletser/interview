
# C++语言基础与对象模型

## new/delete 和 malloc/free 有什么区别？
**15秒简答：**  
`new/delete` 是 C++ **运算符**，除了分配/释放内存，还会调用构造/析构函数，支持类型安全和重载；

`malloc/free` 是 C **函数**，只负责裸内存管理，不会调用对象生命周期函数。

写 C++ 业务代码优先 `new/delete`，更推荐进一步用 RAII 和智能指针替代手写释放。

**3分钟详答：**  
`malloc` **做的是“给你一块指定字节数的原始内存”，返回 `void*`**，所以需要手动转换类型，也不会自动初始化对象。`free` 只是把内存还回去，不会调用析构。 
`new` 则是“**先分配内存，再在这块内存上构造对象**”，因此它和类型绑定，会调用构造函数；`delete` 会先调用析构函数，再释放内存。  



面试里最好再补三点：  

1. `new` 失败默认抛 `std::bad_alloc`，而 `malloc` 失败返回 `nullptr`。  
2. `new/delete` 可以被类重载，适合做对象池、定制分配器；`malloc/free` 不能体现这种面向对象扩展能力。  
3. 混用有风险：`malloc` 出来的内存不能用 `delete` 释放，`new` 出来的对象不能用 `free` 释放，否则是未定义行为。  

**如果资源需要生命周期管理，我尽量不直接写裸 `new/delete`，而是交给对象、容器、智能指针管理**。这就自然引到 RAII。

**可能的追问问题：**  

- `new` 失败和 `malloc` 失败的表现有什么不同？  
  **15秒简答：** `new` 默认抛异常 `std::bad_alloc`，`malloc` 返回 `nullptr`，所以错误处理方式不同。  
- 为什么不建议在现代 C++ 里到处手写 `new/delete`？  
  **15秒简答：** 因为容易泄漏、重复释放、异常路径漏释放；现代 C++ 更推荐 RAII、容器和智能指针托管资源。

## 什么是 RAII？为什么 C++ 特别强调它？
**15秒简答：**  
RAII 就是“资源获取即初始化”，**把资源绑定到对象生命周期**：对象**构造时拿资源**，对象**析构时释放资源**。这样即使函数中途返回或抛异常，也不会漏释放。锁、文件句柄、堆内存都适合这样管理。

**3分钟详答：**   
RAII 的思路是：**不要把“释放资源”写成业务逻辑的一部分，而是写进对象析构函数里**。这样资源的释放就和作用域绑定了。  

最经典例子是加锁：  
进入函数时构造 `std::lock_guard<std::mutex>`，它在构造时加锁；离开作用域时自动析构解锁。这样不管是正常 return、异常退出，还是中间分支，锁都能被释放。  
再比如智能指针，本质也是 RAII：对象离开作用域时自动回收所管理的堆内存。  

**可能的追问问题：**  

- `lock_guard` 为什么是 RAII 的典型例子？  
  **15秒简答：** 因为它在构造时加锁，析构时解锁，锁的生命周期和对象作用域绑定。  
- RAII 只能管理内存吗？  
  **15秒简答：** 不只，锁、文件、socket、数据库连接等都可以，只要资源有获取和释放动作。  
- RAII 和异常安全有什么关系？  
  **15秒简答：** 发生异常时栈展开会调用局部对象析构函数，所以 RAII 能自动完成清理。

## 深拷贝、浅拷贝，以及为什么会引出 Rule of Three/Five/Zero？
**15秒简答：** 
浅拷贝只是复制成员值，如果成员里有裸指针，就会导致多个对象指向同一块资源；

深拷贝会复制资源本身。只要类自己管理资源，就要认真处理拷贝、赋值、析构，C++11 以后还要考虑移动，所以有 Rule of Five；更推荐 Rule of Zero，尽量交给标准库成员管理资源。

**3分钟详答：** 
如果一个类里只有普通值类型成员，比如 `int`、`std::string`、`std::vector`，编译器默认生成的拷贝行为通常没问题，因为这些成员自己就会正确拷贝。  
但如果类里自己维护了裸资源，比如 `char* data` 指向堆内存，默认拷贝就只是把这个地址复制一份，这就是浅拷贝。结果两个对象会指向同一块内存：一边修改会互相影响，更严重的是析构时可能 double free。  

所以只要**类自己管理资源，通常就要显式定义：**  
- 析构函数  
- 拷贝构造  
- 拷贝赋值  
这叫 Rule of Three。  

到 C++11 以后，为了支持移动语义，通常还要考虑：  
- 移动构造  
- 移动赋值  
这就是 Rule of Five。  

但真正更现代的做法是 Rule of Zero：**尽量不要自己管理裸资源，而是把资源交给 `std::string`、`std::vector`、`std::unique_ptr` 这类已经实现好语义的成员。**这样很多特殊成员函数都可以交给编译器默认生成。  

- 能不用裸指针就不用。  
- 真的需要自己管理资源时，先想清楚对象是否可拷贝、是否可移动。  
- 如果语义上不允许拷贝，就 `= delete`。  

**可能的追问问题：**  
- 什么情况下会发生 double free？  
  **15秒简答：** 两个对象因浅拷贝共享同一块堆内存，析构时都去释放它，就会重复释放。  
- Rule of Five 比 Rule of Three 多了什么？  
  **15秒简答：** 多了移动构造和移动赋值，因为 C++11 引入了移动语义。  
- 为什么很多时候更推崇 Rule of Zero？  
  **15秒简答：** 因为把资源管理交给标准库成员更安全，能少写特殊成员函数，也更不容易出错。

## 指针和引用有什么区别？什么时候更推荐引用？
**15秒简答：** 
引用是对象别名，定义时必须初始化，语义上通常不表示“空”；

指针本身是独立对象，可以改指向、可以为空。参数传递里，若函数要求“必须有对象”，优先用引用；若要表达“可空”“可重新绑定”或多态间接访问，才更常用指针。

**3分钟详答：**  

- 引用更像“这个参数一定存在，而且我就是在操作原对象”。  
- 指针更像“我手上拿的是一个地址，它可能为空，也可能 later 改成指向别的对象”。  

从语言层面看：  
1. 引用定义时必须绑定到对象，之后不能改绑。  
2. 指针可以先不指向任何对象，也可以多次修改指向。  
3. 使用时引用语法更自然，不需要显式解引用。  
4. 指针可以做地址运算，引用不行。  

**引用适合表达必然存在的别名关系，指针适合表达可空、可变绑定、间接寻址和所有权封装的底层能力。**

**可能的追问问题：**  
- 为什么很多函数参数喜欢写成 `const T&`？  
  **15秒简答：** 既避免拷贝开销，又表达“函数不会修改参数”，适合传大对象。  
- 引用底层是不是指针？  
  **15秒简答：** 实现上通常会借助指针，但语言语义上引用不是对象，不能简单等同。  
- 返回引用要注意什么？  
  **15秒简答：** **不能返回局部变量引用，**否则会悬空；只能返回仍然有效对象的引用。

# 智能指针与资源管理

## unique_ptr、shared_ptr、weak_ptr 有什么区别？（所有权模型）怎么选？
**15秒简答：**  
`unique_ptr` 独占所有权，开销最小，默认首选；

`shared_ptr` 共享所有权，靠引用计数管理；

`weak_ptr` 不拥有对象，只是观察者，用来打破 `shared_ptr` 循环引用。工程上能独占就别共享，避免不必要的计数和生命周期复杂度。

**3分钟详答：**  
这三者的区别不是“功能多少”，而是**所有权模型不同**。  

`unique_ptr`：  

- 一个对象同一时刻只能有一个拥有者。  
- 不能拷贝，只能移动。  
- 几乎没有引用计数开销。  
所以如果资源的生命周期很清晰，比如某个连接对象、上下文对象只归一个模块所有，优先它。  

`shared_ptr`：  
- 多个对象共同拥有同一资源。  
- 内部有控制块，维护引用计数。  
- 最后一个拥有者离开时自动释放资源。  
它适合“资源确实需要被多方共享”的场景，但代价是额外内存和原子计数成本，而且生命周期更难推理。  

`weak_ptr`：  

- 不增加引用计数。  
- 不能直接解引用，要先 `lock()` 成 `shared_ptr` 再访问。  
它主要用于两种场景：  
1. 打破循环引用。  
2. 做缓存/观察者，表达“我想看这个对象，但不拥有它”。  

面试里建议再补两点，让答案更像工程判断：  
1. **默认优先 `unique_ptr`**，只有在确实有共享生命周期需求时才升级成 `shared_ptr`。  
2. `shared_ptr` 不是“更高级”，它常常意味着更高的复杂度和更难排查的生命周期问题。  

**可能的追问问题：**  

- 为什么说默认优先 `unique_ptr`？  
  **15秒简答：** 因为语义最清晰、开销更小、生命周期更容易推理，符合“能独占就不共享”的原则。  
- `weak_ptr` 为什么不能直接访问对象？  
  **15秒简答：** 因为它不拥有对象，目标对象可能已经释放，必须先 `lock()` 检查是否还活着。  
- `shared_ptr` 的额外开销主要来自哪里？  
  **15秒简答：** 来自控制块、引用计数维护，以及多线程场景下可能涉及的原子操作成本。

## shared_ptr 为什么会有循环引用问题？怎么解决？
**15秒简答：**  
因为 `shared_ptr` 会增加强引用计数，**如果两个对象互相持有 `shared_ptr`**，计数永远不归零，析构就不会触发。

解决方法是在其中一边改成 `weak_ptr`，让它只观察不拥有。

**3分钟详答：**    
`shared_ptr` 的释放条件是“最后一个强引用消失”。如果对象 A 里有一个 `shared_ptr<B>`，对象 B 里又有一个 `shared_ptr<A>`，那么即使外部都不再使用它们了，A 和 B 仍然互相把对方的计数托住，于是两个对象都析构不了，这就是循环引用。  

典型场景有：  
- 双向链表节点互指  
- 父子节点互相持有  
- 观察者模式里相互保存对象  

正确做法通常是：  
- **真正拥有资源的一边用 `shared_ptr`**  
- **仅做反向引用、回指针、观察关系的一边用 `weak_ptr`**  

比如树结构里，父节点持有子节点可以是 `shared_ptr`，子节点回指父节点时用 `weak_ptr`。  
这样当外部不再持有父节点时，强引用链断掉，整棵树就能释放。  

再进一步一点，面试里可以补一句：  
循环引用不是 `shared_ptr` 的 bug，而是共享所有权模型天然带来的问题。所以选型时要先想清楚：**这到底是拥有关系，还是仅仅引用关系？**

**可能的追问问题：**  

- `weak_ptr` 怎么访问对象？  
  **15秒简答：** 通过 `lock()` 获取一个临时 `shared_ptr`，拿到后再安全访问。  
- 所有双向关系都该用 `shared_ptr + weak_ptr` 吗？  
  **15秒简答：** 不一定，关键看谁拥有谁；很多“回指”其实不该是拥有关系。  
- 裸指针也能解决循环引用吗？  
  **15秒简答：** 能表达非拥有关系，但要自己保证对象生命周期；`weak_ptr` 更安全也更明确。

# 面向对象、多态与虚函数

## C++ 多态的实现原理是什么？虚函数表是什么？
**15秒简答：**  
运行时多态靠虚函数机制实现。含虚函数的类通常会有一张虚函数表，对象里有隐藏的虚表指针；通过基类指针/引用调用虚函数时，会在运行时根据实际对象类型分派到正确实现。

**3分钟详答：**  
在 C++ 里，如果成员函数不是虚函数，那么调用哪个版本在编译期就确定了；

而虚函数则支持运行时根据对象真实类型做分派。  

常见实现方式是：  
- 每个含虚函数的类维护一张虚函数表（vtable）。  
- 每个对象内部有一个隐藏指针（vptr）指向所属类的虚表。  
- 当你用基类指针/引用调用虚函数时，程序先通过对象里的 vptr 找到虚表，再取出对应槽位的函数地址执行。  

所以多态成立通常需要三个条件：  
1. 有继承关系。  
2. 基类函数声明为 `virtual`。  
3. 通过基类指针或引用调用。  


- 虚函数调用通常不能像普通函数那样轻易内联，因此会有一点间接调用开销。  
- 但不要把这个开销夸大，大多数业务代码的瓶颈不在这里，关键还是语义正确和设计合理。   

**可能的追问问题：**  

- 为什么必须通过基类指针或引用调用才体现多态？  
  **15秒简答：** 因为对象本体调用时静态类型和动态类型一致，只有基类视角才需要运行时分派。  
- 虚函数的开销主要在哪里？  
  **15秒简答：** 主要是间接调用、可能不易内联，以及一定的对象额外布局成本。  
- 纯虚函数是什么？  
  **15秒简答：** 用 `=0` 声明的虚函数，表示基类只定义接口不提供默认实现，含纯虚函数的类通常是抽象类。

## 为什么基类析构函数通常要声明成 virtual？
**15秒简答：**  
因为如果通过基类指针删除派生类对象，基类析构不写成 `virtual` 会导致只调用基类析构，行为未定义；写成虚析构才能保证派生类析构链完整执行。

**3分钟详答：**  
这题几乎是“虚函数”题后的固定追问。  
场景很常见：  
```cpp
Base* p = new Derived();
delete p;
```
如果 `Base` 的析构函数不是虚函数，那么通过 `Base*` 删除 `Derived` 对象时，程序不一定会正确调用 `Derived` 的析构逻辑，这属于未定义行为。结果可能是派生类资源没有释放、逻辑没有执行，甚至内存破坏。  

如果基类析构函数是 `virtual`，那么 delete 时就会根据对象真实类型，先调用派生类析构，再逐层回到基类析构，保证整个销毁链完整。  

面试里常见一个延伸点:
- **构造函数不能是虚函数**，因为对象都还没构造好，多态基础尚未建立。  
- **析构函数可以而且多态基类通常应该是虚函数**，因为销毁时需要根据动态类型执行正确清理。  

工程上可以这样表述更成熟：  
**只要一个类打算被当作多态基类使用，也就是它会被基类指针/引用持有并通过基类接口操作，那么析构函数几乎总该是虚的。**

**可能的追问问题：**  
- 构造函数为什么不能是虚函数？  
  **15秒简答：** 因为构造阶段对象的动态类型还没完整建立，虚表分派基础不成立。  
- 如果类根本不打算被继承，还需要虚析构吗？  
  **15秒简答：** 不需要，虚析构会带来额外开销，只有多态基类通常才需要。  
- 只有析构函数是 virtual，没有别的虚函数，可以吗？  
  **15秒简答：** 可以，这至少保证通过基类指针删除对象时行为正确。

# C++11/现代C++高频题

## 什么是左值、右值、右值引用？移动语义解决了什么问题？
**15秒简答：**  
**左值一般有稳定身份、可取地址；右值更偏临时对象。**

右值引用 `T&&` 让我们能识别“即将销毁的对象”，把内部资源转移给新对象而不是深拷贝，这就是移动语义，核心价值是减少不必要的拷贝开销。

**3分钟详答：**  
这个题是现代 C++ 的核心题。  
在 C++03 时代，很多临时对象虽然马上就销毁了，但传递、返回、容器扩容时仍可能发生昂贵拷贝，尤其对象内部持有堆内存、文件句柄等资源时，深拷贝代价很高。  

C++11 引入右值引用，本质上是给编译器和程序员一个信号：  
**这个对象是临时值，资源可以“搬走”而不是“复制”。**  

例如一个 `std::vector<std::string>` 扩容时，如果元素支持移动，那么旧空间里的字符串可以把内部 buffer 直接转移到新位置，而不一定重新分配和复制字符内容。  
这就是移动语义的价值：  

1. 提升性能  
2. 降低临时对象成本  
3. 让资源独占类型也能参与返回值传递和容器操作  

面试里别把左值右值讲得太学院派，抓住两个关键词就够：  
- **左值：有名字、可持续存在**  
- **右值：临时结果、即将销毁**  

然后补一句：右值引用并不等于“万能高性能”，它只是让类作者可以定义移动构造和移动赋值，从而把资源转移而非复制。  
真正有价值的是：**理解哪些类型适合移动、移动后对象是什么状态、容器为何受益。**

**可能的追问问题：**  
- 移动之后的对象还能用吗？  
  **15秒简答：** 能，但只保证“有效但值未指定”，可以析构、赋新值，不该依赖原内容。  
- 哪些类型从移动语义里收益大？  
  **15秒简答：** 持有堆内存、句柄、缓存等资源的重对象收益大，像 `int` 这种小对象收益很小。  
- 为什么说移动语义对容器很重要？  
  **15秒简答：** 因为容器扩容、插入、返回对象时经常要搬运元素，移动能显著减少深拷贝成本。

## std::move 和 std::forward 有什么区别？
**15秒简答：**  
`std::move` 是无条件把表达式转成右值，表示“可以搬走”；`std::forward` 是有条件转发，通常配合模板里的转发引用 `T&&` 使用，用来保留实参原本的左值/右值属性。简单说：move 用于显式移动，forward 用于完美转发。

**3分钟详答：**  
这是很容易答混的高频题。  
首先两者本质都不是“真的移动对象”，而是**类型转换工具**。真正移不移动，取决于后续有没有匹配到移动构造、移动赋值等重载。  

`std::move` 的语义很直接：  
- 不管传进来原来是左值还是右值，统统转成右值视角。  
- 适用于你已经决定“这个对象后面不再保留原值语义了，可以把资源交出去”。  

`std::forward` 的使用场景更窄但很重要：  
- 通常出现在函数模板里，参数形式是 `T&&`。  
- 因为模板推导会发生引用折叠，函数体里这个参数本身是有名字的，表达式层面又成了左值。  
- 为了把它原始的值类别保留下去传给下游函数，就需要 `std::forward<T>(arg)`。  

一句话区分：  
- `move`：我主动宣布“把它当右值用”。  
- `forward`：我替调用者原样转交，不擅自改变它的左值/右值属性。  

面试里可以加一个非常加分的坑点：  
**在转发函数里误用 `std::move`，会把本该保持为左值的参数强行搬走，这是常见 bug。**

**可能的追问问题：**  
- `std::move` 会真的搬数据吗？  
  **15秒简答：** 不会，它只是转成右值；真正是否发生移动取决于后续调用的重载。  
- 什么是完美转发？  
  **15秒简答：** 在模板包装层里，把参数的值类别和 const 属性尽量原样传给下游函数。  
- 为什么模板参数 `T&&` 不一定是右值引用？  
  **15秒简答：** 在类型推导场景里它可能是转发引用，左值实参会推导成 `T&`，发生引用折叠。

## 为什么很多类的移动构造函数建议写 noexcept？
**15秒简答：**  
因为标准库容器在搬运元素时更愿意使用“不抛异常的移动”；如果移动构造不是 `noexcept`，像 `vector` 在扩容时可能退回复制，以保证异常安全，性能就可能下降。

**3分钟详答：**  
这个题能很好地区分“背过 move”和“真的理解 STL 行为”的人。  
以 `std::vector` 为例：当它扩容时，需要把旧空间里的元素迁移到新空间。理论上移动比拷贝更便宜，但容器还要保证异常安全。  
如果元素类型的移动构造可能抛异常，那么迁移过程中一旦抛了，容器会更难维持强异常安全保证。于是实现往往会优先使用拷贝，或者在某些条件下不得不承担更复杂的异常处理。  

所以对于一个“移动本来就只是转移资源指针/句柄”的类型，通常移动构造和移动赋值都应该是 `noexcept`。这样标准库就更放心地在扩容、重排时使用移动路径。  

你可以这样总结这题：  
- `noexcept` 不只是语法修饰。  
- 它会直接影响标准库容器是否采用更高效的移动路径。  
- 因此在自定义资源管理类里，这是很实用的性能点。  

**如果一个类型的移动操作本质不会失败，就应尽量标成 `noexcept`，否则你可能“写了移动语义，但容器不敢用”。**

**可能的追问问题：**  
- `vector` 为什么关心移动构造会不会抛异常？  
  **15秒简答：** 因为扩容时要迁移元素，容器要维持异常安全，如果移动可能抛异常就更难保证状态一致。  
- 所有移动构造都必须写 `noexcept` 吗？  
  **15秒简答：** 不是必须，但如果语义上确实不会失败，建议写，这样更利于标准库优化。  
- 如果不写 `noexcept`，最常见后果是什么？  
  **15秒简答：** 某些容器操作可能退回复制而不是移动，性能下降。

# STL容器与工程选型

## vector、list、deque 分别适合什么场景？后端开发里怎么选？
**15秒简答：**  
默认首选 `vector`，因为连续内存、缓存友好、随机访问快；

`list` 适合已知迭代器位置下的频繁插删，但额外指针开销大、缓存不友好；

`deque` 适合首尾频繁插删又需要随机访问。工程里不要因为“插删 O(1)”就盲选 `list`，多数业务场景 `vector` 更快。

**3分钟详答：**  

- `vector` 尾插均摊 O(1)  
- `list` 任意位置插删 O(1)  
- `deque` 首尾插删 O(1)  
这些没错，但还不够工程化。  

真正做后端选型时，要关注的不只是渐进复杂度，还包括：  
1. **内存布局**  
   `vector` 连续存储，CPU 缓存友好；`list` 节点分散，遍历时 cache miss 多。  
2. **真实操作模式**  
   `list` 的 O(1) 插删前提通常是“你已经拿到了那个位置的迭代器”。如果还得先查找，整体不一定划算。  
3. **内存开销**  
   `list` 每个节点至少多两个指针，空间成本高。  
4. **接口需求**  
   需要随机访问就别选 `list`。需要稳定引用/迭代器时可以考虑 `list` 或节点型容器。  

- 默认先想 `vector`。  
- 只有当业务真的有大量中间插删、并且能长期持有有效位置时，才考虑 `list`。  
- 需要双端操作又想保留随机访问能力时考虑 `deque`。  

**可能的追问问题：**  
- 为什么很多时候 `vector` 比 `list` 更快？  
  **15秒简答：** 因为连续内存更缓存友好，虽然理论上中间插删慢，但真实 CPU 上遍历和搬移常常更高效。  
- `list` 的 O(1) 插删有什么前提？  
  **15秒简答：** 前提是已经拿到待插删位置的迭代器，否则先查找的成本可能很高。  
- `deque` 和 `vector` 最大区别是什么？  
  **15秒简答：** `deque` 擅长首尾插删且支持随机访问，但内存不要求整体连续，实现通常是分段存储。

## vector 的底层扩容机制是什么？reserve 和 resize 有什么区别？
**15秒简答：** 
`vector` 空间不够时会申请更大的新内存，把旧元素搬过去，再释放旧空间，所以可能导致迭代器、指针、引用失效。`reserve` 只改容量不改元素个数，`resize` 会改变 size，必要时还会默认构造新元素。

**3分钟详答：** 
`vector` 的核心特点是**连续内存**。正因为要求连续，所以当现有容量不够放新元素时，它不能像链表那样随便挂个节点，而是要：  

1. 申请一块更大的连续内存  
2. 把旧元素迁移过去（拷贝或移动）  
3. 释放旧内存  
4. 更新内部指针  

这就解释了三个经典现象：  
- 扩容有成本  
- 扩容后地址可能变  
- 迭代器/引用/指针可能失效  

至于 `reserve` 和 `resize`：  
- `reserve(n)`：提前预留至少 n 的容量，`size` 不变，不会凭空多出元素。适合已知数据规模时减少扩容次数。  
- `resize(n)`：把逻辑元素个数改成 n。若 n 变大，会新增元素并进行默认构造或值初始化；若 n 变小，只是减少有效元素个数。  

后端场景里最好补一句：  
**如果你大致知道元素规模，提前 `reserve` 能显著减少扩容和搬运成本。**  
再加一句更深的：  
如果元素类型的移动构造是 `noexcept`，扩容迁移通常更容易走移动路径。  

**可能的追问问题：**  

- 扩容后为什么迭代器会失效？  
  **15秒简答：** 因为元素可能整体搬到新内存，原先地址不再有效。  
- `reserve` 会创建新元素吗？  
  **15秒简答：** 不会，它只扩容量，不改 `size`。  
- `resize` 把大小变大时会发生什么？  
  **15秒简答：** 会新增元素，内置类型值初始化，自定义类型会调用相应构造函数。

## map 和 unordered_map 有什么区别？线上如何选？
**15秒简答：**  
`map` 通常基于红黑树，键有序，查找/插入/删除是 O(log n)，支持范围查询；`unordered_map` 基于哈希表，平均 O(1)，但无序，最坏可能退化，还会有 rehash 成本。需要顺序、上下界、范围操作选 `map`；只关心高频单点查找、且 key 分布合适时优先 `unordered_map`。

**3分钟详答：**  
这是后端实习岗几乎必问题，常常还会问“工程里怎么选，而不是书上怎么选”。  
先讲底层：  

- `map` 是关联有序容器，通常用红黑树实现。  
- `unordered_map` 是无序关联容器，通常用哈希表实现。  

因此它们差异主要在四个维度：  
1. **顺序性**  
   `map` 天然按 key 有序，支持 `lower_bound`、区间遍历；`unordered_map` 不保证顺序。  
2. **复杂度**  
   `map` 稳定 O(log n)；`unordered_map` 平均 O(1)，但冲突严重时会退化。  
3. **空间与常数项**  
   哈希表常有桶数组、负载因子、rehash，空间和常数可能更大。  
4. **稳定性与场景特性**  
   对范围查询、排序输出、前缀区间等，`map` 更自然。对高频点查、缓存索引等，`unordered_map` 更常用。  

面试里最加分的是别说“unordered_map 一定更快”。因为真实线上选型还要看：  
- key 分布是否均匀  
- 是否需要自定义哈希  
- 是否担心冲突退化  
- 是否会频繁 rehash  
- 是否关心遍历顺序  
- 内存是否紧张  

成熟回答通常是：  
**没有绝对更优，只有更贴近业务访问模式的容器。**

**可能的追问问题：**  
- `unordered_map` 为什么会退化？  
  **15秒简答：** 因为大量哈希冲突会让桶内元素过多，查找接近线性。  
- 什么是 rehash？  
  **15秒简答：** 当负载因子过高时，哈希表会扩桶并重新分布元素，这会带来额外成本。  
- 什么时候 `map` 明显更合适？  
  **15秒简答：** 需要有序遍历、范围查找、上下界查询时，`map` 更适合。

## set/map 的底层为什么常说是红黑树？红黑树相对普通 BST 的价值是什么？
**15秒简答：**  
因为红黑树是一种近似平衡的二叉搜索树，**能把查找、插入、删除稳定控制在 O(log n)。**

相比普通 BST，不容易因为数据有序插入而退化成链表。

**3分钟详答：**  

普通二叉搜索树理论上查找很快，但如果插入序列接近有序，它就可能退化成一条链，复杂度变成 O(n)。这对标准库容器是不够稳定的。  
红黑树通过节点颜色和几条平衡规则，保证树的高度不会太离谱，因此查找、插入、删除都能维持对数级复杂度。  

为什么 STL 喜欢这种结构？因为关联容器需要：  
- 动态插入删除  
- 保持有序  
- 提供稳定复杂度  
红黑树恰好平衡了这三点。  

面试里不需要背所有旋转细节，但最好能讲出它的工程价值：  
- 它不是“绝对最平衡”的树，但调整成本相对可控。  
- 相比 AVL 树，红黑树在插入删除时通常旋转更少，更适合通用容器场景。  

如果你再补一句“STL 标准并不强制 `map` 一定是红黑树，但常见实现通常如此”，会显得更严谨。

**可能的追问问题：**  
- 红黑树为什么不是严格平衡树？  
  **15秒简答：** 它只保证**近似平衡**，通过约束黑高把高度控制在对数级，不追求每层绝对平衡。  
- 为什么普通 BST 会退化？  
  **15秒简答：** 因为若按有序序列插入，节点会一边倒，树高接近 n。  
- 为什么说红黑树适合关联容器？  
  **15秒简答：** 因为它支持动态维护有序性，同时查改删复杂度比较稳定。

# STL迭代器、失效与算法题

## 什么是迭代器失效？不同容器下有哪些典型场景？
**15秒简答：**  
**迭代器失效就是容器结构变化后，原迭代器、指针或引用不再能安全访问原元素。**

`vector` 扩容最典型，会导致全部迭代器失效；`erase` 往往会让删除点及其后的迭代器失效；`list`/`map` 这类节点型容器通常更稳定，删除一般只影响被删元素本身。

**3分钟详答：**  

核心原因是： 
容器修改后，元素位置、内部缓冲区或节点关系发生变化，原来的迭代器不一定还指向有效位置。  

最常见规律：  
- `vector`：  
  - 扩容时几乎所有迭代器、引用、指针都失效。  
  - `erase` 后，被删位置及后面的迭代器失效。  
- `deque`：  
  - 规则比 `vector` 更复杂，中间插删常影响更大。  
- `list`：  
  - 节点独立存储，插入通常不影响其他迭代器；删除只使被删元素迭代器失效。  
- `map/set`：  
  - 插入通常不让已有迭代器失效；删除通常只影响被删元素。  
- `unordered_map`：  
  - 插入如果触发 rehash，迭代器会整体失效。  


1. 了解容器规则，不拍脑袋写。  
2. 对 `vector` 预留容量，减少扩容。  
3. 删除时使用 `erase` 返回的迭代器更新当前位置。  
4. 需要长期持有稳定位置时考虑节点型容器。  

这个题如果答得好，会让面试官觉得你不是只会刷题，而是知道 STL 真正危险在哪。

**可能的追问问题：**  

- 为什么 `vector` 扩容会导致全部迭代器失效？  
  **15秒简答：** 因为元素整体搬到新连续内存，旧地址都失效了。  
- `map` 插入通常为什么不影响已有迭代器？  
  **15秒简答：** 因为它是节点型结构，插入只调整树链接，不会整体搬迁已有节点。  
- `unordered_map` 哪种场景下最要小心迭代器失效？  
  **15秒简答：** 触发 rehash 时，所有迭代器都可能失效。

## 一边遍历一边删除元素，正确写法是什么？
**15秒简答：**  
不能删完还继续用旧迭代器。对 `vector/list/map` 等，通常写成 `it = c.erase(it);`，因为 `erase` 会返回下一个有效迭代器；如果没删当前元素才 `++it`。

**3分钟详答：**  
这个题非常贴近编码实战。  
很多人第一反应是：

```cpp
for (auto it = v.begin(); it != v.end(); ++it) {
    if (need_delete(*it)) v.erase(it);
}
```
这通常有问题，因为 `erase(it)` 后，`it` 本身可能已经失效，循环里的 `++it` 再用它就是未定义行为。  

正确思路是：  
- 删除当前元素后，立刻接住 `erase` 返回的“下一个有效迭代器”。  
- 如果当前不删除，再正常自增。  

典型写法：
```cpp
for (auto it = c.begin(); it != c.end(); ) {
    if (need_delete(*it)) it = c.erase(it);
    else ++it;
}
```

这个模式特别适合面试，因为它体现的是：  
1. 你理解迭代器失效。  
2. 你知道 STL 设计接口的用意。  

再补一句区分会更完整：  
- 对 `vector`，删除后后续元素前移，所以更要依赖返回值。  
- 对 `map/set`，虽然通常只删当前节点，但统一使用返回值是最稳妥的写法。  

**可能的追问问题：**  
- 为什么 `erase` 要返回迭代器？  
  **15秒简答：** 因为删除后原迭代器可能失效，返回值让你能继续安全遍历。  
- `remove` 和 `erase` 是一回事吗？  
  **15秒简答：** 不是，`remove` 主要是算法层“重排保留元素”，真正删容器元素还要结合 `erase`。  
- `list` 删除时是不是就可以随便写？  
  **15秒简答：** 也不建议，虽然它更稳定，但接住 `erase` 返回值仍是最通用安全的写法。

## STL 中 remove/erase 惯用法是什么？为什么很多人会写错？
**15秒简答：**  
`std::remove` 不会真的删容器元素，它只是把“不需要删除的元素”搬到前面，并返回新的逻辑末尾；

真正删除还要调用容器的 `erase`。所以常见写法是 `v.erase(std::remove(v.begin(), v.end(), x), v.end());`。

**3分钟详答：**  
这是一个特别能区分“只会 API”还是“理解算法与容器分工”的题。  
`remove` 是 `<algorithm>` 里的通用算法，它并不知道底层容器怎么真正释放元素，也不会改容器 `size`。它做的是：  

- 遍历区间  
- 把需要保留的元素前移  
- 返回“新逻辑结尾”迭代器  

因此对 `vector/string/deque` 等容器，如果你只调用 `remove`，容器长度其实没变，后面那一段只是变成了“待忽略垃圾区”。  
所以必须再调用容器的 `erase` 把尾部真正删掉，这就是 erase-remove idiom。  

面试里最好补一句：  
为什么标准库要这样设计？因为算法和容器是分离的。`remove` 只依赖迭代器，不依赖具体容器类型，这体现了 STL 的泛型设计思想。  

如果能说到这一步，面试官通常会觉得你对 STL 的理解比较到位。

**可能的追问问题：**  

- 为什么 `remove` 不直接删元素？  
  **15秒简答：** 因为它是通用算法，只操作迭代器区间，不拥有具体容器的缩容能力。  
- `list` 也一定要用 `std::remove + erase` 吗？  
  **15秒简答：** 不一定，`list` 自己有成员函数 `remove`，能直接按节点删除。  
- 这个惯用法最容易犯什么错？  
  **15秒简答：** 只写了 `remove`，没写 `erase`，结果以为删了其实容器大小没变。

## 迭代器和指针一样吗？为什么 STL 需要迭代器这一层抽象？
**15秒简答：**  
不完全一样。指针是地址，迭代器是“**统一访问容器元素的位置抽象**”。**连续容器的迭代器可能像指针，但树、链表、哈希表容器显然不能只靠原始指针统一表示**。STL 用迭代器把算法和容器解耦。

**3分钟详答：**  
如果只有数组，指针足够表示遍历位置；但 STL 里有 `vector`、`list`、`map`、`unordered_map` 等完全不同结构。  
它们底层分别可能是：  

- 连续数组  
- 双向链表  
- 红黑树  
- 哈希桶  
显然没法要求所有容器都暴露原始地址来让算法工作。  

于是 STL 引入迭代器，作为一种统一接口：  
- `*it` 取元素  
- `++it` 前进  
- 某些容器支持 `--it`  
- 某些容器支持随机跳转 `it + n`  

这样算法只依赖“迭代器能力”，比如随机访问迭代器、双向迭代器，而不依赖具体容器类型。  
这就是为什么 `sort` 需要随机访问迭代器，所以可以作用于 `vector` 但不能直接作用于 `list`。  

面试里如果你能顺手讲出“STL 的核心是容器、算法、迭代器三者解耦”，会很加分。

**可能的追问问题：**  
- 为什么 `sort` 不能直接排序 `list`？  
  **15秒简答：** 因为 `sort` 需要随机访问迭代器，而 `list` 只有双向迭代器。  
- `vector` 的迭代器为什么常常看起来像指针？  
  **15秒简答：** 因为它底层是连续内存，实现上迭代器通常可以很接近原生指针。  
- STL 为什么强调“算法和容器分离”？  
  **15秒简答：** 这样同一套算法可以复用到不同容器上，体现泛型编程的价值。

# 函数对象、lambda 与模板意识

## lambda 表达式有什么价值？在 STL 里常怎么用？
**15秒简答：**  
lambda 能就地定义短小可调用对象，特别适合配合 STL 算法写比较器、过滤条件、回调逻辑。

它减少样板代码，也更容易把局部上下文一起捕获进来。

**3分钟详答：**  
lambda 是 C++11 以后非常实用的特性，面试常问“你项目里怎么用过”。  
在 STL 里，很多算法都需要你提供一个“行为”：  

- `sort` 需要比较规则  

- `find_if` 需要判定条件  

- `for_each` 需要处理动作  

  如果每次都单独写一个函数或仿函数类，代码会很重。lambda 则允许你在调用点直接写出这段逻辑。  

它的工程价值在于：  
1. **局部性好**：逻辑就在使用点。  
2. **减少模板样板代码**。  
3. **能捕获上下文**：不像普通函数那样必须全靠参数传。  



**捕获方式要谨慎。**  
尤其在异步或延迟执行场景里，引用捕获很容易把已经失效的局部变量带进去。  

所以更成熟的回答是：  
lambda 很适合 STL 和回调，但要根据生命周期正确选择值捕获还是引用捕获。

**可能的追问问题：**  

- 值捕获和引用捕获怎么选？  
  **15秒简答：** 生命周期短、同步场景里引用捕获可减少拷贝；异步或延迟执行时更要小心，通常值捕获更安全。  
- lambda 本质上是什么？  
  **15秒简答：** **本质上是编译器生成的匿名函数对象，捕获的变量会成为它的成员。**  
- lambda 为什么适合 STL 算法？  
  **15秒简答：** 因为算法常需要短小的比较/过滤逻辑，lambda 写在调用点最直观。

## 模板的本质是什么？为什么说 STL 是模板库？
**15秒简答：** 
模板本质上是“**编译期代码生成机制**”，不是运行时泛型。

**编译器会根据具体类型实例化出对应代码**。

STL 的容器、算法、迭代器大量依赖模板，才能做到一套接口适配多种类型和容器。

**3分钟详答：** 
C++ 模板不是 Java/C# 那种偏运行时的泛型机制，它更像一份“类型相关的代码蓝图”。  
当你写：

```cpp
std::vector<int>
std::vector<std::string>
```
编译器会为不同类型实例化出对应版本。  

这带来两个好处：  
1. **零额外抽象成本**  
   编译后就是具体类型代码，不必为泛型引入统一基类或运行时装箱。  
2. **强大的复用能力**  
   同一个容器模板能支持任意元素类型；同一个算法模板能工作在不同迭代器上。  

这也是 STL 的基础：  
- 容器是模板  
- 算法是模板  
- 迭代器能力也通过模板约束组合起来  
所以 STL 才能做到“同一套 `sort/find/accumulate` 面向很多数据结构工作”。  

模板也会带来编译报错复杂、代码膨胀、调试难度增加等代价，所以工程里既要会用，也要懂得控制复杂度。

**可能的追问问题：**  
- 模板是编译期还是运行期机制？  
  **15秒简答：** 主要是编译期机制，编译器按具体类型实例化代码。  
- 为什么 STL 这么依赖模板？  
  **15秒简答：** 因为它要支持多类型、多容器复用，同时尽量不牺牲运行时性能。  
- 模板的工程代价有哪些？  
  **15秒简答：** 主要是报错长、编译慢、可能造成代码膨胀和阅读复杂度上升。

# C++语言基础补充

## C 和 C++ 有什么区别？

**15秒简答：**
C 是面向过程语言，C++ 在兼容 C 的基础上增加了面向对象、泛型编程、RAII、异常、函数重载、引用、模板和 STL 等机制。

面试里重点不要只说“C++ 多了类”，而要说 C++ 更强调**类型安全、资源生命周期管理和抽象能力**。

**3分钟详答：**
C 更偏底层过程式编程，核心是函数、指针、结构体和手动内存管理。C++ 在此基础上扩展了很多机制：

1. 支持面向对象，包括封装、继承和多态。
2. 支持泛型编程，模板和 STL 是典型代表。
3. 支持 RAII，可以把资源释放绑定到对象析构，减少泄漏。
4. 支持函数重载、引用、默认参数、命名空间等更强的类型系统能力。
5. 支持异常处理和更复杂的对象生命周期管理。

所以 C++ 不是简单的“C with class”，而是既保留底层控制能力，又提供更高层抽象能力的语言。

**可能的追问问题：**

- C++ 为什么兼容 C？
  **15秒简答：** 因为 C++ 最初是在 C 的基础上发展出来的，兼容 C 可以复用大量已有代码、库和系统接口。
- C++ 的面向对象三大特性是什么？
  **15秒简答：** 封装、继承、多态。封装隐藏实现，继承复用和扩展类型，多态允许通过基类接口调用派生类实现。
- 什么是零成本抽象？
  **15秒简答：** 指 C++ 的抽象机制尽量在编译期完成，运行时开销接近手写底层代码。不是完全没有成本，而是不用时不付费，用时尽量高效。

## struct 和 class 有什么区别？

**15秒简答：**
在 C++ 中，`struct` 和 `class` 的主要区别是默认访问权限不同。`struct` 默认是 `public`，`class` 默认是 `private`。

除此之外，二者都可以有成员函数、构造函数、析构函数、继承和虚函数。

**3分钟详答：**
C++ 里的 `struct` 不再只是 C 语言中的数据集合，它也可以定义成员函数、构造函数、析构函数、继承关系和虚函数。

区别主要有两个：

1. `struct` 成员默认是 `public`。
2. `class` 成员默认是 `private`。
3. `struct` 默认 public 继承，`class` 默认 private 继承。

工程上一般约定：

- 如果只是简单数据聚合，使用 `struct`。
- 如果需要强调封装和行为，使用 `class`。

所以二者语法能力几乎一样，但语义习惯不同。

**可能的追问问题：**

- C++ 的 struct 和 C 的 struct 一样吗？
  **15秒简答：** 不一样。C 的 `struct` 只能放数据成员，C++ 的 `struct` 还可以有成员函数、构造析构、继承和虚函数。
- 什么时候用 struct，什么时候用 class？
  **15秒简答：** 简单数据结构用 `struct`，强调封装和对象行为时用 `class`。
- POD 是什么？
  **15秒简答：** POD 是 Plain Old Data，表示接近 C 风格的简单数据类型，通常没有复杂构造、析构和虚函数，适合和 C 接口或二进制数据交互。

## const 的各种用法怎么区分？

**15秒简答：**
`const` 修饰谁，谁就不能改。

`const int* p` 表示不能通过 `p` 修改指向的值；
`int* const p` 表示 `p` 本身不能改指向；
`const int* const p` 表示指针本身和指向的值都不能改。

**3分钟详答：**
`const` 的核心语义是只读约束。常见写法有：

```cpp
const int* p;        // 指向常量的指针
int* const p;        // 常量指针
const int* const p;  // 指向常量的常量指针
```

还可以修饰成员函数：

```cpp
class A {
public:
    int get() const;
};
```

`const` 成员函数表示该函数不会修改对象的普通成员变量，因此 `const` 对象只能调用 `const` 成员函数。

面试里要注意区分：

- 底层 const：修饰指向的对象。
- 顶层 const：修饰变量本身。

比如 `int* const p` 的 const 是顶层 const，因为指针变量本身不能改。

**可能的追问问题：**

- const 成员函数里能改成员变量吗？
  **15秒简答：** 普通成员变量不能改，但被 `mutable` 修饰的成员变量可以改。
- mutable 是什么？
  **15秒简答：** `mutable` 修饰的成员变量即使在 `const` 成员函数中也能修改，常用于缓存、计数器、懒加载标志等不影响逻辑语义的字段。
- const 对象只能调用什么成员函数？
  **15秒简答：** 只能调用 `const` 成员函数，因为普通成员函数可能修改对象。
- const int* 和 int const* 一样吗？
  **15秒简答：** 一样，都是指向常量 int 的指针，不能通过该指针修改值。

## const 和 constexpr 有什么区别？

**15秒简答：**
`const` 表示变量不能被修改，但不一定是编译期常量；`constexpr` 表示该值或函数可以在编译期求值。

简单说，`const` 强调只读，`constexpr` 强调编译期计算。

**3分钟详答：**
`const` 变量可能是运行期初始化的：

```cpp
int x;
cin >> x;
const int a = x;
```

这里 `a` 是只读变量，但不是编译期常量。

`constexpr` 要求表达式能在编译期计算：

```cpp
constexpr int n = 10;
int arr[n];
```

`constexpr` 常用于：

1. 数组大小。
2. 模板参数。
3. 编译期计算。
4. 替代宏常量。

相比宏，`constexpr` 有类型检查、作用域规则，也更安全。

**可能的追问问题：**

- constexpr 函数有什么特点？
  **15秒简答：** 它可以在编译期求值，也可以在运行期调用。只要实参是编译期常量，就可能在编译期计算。
- constexpr 和宏常量相比有什么优势？
  **15秒简答：** `constexpr` 有类型检查和作用域规则，宏只是文本替换，容易产生副作用和调试困难。
- const 一定是编译期常量吗？
  **15秒简答：** 不一定。运行期初始化的 `const` 变量只是只读，不是编译期常量。

## static 有哪些用法？

**15秒简答：**
`static` 有四种常见用法：

局部静态变量延长生命周期；
全局静态变量或函数限制文件作用域；
静态成员变量属于类而不是对象；
静态成员函数没有 `this` 指针。

**3分钟详答：**
`static` 在不同位置含义不同。

1. 修饰局部变量：
   变量只初始化一次，生命周期到程序结束，但作用域仍在函数内。
2. 修饰全局变量或函数：
   限制链接属性，只在当前文件可见，避免和其他文件符号冲突。
3. 修饰类成员变量：
   所有对象共享一份数据，不属于某个具体对象。
4. 修饰类成员函数：
   属于类本身，没有 `this` 指针，所以不能直接访问普通成员变量。

面试里可以总结为：

**static 要么改变生命周期，要么改变链接属性，要么表示属于类而不是对象。**

**可能的追问问题：**

- 静态局部变量什么时候初始化？
  **15秒简答：** 第一次执行到定义语句时初始化，只初始化一次，生命周期到程序结束。
- C++11 后局部静态变量初始化是否线程安全？
  **15秒简答：** 是线程安全的。多个线程同时进入时，编译器会保证只初始化一次。
- 静态成员函数为什么不能访问普通成员变量？
  **15秒简答：** 因为静态成员函数没有 `this` 指针，而普通成员变量必须依赖具体对象访问。
- 静态成员变量在哪里初始化？
  **15秒简答：** 传统写法是在类外定义和初始化；C++17 后可以使用 `inline static` 在类内初始化。

## inline 和宏有什么区别？

**15秒简答：**
宏是预处理阶段的文本替换，没有类型检查；`inline` 是 C++ 函数机制，有类型检查、作用域和调试信息。

`inline` 只是建议编译器内联，不保证一定展开。

**3分钟详答：**
宏在预处理阶段展开，本质是简单替换：

```cpp
#define SQUARE(x) x * x
```

它没有类型检查，也容易出现优先级和重复求值问题。

`inline` 函数仍然是函数：

```cpp
inline int square(int x) {
    return x * x;
}
```

它有类型检查、作用域规则，也更容易调试。

`inline` 的另一个作用是允许函数定义放在头文件中，被多个翻译单元包含时不会产生重复定义错误。

但是否真正内联由编译器决定，不由关键字强制决定。

**可能的追问问题：**

- inline 函数一定会内联吗？
  **15秒简答：** 不一定。是否内联由编译器根据函数复杂度、优化级别等因素决定。
- 类内定义的成员函数默认是 inline 吗？
  **15秒简答：** 是的，类内部直接定义的成员函数默认具有 inline 语义。
- inline 可以解决头文件重复定义问题吗？
  **15秒简答：** 可以用于函数定义。多个翻译单元包含同一个 inline 函数定义时，链接器允许合并。
- 宏有什么典型坑？
  **15秒简答：** 没有类型检查，容易有优先级问题和重复求值问题，比如 `SQUARE(a + b)` 展开后结果可能不符合预期。

## explicit 的作用是什么？

**15秒简答：**
`explicit` 用来禁止构造函数发生隐式类型转换，避免编译器自动把某个类型转换成类对象。

一般单参数构造函数建议加 `explicit`。

**3分钟详答：**
如果一个类有单参数构造函数：

```cpp
class A {
public:
    A(int x) {}
};
```

那么下面代码可能成立：

```cpp
A a = 10;
```

这其实发生了从 `int` 到 `A` 的隐式转换。很多时候这种转换并不是程序员真正想要的。

加上 `explicit` 后：

```cpp
class A {
public:
    explicit A(int x) {}
};
```

就只能显式构造：

```cpp
A a(10);
```

不能再隐式转换：

```cpp
A a = 10; // 错
```

它的价值是让代码意图更明确，减少隐藏 bug。

**可能的追问问题：**

- 哪些构造函数容易发生隐式转换？
  **15秒简答：** 单参数构造函数，或者除了第一个参数外其他参数都有默认值的构造函数。
- 为什么建议单参数构造函数加 explicit？
  **15秒简答：** 可以防止编译器进行不明显的自动类型转换，避免语义错误。
- explicit 可以用于转换函数吗？
  **15秒简答：** 可以。C++11 后转换运算符也可以加 `explicit`，例如 `explicit operator bool()`。

## nullptr 和 NULL 有什么区别？

**15秒简答：**
`NULL` 本质上通常是 `0` 或 `0L`，可能被当成整数；`nullptr` 是 C++11 引入的空指针字面量，类型是 `std::nullptr_t`，能更准确表示空指针。

**3分钟详答：**
`NULL` 在 C++ 里经常定义为整数 0，因此在函数重载时容易出问题：

```cpp
void f(int);
void f(int*);

f(NULL);     // 可能调用 f(int)
f(nullptr);  // 调用 f(int*)
```

`nullptr` 的类型是 `std::nullptr_t`，可以隐式转换成任意指针类型，但不会被当成普通整数。

所以现代 C++ 中表示空指针应优先使用 `nullptr`，而不是 `NULL` 或 `0`。

**可能的追问问题：**

- nullptr 的类型是什么？
  **15秒简答：** 是 `std::nullptr_t`。
- nullptr 能转换成 int 吗？
  **15秒简答：** 不能隐式转换成普通整数类型，所以能避免 `NULL` 的重载歧义。
- 为什么不推荐用 0 表示空指针？
  **15秒简答：** 因为 0 同时是整数，语义不清晰，在重载和模板场景中容易产生歧义。

## sizeof 和 strlen 有什么区别？

**15秒简答：**
`sizeof` 是运算符，返回对象或类型占用的字节数；`strlen` 是函数，计算 C 字符串中 `\0` 之前的字符个数。

**3分钟详答：**
例如：

```cpp
char arr[] = "abc";
```

则：

```cpp
sizeof(arr); // 4，包括 '\0'
strlen(arr); // 3，不包括 '\0'
```

`sizeof` 常在编译期确定结果，作用于数组时得到整个数组大小。
`strlen` 需要运行时从字符首地址开始遍历，直到遇到 `\0`。

如果是指针：

```cpp
char* p = arr;
sizeof(p); // 64位系统通常是8
strlen(p); // 3
```

所以 `sizeof` 关心对象存储大小，`strlen` 关心字符串逻辑长度。

**可能的追问问题：**

- 指针用 sizeof 得到什么？
  **15秒简答：** 得到指针变量本身大小，64 位系统通常是 8 字节。
- 数组作为函数参数会发生什么？
  **15秒简答：** 数组参数会退化成指针，所以函数里 `sizeof(arr)` 得到的是指针大小。
- sizeof(std::string) 是字符串长度吗？
  **15秒简答：** 不是。它是 `std::string` 对象本身大小，字符串长度要用 `.size()` 或 `.length()`。

------

## volatile 有什么作用？

**15秒简答：**
`volatile` 告诉编译器变量可能被程序控制流之外的因素修改，不要对它的读写做过度优化。

但 `volatile` 不能保证线程安全，不能替代锁或 `atomic`。

**3分钟详答：**
`volatile` 常见于：

1. 硬件寄存器。
2. 中断处理变量。
3. 信号处理相关变量。

它的核心作用是防止编译器把变量读写优化掉。例如编译器不能假设一个 `volatile` 变量的值在两次读取之间一定不变。

但在多线程中，`volatile` 不提供原子性，也不提供正确的同步顺序。所以多个线程读写共享变量时，应该使用 `std::atomic`、`mutex` 等同步机制。

**可能的追问问题：**

- volatile 能替代锁吗？
  **15秒简答：** 不能。它不保证原子性和线程同步。
- volatile 和 atomic 的区别是什么？
  **15秒简答：** `volatile` 主要限制编译器优化，`atomic` 提供原子操作和内存序语义。
- 嵌入式里为什么常见 volatile？
  **15秒简答：** 因为硬件寄存器或中断变量可能随时被外部改变，必须让编译器每次真实访问内存。

------

# 指针、数组与声明补充

## 指针数组和数组指针怎么区分？

**15秒简答：**
`int* arr[5]` 是指针数组，`arr` 先和 `[5]` 结合，说明它是数组，数组元素是 `int*`。

`int (*p)[5]` 是数组指针，`p` 先和 `*` 结合，说明它是指针，指向一个包含 5 个 int 的数组。

**3分钟详答：**
判断复杂声明时，从变量名开始看，优先看括号。

```cpp
int* arr[5];
```

`arr` 先和 `[5]` 结合，所以 `arr` 是一个数组。数组里每个元素类型是 `int*`，因此叫指针数组。

```cpp
int (*p)[5];
```

括号让 `p` 先和 `*` 结合，所以 `p` 是一个指针。它指向的对象是 `int[5]`，因此叫数组指针。

使用例子：

```cpp
int a[5] = {1, 2, 3, 4, 5};
int (*p)[5] = &a;

(*p)[0]; // 1
```

**可能的追问问题：**

- int (*p)[5] 怎么访问元素？
  **15秒简答：** 用 `(*p)[i]` 访问，因为 `p` 是指向数组的指针，先解引用得到数组。
- 二维数组传参为什么常写成 int (*p)[n]？
  **15秒简答：** 因为二维数组退化时只退化第一维，函数必须知道每一行有多少列。
- 数组名什么时候不会退化成指针？
  **15秒简答：** 在 `sizeof(arr)`、`&arr`、`decltype(arr)` 等场景下不会退化。

------

## 数组名和指针完全一样吗？

**15秒简答：**
不完全一样。数组名在大多数表达式中会退化成指向首元素的指针，但数组名本身不是普通指针。

`sizeof(arr)` 得到整个数组大小，`sizeof(p)` 得到指针大小。

**3分钟详答：**
例如：

```cpp
int arr[10];
int* p = arr;
```

在大多数表达式中，`arr` 会退化成 `&arr[0]`，所以可以赋给 `int*`。

但它们不是同一种东西：

```cpp
sizeof(arr); // 10 * sizeof(int)
sizeof(p);   // 指针大小，64位通常8字节
```

再比如：

```cpp
&arr
```

它的类型是 `int (*)[10]`，表示指向整个数组的指针；
而 `arr` 退化后是 `int*`，表示指向第一个元素的指针。

它们地址值可能相同，但类型和指针加一的跨度不同。

**可能的追问问题：**

- &arr 和 arr 有什么区别？
  **15秒简答：** `arr` 通常退化成首元素地址，类型是 `int*`；`&arr` 是整个数组的地址，类型是 `int (*)[N]`。
- 为什么函数参数里数组大小丢失？
  **15秒简答：** 因为数组作为函数参数会退化成指针，函数只拿到首元素地址，拿不到数组总长度。
- 如何让函数知道数组大小？
  **15秒简答：** 可以额外传长度、使用模板引用参数、`std::array` 或 `std::vector`。

------

## 野指针和悬空指针是什么？

**15秒简答：**
野指针是没有初始化或指向未知地址的指针；悬空指针是原来指向有效对象，但对象生命周期结束后仍保留旧地址的指针。

二者解引用都可能导致未定义行为。

**3分钟详答：**
野指针常见于：

```cpp
int* p;
*p = 10; // 危险
```

因为 `p` 没有初始化，不知道指向哪里。

悬空指针常见于：

```cpp
int* p = new int(10);
delete p;
*p = 20; // use after free
```

`delete` 释放了内存，但 `p` 仍保存旧地址。这个地址已经不再有效。

还有一种常见错误：

```cpp
int* f() {
    int x = 10;
    return &x;
}
```

返回局部变量地址，函数结束后 `x` 生命周期结束，返回的指针悬空。

**可能的追问问题：**

- 如何避免野指针？
  **15秒简答：** 指针定义时初始化，不使用时置为 `nullptr`，更推荐用智能指针和容器管理资源。
- delete p 后为什么建议 p = nullptr？
  **15秒简答：** 因为 `delete` 不会自动修改指针变量，置空可以避免后续误用旧地址。
- 返回局部变量地址为什么危险？
  **15秒简答：** 局部变量在函数返回后被销毁，返回地址会变成悬空指针。

------

## 函数指针怎么写？和回调有什么关系？

**15秒简答：**
函数指针保存函数入口地址，可以通过它间接调用函数。回调函数本质上就是把函数作为参数传给另一个模块，在合适时机由对方调用。

**3分钟详答：**
函数指针写法：

```cpp
int add(int a, int b) {
    return a + b;
}

int (*fp)(int, int) = add;
```

调用：

```cpp
int ret = fp(1, 2);
```

函数指针常用于回调。例如某个框架需要在事件发生时调用用户提供的函数，就可以保存函数指针。

现代 C++ 中更常用：

1. 函数指针。
2. 函数对象。
3. lambda。
4. `std::function`。

函数指针最轻量，但只能保存普通函数地址；`std::function` 更灵活，但有额外开销。

**可能的追问问题：**

- 函数指针和 std::function 有什么区别？
  **15秒简答：** 函数指针只能保存普通函数地址；`std::function` 可以包装函数、lambda、仿函数等，但有类型擦除开销。
- lambda 能不能转成函数指针？
  **15秒简答：** 没有捕获的 lambda 可以转换成函数指针，有捕获的 lambda 不能直接转换。
- 成员函数指针和普通函数指针一样吗？
  **15秒简答：** 不一样。成员函数需要依赖对象调用，内部涉及 `this` 指针，类型和调用方式都不同。

------

# 对象模型与面向对象补充

## 构造函数和析构函数的调用顺序是什么？

**15秒简答：**
构造顺序是：先构造基类，再构造成员变量，最后执行派生类构造函数体。

析构顺序相反：先执行派生类析构函数体，再析构成员变量，最后析构基类。

**3分钟详答：**
对象构造时，必须先把它依赖的部分构造好。

对于派生类对象：

1. 先构造基类子对象。
2. 再按照成员变量声明顺序构造成员变量。
3. 最后执行派生类构造函数体。

析构时顺序完全相反：

1. 先执行派生类析构函数体。
2. 再按照成员声明顺序的逆序析构成员变量。
3. 最后析构基类子对象。

注意，成员变量初始化顺序由**声明顺序**决定，不由初始化列表书写顺序决定。

**可能的追问问题：**

- 成员变量初始化顺序由什么决定？
  **15秒简答：** 由成员变量在类中的声明顺序决定。
- 初始化列表顺序重要吗？
  **15秒简答：** 实际初始化顺序不由列表书写顺序决定，但建议书写顺序和声明顺序一致，避免误解和潜在 bug。
- 构造函数里调用虚函数会发生多态吗？
  **15秒简答：** 不会表现出派生类多态，因为构造基类部分时派生类还没构造完成。

------

## 构造函数和析构函数中可以调用虚函数吗？

**15秒简答：**
语法上可以，但不建议。构造和析构期间调用虚函数不会表现出完整的动态多态，通常只会调用当前构造或析构层级的版本。

**3分钟详答：**
构造函数执行时，对象还没有完全构造完成。
如果基类构造函数中调用虚函数，此时派生类部分还没构造，所以不会分派到派生类版本。

析构函数中也类似。
当基类析构函数执行时，派生类部分已经析构完成，也不能再安全调用派生类版本。

所以构造和析构函数中调用虚函数容易产生误解，尤其是依赖派生类资源时可能出错。

面试中可以总结：

**构造和析构期间，对象的动态类型不应被当作完整派生类来依赖。**

**可能的追问问题：**

- 构造函数为什么不能是虚函数？
  **15秒简答：** 因为虚函数机制依赖对象已经构造完成，而构造函数执行时对象动态类型还不完整。
- 析构函数为什么可以是虚函数？
  **15秒简答：** 因为通过基类指针删除派生类对象时，需要根据真实类型调用完整析构链。
- 析构函数里调用虚函数会怎样？
  **15秒简答：** 通常只会调用当前析构层级的版本，不会再分派到已经析构掉的派生类部分。

------

## 重载、重写、隐藏有什么区别？

**15秒简答：**
重载是同一作用域中函数名相同、参数不同；
重写是派生类重新实现基类虚函数；
隐藏是派生类定义同名函数后，把基类同名函数隐藏掉。

**3分钟详答：**
重载 overload：

```cpp
void f(int);
void f(double);
```

同一作用域，函数名相同，参数列表不同。

重写 override：

```cpp
class Base {
public:
    virtual void f(int);
};

class Derived : public Base {
public:
    void f(int) override;
};
```

要求基类函数是虚函数，并且派生类函数签名匹配。

隐藏 hide：

```cpp
class Base {
public:
    void f(int);
};

class Derived : public Base {
public:
    void f(double);
};
```

派生类的 `f(double)` 会隐藏基类的 `f(int)`，即使参数不同。

C++11 后建议重写虚函数时加 `override`，让编译器检查是否真的重写成功。

**可能的追问问题：**

- 函数重载看返回值吗？
  **15秒简答：** 不看。不能只靠返回值不同构成重载。
- 重写必须参数完全一样吗？
  **15秒简答：** 基本要一致，包括函数名、参数列表、const 限定等，否则可能不是重写。
- 派生类同名函数为什么会隐藏基类函数？
  **15秒简答：** 因为名字查找会先在派生类作用域中找，同名函数会隐藏基类同名函数。

------

## override 和 final 有什么作用？

**15秒简答：**
`override` 表示派生类函数必须重写基类虚函数，签名不匹配会编译报错。

`final` 可以禁止类继续被继承，或者禁止虚函数继续被重写。

**3分钟详答：**
`override` 的价值是让编译器帮忙检查：

```cpp
class Base {
public:
    virtual void f() const;
};

class Derived : public Base {
public:
    void f() override; // 报错，因为少了 const
};
```

如果不写 `override`，这个错误可能变成隐藏或新函数定义，导致多态失效。

`final` 可以修饰类：

```cpp
class A final {};
```

表示 A 不能再被继承。

也可以修饰虚函数：

```cpp
virtual void f() final;
```

表示后续派生类不能继续重写该函数。

**可能的追问问题：**

- 为什么推荐写 override？
  **15秒简答：** 可以让编译器检查是否真的重写了基类虚函数，避免签名写错。
- final 对优化有没有帮助？
  **15秒简答：** 可能有。编译器知道函数不会再被重写时，某些情况下可以进行去虚化优化。
- final 修饰类和修饰函数有什么区别？
  **15秒简答：** 修饰类表示类不能被继承；修饰函数表示该虚函数不能继续被重写。

------

## 虚继承解决什么问题？

**15秒简答：**
虚继承主要解决菱形继承中公共基类被重复继承的问题，保证最终派生类中只有一份虚基类子对象。

**3分钟详答：**
菱形继承结构：

```cpp
class A {};
class B : public A {};
class C : public A {};
class D : public B, public C {};
```

此时 `D` 中会有两份 `A` 子对象，一份来自 `B`，一份来自 `C`。
访问 `A` 的成员时可能产生二义性。

虚继承写法：

```cpp
class A {};
class B : virtual public A {};
class C : virtual public A {};
class D : public B, public C {};
```

这样 `D` 中只有一份共享的 `A` 子对象。

代价是对象布局更复杂，可能引入虚基类指针或偏移信息。

**可能的追问问题：**

- 菱形继承为什么会有二义性？
  **15秒简答：** 因为最终派生类中可能有两份公共基类子对象，访问公共基类成员时不知道使用哪一份。
- 虚继承对对象布局有什么影响？
  **15秒简答：** 会引入额外的虚基类指针或偏移信息，对象布局更复杂，访问可能有额外开销。
- 虚基类由谁负责构造？
  **15秒简答：** 由最底层的最终派生类负责构造。

------

## 空类大小为什么不是 0？

**15秒简答：**
空类对象大小至少是 1 字节，因为 C++ 要保证不同对象具有不同地址。

如果空类大小为 0，多个对象地址就无法区分。

**3分钟详答：**
例如：

```cpp
class A {};
sizeof(A); // 通常是 1
```

虽然 `A` 没有任何成员，但对象必须能被取地址：

```cpp
A a1, a2;
&a1 != &a2;
```

所以编译器至少给它分配 1 字节。

如果类中有虚函数：

```cpp
class B {
public:
    virtual void f();
};
```

对象中通常会多一个虚表指针 `vptr`，所以 64 位系统下 `sizeof(B)` 常见是 8 字节。

静态成员变量不属于单个对象，不计入对象大小。

**可能的追问问题：**

- 类里有虚函数后大小是多少？
  **15秒简答：** 通常至少包含一个虚表指针，64 位系统常见是 8 字节，但具体取决于编译器和对齐。
- 类里有静态成员变量会不会增加对象大小？
  **15秒简答：** 不会。静态成员属于类，不属于具体对象。
- padding 是什么？
  **15秒简答：** 编译器为了满足内存对齐要求，在成员之间或对象末尾插入的空字节。

------

## this 指针是什么？

**15秒简答：**
`this` 是非静态成员函数隐含的指针，指向当前调用该成员函数的对象。

静态成员函数没有 `this` 指针。

**3分钟详答：**
当调用：

```cpp
obj.f();
```

实际可以理解为：

```cpp
A::f(&obj);
```

普通成员函数内部能访问成员变量，是因为编译器隐式传入了 `this` 指针。

```cpp
class A {
    int x;
public:
    void set(int v) {
        this->x = v;
    }
};
```

静态成员函数属于类本身，不依赖具体对象，所以没有 `this` 指针，也不能直接访问普通成员变量。

**可能的追问问题：**

- this 可以为 nullptr 吗？
  **15秒简答：** 正常情况下不应该为空。通过空指针调用非静态成员函数是未定义行为。
- 静态成员函数为什么没有 this？
  **15秒简答：** 因为静态成员函数属于类，不属于某个具体对象。
- 成员函数如何区分不同对象的数据？
  **15秒简答：** 通过隐式传入的 `this` 指针访问当前对象的数据成员。

------

## friend 友元是什么？

**15秒简答：**
`friend` 可以让某个函数或类访问当前类的私有成员和保护成员。

它适合紧密协作场景，但会破坏封装，不能滥用。

**3分钟详答：**
友元函数不是类成员，但可以访问类的私有数据：

```cpp
class A {
private:
    int x;
public:
    friend void print(const A& a);
};

void print(const A& a) {
    cout << a.x;
}
```

常见场景是重载 `operator<<`：

```cpp
friend ostream& operator<<(ostream& os, const A& a);
```

因为左操作数是 `ostream`，不能写成类成员函数，同时又可能需要访问类内部数据，所以常写成友元。

**可能的追问问题：**

- 友元函数是不是成员函数？
  **15秒简答：** 不是。它只是被授权访问类私有成员的普通函数。
- 友元关系能不能继承？
  **15秒简答：** 不能。友元关系不能继承，也不是双向的。
- 为什么 operator<< 常写成友元？
  **15秒简答：** 因为左操作数是 `ostream`，通常不能作为类成员函数实现。

------

# 内存管理补充

## 栈上对象和堆上对象有什么区别？

**15秒简答：**
栈上对象自动分配和释放，生命周期随作用域结束；堆上对象通过 `new` 或 `malloc` 动态分配，需要手动释放或交给智能指针管理。

**3分钟详答：**
栈对象：

```cpp
void f() {
    A a;
}
```

函数结束后 `a` 自动析构，不需要手动释放。

堆对象：

```cpp
A* p = new A();
delete p;
```

生命周期更灵活，但容易出现内存泄漏、重复释放、悬空指针等问题。

所以现代 C++ 更推荐：

```cpp
auto p = std::make_unique<A>();
```

用 RAII 和智能指针管理堆资源。

**可能的追问问题：**

- 栈为什么快？
  **15秒简答：** 栈分配通常只是移动栈指针，释放时回退栈指针，成本很低。
- 堆为什么容易产生碎片？
  **15秒简答：** 堆内存频繁申请释放，不同大小空闲块分散，可能导致连续空间不足。
- 栈溢出和内存泄漏分别是什么？
  **15秒简答：** 栈溢出是栈空间耗尽，内存泄漏是堆内存申请后没有释放。

------

## C++ 程序内存分区有哪些？

**15秒简答：**
常见分区包括代码区、全局/静态区、常量区、堆区和栈区。

局部变量通常在栈上，`new` 出来的对象在堆上，全局变量和静态变量在静态存储区。

**3分钟详答：**
常见内存区域：

1. 代码区：存放程序指令。
2. 全局/静态区：存放全局变量、静态变量。
3. 常量区：存放字符串字面量、只读常量数据。
4. 堆区：动态申请的内存。
5. 栈区：函数参数、局部变量、返回地址等。

例如：

```cpp
int g = 10; // 全局/静态区

void f() {
    static int s = 1;     // 全局/静态区
    int x = 2;            // 栈区
    int* p = new int(3);  // p在栈上，*p在堆上
}
```

面试重点是区分**指针变量本身在哪里**和**指针指向的对象在哪里**。

**可能的追问问题：**

- 字符串字面量在哪里？
  **15秒简答：** 通常在只读常量区或只读数据段，不能修改。
- static 局部变量在哪里？
  **15秒简答：** 在静态存储区，不在栈上，生命周期持续到程序结束。
- new 出来的对象本体在哪里？指针变量在哪里？
  **15秒简答：** 对象本体在堆上，指针变量的位置取决于它自己的定义位置。

## new[]/delete[] 和 new/delete 为什么不能混用？

**15秒简答：**
`new[]` 分配数组并构造多个元素，必须用 `delete[]` 逐个析构并释放。

`new` 和 `delete`、`new[]` 和 `delete[]` 必须配对使用，混用是未定义行为。

**3分钟详答：**
例如：

```cpp
A* p = new A[10];
delete[] p;
```

`new A[10]` 会构造 10 个对象，`delete[]` 会调用 10 次析构函数。

如果错误写成：

```cpp
delete p;
```

可能只调用一次析构，或者破坏运行时记录的数组信息，属于未定义行为。

即使是内置类型数组，也必须配对：

```cpp
int* p = new int[10];
delete[] p;
```

因为分配和释放形式必须一致。

**可能的追问问题：**

- 内置类型数组也必须用 delete[] 吗？
  **15秒简答：** 必须。即使没有析构函数，也要保证分配释放形式匹配。
- malloc 出来的内存可以 delete 吗？
  **15秒简答：** 不可以。`malloc/free` 配对，`new/delete` 配对，混用是未定义行为。
- delete nullptr 安全吗？
  **15秒简答：** 安全。对空指针执行 `delete` 没有效果。

## placement new 是什么？

**15秒简答：**
placement new 是在一块已经分配好的内存上构造对象，不负责分配内存。

常用于内存池、对象池、STL 容器底层实现。

**3分钟详答：**
普通 `new` 做两件事：

1. 分配内存。
2. 构造对象。

placement new 只做第二件事：

```cpp
char buffer[sizeof(A)];
A* p = new (buffer) A();
p->~A();
```

这里内存来自 `buffer`，placement new 只是在这块内存上调用构造函数。

由于它不负责释放内存，所以对象不用时需要手动调用析构函数：

```cpp
p->~A();
```

这体现了对象生命周期和内存分配可以分离。

**可能的追问问题：**

- placement new 后为什么要手动调用析构？
  **15秒简答：** 因为它只构造对象，不管理内存释放，对象生命周期结束要手动析构。
- 它和普通 new 的区别是什么？
  **15秒简答：** 普通 new 分配内存并构造对象；placement new 在已有内存上构造对象。
- STL allocator 和 placement new 有什么关系？
  **15秒简答：** 容器底层常先分配原始内存，再用 placement new 在指定位置构造元素。

## 内存泄漏怎么排查？

**15秒简答：**
内存泄漏是堆内存申请后没有释放。排查时可以结合 RAII、智能指针、代码审查、日志、Valgrind、ASan、Visual Studio 诊断工具等。

**3分钟详答：**
内存泄漏常见原因：

1. `new` 后忘记 `delete`。
2. 异常或提前 return 导致释放逻辑没执行。
3. 容器中保存裸指针但没有释放指向对象。
4. `shared_ptr` 循环引用。

排查思路：

1. 看内存是否持续增长。
2. 检查所有权关系是否清晰。
3. 用 ASan、Valgrind 等工具定位泄漏位置。
4. 用智能指针和 RAII 改造资源管理。

根本解决方式是减少裸 `new/delete`，让对象生命周期自动管理。

**可能的追问问题：**

- 什么是 double free？
  **15秒简答：** 同一块内存被释放两次，会导致未定义行为，可能破坏堆结构。
- 什么是 use after free？
  **15秒简答：** 内存释放后仍然通过旧指针访问它。
- ASan 能查什么问题？
  **15秒简答：** 可以检测越界访问、释放后使用、重复释放、栈/堆缓冲区溢出等问题。

# STL补充

## push_back 和 emplace_back 有什么区别？

**15秒简答：**
`push_back` 是把已有对象拷贝或移动进容器；`emplace_back` 是在容器尾部直接用参数原地构造对象。

`emplace_back` 可能减少临时对象和移动成本，但不一定永远更快。

**3分钟详答：**
例如：

```cpp
vector<pair<int, string>> v;
v.push_back(pair<int, string>(1, "abc"));
v.emplace_back(1, "abc");
```

`push_back` 需要先构造一个临时 `pair`，再移动或拷贝进容器。
`emplace_back` 直接把参数传给构造函数，在容器内部原地构造。

但如果已经有一个对象：

```cpp
A a;
v.push_back(a);
v.emplace_back(a);
```

这时 `emplace_back` 不一定有明显优势。

所以面试里不要绝对说 `emplace_back` 一定更快，而要看传入参数和对象类型。

**可能的追问问题：**

- emplace_back 一定比 push_back 快吗？
  **15秒简答：** 不一定。只有能避免临时对象或额外移动时才更有优势。
- 什么场景差异明显？
  **15秒简答：** 构造成本高、参数较多、对象不可拷贝或希望原地构造时差异更明显。
- emplace_back 会导致扩容吗？
  **15秒简答：** 会。如果容量不足，仍然会触发 vector 扩容和元素搬移。

------

## vector 的 size 和 capacity 有什么区别？

**15秒简答：**
`size` 是当前实际元素个数；`capacity` 是当前已分配空间最多能容纳的元素个数。

`reserve` 改容量，不改元素个数；`resize` 改元素个数。

**3分钟详答：**
例如：

```cpp
vector<int> v;
v.reserve(100);
```

此时：

```cpp
v.size();     // 0
v.capacity(); // 至少100
```

`reserve` 只是提前申请空间，不会创建元素。

```cpp
v.resize(100);
```

此时 `size` 变成 100，会真正创建元素。

`capacity` 的存在是为了减少频繁扩容。如果每次插入都重新分配，效率很低，所以 vector 会预留更多空间。

**可能的追问问题：**

- clear 会不会释放 capacity？
  **15秒简答：** 通常不会。`clear` 只清空元素，`capacity` 一般保留。
- 如何释放 vector 多余容量？
  **15秒简答：** 可以用 `vector<T>().swap(v)`，也可以尝试 `v.shrink_to_fit()`。
- shrink_to_fit 一定释放内存吗？
  **15秒简答：** 不一定。它只是请求，具体是否释放由标准库实现决定。

------

## vector 里存指针要注意什么？

**15秒简答：**
`vector<T*>` 只管理指针本身，不管理指针指向的对象。

如果对象是 `new` 出来的，删除 vector 元素不会自动释放对象，更推荐使用 `vector<unique_ptr<T>>`。

**3分钟详答：**
例如：

```cpp
vector<A*> v;
v.push_back(new A());
v.clear();
```

`clear` 只删除 vector 里的指针值，不会自动 `delete` 指针指向的对象，所以会内存泄漏。

更安全的写法：

```cpp
vector<unique_ptr<A>> v;
v.push_back(make_unique<A>());
```

这样 vector 清空或析构时，`unique_ptr` 会自动释放对象。

如果对象需要共享所有权，可以用 `shared_ptr`，但要注意循环引用和生命周期复杂度。

**可能的追问问题：**

- vector<T*> 删除元素会不会自动 delete？
  **15秒简答：** 不会。删除的是指针值，不是指针指向的对象。
- vector<unique_ptr> 能不能拷贝？
  **15秒简答：** 不能拷贝，因为 `unique_ptr` 独占所有权，只能移动。
- 智能指针放进容器要注意什么？
  **15秒简答：** `unique_ptr` 要用移动语义，`shared_ptr` 要注意共享所有权和循环引用。

## string 的底层是什么？SSO 是什么？

**15秒简答：**
`std::string` 通常管理一段字符缓冲区。很多实现有 SSO，也就是小字符串优化，短字符串直接存在 string 对象内部，避免堆分配。

**3分钟详答：**
`std::string` 和 `char*` 最大区别是：

- `string` 自动管理内存。
- 支持拷贝、移动、拼接、比较、长度查询。
- 不需要手动维护 `\0` 和释放内存。

SSO 是 Small String Optimization。
当字符串很短时，不单独在堆上分配内存，而是直接把内容存在 `string` 对象内部的小缓冲区中。

这样可以减少小字符串频繁堆分配，提高性能。

但 SSO 是实现优化，不同标准库实现细节可能不同。

**可能的追问问题：**

- string 和 char* 区别是什么？
  **15秒简答：** `string` 自动管理内存并提供丰富接口；`char*` 只是字符指针，需要手动管理内存和结尾。
- c_str() 返回的指针什么时候失效？
  **15秒简答：** 当 string 被修改、扩容或析构后，之前的 `c_str()` 指针可能失效。
- string 是否以 \0 结尾？
  **15秒简答：** `c_str()` 返回的字符数组以 `\0` 结尾，但 string 本身也可以包含中间的 `\0` 字符。

## set 和 unordered_set 有什么区别？

**15秒简答：**
`set` 通常基于红黑树，元素有序，查找、插入、删除是 O(log n)。

`unordered_set` 基于哈希表，元素无序，平均 O(1)，最坏 O(n)。

**3分钟详答：**
`set` 的特点：

1. 自动排序。
2. 支持有序遍历。
3. 支持 `lower_bound`、`upper_bound`。
4. 复杂度稳定 O(log n)。

`unordered_set` 的特点：

1. 不保证顺序。
2. 平均查找更快。
3. 依赖哈希函数和负载因子。
4. 冲突严重时可能退化。

如果需要顺序或范围操作，选 `set`。
如果只需要高频查找并且 key 哈希分布较好，选 `unordered_set`。

**可能的追问问题：**

- 为什么 set 不能直接修改元素值？
  **15秒简答：** 因为元素值决定树中的排序位置，直接修改会破坏有序结构。
- unordered_set 什么时候会 rehash？
  **15秒简答：** 当元素增多导致负载因子超过阈值时，会扩桶并重新分布元素。
- 自定义类型放进 unordered_set 需要什么？
  **15秒简答：** 需要提供哈希函数和相等比较函数。

## priority_queue 底层是什么？

**15秒简答：**
`priority_queue` 是容器适配器，底层通常是 `vector` 加堆算法实现。默认是大根堆。

取堆顶 O(1)，插入和删除堆顶 O(log n)。

**3分钟详答：**
`priority_queue` 默认让最大元素在堆顶：

```cpp
priority_queue<int> pq;
```

底层常用 `vector` 保存元素，再通过 `push_heap`、`pop_heap` 维护堆结构。

如果要小根堆：

```cpp
priority_queue<int, vector<int>, greater<int>> pq;
```

堆只保证堆顶是最大或最小，不保证整体有序。所以不能像有序数组一样直接遍历得到排序结果。

**可能的追问问题：**

- 怎么改成小根堆？
  **15秒简答：** 使用 `priority_queue<int, vector<int>, greater<int>>`。
- 堆和二叉搜索树有什么区别？
  **15秒简答：** 堆只保证父子节点优先级关系，二叉搜索树保证左小右大的全局有序关系。
- priority_queue 能不能遍历有序结果？
  **15秒简答：** 不能直接有序遍历，只能不断 `top()` 和 `pop()` 得到优先级顺序。

## stack 和 queue 为什么叫容器适配器？

**15秒简答：**
因为 `stack` 和 `queue` 本身不是独立底层容器，而是在已有容器基础上封装出受限接口。

它们默认底层通常使用 `deque`。

**3分钟详答：**
`stack` 只暴露后进先出接口：

```cpp
push
pop
top
```

它可以基于 `deque`、`vector`、`list` 等容器实现。

`queue` 只暴露先进先出接口：

```cpp
push
pop
front
back
```

它通常基于 `deque`，因为需要尾部插入和头部删除。

容器适配器的特点是：限制底层容器的部分能力，只提供特定数据结构语义。

**可能的追问问题：**

- stack 默认底层容器是什么？
  **15秒简答：** 通常是 `deque`。
- 能不能用 vector 实现 stack？
  **15秒简答：** 可以，因为 stack 只需要尾部插入、尾部删除和访问尾部。
- queue 为什么通常不用 vector 作为默认底层？
  **15秒简答：** 因为 queue 需要头部删除，vector 头删效率低。

## sort 和 stable_sort 有什么区别？

**15秒简答：**
`sort` 不保证相等元素原有顺序；`stable_sort` 保证相等元素的相对顺序不变。

`sort` 通常更快，`stable_sort` 可能需要更多额外空间。

**3分钟详答：**
如果有数据：

```cpp
{score: 90, name: A}
{score: 90, name: B}
```

按照 score 排序后，`stable_sort` 会保持 A 在 B 前面。
`sort` 不保证这一点。

`std::sort` 通常要求随机访问迭代器，所以可以用于 `vector`、`deque`，但不能直接用于 `list`。

比较函数必须满足严格弱序，不能写出前后矛盾的比较逻辑，否则排序结果可能异常。

**可能的追问问题：**

- sort 要求什么迭代器？
  **15秒简答：** 要求随机访问迭代器。
- 为什么 list 不能直接用 std::sort？
  **15秒简答：** 因为 list 只有双向迭代器，不支持随机访问。list 有自己的 `list::sort()`。
- 比较函数要满足什么规则？
  **15秒简答：** 要满足严格弱序，不能出现自相矛盾的比较关系。

# 类型转换补充

## C++ 四种 cast 有什么区别？

**15秒简答：**
C++ 有四种常见强制类型转换：

`static_cast` 用于常规转换；
`dynamic_cast` 用于多态类型运行时转换；
`const_cast` 用于去掉或添加 const；
`reinterpret_cast` 用于底层二进制重新解释，风险最高。

**3分钟详答：**
`static_cast`：

```cpp
double d = 3.14;
int x = static_cast<int>(d);
```

适合基本类型转换、父子类指针的安全方向转换等。

`dynamic_cast`：

```cpp
Derived* d = dynamic_cast<Derived*>(basePtr);
```

用于多态类向下转换，会进行运行时类型检查。

`const_cast`：

```cpp
const int* p;
int* q = const_cast<int*>(p);
```

用于去掉 const，但如果原对象本身就是 const，修改它仍然是未定义行为。

`reinterpret_cast`：

```cpp
long x = reinterpret_cast<long>(p);
```

进行低层次重新解释，风险很高，应该少用。

**可能的追问问题：**

- dynamic_cast 依赖什么机制？
  **15秒简答：** 依赖 RTTI，通常要求基类至少有一个虚函数。
- dynamic_cast 失败返回什么？
  **15秒简答：** 指针转换失败返回 `nullptr`，引用转换失败抛 `std::bad_cast`。
- 为什么不推荐 C 风格强转？
  **15秒简答：** C 风格强转语义不清，可能混合多种转换行为，不利于代码审查。

## 隐式类型转换有哪些坑？

**15秒简答：**
隐式转换可能发生在构造函数、转换运算符、算术类型提升、指针到 bool 等场景中。

它会让代码看起来能编译，但实际语义不符合预期。

**3分钟详答：**
常见隐式转换来源：

1. 单参数构造函数。
2. 类型转换运算符。
3. 基本类型自动提升。
4. 指针到 bool。
5. 派生类指针到基类指针。

例如：

```cpp
class A {
public:
    A(int) {}
};

void f(A a);

f(10); // int 隐式转换成 A
```

如果这不是期望行为，应该把构造函数写成：

```cpp
explicit A(int) {}
```

现代 C++ 还推荐用 `{}` 初始化减少窄化转换。

**可能的追问问题：**

- explicit 为什么重要？
  **15秒简答：** 可以阻止构造函数被用于隐式类型转换，减少隐藏 bug。
- operator bool 有什么风险？
  **15秒简答：** 可能让对象在条件判断或算术表达式中发生意外转换，现代 C++ 常写成 `explicit operator bool()`。
- {} 初始化有什么好处？
  **15秒简答：** 花括号初始化更严格，会禁止窄化转换。

# 现代C++补充

## auto 和 decltype 有什么区别？

**15秒简答：**
`auto` 根据初始化表达式推导变量类型，通常会忽略顶层 const 和引用。

`decltype` 根据表达式形式推导类型，可以保留引用和 const 信息。

**3分钟详答：**
`auto` 常用于简化复杂类型：

```cpp
auto it = v.begin();
```

但它会有类型衰减：

```cpp
const int x = 10;
auto a = x; // a 是 int，不是 const int
```

如果想保留引用，需要写：

```cpp
auto& b = x;
```

`decltype` 更关注表达式本身：

```cpp
int x = 0;
decltype(x) a;   // int
decltype((x)) b = x; // int&
```

`decltype((x))` 是引用类型，因为 `(x)` 是左值表达式。

**可能的追问问题：**

- decltype((x)) 和 decltype(x) 一样吗？
  **15秒简答：** 不一样。`decltype(x)` 是变量声明类型，`decltype((x))` 通常得到左值引用类型。
- auto& 和 const auto& 有什么区别？
  **15秒简答：** `auto&` 是可修改引用，`const auto&` 是只读引用，可以绑定临时对象。
- 什么时候用 decltype(auto)？
  **15秒简答：** 当需要保留返回表达式的引用属性和 const 属性时使用。

------

## 什么是 RVO 和 NRVO？

**15秒简答：**
RVO 和 NRVO 是编译器对返回对象的优化，可以避免不必要的拷贝或移动，直接在目标位置构造返回值。

C++17 后某些场景下拷贝省略是强制保证的。

**3分钟详答：**
例如：

```cpp
A makeA() {
    return A();
}
```

编译器可以直接在调用者接收返回值的位置构造 `A`，避免临时对象。

NRVO 是返回具名局部对象：

```cpp
A makeA() {
    A a;
    return a;
}
```

编译器可能把 `a` 直接构造在返回值位置。

需要注意：

```cpp
return std::move(a);
```

不一定更好，因为它可能阻止 NRVO，让代码退化成移动构造。

**可能的追问问题：**

- 返回局部对象会不会一定拷贝？
  **15秒简答：** 不一定。编译器通常会进行 NRVO 或移动优化。
- return std::move(local) 一定好吗？
  **15秒简答：** 不一定，可能阻止 NRVO，反而降低优化机会。
- C++17 对拷贝省略有什么变化？
  **15秒简答：** 某些返回临时对象的场景强制保证拷贝省略。

------

## lambda 捕获值和捕获引用有什么区别？

**15秒简答：**
值捕获会把变量拷贝进 lambda 对象，更安全但可能有拷贝成本；引用捕获不拷贝，但如果 lambda 延迟执行，容易出现悬空引用。

**3分钟详答：**
值捕获：

```cpp
int x = 10;
auto f = [x]() { return x; };
```

lambda 内部保存一份 `x` 的副本。

引用捕获：

```cpp
auto f = [&x]() { return x; };
```

lambda 保存对外部变量的引用。

如果 lambda 立即执行，引用捕获通常没问题。
但如果 lambda 被保存起来、异步执行或传给线程，引用捕获很危险，因为原变量可能已经销毁。

捕获 `this` 也要小心。如果对象已经析构，lambda 再访问 `this` 就会悬空。

**可能的追问问题：**

- lambda 本质是什么？
  **15秒简答：** 本质是编译器生成的匿名函数对象，捕获变量会成为对象成员。
- [=] 和 [&] 有什么风险？
  **15秒简答：** `[=]` 可能产生不必要拷贝，`[&]` 在异步场景中容易悬空引用。
- 捕获 this 要注意什么？
  **15秒简答：** 要保证 lambda 执行时对象仍然存活，否则 `this` 会悬空。

------

## std::function 和函数指针有什么区别？

**15秒简答：**
函数指针只能保存普通函数地址；`std::function` 是通用可调用对象包装器，可以保存普通函数、lambda、仿函数和绑定表达式。

但 `std::function` 有额外类型擦除开销。

**3分钟详答：**
函数指针：

```cpp
int (*fp)(int, int);
```

只能指向函数。

`std::function`：

```cpp
std::function<int(int, int)> f;
```

可以保存：

1. 普通函数。
2. lambda。
3. 函数对象。
4. `std::bind` 结果。

它更灵活，适合回调接口。但因为使用类型擦除，可能有间接调用和额外内存分配，不适合极端性能敏感路径滥用。

**可能的追问问题：**

- std::bind 是什么？
  **15秒简答：** 可以把函数和部分参数绑定起来，生成新的可调用对象。
- lambda 和仿函数有什么关系？
  **15秒简答：** lambda 本质上就是编译器生成的匿名仿函数对象。
- std::function 是否有性能成本？
  **15秒简答：** 有，主要是类型擦除、间接调用和可能的动态分配。

# 多线程与并发补充

## 进程和线程有什么区别？

**15秒简答：**
进程是资源分配的基本单位，线程是 CPU 调度的基本单位。

同一进程内多个线程共享地址空间，但每个线程有自己的栈和寄存器上下文。

**3分钟详答：**
进程拥有独立的地址空间、文件描述符、虚拟内存等资源。
线程运行在进程内部，共享进程的大部分资源。

线程共享：

1. 堆。
2. 全局变量。
3. 静态变量。
4. 文件描述符。
5. 进程地址空间。

线程私有：

1. 栈。
2. 寄存器上下文。
3. 程序计数器。
4. 线程局部存储。

所以多线程通信方便，但共享数据需要同步。多进程隔离性更好，但通信成本更高。

**可能的追问问题：**

- 线程之间共享哪些资源？
  **15秒简答：** 共享地址空间、堆、全局变量、静态变量、文件描述符等。
- 线程私有哪些资源？
  **15秒简答：** 私有栈、寄存器上下文、程序计数器和线程局部存储。
- 进程切换和线程切换哪个开销大？
  **15秒简答：** 通常进程切换开销更大，因为涉及地址空间切换。

------

## 多线程中堆变量是否共享？

**15秒简答：**
同一进程的多个线程共享进程地址空间，所以堆区是共享的。

但某个堆对象是否真的被多个线程访问，取决于多个线程是否拿到了同一个地址。

**3分钟详答：**
例如：

```cpp
int* p = new int(10);
```

如果多个线程都拿到 `p`，它们访问的是同一块堆内存。

```cpp
void f(int* p) {
    (*p)++;
}
```

多个线程同时执行 `f(p)`，就会并发写同一个对象，需要同步。

但如果每个线程都自己 `new` 一块内存，它们的对象虽然都在堆区，但不是同一个对象。

所以更准确说：

**堆区对线程共享，但具体对象是否共享，取决于地址是否被共享。**

**可能的追问问题：**

- 栈变量是否共享？
  **15秒简答：** 每个线程有自己的栈，普通局部变量默认不共享，但如果把地址传给其他线程，也可能被访问。
- 全局变量是否共享？
  **15秒简答：** 共享。同一进程内所有线程都能访问全局变量。
- thread_local 是什么？
  **15秒简答：** `thread_local` 修饰的变量每个线程都有自己独立的一份。

------

## 什么是数据竞争？

**15秒简答：**
多个线程同时访问同一内存位置，其中至少一个是写操作，并且没有同步保护，就会产生数据竞争。

C++ 中数据竞争会导致未定义行为。

**3分钟详答：**
例如：

```cpp
int x = 0;

void f() {
    x++;
}
```

如果多个线程同时执行 `x++`，就有数据竞争。
因为 `x++` 不是单条不可分割操作，通常包含读、加、写三个步骤。

解决方法：

1. 使用 `mutex`。
2. 使用 `atomic`。
3. 避免共享可变数据。
4. 使用线程局部变量。

只读共享通常安全，但只要有线程写，其他线程读或写，就需要同步。

**可能的追问问题：**

- 数据竞争和竞态条件一样吗？
  **15秒简答：** 不完全一样。数据竞争是未同步并发访问同一内存；竞态条件是结果依赖执行时序的更广义问题。
- 如何避免数据竞争？
  **15秒简答：** 用锁、原子变量、线程局部存储，或者设计上避免共享可变状态。
- 为什么只读共享通常安全？
  **15秒简答：** 因为没有线程修改数据，不会产生读写冲突。

------

## mutex、lock_guard、unique_lock 有什么区别？

**15秒简答：**
`mutex` 是锁本身；`lock_guard` 是简单 RAII 锁，构造加锁、析构解锁；`unique_lock` 更灵活，支持延迟加锁、手动解锁和移动，常配合条件变量使用。

**3分钟详答：**
直接使用 `mutex`：

```cpp
mtx.lock();
// 临界区
mtx.unlock();
```

容易因为异常或提前 return 忘记解锁。

`lock_guard`：

```cpp
lock_guard<mutex> lock(mtx);
```

进入作用域加锁，离开作用域自动解锁，简单安全。

`unique_lock`：

```cpp
unique_lock<mutex> lock(mtx);
```

功能更强，可以：

1. 延迟加锁。
2. 手动 unlock。
3. 转移锁所有权。
4. 配合 `condition_variable`。

如果只是简单临界区，优先 `lock_guard`；需要灵活控制锁，使用 `unique_lock`。

**可能的追问问题：**

- 为什么推荐 RAII 管理锁？
  **15秒简答：** 因为作用域结束会自动解锁，避免异常或提前返回导致忘记解锁。
- unique_lock 为什么能配合条件变量？
  **15秒简答：** 因为条件变量等待时需要临时释放锁，被唤醒后重新加锁，`unique_lock` 支持这种操作。
- 手动 lock/unlock 有什么风险？
  **15秒简答：** 容易在异常、return 或复杂分支中漏解锁，导致死锁。

------

## 条件变量 condition_variable 怎么用？

**15秒简答：**
条件变量用于线程等待某个条件成立。等待线程持有 `unique_lock`，调用 `wait` 时释放锁并阻塞，被通知后重新加锁并检查条件。

必须防止虚假唤醒。

**3分钟详答：**
典型写法：

```cpp
std::mutex mtx;
std::condition_variable cv;
bool ready = false;

std::unique_lock<std::mutex> lock(mtx);
cv.wait(lock, [] { return ready; });
```

`wait` 做了三件事：

1. 检查条件。
2. 条件不满足时释放锁并阻塞。
3. 被唤醒后重新加锁并再次检查条件。

必须用谓词或 while 检查条件，因为可能发生虚假唤醒。

通知端：

```cpp
{
    std::lock_guard<std::mutex> lock(mtx);
    ready = true;
}
cv.notify_one();
```

**可能的追问问题：**

- 为什么 wait 要配合锁？
  **15秒简答：** 因为条件的检查和修改需要互斥保护，避免错过通知或读到不一致状态。
- 什么是虚假唤醒？
  **15秒简答：** 线程可能在条件不满足时从 wait 返回，所以醒来后必须重新检查条件。
- notify_one 和 notify_all 区别？
  **15秒简答：** `notify_one` 唤醒一个等待线程，`notify_all` 唤醒所有等待线程。

------

## 什么是死锁？如何避免？

**15秒简答：**
死锁是多个线程互相等待对方持有的资源，导致都无法继续执行。

避免方式包括固定加锁顺序、减少锁粒度、使用 `std::lock`、避免持锁调用外部逻辑等。

**3分钟详答：**
经典死锁：

```cpp
线程1：锁A -> 等锁B
线程2：锁B -> 等锁A
```

死锁四个必要条件：

1. 互斥。
2. 占有且等待。
3. 不可抢占。
4. 循环等待。

破坏其中一个条件就可以避免死锁。

工程中常用方法：

1. 多把锁按固定顺序加锁。
2. 使用 `std::lock` 一次性锁多把锁。
3. 减少锁持有时间。
4. 不在持锁状态下调用外部未知代码。
5. 使用超时锁。

**可能的追问问题：**

- 死锁四个必要条件是什么？
  **15秒简答：** 互斥、占有且等待、不可抢占、循环等待。
- 两把锁如何安全加锁？
  **15秒简答：** 固定加锁顺序，或者使用 `std::lock` 同时锁住多把锁。
- std::lock 有什么用？
  **15秒简答：** 可以同时锁定多个互斥量，并帮助避免加锁顺序不同造成死锁。

------

## atomic 和 mutex 有什么区别？

**15秒简答：**
`atomic` 适合单个变量的简单原子操作；`mutex` 适合保护复杂临界区或多个变量之间的一致性。

`atomic` 不能完全替代锁。

**3分钟详答：**
例如：

```cpp
atomic<int> count = 0;
count++;
```

这是线程安全的。

但如果有多个变量：

```cpp
int a;
int b;
```

它们之间需要保持某种关系，比如 `a + b` 始终等于固定值，仅靠两个 atomic 不一定够，因为其他线程可能看到中间状态。

这时用 `mutex` 更合适：

```cpp
lock_guard<mutex> lock(mtx);
// 同时修改 a 和 b
```

所以选择标准：

- 单变量计数、状态标记：可以考虑 atomic。
- 多变量一致性、复杂数据结构：通常用 mutex。

**可能的追问问题：**

- atomic++ 是线程安全吗？
  **15秒简答：** 是，对单个 atomic 变量的自增是原子操作。
- 多个变量的一致性用 atomic 够吗？
  **15秒简答：** 不一定。多个变量之间有约束时通常需要锁保护整体临界区。
- CAS 是什么？
  **15秒简答：** CAS 是 Compare-And-Swap，比较当前值是否等于预期值，相等则替换，是无锁编程常见基础。

------

## shared_ptr 是线程安全的吗？

**15秒简答：**
`shared_ptr` 的引用计数操作通常是线程安全的，但它指向的对象不一定线程安全。

多个线程访问同一个对象时，仍然需要同步。

**3分钟详答：**
需要区分两层安全：

1. 控制块引用计数安全。
   不同 `shared_ptr` 对象拷贝、析构时，引用计数不会乱。
2. 被管理对象不自动线程安全。
   如果多个线程通过 `shared_ptr` 同时修改同一个对象，仍然会数据竞争。

还有一个细节：

多个线程操作**同一个 shared_ptr 变量本身**，比如同时 `reset`，也需要额外同步。

所以不能简单说 `shared_ptr` 完全线程安全，只能说它的引用计数管理是线程安全的。

**可能的追问问题：**

- 引用计数安全和对象安全有什么区别？
  **15秒简答：** 引用计数安全只保证 shared_ptr 生命周期管理不乱；对象安全需要对象自身同步。
- 多线程同时 reset 同一个 shared_ptr 变量安全吗？
  **15秒简答：** 不安全。多个线程读写同一个 shared_ptr 变量本身需要同步。
- weak_ptr::lock 为什么安全？
  **15秒简答：** `lock()` 会尝试生成临时 `shared_ptr`，如果对象还活着就安全增加引用计数，否则返回空。

------

# 编译链接补充

## C++ 从源码到可执行文件经历哪些阶段？

**15秒简答：**
主要经历预处理、编译、汇编、链接四个阶段。

预处理处理宏和头文件；编译生成汇编；汇编生成目标文件；链接把目标文件和库合成可执行文件。

**3分钟详答：**
四个阶段：

1. 预处理
   展开头文件、宏替换、条件编译，生成预处理后的源码。
2. 编译
   进行语法语义分析和优化，把源码转成汇编代码。
3. 汇编
   把汇编代码转成机器码目标文件，比如 `.o` 或 `.obj`。
4. 链接
   把多个目标文件和库合并，解析符号引用，生成最终可执行文件。

编译错误一般发生在单个源文件处理阶段；链接错误通常发生在多个目标文件合并阶段，比如函数只有声明没有定义。

**可能的追问问题：**

- 头文件重复包含怎么解决？
  **15秒简答：** 使用 include guard 或 `#pragma once`。
- 静态链接和动态链接区别？
  **15秒简答：** 静态链接把库代码合进可执行文件，动态链接在运行时加载共享库。
- 什么是符号表？
  **15秒简答：** 符号表记录函数、全局变量等符号名称和地址信息，链接器用它解析引用。

------

## 静态库和动态库有什么区别？

**15秒简答：**
静态库在链接阶段被复制进可执行文件，运行时不依赖库文件；动态库在运行时加载，多个程序可以共享，但部署时要保证库存在。

**3分钟详答：**
静态库：

- Linux 下通常是 `.a`。
- Windows 下通常是 `.lib`。
- 链接后代码进入可执行文件。
- 可执行文件更大，但部署相对简单。

动态库：

- Linux 下通常是 `.so`。
- Windows 下通常是 `.dll`。
- 运行时加载。
- 多个程序可以共享同一个库，升级方便。

缺点是运行环境必须能找到对应版本的动态库，否则程序启动失败。

**可能的追问问题：**

- Linux 下静态库和动态库后缀是什么？
  **15秒简答：** 静态库是 `.a`，动态库是 `.so`。
- 动态库找不到怎么办？
  **15秒简答：** 检查库路径、`LD_LIBRARY_PATH`、rpath、系统库缓存和版本是否匹配。
- 动态链接有什么好处？
  **15秒简答：** 多程序共享库代码，节省空间，库可以单独升级。

------

## 头文件里为什么不要定义普通全局变量？

**15秒简答：**
头文件会被多个 `.cpp` 包含，如果在头文件中定义普通全局变量，多个翻译单元会各自产生定义，链接时可能重复定义。

应该在头文件声明，在一个 `.cpp` 中定义。

**3分钟详答：**
错误写法：

```cpp
// a.h
int g = 10;
```

如果多个 `.cpp` 都包含 `a.h`，每个目标文件里都会有一个 `g` 的定义，链接时可能报重复定义。

正确写法：

```cpp
// a.h
extern int g;

// a.cpp
int g = 10;
```

`extern` 是声明，不分配存储空间。真正定义只放一处。

C++17 后可以用 `inline variable`：

```cpp
inline int g = 10;
```

允许在头文件定义并由链接器合并。

**可能的追问问题：**

- 声明和定义有什么区别？
  **15秒简答：** 声明告诉编译器符号存在，定义真正分配存储空间或提供函数实现。
- \#pragma once 能解决全局变量重复定义吗？
  **15秒简答：** 不能解决多个 `.cpp` 分别包含导致的链接重复定义，它只防止同一翻译单元重复包含。
- inline variable 是什么？
  **15秒简答：** C++17 引入的内联变量，允许变量定义放在头文件中，并在链接时合并。

# 异常与错误处理补充

## C++ 异常机制是什么？

**15秒简答：**
异常用于处理错误，通过 `throw` 抛出异常，沿调用栈寻找匹配的 `catch`。

异常可以把错误处理和正常逻辑分离，但也会带来控制流复杂和异常安全问题。

**3分钟详答：**
异常基本形式：

```cpp
try {
    throw runtime_error("error");
} catch (const exception& e) {
    cout << e.what();
}
```

当异常抛出后，程序会沿调用栈向上寻找匹配的 `catch`。
这个过程中，已经构造完成的局部对象会被析构，这叫栈展开。

RAII 和异常关系非常密切，因为即使发生异常，局部对象析构函数也会执行，从而释放资源。

析构函数中不建议抛异常，因为如果在栈展开过程中再次抛异常，可能导致 `std::terminate`。

**可能的追问问题：**

- 析构函数里能不能抛异常？
  **15秒简答：** 不建议。析构函数抛异常可能导致 `std::terminate`。
- 什么是栈展开？
  **15秒简答：** 异常抛出后沿调用栈查找 catch，并析构沿途局部对象的过程。
- RAII 为什么和异常安全关系密切？
  **15秒简答：** 因为异常发生时局部对象会自动析构，RAII 能保证资源释放。

------

## noexcept 有什么作用？

**15秒简答：**
`noexcept` 表示函数承诺不抛异常。如果违反承诺，会调用 `std::terminate`。

它不只是说明文档，还会影响标准库优化，比如 vector 扩容时是否使用移动构造。

**3分钟详答：**
例如：

```cpp
A(A&& other) noexcept;
```

如果移动构造是 `noexcept`，`vector` 扩容搬移元素时更愿意使用移动。
如果移动构造可能抛异常，标准库为了保证异常安全，可能退回使用拷贝。

所以对于资源转移类，如果移动操作不会失败，应该尽量标记 `noexcept`。

`noexcept` 也可以用于函数类型判断：

```cpp
noexcept(f());
```

判断表达式是否声明为不抛异常。

**可能的追问问题：**

- noexcept 是不是只影响文档说明？
  **15秒简答：** 不是，它会影响编译器优化和标准库容器选择移动还是拷贝。
- 析构函数默认是不是 noexcept？
  **15秒简答：** 通常析构函数默认是 `noexcept`，除非成员或基类析构可能抛异常。
- noexcept 函数真的抛异常会怎样？
  **15秒简答：** 会调用 `std::terminate` 终止程序。

# 哈希与算法复杂度补充

## 哈希冲突有哪些解决方式？

**15秒简答：**
常见解决方式有链地址法和开放寻址法。

链地址法是在同一个桶里挂链表或其他结构；开放寻址法是在数组中继续探测下一个可用位置。

**3分钟详答：**
链地址法：

```text
bucket[3] -> key1 -> key2 -> key3
```

优点是实现直观，删除简单。缺点是需要额外指针或节点空间，缓存不一定友好。

开放寻址法：

如果当前位置被占用，就按照规则寻找下一个位置，例如：

1. 线性探测。
2. 二次探测。
3. 双重哈希。

优点是数据在数组内，缓存友好。缺点是删除和高负载因子处理更复杂。

哈希表性能依赖哈希函数、负载因子和冲突处理方式。

**可能的追问问题：**

- 链地址法和开放寻址法优缺点？
  **15秒简答：** 链地址法删除简单但有额外节点开销；开放寻址缓存友好但删除复杂，对负载因子敏感。
- 负载因子过高会怎样？
  **15秒简答：** 冲突增加，查找和插入效率下降，严重时接近 O(n)。
- 为什么要 rehash？
  **15秒简答：** 为了扩充桶数量、降低负载因子、减少哈希冲突。

## 红黑树和 AVL 树有什么区别？

**15秒简答：**
AVL 树更严格平衡，查找性能更稳定；红黑树是近似平衡，插入删除调整成本通常更低。

STL 的 `map/set` 通常更适合用红黑树。

**3分钟详答：**
AVL 树要求左右子树高度差不能超过 1，因此更接近严格平衡。
优点是查找路径更短，缺点是插入删除后可能需要更多旋转调整。

红黑树通过颜色规则保证近似平衡，不追求绝对高度平衡。
它的查找、插入、删除都能稳定在 O(log n)，同时插入删除调整成本相对可控。

对于通用关联容器，插入删除和查找都很重要，所以红黑树更常见。

**可能的追问问题：**

- 红黑树查找为什么是 O(log n)？
  **15秒简答：** 因为红黑树通过颜色规则限制树高，使高度保持在 O(log n)。
- 红黑树是不是严格平衡？
  **15秒简答：** 不是，它是近似平衡。
- 为什么数据库索引常用 B+ 树而不是红黑树？
  **15秒简答：** B+ 树一个节点能存多个 key，树高更低，磁盘 IO 更少，更适合外存场景。

## 二分查找为什么是 O(log n)？

**15秒简答：**
因为每次比较都能排除一半数据，问题规模从 n 变成 n/2，所以最多比较 `log2 n` 次。

**3分钟详答：**
二分查找的前提是数据有序，并且能随机访问。

每轮取中间位置：

```cpp
mid = l + (r - l) / 2;
```

如果目标比中间值小，就去左半部分；如果目标比中间值大，就去右半部分。

每次搜索区间减半：

```text
n -> n/2 -> n/4 -> n/8 ...
```

当规模减到 1 时，次数约为 `log2 n`。

所以时间复杂度是 O(log n)。

**可能的追问问题：**

- 二分查找前提是什么？
  **15秒简答：** 数据有序，并且最好支持随机访问。
- 左闭右闭和左闭右开有什么区别？
  **15秒简答：** 左闭右闭是 `[l, r]`，循环常用 `l <= r`；左闭右开是 `[l, r)`，循环常用 `l < r`。
- 二分为什么容易死循环？
  **15秒简答：** 因为边界更新没有排除 `mid`，导致区间不缩小。

## 快排平均 O(nlogn)，最坏 O(n²) 的原因是什么？

**15秒简答：**
快排每次分区如果比较均匀，递归深度是 O(log n)，每层处理 O(n)，所以平均 O(nlogn)。

如果 pivot 每次都选到极端值，递归深度退化成 O(n)，总复杂度变成 O(n²)。

**3分钟详答：**
快排核心是分区。
选一个 pivot，把小于 pivot 的放左边，大于 pivot 的放右边，然后递归处理左右区间。

如果每次 pivot 都接近中位数：

```text
n -> n/2 -> n/4 -> ...
```

递归深度是 O(log n)，每层总处理 n 个元素，所以是 O(nlogn)。

如果数组已经有序，而 pivot 总选第一个或最后一个：

```text
n -> n-1 -> n-2 -> ...
```

递归深度变成 O(n)，每层处理规模逐渐减少，总复杂度退化到 O(n²)。

**可能的追问问题：**

- 如何优化快排？
  **15秒简答：** 随机选 pivot、三数取中、三路快排、小数组改插入排序。
- 快排是稳定排序吗？
  **15秒简答：** 通常不是，因为分区交换可能改变相等元素的相对顺序。
- 快排和归并排序怎么选？
  **15秒简答：** 快排原地、缓存友好、平均快；归并稳定、最坏 O(nlogn)，但需要额外空间。

# C++高频补充题

## make_shared 和 shared_ptr(new T) 有什么区别？

**15秒简答：**
`make_shared<T>()` 通常只进行一次内存分配，同时分配对象和控制块；`shared_ptr<T>(new T)` 通常需要两次分配，一次分配对象，一次分配控制块。

工程上更推荐 `make_shared`，因为它更高效、更异常安全。

**3分钟详答：**
`shared_ptr` 管理对象时，除了对象本身，还需要一个控制块。控制块里保存引用计数、弱引用计数、删除器等信息。

如果写：

```cpp
shared_ptr<A> p(new A());
```

通常会先 `new A()` 分配对象，再为 `shared_ptr` 的控制块分配内存，可能有两次内存分配。

如果写：

```cpp
auto p = make_shared<A>();
```

标准库可以把对象和控制块放在一块连续内存中，通常只分配一次，效率更高，也更缓存友好。

另外，`make_shared` 更异常安全。
如果在复杂表达式中先 `new` 出对象，但 `shared_ptr` 控制块创建时抛异常，可能导致裸指针泄漏。`make_shared` 能减少这种风险。

但 `make_shared` 也有一个缺点：对象和控制块在同一块内存中。如果还有 `weak_ptr` 存在，即使强引用计数为 0，对象已经析构，控制块仍要保留，这块整体内存可能不能立即释放。

**可能的追问问题：**

- make_shared 为什么更高效？
  **15秒简答：** 因为它通常把对象和控制块一次性分配，减少内存分配次数。
- make_shared 有没有缺点？
  **15秒简答：** 有。如果 weak_ptr 长时间存在，控制块不能释放，对象所在的整块内存可能也不能立刻归还。
- 什么时候不能用 make_shared？
  **15秒简答：** 需要自定义 deleter，或者希望对象内存和控制块分开管理时，不适合用 `make_shared`。

------

## enable_shared_from_this 是什么？

**15秒简答：**
`enable_shared_from_this` 允许对象在自己的成员函数中安全地获取指向自己的 `shared_ptr`。

它主要解决的问题是：不要在对象内部直接用 `shared_ptr<T>(this)`，否则可能产生两个独立控制块，导致重复释放。

**3分钟详答：**
错误写法：

```cpp
class A {
public:
    shared_ptr<A> getPtr() {
        return shared_ptr<A>(this);
    }
};
```

如果这个对象本来已经被一个 `shared_ptr` 管理，再用 `shared_ptr<A>(this)` 会创建一个新的控制块。
结果是两个 `shared_ptr` 都以为自己拥有对象，最后可能 delete 两次。

正确写法：

```cpp
class A : public enable_shared_from_this<A> {
public:
    shared_ptr<A> getPtr() {
        return shared_from_this();
    }
};
```

使用前提是对象必须已经被 `shared_ptr` 管理：

```cpp
auto p = make_shared<A>();
auto q = p->getPtr();
```

这时 `q` 和 `p` 共享同一个控制块，引用计数正确增加，不会重复释放。

面试里重点说清楚：

**`enable_shared_from_this` 不是让普通对象凭空变成 shared_ptr，而是让已经被 shared_ptr 管理的对象安全获取同控制块的 shared_ptr。**

**可能的追问问题：**

- 为什么不能直接 shared_ptr(this)？
  **15秒简答：** 因为会创建新的控制块，可能导致同一个对象被释放两次。
- shared_from_this() 什么时候会出错？
  **15秒简答：** 如果对象还没有被 `shared_ptr` 管理就调用，可能抛出 `std::bad_weak_ptr`。
- enable_shared_from_this 常用于什么场景？
  **15秒简答：** 常用于异步回调中延长对象生命周期，防止回调执行时对象已经析构。

------

## weak_ptr 的 expired() 和 lock() 有什么区别？

**15秒简答：**
`expired()` 只检查对象是否已经释放；`lock()` 会尝试把 `weak_ptr` 提升成 `shared_ptr`。

真正访问对象时应该用 `lock()`，因为它能在对象仍存在时临时增加强引用，保证访问期间对象不被释放。

**3分钟详答：**
`weak_ptr` 不拥有对象，不增加强引用计数。它主要用于观察 `shared_ptr` 管理的对象，或者解决循环引用。

```cpp
weak_ptr<A> wp = sp;
```

检查对象是否还活着：

```cpp
if (!wp.expired()) {
    // 只能说明检查这一刻对象还在
}
```

但 `expired()` 有一个问题：
在多线程环境下，检查完之后对象可能马上被其他线程释放。
所以访问对象时更推荐：

```cpp
if (auto sp = wp.lock()) {
    sp->doSomething();
}
```

`lock()` 如果成功，会返回一个 `shared_ptr`，强引用计数增加。这样在 `sp` 生命周期内，对象不会被释放。
如果对象已经释放，`lock()` 返回空 `shared_ptr`。

所以：

- `expired()` 适合判断状态。
- `lock()` 适合安全访问对象。

**可能的追问问题：**

- weak_ptr 能不能直接解引用？
  **15秒简答：** 不能。因为它不拥有对象，对象可能已经释放，必须先 `lock()` 成 `shared_ptr`。
- expired() 为 false 后一定安全吗？
  **15秒简答：** 不一定。多线程下检查后对象可能马上被释放，所以访问时应该用 `lock()`。
- weak_ptr 会影响对象生命周期吗？
  **15秒简答：** 不会。`weak_ptr` 不增加强引用计数，不阻止对象析构。

## extern "C" 的作用是什么？

**15秒简答：**
`extern "C"` 用来告诉 C++ 编译器按照 C 语言方式处理函数名，避免 C++ 名字改编，方便 C 和 C++ 代码互相调用。

常用于 C++ 调用 C 库，或者 C 调用 C++ 暴露的接口。

**3分钟详答：**
C++ 支持函数重载，所以编译器会对函数名进行名字改编。
例如：

```cpp
void f(int);
void f(double);
```

这两个函数在链接阶段需要不同符号名，否则无法区分。

但 C 语言不支持函数重载，函数符号名通常就是原始函数名。
如果 C++ 代码要调用 C 编译出来的函数，必须避免名字不一致。

写法：

```cpp
extern "C" {
    void c_func();
}
```

或者：

```cpp
extern "C" void c_func();
```

如果是头文件同时被 C 和 C++ 使用，常见写法是：

```cpp
#ifdef __cplusplus
extern "C" {
#endif

void c_func();

#ifdef __cplusplus
}
#endif
```

这样 C++ 编译器会用 C 方式链接，C 编译器则正常处理。

**可能的追问问题：**

- extern "C" 能修饰类吗？
  **15秒简答：** 不能按 C 接口方式导出 C++ 类，因为 C 没有类、重载、成员函数等机制。通常只暴露普通函数接口。
- 为什么 C++ 需要名字改编？
  **15秒简答：** 因为 C++ 支持函数重载、命名空间、类成员函数等，链接符号需要包含更多类型信息。
- C 调用 C++ 代码时要注意什么？
  **15秒简答：** 通常要用 `extern "C"` 暴露普通函数接口，不能直接暴露 C++ 类对象和异常。

## delete this 可不可以？

**15秒简答：**
语法上可以，但非常危险。只有当对象确实是在堆上创建，并且调用 `delete this` 后不再访问任何成员时，才可能安全。

普通业务代码不推荐使用 `delete this`。

**3分钟详答：**
`delete this` 的含义是对象自己释放自己：

```cpp
class A {
public:
    void destroy() {
        delete this;
    }
};
```

如果对象是这样创建的：

```cpp
A* p = new A();
p->destroy();
```

理论上可以释放。

但如果对象是在栈上：

```cpp
A a;
a.destroy(); // 错误
```

此时 `delete this` 会释放栈对象，属于未定义行为。

还有一个风险是：

```cpp
void destroy() {
    delete this;
    this->x = 1; // 错误，对象已经被释放
}
```

`delete this` 之后不能再访问成员变量、成员函数，也不能让外部继续使用原来的指针。

所以它只适合非常特殊的对象生命周期设计，比如引用计数对象、自管理对象。现代 C++ 更推荐用智能指针管理生命周期。

**可能的追问问题：**

- delete this 后还能访问成员吗？
  **15秒简答：** 不能。对象已经析构并释放，再访问就是 use after free。
- 栈对象能 delete this 吗？
  **15秒简答：** 不能。栈对象不是 `new` 出来的，不能用 `delete` 释放。
- delete this 常见于什么场景？
  **15秒简答：** 少数自管理生命周期对象中可能出现，但现代 C++ 更推荐智能指针。

## 纯虚函数和抽象类是什么？

**15秒简答：**
纯虚函数是用 `= 0` 声明的虚函数，表示基类只定义接口，不提供普通实现。

包含纯虚函数的类是抽象类，不能直接实例化，只能作为接口或基类使用。

**3分钟详答：**
纯虚函数写法：

```cpp
class Shape {
public:
    virtual void draw() = 0;
};
```

`Shape` 中有纯虚函数，所以它是抽象类：

```cpp
Shape s; // 错误，抽象类不能实例化
```

派生类必须实现纯虚函数，才能变成具体类：

```cpp
class Circle : public Shape {
public:
    void draw() override {
        // draw circle
    }
};
```

抽象类常用于定义接口，让不同派生类提供不同实现。

面试里可以这样说：

**纯虚函数强调接口约束，抽象类用于表达“这个类型只作为基类，不应该直接创建对象”。**

**可能的追问问题：**

- 抽象类能有构造函数吗？
  **15秒简答：** 可以。抽象类不能直接实例化，但派生类构造时仍然要构造基类部分。
- 纯虚函数可以有函数体吗？
  **15秒简答：** 可以，但仍然是纯虚函数，类仍然是抽象类。
- 抽象类可以有普通成员变量吗？
  **15秒简答：** 可以。抽象类不等于只有接口，也可以包含数据成员和普通成员函数。

------

## 纯虚析构函数为什么也要有定义？

**15秒简答：**
纯虚析构函数可以让类变成抽象类，但析构函数最终一定会被调用，所以即使是纯虚析构函数，也必须提供函数定义。

**3分钟详答：**
写法：

```cpp
class Base {
public:
    virtual ~Base() = 0;
};

Base::~Base() {}
```

虽然析构函数被声明为纯虚函数，但派生类对象析构时，析构顺序是：

1. 先调用派生类析构函数。
2. 再调用基类析构函数。

所以 `Base::~Base()` 最终一定会被调用。
如果只有声明没有定义，链接阶段会找不到基类析构函数定义，导致链接错误。

纯虚析构函数的常见目的：

1. 让基类成为抽象类。
2. 保证通过基类指针删除派生类对象时析构链正确。

**可能的追问问题：**

- 纯虚析构函数会让类变成抽象类吗？
  **15秒简答：** 会。只要类中有纯虚函数，包括纯虚析构函数，它就是抽象类。
- 纯虚析构函数为什么还会被调用？
  **15秒简答：** 因为派生类析构后，必须继续析构基类子对象。
- 多态基类析构函数为什么要 virtual？
  **15秒简答：** 为了通过基类指针删除派生类对象时，能正确调用派生类和基类析构函数。

## 运算符重载要注意什么？

**15秒简答：**
运算符重载不能改变运算符优先级、结合性和操作数个数，也不能创造新的运算符。

重载时要符合原运算符语义，避免让代码难以理解。

**3分钟详答：**
运算符重载是让自定义类型像内置类型一样使用。

例如复数加法：

```cpp
class Complex {
public:
    double real, imag;

    Complex operator+(const Complex& other) const {
        return {real + other.real, imag + other.imag};
    }
};
```

常见注意点：

1. 不要滥用，语义要自然。
2. 不能改变优先级和结合性。
3. 不能改变操作数个数。
4. 至少有一个操作数是自定义类型。
5. 赋值、下标、函数调用、成员访问箭头等运算符通常重载为成员函数。
6. `operator<<` 通常重载为普通函数或友元函数。

例如：

```cpp
ostream& operator<<(ostream& os, const A& a);
```

因为左操作数是 `ostream`，不能写成 `A` 的普通成员函数来满足 `cout << a` 的形式。

**可能的追问问题：**

- 哪些运算符不能重载？
  **15秒简答：** 例如 `.`、`.*`、`::`、`?:`、`sizeof` 不能重载。
- operator= 应该返回什么？
  **15秒简答：** 通常返回 `*this` 的引用，即 `A&`，以支持连续赋值。
- operator++ 前置和后置怎么区分？
  **15秒简答：** 前置是 `operator++()`，后置是 `operator++(int)`，其中 `int` 是占位参数。

## 模板特化和偏特化是什么？

**15秒简答：**
模板特化是为某个具体类型提供专门实现；偏特化是为一类满足特定形式的模板参数提供专门实现。

函数模板只能全特化，类模板可以全特化和偏特化。

**3分钟详答：**
普通模板：

```cpp
template <typename T>
class A {
public:
    void print() {
        cout << "general";
    }
};
```

全特化：

```cpp
template <>
class A<int> {
public:
    void print() {
        cout << "int";
    }
};
```

表示 `A<int>` 使用专门版本。

偏特化：

```cpp
template <typename T>
class A<T*> {
public:
    void print() {
        cout << "pointer";
    }
};
```

表示所有指针类型都使用这个版本，比如 `A<int*>`、`A<double*>`。

偏特化常用于类型萃取、智能指针、容器适配、模板元编程等场景。

注意：
函数模板不支持偏特化，但可以通过函数重载达到类似效果。

**可能的追问问题：**

- 全特化和偏特化区别是什么？
  **15秒简答：** 全特化针对一个完全确定的类型，偏特化针对一类类型模式。
- 函数模板可以偏特化吗？
  **15秒简答：** 不可以。函数模板只能全特化，类似需求通常用函数重载解决。
- 模板特化有什么用？
  **15秒简答：** 可以为特殊类型提供更合适或更高效的实现。

## typename 和 class 在模板参数里有什么区别？

**15秒简答：**
在模板参数列表中，`typename` 和 `class` 基本没有区别，都表示类型参数。

但在依赖类型场景中，`typename` 还有一个特殊作用：告诉编译器后面的名字是一个类型。

**3分钟详答：**
下面两种写法等价：

```cpp
template <typename T>
class A {};
template <class T>
class A {};
```

这里 `typename` 和 `class` 都表示 T 是一个类型参数。

但在模板内部，如果某个类型依赖模板参数，编译器可能不知道它到底是类型还是静态成员变量。

例如：

```cpp
template <typename T>
void f() {
    typename T::value_type x;
}
```

这里 `T::value_type` 依赖于模板参数 `T`。
必须加 `typename` 告诉编译器：`T::value_type` 是一个类型。

如果不加，编译器可能无法解析。

所以：

- 模板参数列表里，`typename` 和 `class` 基本等价。
- 模板内部依赖类型前，必须用 `typename` 明确说明。

**可能的追问问题：**

- template 和 template 哪个更推荐？
  **15秒简答：** 类型参数两者都可以，现代 C++ 中很多人更习惯用 `typename`，语义更直接。
- 什么是依赖类型？
  **15秒简答：** 依赖模板参数才能确定的类型，比如 `T::value_type`。
- 为什么有时必须写 typename？
  **15秒简答：** 因为编译器在模板定义阶段不知道某个依赖名字是不是类型，需要 `typename` 明确说明。
