# 操作系统：进程、线程与同步

## 进程和线程有什么区别？在后端开发里该怎么选？

**15秒简答：**  
进程是资源分配的基本单位，线程是 CPU 调度的基本单位。

进程有**独立地址空间，隔离强但切换和通信成本**更高；

线程**共享进程内存，通信方便、创建和切换更轻**，但更容易出现竞态和崩溃扩散。

后端里一般**同一服务内部优先多线程**，**隔离性要求高或需要容灾时会用多进程。**

**3分钟详答：**  
进程可以理解为“资源容器”，它拥有**独立的虚拟地址空间、文件描述符表、信号处理上下文**等；

线程则是“执行路径”，同一个进程内的线程共享代码段、堆、全局变量、打开的文件等资源，但每个线程有自己**独立的栈、寄存器上下文、线程局部存储**。  

在工程上，二者最核心的区别有四点：  
第一，**隔离性**。一个进程崩了，不一定影响别的进程；而一个线程写野指针，往往能把整个进程带崩。  
第二，**通信成本**。线程之间共享地址空间，传指针和共享队列就能通信；进程之间通常要借助管道、共享内存、socket 等 IPC。  
第三，**切换成本**。线程切换通常更轻，因为同进程线程共享大部分资源；而进程切换涉及地址空间切换，代价更高。  
第四，**可控性**。多进程更适合做隔离、权限边界和容灾；多线程更适合做高吞吐任务处理。  

在后端服务里，如果是一个 RPC 服务或 HTTP 服务，常见做法是：主进程负责启动和资源初始化，工作线程池负责处理请求。因为线程共享内存，任务队列、缓存、连接池更容易复用。  
但如果是对稳定性要求很高的场景，比如网关、Nginx、Redis 这种非常注重单个工作单元故障隔离的服务，往往会采用多进程或“多进程 + 事件驱动”的方式。  

面试里不要只背定义，最好顺手补一句：**线程不是越多越好**。CPU 密集型更看核心数；I/O 密集型可以多一些线程，但线程过多会导致调度开销和锁竞争变重。

**可能追问：**

- 为什么线程切换通常比进程切换轻？  
  - 15秒简答：同进程线程共享地址空间和大部分资源，切换时主要保存/恢复寄存器、栈等执行上下文；进程切换通常还涉及页表切换、TLB 失效等额外开销，所以更重。
- 多进程相比多线程的优势是什么？  
  - 15秒简答：隔离性强，单个工作单元崩溃不容易拖死整个服务；权限控制更自然，也更适合把不同模块拆开独立部署。
- 多线程相比多进程的优势是什么？  
  - 15秒简答：创建和切换成本通常更低，线程间共享数据方便，适合同一服务内部做任务并行和请求处理。

## 【高频】进程间通信有哪些方式？各自适合什么场景？

**15秒简答：**  
常见 IPC 有管道、消息队列、共享内存、信号量、socket。

**共享内存最快**，但要自己处理同步；

管道适合有亲缘关系的进程做简单单向通信；

socket 最通用，既能本机也能跨机；消息队列适合按消息边界传输；信号量更偏同步而不是传数据。

**3分钟详答：**   

**第一类：管道 / 命名管道。**  
匿名管道通常用于父子进程，特点是简单、半双工、字节流；命名管道适用范围更广，但本质上还是本机 IPC，适合做简单数据传输。优点是使用简单，缺点是功能有限、吞吐也一般。  

**第二类：消息队列。**  
它是内核维护的消息链表，天然有消息边界。相比管道，消息队列更适合“一条条消息”通信，写业务逻辑更直观。但它仍然是本机 IPC，性能和容量也有限。  

**第三类：共享内存。**  
这是本机多进程中效率最高的一类，因为数据不用在内核和用户态之间来回复制多次，多个进程直接映射到同一块物理内存。问题在于：共享内存只解决“看见同一份数据”，不解决“怎么安全地一起改”，所以通常要配合互斥锁、信号量、原子变量一起用。  

**第四类：信号量。**  
它更偏同步原语，不是拿来传大块业务数据的，而是控制访问顺序，比如控制共享资源同时最多允许几个进程进入。  

**第五类：socket。**  
这是面试里最值得强调的，因为它最通用：同机进程可以用 Unix Domain Socket，跨机进程可以用 TCP/UDP socket。很多后端服务、微服务、代理程序底层都是 socket 通信。它的优点是统一、通用、可扩展到分布式；缺点是相对共享内存来说开销更大。  

真正回答时可以给出选型逻辑：  
- **大量数据 + 本机高性能**：共享内存  
- **简单父子进程通信**：管道  
- **需要按消息组织**：消息队列  
- **需要统一成本机 / 跨机通信**：socket  
- **需要控制并发访问**：信号量  

**可能追问：**
- 为什么共享内存快？  
  - 15秒简答：因为**多个进程直接访问同一块物理内存**，省去了很多中间复制，内核主要负责建立映射，不用每次都搬运大块数据。
- 共享内存为什么还需要同步机制？  
  - 15秒简答：因为“共享”只保证大家能看到同一份数据，不保证并发读写安全；多个进程同时修改会有竞态条件，所以要配锁、信号量或原子操作。
- 本机进程通信为什么很多时候也会用 socket？  
  - 15秒简答：因为接口统一、工程实现简单，而且未来容易扩展到跨机，不用改通信模型。

## 线程同步方式有哪些？互斥锁、自旋锁、读写锁、条件变量、信号量分别适合什么场景？

**15秒简答：**  
互斥锁适合一般临界区保护；

自旋锁适合锁持有时间很短、线程切换代价大时；

读写锁适合读多写少；条件变量用于“等条件成立”；

信号量适合控制资源数量。核心不是背定义，而是看临界区长度、读写比例和是否需要阻塞等待。

**3分钟详答：**  
这题是后端和客户端岗位都很喜欢问的，因为它直接对应代码里的并发控制。  
可以从“锁解决什么问题”来理解：线程同步本质是在解决 **竞态条件、可见性、执行顺序** 这几类问题。  

**互斥锁** 最常见，它的语义是“同一时刻只有一个线程进入临界区”。优点是语义直白，适用面最广；缺点是如果竞争激烈，会让线程阻塞和唤醒，带来上下文切换开销。  

**自旋锁** 的思路是不睡眠，而是在原地循环等待。它适合锁持有时间特别短的场景，比如内核或高性能运行时里很短的临界区。因为如果马上就能拿到锁，睡眠再唤醒反而更亏；但如果持锁时间长，自旋会空耗 CPU。  

**读写锁** 允许多个读者并发进入，但写者独占。它适合缓存、配置读取、路由表这种“读远多于写”的场景。如果写很多，读写锁反而可能退化。  

**条件变量** 不是用来互斥的，而是用来“等待某个条件变为真”。典型场景是生产者消费者：队列空了，消费者睡眠；生产者放入数据后发通知把它唤醒。条件变量通常要配合互斥锁使用，因为“检查条件 + 进入等待”必须是原子的。  

**信号量** 可以看作带计数的许可系统。二值信号量类似互斥，计数信号量适合做“资源池”控制，比如数据库连接池允许同时最多 N 个连接被借出。  

面试里再多说一句会很加分：  
- 互斥锁适合通用场景  
- 自旋锁适合短临界区  
- 读写锁适合读多写少  
- 条件变量适合等待事件  
- 信号量适合控制配额  

这就是典型的“按场景选同步原语”。

**可能追问：**
- 条件变量为什么一定要和互斥锁一起用？  
  - 15秒简答：因为需要保证“判断条件”和“进入等待”之间没有竞态，否则可能发生通知丢失。
- 自旋锁什么时候不适合用？  
  - 15秒简答：临界区较长、竞争激烈、单核环境或者线程可能被抢占时，自旋会白白浪费 CPU。
- 读写锁为什么在写多场景下可能不划算？  
  - 15秒简答：因为写锁独占，读写切换还要维护额外状态，写多时并发优势发挥不出来，反而增加开销。

## 什么是死锁？产生死锁的必要条件是什么？怎么避免？

**15秒简答：**  
死锁就是多个线程或进程互相等待对方持有的资源，最终谁也走不下去。

经典四个必要条件是互斥、占有且等待、不可抢占、循环等待。避免方法通常是破坏其中至少一个条件，比如统一加锁顺序、一次性申请资源、加超时和降级策略。

**3分钟详答：**  
死锁是并发面试里的常驻题，面试官并不只想听四个条件，更想看你有没有工程意识。  
先说定义：如果多个执行单元都在等待别人释放资源，而这种等待形成闭环，就进入死锁。和“慢”“卡”不同，死锁是逻辑上无法前进。  

经典四个必要条件：  
1. **互斥**：资源同一时刻只能一个人用  
2. **占有且等待**：已经拿到一部分资源，又去等其他资源  
3. **不可抢占**：资源不能被别人强行拿走  
4. **循环等待**：A 等 B，B 等 C，C 又等 A  

只要破坏掉其中一个，死锁就不会成立。工程上最常用的做法有：  

**第一，统一加锁顺序。**  
比如所有线程都先锁 A 再锁 B，就不会出现一个线程先拿 A 等 B，另一个线程先拿 B 等 A。这个方法最实用，也是面试里最推荐先答的。  

**第二，一次性申请全部资源。**  
拿不到就都不拿，避免“拿着一个等另一个”。  

**第三，引入超时、try_lock、回退机制。**  
就算冲突了，也不要永久等下去，而是失败后释放已拿到的资源，稍后重试。工程里这比纯理论更常见。  

**第四，减少锁粒度和锁嵌套。**  
很多死锁都是因为临界区设计太大、调用层次太深。  

如果面试官继续追问，可以补充：  
死锁和活锁、饥饿要区分开。死锁是永远不动；活锁是大家都在动，但策略冲突导致谁也做不成；饥饿是有人长期拿不到资源。  

**可能追问：**
- 怎么排查死锁？  
  - 15秒简答：先看线程栈，确认哪些线程阻塞在锁上；再看锁顺序是否形成环；线上常用日志、线程 dump、死锁检测工具辅助定位。
- 统一加锁顺序为什么能避免死锁？  
  - 15秒简答：因为它破坏了循环等待条件，所有线程都按同一顺序拿锁，就很难形成资源环。
- try_lock 有什么意义？  
  - 15秒简答：它让线程拿不到锁时可以主动失败、回退或重试，避免无限等待，常用于降低死锁风险。

# 操作系统：内存管理与系统调用

## 什么是虚拟内存？为什么操作系统要引入它？

**15秒简答：**  
虚拟内存就是**让每个进程看到连续、独立的虚拟地址空间，**再由**操作系统和硬件把虚拟地址映射到物理内存。**

它的意义是**隔离进程、提高内存利用率、支持按需加载和大程序运行。**

**3分钟详答：**  
如果没有虚拟内存，程序直接操作物理地址，会有三个大麻烦：  
第一，不同进程很难安全共存，因为可能写到别人的内存；  
第二，程序需要关心自己被装载到哪里，编程复杂；  
第三，物理内存有限，无法优雅支持“程序比内存大”的情况。  

引入虚拟内存以后，每个进程都有自己独立的虚拟地址空间。程序只管用虚拟地址，CPU 在访问时通过页表把虚拟地址翻译成物理地址。  
它带来的几个核心好处是：  

**1. 隔离性。**  
每个进程都以为自己独占一大片内存，访问权限也由页表控制，非法访问会触发异常。  

**2. 按需加载。**  
程序运行时并不是一次性把所有代码和数据都放进物理内存，而是“访问到哪一页，再加载哪一页”，这就是分页和缺页处理的基础。  

**3. 提高利用率。**  
多个进程可以共享只读代码页，比如同一个动态库；还能通过换页把暂时不用的数据换到磁盘上。  

**4. 简化编程模型。**  
应用程序不需要感知自己真正在哪块物理内存，只看到逻辑上连续的地址空间。  

虚拟内存不是为了“变快”，而是为了**隔离、抽象和更高的资源利用率**。当然它也会引入地址翻译开销，所以才需要 TLB 来加速。

**可能追问：**
- 虚拟内存一定比直接访问物理内存快吗？  
  - 15秒简答：不一定。虚拟内存的主要价值是隔离和抽象，不是单纯加速；它会增加地址翻译成本，所以需要 TLB 做缓存。
- 一个进程的虚拟地址空间为什么可以比物理内存大？  
  - 15秒简答：因为虚拟地址只是逻辑空间，不要求全部同时映射到物理内存，可以按需加载，暂时不用的页也可以不驻留在内存里。
- 共享库为什么能被多个进程复用？  
  - 15秒简答：因为相同的只读代码页可以映射到多个进程的虚拟地址空间，底层共享同一份物理页。

## 【高频】页表、TLB、缺页异常分别是什么？它们之间是什么关系？

**15秒简答：**  
页表负责虚拟地址到物理地址的映射；

TLB 是页表项的高速缓存，用来减少地址翻译开销；

访问页面时如果页表发现该页不在内存或权限不满足，就会触发缺页异常，由操作系统处理。

**3分钟详答：**  
这题本质上是在问“CPU 访问一次内存，背后发生了什么”。  
程序给出的是虚拟地址，CPU 不能直接拿它去访问物理内存，而是要做地址翻译。页表就是这个翻译规则，它通常以“页”为单位记录虚拟页号对应到哪个物理页框，以及访问权限、是否存在、是否脏页等状态。  

但如果每次访问内存都先查页表，开销太大，所以硬件里会有 **TLB**，也就是快表。TLB 缓存最近使用过的一小部分页表项。  
访问流程通常是：  
1. CPU 先查 TLB  
2. 命中则直接拿到物理页框号  
3. 未命中则继续查页表  
4. 查到后回填 TLB，再完成访问  

如果查页表时发现该页没有映射到物理内存，或者权限不允许，比如写只读页，就会触发 **缺页异常**。  
缺页异常不一定是坏事，它分两类理解：  

- **合法但未装入内存**：比如按需加载，OS 会把对应页从磁盘调入，再更新页表继续执行  
- **非法访问**：比如野指针访问未分配区域，OS 会直接杀掉进程或发信号  

面试时再加一句更完整：  
TLB 的存在是为了把“页表查找”从常规内存访问尽量变成一次高速缓存命中，否则虚拟内存机制的性能损耗会很明显。  

**可能追问：**
- TLB miss 一定会触发缺页异常吗？  
  - 15秒简答：不一定。TLB miss 只是快表没命中，继续查页表即可；只有页表也发现该页不存在或权限有问题，才会触发缺页异常。
- 缺页异常一定是错误吗？  
  - 15秒简答：不一定。按需加载时的缺页是正常机制，非法访问导致的缺页才是错误。
- 为什么页表不能做成一个简单数组直接查到底？  
  - 15秒简答：地址空间太大，简单平铺会非常浪费内存，所以实际会用多级页表等结构降低空间开销。

## 【高频】fork 为什么通常很快？什么是写时复制（COW）？

**15秒简答：**  
`fork` 快的关键不是把父进程内存全量复制一遍，而是先复制页表，让父子进程共享物理页，并把这些页标成只读。只有某一方真正写入时，才会复制出新页，这就是写时复制。

**3分钟详答：**  
这是 Linux 和服务端面试里特别高频的一题。很多人会说“fork 会复制父进程”，但如果真按字节全复制，代价会非常大，所以现代系统不会这么做。  
更准确地说，`fork` 创建子进程时，**先复制的是进程元信息和页表**，而不是立刻复制整片用户态内存。父子进程最开始会共享同一批物理页，这些共享页会被标记成只读。  

接下来如果父子进程都只读，那就一直共用这批页，不发生实际拷贝。  
只有当某一方尝试写某个共享页时，CPU 会因为写只读页触发异常，内核捕获后为写入方分配一个新的物理页，把旧内容复制过去，再更新页表，让它写自己的副本。这就是 **Copy-On-Write，写时复制**。  

这样做的好处是：  
- 大多数 `fork` 后很快会接 `exec`，根本不需要复制整份内存  
- 即使不 `exec`，也只有真正被修改的页才会发生复制  
- 因此 `fork` 的主要成本变成了复制页表和创建任务结构，而不是复制整片地址空间  

这也是为什么很多服务框架会利用 `fork` 快速派生工作进程。  

面试时建议多补一句：  
**COW 的前提是“先共享、后写时分离”**。它既提升性能，也节省内存，但如果 `fork` 后父子都大量写内存，那复制成本还是会显现。

**可能追问：**
- `fork` 后父子进程的变量值一开始相同吗？  
  - 15秒简答：是的，逻辑上看是一份拷贝；但底层很多页一开始是共享的，直到一方写入才真正分离。
- 为什么很多程序 `fork` 后马上 `exec`？  
  - 15秒简答：因为只想复用父进程的启动上下文来创建新进程，然后直接装载新程序，这样能最大化利用 COW，避免无意义的内存复制。
- COW 和共享内存有什么区别？  
  - 15秒简答：COW 是“先共享，写时分家”；共享内存是“设计上就要求多进程长期共同读写同一片内存”。

## 什么是 mmap？它和 read/write 相比有什么特点？

**15秒简答：**  
`mmap` 是把文件或设备映射进进程地址空间，之后就能像访问内存一样访问文件内容。

它适合大文件、随机访问和共享映射场景；相比 `read/write`，它可以减少一次显式拷贝和系统调用次数，但也会引入缺页、映射管理和一致性问题。

**3分钟详答：**  

传统 `read/write` 模式是：程序发起系统调用，内核把文件内容拷到内核缓冲区，再拷到用户缓冲区，用户再处理。  
而 `mmap` 的思路是：把文件的某个区域直接映射到进程虚拟地址空间。程序访问这段地址时，操作系统按页把数据调入内存。这样应用层看起来像在读内存，不再需要反复 `read`。  

它的典型优点有：  
- **适合大文件和随机访问**，不必手动维护复杂的偏移读取逻辑  
- **减少系统调用次数**  
- **便于多个进程共享同一份映射**，尤其是 `MAP_SHARED` 场景  
- 在某些场景下能减少不必要的数据搬运  

但它也不是银弹：  
- 首次访问某页可能触发缺页异常，延迟不一定稳定  
- 映射区生命周期和地址管理更复杂  
- 文件被截断或外部修改时要小心一致性问题  
- 不适合特别简单的小块顺序读写场景，代码复杂度可能得不偿失  

面试时可以给一个很实用的对比结论：  
**顺序小文件读写，用 `read/write` 更直观；大文件、随机访问、共享访问，用 `mmap` 更常见。**

**可能追问：**
- `MAP_SHARED` 和 `MAP_PRIVATE` 的区别是什么？  
  - 15秒简答：`MAP_SHARED` 对映射区的修改会反映到底层对象，多个映射者可见；`MAP_PRIVATE` 是私有写时复制，修改只对当前进程可见。
- `mmap` 一定比 `read` 快吗？  
  - 15秒简答：不一定。它更适合大文件和随机访问；如果数据很小、只读一次，`read` 可能更简单更稳定。
- 为什么说 `mmap` 也可能触发缺页异常？  
  - 15秒简答：因为文件页通常不是一开始就全部装入内存，而是在访问到对应虚拟页时按需加载。

## 什么是用户态和内核态？一次系统调用大致发生了什么？

**15秒简答：**  
用户态运行应用代码，权限受限；

内核态运行操作系统核心代码，能访问受保护资源。系统调用本质上是**从用户态受控地进入内核态，请求内核帮应用完成文件、网络、进程管理等敏感操作，然后再返回用户态。**

**3分钟详答：**

操作系统为什么要区分两种态？因为不能让普通应用随便改页表、关中断、直接操作硬件，否则整个系统就不安全了。  
所以 CPU 会提供不同特权级。日常业务代码运行在 **用户态**，权限较低；操作系统核心服务运行在 **内核态**，权限更高。  

当应用需要做一些敏感操作时，比如打开文件、收发网络包、创建进程、申请映射，它不能自己直接操作硬件，而是通过 **系统调用** 这个正式入口把请求交给内核。  
系统调用的大致过程可以这么讲：  

1. 用户态准备参数  
2. 通过特定指令陷入内核态  
3. CPU 切换到内核栈，保存现场  
4. 内核根据系统调用号找到对应处理逻辑  
5. 完成检查、资源访问和结果填充  
6. 返回用户态，恢复执行  

为什么系统调用比普通函数调用重？  
因为它涉及特权级切换、上下文保存恢复，有时还伴随调度、拷贝和阻塞等待。  

工程上这会影响很多设计：  
- 为什么高性能程序会减少系统调用次数  
- 为什么网络库爱用批量读写、事件驱动  
- 为什么用户态缓存和对象池常常能带来收益  

面试里再补一句：  
**用户态和内核态不是“哪个更高级”，而是权限和职责不同。** 这样表述会更稳。

**可能追问：**
- 系统调用和普通函数调用的主要区别是什么？  
  - 15秒简答：普通函数调用仍在用户态执行；系统调用会触发用户态到内核态的切换，由内核代为完成受保护操作。
- 为什么频繁系统调用会影响性能？  
  - 15秒简答：因为有特权级切换、上下文保存恢复，还可能伴随阻塞、唤醒和数据拷贝，成本远高于普通函数调用。
- 应用程序能不能直接访问硬件？  
  - 15秒简答：通常不能，必须通过操作系统提供的接口间接访问，这样系统才能保证安全和隔离。

# 操作系统：I/O 模型与高并发

## 阻塞、非阻塞、同步、异步分别是什么意思？它们容易混在哪？

**15秒简答：**  
阻塞/非阻塞描述的是调用方在结果没准备好时会不会被挂住；

同步/异步描述的是结果就绪后谁来处理完成这件事。同步 I/O 往往要自己主动收结果，异步 I/O 则由系统在完成后通知你。面试里最常见的混淆是把“非阻塞”误说成“异步”。

**3分钟详答：**  
这是网络编程里特别爱问的概念题，因为很多人把四个词混成两组。  
你可以这样拆：  

**阻塞 / 非阻塞** 看的是“当前这次调用会不会把线程卡住”。  
比如一个阻塞式 `read`，如果数据没到，线程就睡在那里；

非阻塞 `read` 如果没数据，会立刻返回 `EAGAIN`，线程不会挂住。  

**同步 / 异步** 看的是“谁来负责等待和收尾”。  
同步 I/O 的本质是：应用自己关注这个 I/O 什么时候能做、什么时候完成。  
异步 I/O 则是：应用把请求交出去，真正完成后由系统主动通知，应用不需要自己反复轮询。  

所以常见组合是：  
- **同步阻塞**：最直观，比如阻塞式 `read`  
- **同步非阻塞**：比如非阻塞 socket + `epoll`，线程不在单个 fd 上阻塞，但 I/O 完成这件事仍是应用自己根据就绪事件去处理  
- **异步非阻塞**：典型是内核完成 I/O 后异步通知  

为什么说 `epoll` 通常仍属于同步 I/O？  
因为 `epoll` 告诉你“这个 fd 可以读/写了”，但真正的 `read/write` 还是你自己调，数据搬运和收尾逻辑还是应用自己负责。它是 **I/O 就绪通知**，不是把整个 I/O 生命周期都托管给内核。  

面试时把这句说出来，通常会比较加分：  
**非阻塞不等于异步，多路复用也不等于异步。**

**可能追问：**
- `epoll` 是同步还是异步？  
  - 15秒简答：通常归类为同步 I/O 的就绪通知模型，因为内核只是通知你“能读/能写了”，真正的读写还是应用自己做。
- 非阻塞 socket 为什么还可能需要循环重试？  
  - 15秒简答：因为它只是保证“现在拿不到结果就立即返回”，并不保证这次一定完成，所以应用要配合事件通知或重试机制。
- 同步阻塞和同步非阻塞哪个更好？  
  - 15秒简答：没有绝对好坏。连接少、逻辑简单时阻塞模型足够；高并发网络服务一般更偏同步非阻塞 + I/O 多路复用。

## 【高频】select、poll、epoll 有什么区别？为什么高并发服务更偏向 epoll？

**15秒简答：**  
三者都是 I/O 多路复用。`select` 有 fd 数量上限且每次都要重置集合；`poll` 去掉了固定上限，但本质上仍要遍历整个监听集合；`epoll` 把“关注集合”和“就绪集合”分开管理，适合大量连接、少量活跃的场景，所以 Linux 高并发服务更常用它。

**3分钟详答：**  
这题基本可以算 Linux 网络面试的必考题。  
它们三者都在做同一件事：一个线程同时监听多个文件描述符，看看哪些准备好了。区别主要在 **数据结构、内核交互方式、扩展性**。  

**select** 最老。它用位图表示监听集合，调用时把整个位图从用户态拷到内核态，内核检查完后再把结果拷回来。问题是：  
- 有 `FD_SETSIZE` 限制，默认常见是 1024  
- 每次调用都要重新设置监听集合  
- 内核通常要线性扫描全部 fd  

**poll** 比 `select` 稍好一点。它用数组而不是位图，去掉了固定 fd 上限，但本质问题没变：每次调用还是把整个监听列表交给内核，返回后应用还要遍历看看哪些就绪。  

**epoll** 是 Linux 专门为大规模 fd 场景设计的。它把工作拆成两步：  

1. 用 `epoll_ctl` 把“我关心哪些 fd、关心什么事件”注册进内核  
2. 用 `epoll_wait` 只取回当前已经就绪的那部分 fd  

这带来两个关键好处：  
- 不需要每次都把整个监听集合来回搬  
- 返回的是“就绪列表”，不是让你自己扫全量集合  

因此在 **连接很多、但每时每刻活跃连接只占一小部分** 的典型后端场景下，`epoll` 的优势非常明显。  
但面试里不要绝对化地说“epoll 一定最快”。如果 fd 很少、而且大部分都很活跃，`select/poll` 也能满足需求，代码甚至更简单；而且 `epoll` 是 Linux 特有，不像 `select/poll` 更通用。  

一个很稳的回答方式是：  
**高并发服务偏向 epoll，不是因为名字高级，而是因为它更适合海量连接、稀疏就绪的事件驱动模型。**

**可能追问：**
- `select` 为什么有 1024 限制？  
  - 15秒简答：因为它使用固定大小的位图集合表示 fd，受 `FD_SETSIZE` 限制，默认常见值就是 1024。
- `poll` 比 `select` 强在哪里？  
  - 15秒简答：`poll` 去掉了固定 fd 上限，接口表达更灵活，但仍然要把整个监听集合传给内核并遍历检查。
- `epoll` 一定是 O(1) 吗？  
  - 15秒简答：面试里别机械背复杂度，更准确地说它避免了每次全量扫描，通常在大量连接、少量活跃场景下表现更好。

## 【高频】epoll 的 LT 和 ET 有什么区别？为什么 ET 一般要配合非阻塞并且读到 EAGAIN？

**15秒简答：**  
LT 是“只要缓冲区里还有数据，就会反复通知”；ET 是“状态从无到有变化时通知一次”。ET 减少重复通知，但如果你一次没把数据读完，后面可能收不到新通知，所以一般要把 fd 设成非阻塞，并循环读到 `EAGAIN` 为止。

**3分钟详答：**  
这是 `epoll` 里最有区分度的一道题，因为会背“LT/ET 区别”还不够，面试官更想看你知不知道为什么 ET 容易踩坑。  

**LT，Level Triggered，水平触发。**  
只要条件一直满足，比如 socket 接收缓冲区里还有数据没读完，`epoll_wait` 下次还会继续告诉你这个 fd 可读。它更像“提醒式”，编程更稳，不容易漏事件。  

**ET，Edge Triggered，边沿触发。**  
只有状态发生变化时才通知，比如从“不可读”变成“可读”时通知一次。如果你收到通知后只读了一半，缓冲区里虽然还剩数据，但因为“可读状态”没有再次发生边沿变化，后续就不一定再提醒了。  

这就是为什么 ET 模式通常要求两件事：  
**第一，把 fd 设成非阻塞。**  
因为你要在一次事件回调里尽可能把能读的数据都读完；如果是阻塞 fd，读到一半没更多数据时线程可能直接卡住，整个事件循环就废了。  

**第二，循环读/写直到返回 `EAGAIN`。**  
`EAGAIN` 的意思是“当前已经没有更多可读/可写数据了”，你才能放心退出本轮处理。这样才能避免把未处理数据遗留在缓冲区却收不到下一次通知。  

工程上怎么选？  
- 想要稳定、实现简单：LT  
- 想减少重复通知、追求更高事件处理效率：ET，但实现必须足够严谨  

很多成熟网络库都是在细节处理到位的前提下使用 ET，不然很容易出现“连接偶尔卡死、消息读不全”的隐蔽 bug。

**可能追问：**
- LT 会不会更容易写？  
  - 15秒简答：会。因为没读完也还会继续收到通知，不容易漏事件，所以大多数初版服务器更容易先用 LT 写对。
- ET 为什么通常要配合非阻塞？  
  - 15秒简答：因为 ET 期望你一次把缓冲区尽量处理干净，若 fd 是阻塞的，读到没数据时线程可能挂住，影响整个事件循环。
- `EAGAIN` 代表什么？  
  - 15秒简答：表示在非阻塞模式下，这次操作现在做不了，不是致命错误，通常意味着当前缓冲区已经被你读空或暂时不能继续写。

## 【高频】什么是零拷贝？常见零拷贝方案有哪些？

**15秒简答：**  
零拷贝不是完全没有拷贝，而是尽量减少用户态和内核态之间不必要的数据复制与上下文切换。常见方案有 `sendfile`、`mmap + write`、DMA、以及网卡支持下的更深层 offload。后端里最常见的理解是“文件发送时避免先读到用户态再写回内核”。

**3分钟详答：**  
这题在服务器开发、网关、存储、CDN 相关岗位里出现频率很高。  
传统文件发送路径通常是：  
1. 磁盘数据进内核缓冲区  
2. 从内核拷到用户缓冲区  
3. 用户再调用 `write`  
4. 又从用户拷回内核 socket 缓冲区  
5. 最后交给网卡发送  

这里最明显的问题是：数据只是“路过”应用层，但应用并没有真正处理内容，却发生了多次内存复制和系统调用。  

所谓 **零拷贝**，更准确地说是**减少 CPU 参与的数据搬运**。  
常见方案有：  

**1. `sendfile`**  
适合“磁盘文件 -> socket”这种场景。应用只告诉内核“把这个文件发到这个连接上”，内核直接完成搬运，避免数据先拷到用户态。Nginx、静态文件服务常用。  

**2. `mmap + write`**  
把文件映射进用户地址空间，省掉一次 `read` 复制，但最终写 socket 时通常仍需要把数据送到内核网络栈，所以它不一定比 `sendfile` 更优。  

**3. DMA 和硬件 offload**  
磁盘到内存、内存到网卡本身很多是 DMA 参与，减少 CPU 搬运。更高级的网卡/内核能力还会进一步减轻 CPU 负担。  

面试里要注意一个坑：  
别把零拷贝讲成“数据在机器里完全没有移动”。真实情况是很多时候仍然有 DMA 搬运，所谓零拷贝更多是指**减少用户态参与的显式拷贝**。  

工程上收益主要体现在：  
- 降低 CPU 占用  
- 减少内存带宽浪费  
- 减少系统调用与上下文切换  
- 提升大文件传输吞吐  

**可能追问：**
- `sendfile` 为什么适合静态文件服务？  
  - 15秒简答：因为应用层通常不需要修改文件内容，直接让内核把文件送到 socket，能减少一次用户态中转和系统调用。
- `mmap + write` 算不算零拷贝？  
  - 15秒简答：一般算“减少拷贝”的一种，但未必是最彻底的，因为写 socket 时通常仍有数据进入内核网络发送路径。
- 零拷贝的核心收益是什么？  
  - 15秒简答：减少 CPU 搬运、系统调用和上下文切换，提高吞吐并降低延迟抖动。

## 【高频】什么是惊群现象？在服务端里通常怎么避免？

**15秒简答：**  
惊群就是一个事件到来时唤醒了很多等待者，但最后只有少数甚至一个真正拿到资源，其余线程或进程被白白唤醒。它会浪费 CPU、加重调度开销。常见规避方式是减少无效唤醒、合理使用事件分发、连接负载均衡和 one-shot 机制。

**3分钟详答：**  
惊群现象在网络服务、连接接入、多线程 epoll 场景里都可能出现。  
典型例子是：多个工作线程都在等同一个监听 socket 上的新连接，结果来了一个连接，内核把很多线程都唤醒了，但最终只有一个线程成功 `accept` 到连接，其他线程被唤醒后又睡回去。  

问题在于：这些“陪跑线程”虽然什么也没干成，却消耗了调度、缓存、锁竞争等成本。连接量一大，系统就会出现很多无效唤醒。  

为什么会发生？本质上是：  
**一个事件被广播给了过多候选者，而资源最终只能由少数人消费。**  

在工程里，常见的缓解方式有：  
- 让一个专门线程负责 `accept`，再把连接分发给工作线程  
- 使用更细粒度的事件分发策略，避免所有线程同时盯同一个 fd  
- 合理使用 `EPOLLONESHOT` 等机制，避免一个 fd 被多个线程同时处理  
- 多进程 / 多线程场景结合负载均衡和 SO_REUSEPORT 等机制，让连接更均匀落到不同工作单元  

回答这题时最好体现一个意识：  
**惊群不是“功能错了”，而是“性能浪费了”。**  
程序可能还能工作，但吞吐和延迟会变差。

**可能追问：**
- 惊群为什么会影响性能？  
  - 15秒简答：因为大量线程被无效唤醒后又竞争又休眠，带来额外调度、缓存失效和锁开销。
- 一个线程专门做 `accept` 有什么好处？  
  - 15秒简答：它把连接接入和业务处理分开，能减少多个工作线程同时抢监听 socket 导致的无效唤醒。
- `EPOLLONESHOT` 的作用是什么？  
  - 15秒简答：一次事件通知后先把 fd 置为“仅本次有效”，处理完再重新注册，避免同一个 fd 被多个线程重复并发处理。

# 计算机网络：TCP / UDP 与传输层

## 【高频】TCP 和 UDP 有什么区别？各自适合什么场景？

**15秒简答：**  
TCP 面向连接、可靠、字节流、有拥塞控制和流量控制；UDP 无连接、不保证可靠、面向报文，开销更小、时延更低。需要可靠性和顺序的一般选 TCP；更看重实时性、能容忍丢包或自己实现可靠性的场景常选 UDP。

**3分钟详答：**  
这题是网络基础中的绝对高频题。  
最稳的答法不是背表格，而是先说二者定位：  
TCP 是“可靠传输服务”，UDP 是“尽力而为的报文投递服务”。  

**TCP 的特点：**  
- 建立连接后再通信  
- 保证数据按序、可靠到达  
- 面向字节流，没有天然消息边界  
- 有重传、确认、流量控制、拥塞控制  
- 更适合文件传输、Web、RPC、数据库连接等要求正确性的场景  

**UDP 的特点：**  
- 不需要建立连接  
- 面向报文，保留消息边界  
- 不保证到达、不保证顺序、不保证不重复  
- 协议头更小，时延更低  
- 更适合音视频、实时游戏、DNS、日志上报、服务发现这类更看重实时性或可容忍丢包的场景  

但实际工程里不能把 UDP 说成“没用的简化版 TCP”。  
很多高性能协议都会基于 UDP 自己补可靠性和拥塞控制，比如 QUIC。因为这样应用可以更灵活地决定重传策略、连接迁移、流控制粒度等。  

面试里补一句会显得更成熟：  
**TCP 和 UDP 不是谁更高级，而是服务目标不同。**  
如果业务要求“绝不能错、绝不能乱”，TCP 更自然；如果业务要求“宁可丢一点也不能卡”，UDP 反而更合适。

**可能追问：**
- 为什么说 TCP 是字节流，UDP 是报文？  
  - 15秒简答：TCP 只保证字节序列连续到达，不保留应用层一次次发送的边界；UDP 一次发送对应一个报文，边界天然保留。
- UDP 能不能实现可靠传输？  
  - 15秒简答：能，应用层自己加序号、确认、重传、窗口和拥塞控制就行，QUIC 就是典型例子。
- DNS 为什么很多时候用 UDP？  
  - 15秒简答：因为请求响应通常很小、对时延敏感，用 UDP 开销更低；但报文过大或某些场景下也会回退到 TCP。

## 【高频】TCP 为什么是三次握手？两次不行吗？

**15秒简答：**  
三次握手的核心作用是双方都确认“自己能发、也能收”，并同步初始序列号。两次握手只能保证客户端知道服务端能收能发，但服务端无法确认客户端是否收到了自己的应答，也就无法可靠确认连接建立。

**3分钟详答：**  
这题几乎每个后端面试都会问。  
回答时建议抓住两个关键词：**双向可达** 和 **序列号同步**。  

第一次握手：客户端发 SYN，表示“我想建立连接，这是我的初始序列号”。  
第二次握手：服务端回 SYN + ACK，表示“我收到了你的 SYN，也告诉你我的初始序列号”。  
第三次握手：客户端再回 ACK，表示“我收到了你的应答，你的序列号我也知道了，连接现在对双方都确认成立”。  

为什么不是两次？  
因为 TCP 连接不是“我单方面想连就算建立”，而是要确认双方收发链路都正常。  
如果只有两次，服务端在发出 SYN+ACK 后就认为连接建立，但它并不知道客户端是否真的收到了这个应答。第三次握手就是客户端对服务端的最终确认。  

再进一步，三次握手还能避免“旧的、失效的连接请求报文”引起的混乱。  
如果网络里滞留了一份过期 SYN，服务端收到后回了 SYN+ACK，但客户端其实并没有发起这次连接。如果没有第三次握手确认，服务端就可能白白建立一个无效连接。  

面试时最好加一句：  
**三次握手不只是为了礼貌确认，而是为了可靠建立双向通信状态并同步双方初始序列号。**

**可能追问：**
- 第三次握手可以带数据吗？  
  - 15秒简答：可以，在一些实现里客户端第三次 ACK 后就可以携带数据，但本质上它首先还是在完成连接确认。
- SYN 泛洪攻击是什么？  
  - 15秒简答：攻击者大量发送 SYN 但不完成后续握手，占满服务端半连接队列，导致正常用户连不上。
- 什么是半连接队列？  
  - 15秒简答：就是服务端收到 SYN 后但还没完成三次握手前，保存这些未完全建立连接请求的队列。

## 【高频】TCP 为什么是四次挥手？TIME_WAIT 有什么意义？

**15秒简答：**  
四次挥手是因为 TCP 是全双工的，双方关闭发送方向通常要分别进行：一方发 FIN 表示“我不发了”，对方先 ACK；等对方也发完自己的数据，再单独发 FIN。`TIME_WAIT` 的作用主要有两个：确保最后一个 ACK 能重发成功，以及让旧连接中的迟到报文在网络中自然消失。

**3分钟详答：**  
这是 TCP 里和三次握手并列的高频题。  
先说为什么通常是四次。TCP 是全双工协议，A 不发数据了，不代表 B 也不发了。  
所以当 A 发送 FIN 时，只表示“我的发送方向关闭了”。B 收到后先回 ACK，表示“我知道你不再发了”，但 B 可能还有剩余数据没发完。等 B 也发完，才会再发自己的 FIN。于是就形成了四次挥手。  

为什么有时抓包看起来像三次？  
因为 B 的 ACK 和它自己的 FIN 可能会合并发送，但协议语义上仍然是两个动作。  

再说 `TIME_WAIT`。  
主动关闭连接的一方在发出最后 ACK 后，通常不会立刻把连接状态删掉，而是进入 `TIME_WAIT`。它的意义主要有两个：  
**第一，防止最后一个 ACK 丢失。**  
如果对方没收到 ACK，会重发 FIN；此时处于 `TIME_WAIT` 的一方还能重发 ACK。  
**第二，避免旧连接中的延迟报文污染新连接。**  
保留一段时间，让网络里可能残留的旧报文自然过期，避免同样四元组的新连接误收到旧数据。  

面试时很容易被追问“为什么是 2MSL”，你可以答：  
因为要覆盖报文在网络中的最大生存时间及其来回过程，保证旧报文不会混入新连接。  

工程上再补一句：  
大量短连接服务里 `TIME_WAIT` 会很多，这是正常现象；真正要做的是优化连接复用和内核参数，而不是简单粗暴地关闭它。

**可能追问：**
- 什么是主动关闭方？为什么 `TIME_WAIT` 多出现在它身上？  
  - 15秒简答：主动先发起关闭的一方通常进入 `TIME_WAIT`，因为它负责发送最后 ACK，需要保留状态来处理对方可能重传的 FIN。
- `CLOSE_WAIT` 和 `TIME_WAIT` 有什么区别？  
  - 15秒简答：`CLOSE_WAIT` 说明本端收到了对方 FIN，但应用还没把自己的连接彻底关掉；`TIME_WAIT` 说明本端已完成最后 ACK，正在等待旧报文消失。
- 服务端出现很多 `TIME_WAIT` 一定是异常吗？  
  - 15秒简答：不一定。高并发短连接场景很常见，关键看是否影响端口复用、资源占用和业务吞吐。

## 【高频】TCP 是怎么保证可靠传输的？

**15秒简答：**  
TCP 的可靠性不是靠单一机制，而是靠一整套组合：序列号保证有序，ACK 确认已收，超时重传处理丢包，校验和检查损坏，接收端去重和重排处理乱序，滑动窗口做流量控制，再配合拥塞控制避免把网络打爆。

**3分钟详答：**  
这题回答得好不好，能很明显区分“背过八股”和“理解过 TCP”。  
可靠传输至少包含四层意思：**不丢、不错、不乱、不重**。  
TCP 是靠多个机制共同完成的。  

**1. 序列号**  
发送的每个字节都有序号，接收方根据序号判断数据位置，才能识别乱序、重复和缺失。  

**2. ACK 确认**  
接收方会确认自己已经连续收到的数据范围，发送方知道哪些已经成功到达。  

**3. 超时重传 / 快速重传**  
如果发送后迟迟收不到确认，发送方会认为可能丢包并重传。某些实现里还会根据重复 ACK 提前重传，提高效率。  

**4. 校验和**  
发现报文在传输过程中是否损坏，损坏则丢弃，由发送方后续重传。  

**5. 接收端重排与去重**  
网络可能导致乱序到达，TCP 会根据序列号重新排序；重复到达的报文会被识别并丢弃。  

**6. 滑动窗口**  
接收方通过窗口告诉发送方“我现在还能接收多少”，避免发送太快把接收方压垮。  

但这里要强调：  
**流量控制解决的是接收方吃不消，拥塞控制解决的是网络吃不消。**  
很多人把这两者混在一起。  

所以面试里一个比较完整的总结是：  
**TCP 通过序列号、确认应答、重传机制、校验和、滑动窗口以及相关控制算法，实现面向连接的可靠有序字节流传输。**

**可能追问：**
- TCP 为什么还能乱序到达？  
  - 15秒简答：因为网络中的不同分组可能走不同路径或遭遇不同排队时延，所以到达顺序和发送顺序不一定一致。
- 如果 ACK 丢了怎么办？  
  - 15秒简答：发送方在超时后会重传未确认数据；接收方如果已经收到过，会根据序列号去重并再次回复 ACK。
- 校验和能完全保证不出错吗？  
  - 15秒简答：它能显著降低错误漏检概率，但从工程上讲可靠性是多个机制共同保证的，不靠单一校验。

## 【高频】流量控制和拥塞控制有什么区别？

**15秒简答：**  
流量控制关注的是“接收方来不来得及处理”，本质是端到端的接收能力匹配；拥塞控制关注的是“网络本身会不会被打爆”，本质是发送速率对网络状态的适应。前者看接收窗口，后者看拥塞窗口。

**3分钟详答：**  
这是 TCP 面试里非常高频的对比题。  
很多人会把两者都说成“控制发送速度”，这不算错，但太浅。真正区分它们的是控制对象不同。  

**流量控制** 是为了保护接收方。  
比如服务端处理慢、接收缓冲区快满了，就通过接收窗口告诉发送方“你先少发点”。  
它解决的是“对端应用 / 缓冲区扛不住”的问题。  

**拥塞控制** 是为了保护网络。  
就算接收方很强，如果链路、交换机、路由器已经很忙，再继续猛发也会导致排队、丢包、重传，整体更慢。  
所以 TCP 会通过慢启动、拥塞避免等策略逐步试探网络容量，而不是一上来就打满。  

简化理解：  
- **流量控制**：别把对端撑爆  
- **拥塞控制**：别把网络撑爆  

在实现上，发送方真正能发多少，通常受到多个窗口共同约束，工程上常把它理解为受接收窗口和拥塞窗口共同限制。  

面试时再多说一句：  
如果网络丢包了，不一定就是接收方处理慢，也可能是链路拥塞；这就是为什么 TCP 需要同时有两套控制机制，而不是只靠接收方说“我还能收多少”。

**可能追问：**
- 接收窗口是谁告诉谁的？  
  - 15秒简答：由接收方通告给发送方，表示自己当前还能接收多少数据。
- 拥塞控制为什么不能只看是否丢包？  
  - 15秒简答：因为等明显丢包再降速往往已经晚了，现代实现通常会结合时延、ACK 行为等信号更早判断网络压力。
- 发送方真正可发送的上限受什么影响？  
  - 15秒简答：通常同时受接收方通告窗口和发送方的拥塞窗口约束，两者中更小的那个更关键。

## 【高频】为什么 TCP 会有粘包和拆包？怎么解决？

**15秒简答：**  
因为 TCP 是字节流，没有消息边界。应用层一次 `send`，对端不一定一次 `recv` 就按原样拿到，可能多条粘在一起，也可能一条被拆成多次收到。解决方法是在应用层自己设计边界，比如固定长度、分隔符、长度字段加消息体。

**3分钟详答：**  
这题几乎是所有服务端项目都会被问到，尤其你简历里写了聊天室、RPC、WebServer、网络库时更容易被追问。  
关键点是：**粘包拆包不是 TCP 出 bug，而是 TCP 的正常语义。**  
因为 TCP 提供的是可靠有序的字节流，不关心你应用层认为“一条消息”长什么样。  

导致粘包 / 拆包的原因主要有：  
- 发送方多次写入的数据，被 TCP 合并后一起发出  
- 接收方一次读取时，缓冲区里已经有多条消息  
- 一条很长的消息可能被分成多次到达  
- 接收方每次读取长度有限，天然可能把一条消息拆开  

所以解决方案一定在 **应用层协议设计**。常见做法有：  

**1. 固定长度协议**  
每条消息长度固定，读取简单，但空间利用率差。  

**2. 分隔符协议**  
比如一条消息以 `\r\n` 结尾，适合文本协议。缺点是正文里若可能出现分隔符，需要转义或额外处理。  

**3. 长度字段 + 消息体**  
最通用。先读取固定长度消息头，得到 body 长度，再按长度继续收完整消息。这是很多 RPC、IM、自定义二进制协议的常见方案。  

面试里最好说一句：  
服务端代码不能假设“一次 `recv` 对应一条完整业务消息”，而应该维护接收缓冲区和解包状态机。  

**可能追问：**
- UDP 会有粘包问题吗？  
  - 15秒简答：通常没有，因为 UDP 面向报文，消息边界由协议保留；但如果应用自己把一个大消息拆成多报文，仍然需要重组逻辑。
- 为什么长度字段方案更常见？  
  - 15秒简答：因为它适合二进制数据、效率高、边界清晰，不依赖正文内容里不能出现某些特殊字符。
- 服务端为什么要有接收缓冲区？  
  - 15秒简答：因为一次 `recv` 可能只拿到半条消息或多条消息，必须先缓存，再按协议持续解包。

# 计算机网络：HTTP / HTTPS 与应用层

## 【高频】HTTP 和 HTTPS 有什么区别？TLS 握手大致做了什么？

**15秒简答：**  
HTTPS 本质上就是 HTTP 跑在 TLS 之上。HTTP 明文传输，HTTPS 能提供机密性、完整性和身份认证。TLS 握手大致做三件事：协商参数、验证服务端身份、建立会话密钥，之后业务数据用对称加密传输。

**3分钟详答：**  
这是应用层里最常见的一题。  
很多人只会说“HTTPS 更安全”，但面试里更希望你说清楚：到底多了什么。  

HTTP 本身是应用层语义，负责定义请求方法、状态码、报文结构等；它默认不加密，所以中间人能看到和篡改内容。  
HTTPS 则是在 HTTP 下面加了一层 TLS。这样在真正传输业务请求前，客户端和服务端会先完成一次安全协商。  

TLS 握手可以概括为三件事：  
**1. 协商版本和加密套件**  
双方先确定用什么协议版本、什么密钥交换算法、什么加密算法。  

**2. 认证服务端身份**  
服务端把证书发给客户端，客户端验证证书链、域名、有效期等，确认自己连的确实是目标站点，而不是中间人。  

**3. 建立会话密钥**  
双方通过密钥交换得到共享密钥，后续大量业务数据一般使用对称加密传输，因为它性能更高。  

所以 HTTPS 的核心收益有三点：  
- **机密性**：别人窃听也看不懂  
- **完整性**：数据被改动能发现  
- **身份认证**：确认服务端身份  

代价主要是：握手有额外时延，证书管理有成本，加解密会带来一定 CPU 开销。  
但现代 TLS 和硬件已经很成熟，这些成本通常是值得的。  

面试里再补一句更完整：  
**HTTPS 不是一个新应用协议，而是 HTTP over TLS。**

**可能追问：**
- HTTPS 用的是对称加密还是非对称加密？  
  - 15秒简答：两者都会用。握手阶段会用非对称密码学做身份认证和密钥协商，真正传业务数据时主要用对称加密。
- 证书主要解决什么问题？  
  - 15秒简答：解决“我连到的到底是不是目标服务器”这个身份认证问题，防止中间人伪装。
- HTTPS 为什么比 HTTP 慢一点？  
  - 15秒简答：因为多了 TLS 握手、证书校验和加解密开销，但现代协议优化后通常已经可接受。

## 【高频】GET 和 POST 有什么区别？什么是安全、幂等？

**15秒简答：**  
GET 常用于获取资源，POST 常用于提交数据或触发服务端处理。更重要的区别在于 HTTP 语义：GET 通常是安全且幂等的，POST 通常不是。安全指不应产生副作用，幂等指同样请求重复执行多次，结果应与执行一次一致。

**3分钟详答：**  
这题表面简单，其实很适合体现你对 HTTP 语义的理解。  
很多人只会说“GET 放 URL，POST 放 body”，这只是常见用法，不是本质。  
更核心的区别是它们在协议语义上的定位。  

**GET** 一般表示“获取资源表示”，它应该是 **安全** 的，也就是不应修改服务端状态；同时通常也是 **幂等** 的，多次请求同一资源，不应该因为重复调用而产生额外副作用。  

**POST** 更像“向资源提交数据进行处理”，它常用于创建资源、提交表单、触发任务等。它通常 **不保证幂等**，同一个 POST 重复提交两次，可能创建两笔订单、触发两次扣款。  

安全和幂等经常一起问：  
- **安全**：偏语义层，表示这个请求应当是只读的  
- **幂等**：偏重复执行效果，执行一次和执行多次对系统最终状态一致  

所以：  
- GET：通常安全、幂等  
- PUT / DELETE：通常不安全，但幂等  
- POST：通常既不安全，也不幂等  

工程上这个区别很重要，因为它会影响：  
- 是否能安全重试  
- 代理和缓存的处理方式  
- 接口设计是否容易出错  

面试里如果能补一句会更好：  
**“能不能把参数放到 body 或 URL 里”不是 GET/POST 的本质区别，协议语义和幂等性才更重要。**

**可能追问：**
- GET 一定不能带请求体吗？  
  - 15秒简答：规范层面不是绝对禁止，但工程上通常不依赖这种做法，因为很多中间件、框架和缓存系统对它支持并不一致。
- 为什么幂等性对后端很重要？  
  - 15秒简答：因为网络请求可能超时重试，如果接口不是幂等的，重复提交就可能导致重复扣费、重复创建等问题。
- POST 能不能做成幂等？  
  - 15秒简答：可以，常见做法是引入业务唯一键或幂等键，让服务端识别“这其实是同一次请求”。

## 【高频】HTTP/1.1、HTTP/2、HTTP/3 有什么区别？

**15秒简答：**  
HTTP/1.1 主要特点是持久连接，但同一连接上的并发能力有限；HTTP/2 引入二进制分帧、头部压缩和多路复用，提高了同连接并发效率；HTTP/3 则把 HTTP 跑在 QUIC 之上，底层改用 UDP，重点解决 TCP 层的队头阻塞问题，并把加密集成得更紧密。

**3分钟详答：**  
这题非常适合体现“你不是只会背老八股”。  
可以按“连接复用能力”和“阻塞位置”来回答。  

**HTTP/1.1**  
它相比早期版本的重要改进是 **持久连接**，也就是一个 TCP 连接可以复用多个请求，不必每次都重新握手。  
但它在单连接上的请求并发能力仍有限，实际工程里常靠多连接并发来缓解阻塞问题。  

**HTTP/2**  
HTTP/2 保持了 HTTP 语义不变，但传输表达方式变了：  
- 使用 **二进制分帧**  
- 引入 **头部压缩**  
- 支持 **多路复用**，多个请求响应可以并发跑在一条连接上  

这样能显著减少应用层的队头阻塞和重复连接成本。  
但要注意一个关键点：HTTP/2 仍然跑在 TCP 上，所以如果底层某个 TCP 报文丢了，整个连接上的流在那个丢包点仍会受到影响，这就是 TCP 层的队头阻塞。  

**HTTP/3**  
HTTP/3 把 HTTP 映射到 **QUIC** 上，QUIC 基于 UDP 实现可靠传输、多路复用和更紧密集成的加密。  
它的意义是：每个流之间更独立，底层不再共享一个单一 TCP 字节流，因此能明显缓解 TCP 级别的队头阻塞。  

一个比较稳的总结是：  
- HTTP/1.1：持久连接，但并发能力有限  
- HTTP/2：单连接多路复用，更高效  
- HTTP/3：进一步把多路复用放到 QUIC/UDP 上，缓解 TCP 层队头阻塞  

**可能追问：**
- HTTP/2 已经多路复用了，为什么还需要 HTTP/3？  
  - 15秒简答：因为 HTTP/2 的多路复用仍建立在单条 TCP 字节流上，底层一旦丢包，多个流仍可能一起受影响；HTTP/3 主要是在传输层缓解这个问题。
- HTTP/1.1 的 keep-alive 有什么意义？  
  - 15秒简答：让多个请求复用同一个 TCP 连接，减少反复建连和挥手的成本，降低时延。
- QUIC 为什么基于 UDP？  
  - 15秒简答：因为这样应用层协议可以自己控制可靠性、重传和多路复用逻辑，避免被 TCP 固定语义限制。

## 【高频】从浏览器输入一个 URL 到页面返回，中间大致发生了什么？

**15秒简答：**  
简化流程是：解析 URL -> 查缓存和 DNS 得到 IP -> 建立连接（HTTPS 还要 TLS 握手）-> 发送 HTTP 请求 -> 服务端处理并返回响应 -> 浏览器解析 HTML、再继续请求 CSS/JS/图片等资源 -> 渲染页面。面试里重点是能把 DNS、TCP/HTTPS、HTTP、渲染这几层串起来。

**3分钟详答：**  
这是特别经典的一道综合题，几乎能把网络基础、HTTP 和浏览器工作流程串起来。  
一个完整但不冗长的回答可以按下面顺序讲：  

**1. 解析 URL**  
浏览器先识别协议、主机名、端口、路径等。  

**2. DNS 解析**  
浏览器会先看本地缓存、系统缓存等，若没有，再通过 DNS 把域名解析成 IP 地址。  

**3. 建立传输连接**  
如果是 HTTP，就建立 TCP 连接；如果是 HTTPS，通常先建立 TCP，再做 TLS 握手，然后才能安全传输 HTTP 请求。  
如果是 HTTP/3，则底层会用 QUIC/UDP。  

**4. 发送 HTTP 请求**  
浏览器构造请求行、请求头、可能的请求体发给服务端。  

**5. 服务端处理请求**  
服务端经过网关、负载均衡、应用服务、缓存、数据库等组件，最终生成响应报文返回。  

**6. 浏览器接收响应并解析**  
收到 HTML 后，浏览器会边解析边发现更多外部资源，比如 CSS、JS、图片，再继续发请求。  

**7. 渲染页面**  
浏览器构建 DOM、CSSOM，进行布局、绘制，最终把页面显示出来；JS 还可能改变 DOM，触发后续更新。  

如果是面后端岗位，重点不一定是浏览器渲染细节，而是你能不能清楚讲出：  
**DNS -> 建连 -> TLS -> HTTP 请求响应 -> 服务端处理** 这条主线。  

面试时很加分的一句话是：  
**“这个问题不是要求把浏览器内核讲透，而是考察我能否把多层协议串成一个完整链路。”**

**可能追问：**
- DNS 一定用 UDP 吗？  
  - 15秒简答：多数普通查询常用 UDP，因为快且报文小；但报文过大、区域传送或某些特殊场景下也会使用 TCP。
- 为什么浏览器会并发请求很多资源？  
  - 15秒简答：因为一个页面通常由 HTML、CSS、JS、图片等多个资源组成，并发请求能减少总加载时延。
- HTTPS 下最先发的是 HTTP 还是 TLS？  
  - 15秒简答：先做 TLS 握手，建立安全会话后才会发送真正的 HTTP 请求数据。
