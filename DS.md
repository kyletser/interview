# 数据结构与算法

## 如何分析一个算法的时间复杂度和空间复杂度？


时间复杂度看“核心操作执行了多少次”，顺序结构取加法，嵌套循环取乘法，递归写递推式；

空间复杂度除了显式申请的容器，还要把递归栈算进去。面试里优先给最坏复杂度，必要时再补平均复杂度和均摊复杂度。

## 数组和链表怎么选？各自的优缺点是什么？



数组**连续存储**，随机访问快，按下标访问是 `O(1)`，但中间插入删除通常要搬移元素；

链表**插入删除更灵活，但随机访问慢，**要顺着指针找。需要**高频读、按下标访问选数组**；需要**频繁插删、节点稳定性强时选链表**。

**可能的追问：**  

1. 链表删除为什么常说是 `O(1)`，又经常被说成不一定快？  
    **15秒答：** 已知待删节点前驱时删除是 `O(1)`；但很多题里你得先找到它，查找过程是 `O(n)`，所以整体不一定快。

2. 双向链表相比单链表的代价是什么？  
    **15秒答：** 好处是支持从当前节点 `O(1)` 删除和反向遍历；代价是多一个前驱指针，空间更大，实现也更容易出错。

## 哈希表为什么查找快？什么时候会退化？


哈希表本质是“**键先经过哈希函数映射到桶，再在桶里找**”，平均查找接近 `O(1)`。

但如果**哈希冲突严重、装载因子过高、哈希函数分布差**，或者被恶意构造输入攻击，就可能退化到 `O(n)`。

哈希表快的前提是：  

- 哈希函数分布足够均匀；  

- 桶数量和元素数量比例合理；  

- 冲突处理策略得当；  

- 必要时及时扩容。  
  面试官有时还会追问“哈希表和红黑树怎么选”。这里的关键不是说谁绝对更强，而是知道：**哈希表平均快，但不保序，最坏会退化；**

  平衡树是 `O(logn)`，但有序，支持范围查询。

**可能的追问：**  

1. 哈希冲突常见的解决方式有哪些？  
**15秒答：** 主要是**链地址法和开放定址法**；前者桶里挂链表或其他结构，后者冲突后继续探测下一个位置。

2. 什么是装载因子？  
**15秒答：** 装载因子一般指**元素个数除以桶数**，越高说明越拥挤，冲突概率越大；超过阈值通常要扩容。

3. 哈希表和红黑树怎么选？  
**15秒答：** 查单点、计数、去重优先哈希；需要有序遍历、范围查询、前驱后继时选红黑树这类平衡搜索树。

# 线性结构与常用技巧

## 栈、队列、单调栈、单调队列分别适合什么场景？

**栈是后进先出，适合括号匹配、函数调用、表达式处理；队列是先进先出，适合层序遍历和按顺序处理任务；单调栈适合找“下一个更大/更小元素”；单调队列适合维护滑动窗口最值。**

**3分钟详答：**   
栈的特点是后进先出，所以天然适合“最近的未处理对象”。比如括号匹配，本质上就是当前右括号去匹配最近的左括号；再比如中序遍历的迭代写法，也是借助栈保存回溯路径。  
队列先进先出，适合**按层、按时间顺序推进**。二叉树层序遍历、图的 BFS、生产者消费者模型，背后都是这个思想。  
单调栈是在栈的基础上增加“栈内元素单调”的约束。它最适合解决这样的问题：对每个元素，找左边或右边第一个比它大/小的元素。因为每个元素最多进栈出栈一次，所以整体是 `O(n)`。  
单调队列则常用于滑动窗口最值。普通窗口每次都重新求最大值会退化成 `O(nk)`，但如果维护一个单调递减队列，队头始终是窗口最大值，整体可以降到 `O(n)`。  

# 排序与查找

## 常见排序算法怎么比较？快排、归并、堆排各自适合什么场景？

| 算法 |      最坏      |   最好   | 稳定 | 原地 |
| :--: | :------------: | :------: | :--: | :--: |
| 冒泡 |     O(n²)      |   O(n)   |  ✅   |  ✅   |
| 插入 |     O(n²)      |   O(n)   |  ✅   |  ✅   |
| 选择 |     O(n²)      |  O(n²)   |  ❌   |  ✅   |
| 希尔 | O(n²)~O(nlogn) |    —     |  ❌   |  ✅   |
| 快排 |     O(n²)      | O(nlogn) |  ❌   |  ✅   |
| 归并 |    O(nlogn)    | O(nlogn) |  ✅   |  ❌   |
| 堆排 |    O(nlogn)    | O(nlogn) |  ❌   |  ✅   |
| 计数 |     O(n+k)     |  O(n+k)  |  ✅   |  ❌   |
| 桶排 |     O(n²)      |  O(n+k)  |  ✅   |  ❌   |
| 基数 |     O(n*k)     |  O(n*k)  |  ✅   |  ❌   |

快排的优点是平均 `O(nlogn)`，原地分区，常数小，缓存局部性也不错，所以实践中很常用；缺点是最坏可能退化到 `O(n^2)`，而且它不稳定。  
归并排序也是 `O(nlogn)`，而且稳定。它非常适合链表排序，因为链表分割和合并都自然；也适合外排序，因为可以把大文件分块排序后归并。但缺点是通常需要 `O(n)` 额外空间。  
堆排序最吸引人的地方是最坏也能稳定在 `O(nlogn)`，并且空间复杂度 `O(1)`；问题是它不稳定，而且常数通常不如快排，局部性也差一些。  
如果再上升一层，排序算法的“工程现实”是：真实标准库往往不是只用一种，而是混合策略。比如小区间切换插入排序，快排过深时切堆排，既兼顾平均性能，也避免最坏退化。  

**可能的追问：**  

1. 链表排序为什么常想到归并？  
    **15秒答：** 因为链表不支持随机访问，不适合快排那种基于下标交换；而归并只需要分裂和合并链，特别自然。

2. 堆排序为什么工程里没快排常见？  
    **15秒答：** 因为虽然最坏复杂度更稳，但常数和 cache 友好性通常不如快排，实际跑起来常常更慢。

## 为什么快排这么高频？它的最坏情况怎么避免？


最坏情况通常发生在 pivot 选得很差、划分极不均衡时；常见优化有随机选 pivot、三数取中、三路快排、小区间切插排、尾递归优化。

**3分钟详答：**

快排的核心在于 partition：选一个基准值，把数组分成“小于它”和“大于它”的两部分，然后递归处理两边。只要 partition 写得对，整个算法就成立。  
为什么最坏会变成 `O(n^2)`？因为如果每次选到的 pivot 都很偏，比如原数组接近有序，而你总选第一个元素做 pivot，那么划分会变成 `1` 和 `n-1`，递归树就退化了。  
为了避免这种情况，常见优化有：  

- **随机化 pivot**：降低总碰到极端划分的概率；  
- **三数取中**：从左中右取中位数，避免明显偏斜；  
- **三路快排**：处理大量重复元素时更高效；  
- **小区间转插入排序**：减少递归开销；  
- **尾递归优化**：先递归较短一边，控制栈深。  
面试里如果你能把“为什么平均快”和“为什么最坏会退化”都讲清楚，再补一两个工程优化点，基本就不只是会背答案，而是知道快排为什么在真实系统里常被改造使用。

**可能的追问：**  

1. 快排为什么平均性能通常很好？  
**15秒答：** 因为平均能把问题比较均匀地划分，递归深度接近 `logn`，同时原地分区、常数较小，所以实际表现常常很好。

2. 三路快排适合什么场景？  
**15秒答：** 适合重复元素很多的场景，把小于、等于、大于 pivot 的元素分三段，避免重复元素导致大量无效递归。

3. 快排是稳定排序吗？  
**15秒答：** 不是，因为 partition 过程中相等元素的相对顺序通常会被打乱。

# 动态规划、贪心与搜索

## 动态规划的本质是什么？


动态规划的本质是“把有重叠子问题的最优化问题拆成状态转移”，核心是先定义状态，再写转移方程，最后处理初始化和遍历顺序。不会 DP 的人常常不是不会做题，而是状态没定义对。

- **最优子结构**：原问题的最优解能由子问题最优解推出；  
- **重叠子问题**：同样的子问题会被反复计算，所以 DP 可以理解成“带缓存的递推”。  

**可能的追问：**  
1. 动态规划和递归有什么关系？  
**15秒答：** DP 可以看成把递归中的重复子问题保存下来，避免重复计算；记忆化搜索是自顶向下，表格法是自底向上。
3. 怎么判断一个问题适不适合 DP？  
**15秒答：** 看是否有最优子结构和重叠子问题；如果局部决策会影响全局，且子问题会反复出现，就很可能适合 DP。

## 贪心什么时候能用？面试里如何证明贪心是对的？

贪心能用的关键不是“看起来每次选最优”，而是要满足局部最优能推出全局最优。

面试里常见证明方法是交换论证、归纳证明，或者说明问题具有贪心选择性质。

**可能的追问：**  

1. 贪心和动态规划的区别是什么？  
**15秒答：** 贪心只做局部最优选择，不回头；DP 会系统比较多种状态和决策，更通用但通常更重。

2. 为什么有些题看起来能贪心，实际不行？  
**15秒答：** 因为局部最优未必能推出全局最优，当前一步看起来最赚，可能会破坏后面更大的收益空间。

3. 什么是交换论证？  
**15秒答：** 就是假设一个最优解没有采用贪心选择，再把它对应部分替换成贪心选择，证明结果不会更差，从而说明贪心是安全的。

## 回溯和剪枝是什么？它和 DFS 有什么关系？

回溯可以理解为“带状态恢复的 DFS”，适合枚举所有可能方案；

剪枝就是提前判断某条分支不可能成为答案，从而不再继续搜。回溯的关键是路径选择、状态恢复和剪枝条件。

**可能的追问：**  
1. 回溯和暴力枚举有什么区别？  
**15秒答：** 回溯本质上也是枚举，但它有明确的决策树结构和状态恢复机制，并且通常会配合剪枝减少无效搜索。

2. 剪枝为什么能加速？  
**15秒答：** 因为它提前排除了不可能得到有效答案的分支，避免把整棵搜索树都走完。

3. 回溯一定是指数复杂度吗？  
**15秒答：** 很多经典回溯题最坏确实是指数级，但实际性能高度依赖剪枝质量和问题约束。

# 经典设计与后端常考结构

## LRU 为什么要用“哈希表 + 双向链表”实现？

因为 LRU 要同时满足两件事：**`O(1)` 找到某个 key，以及 `O(1)` 把它移动到“最近使用”位置并在满了时淘汰最久未使用节点。**

哈希表负责定位，双向链表负责维护访问顺序。

- 访问一个 key 时，要能快速找到它；  
- 找到后，还要把它标记成最近使用；  
- 插入新 key 时，如果满了，要快速淘汰最久没用的那个。  
如果只有哈希表，查找快，但无法维护使用顺序；如果只有链表，顺序有了，但查找一个 key 要线性扫描。  
所以标准解法就是组合：  
- **哈希表**：`key -> 链表节点地址`，保证查找 `O(1)`；  
- **双向链表**：头部表示最近使用，尾部表示最久未使用，节点移动和删除都是 `O(1)`。  
为什么必须是双向链表？因为你需要在已知节点位置时把它从中间摘掉，再插到头部；单链表删除中间节点通常还得找前驱。  
这题高频的原因在于它特别能体现“多个数据结构协同设计”的能力。你如果能再补一句“真实缓存场景下 LRU 是利用局部性原理”，会更有后端味道。

**可能的追问：**  

1. LRU 和 LFU 的区别是什么？  
  **15秒答：** LRU 按最近是否使用淘汰，LFU 按使用频率淘汰；LFU 通常实现更复杂，因为要维护频次和同频次内的顺序。

## 布隆过滤器是什么？为什么后端岗位爱问它？


布隆过滤器是一种**用位图和多个哈希函数实现的集合判定结构**，特点是“**不存在一定准**，存在可能误判”。它特别适合**大规模判重、缓存穿透防护、黑名单过滤，**因为空间效率高。

**3分钟详答：**

它的核心做法是：准备一个位数组，对每个元素用多个哈希函数映射出多个位置，把这些位都置为 1。查询某个元素时，也用同样的哈希函数去看对应位：  

- 只要有一个位是 0，就说明它一定没出现过；  

- 如果全是 1，只能说明它“可能出现过”，因为不同元素可能把这些位恰好都打满了。  
  所以布隆过滤器有两个关键性质：  

- **不会漏判不存在**；  

- **可能误判存在**。  

  它为什么高频？因为在后端场景里特别实用。比如缓存穿透：如果大量请求访问根本不存在的数据，直接打数据库会很危险。此时可以先过布隆过滤器，如果判定一定不存在，就直接拦掉。  
  它的缺点也要会说：  

- 只能做近似判定；  

- 普通布隆过滤器删除不方便；  

- 误判率和位数组大小、哈希函数个数、数据规模有关。  
如果能再补一句“它适合做前置过滤，不适合做最终一致的精确存储”，这个回答就很完整了。

**可能的追问：**  

1. 为什么布隆过滤器会误判？  
**15秒答：** 因为不同元素经过多个哈希后可能碰巧把同一组 bit 都置成了 1，查询时就会误以为目标元素存在。

2. 布隆过滤器为什么通常不能直接删除？  
**15秒答：** 因为一个 bit 可能被多个元素共享，直接清零可能把别的元素的信息也删掉；要支持删除通常得用计数布隆过滤器。

3. 布隆过滤器适合做最终存储吗？  
**15秒答：** 不适合，它是近似判定结构，适合做前置过滤、判重、风控拦截，不适合作为最终准确数据源。

# 树与索引结构补充

## 满二叉树、完全二叉树、平衡二叉树、二叉搜索树分别是什么？


满二叉树强调“每个非叶子节点都有两个孩子”；完全二叉树强调“除了最后一层，前面都满，最后一层从左到右连续”；平衡二叉树强调“左右子树高度差不能太大”；二叉搜索树强调“左小右大”的有序性。它们关注点不同：有的看形状，有的看平衡，有的看值的顺序。

**可能的追问：**  

1. 堆为什么通常要求是完全二叉树？  
**15秒答：** 因为完全二叉树适合顺序存储，用数组就能根据下标快速定位父子节点，空间利用率也高。

2. 二叉搜索树一定是平衡的吗？  
**15秒答：** 不一定。普通 BST 只要求左小右大，不限制高度，插入顺序不好时会退化成链表。

3. 满二叉树和完全二叉树谁要求更严格？  
**15秒答：** 满二叉树更严格。满二叉树一定是完全二叉树，但完全二叉树不一定是满二叉树。

## 红黑树有哪些核心性质？为什么它能保证操作复杂度是 O(logn)？


红黑树通过“**节点着色 + 路径黑高约束**”限制树不能太歪，虽然不像 AVL 那样绝对平衡，但能保证最长路径不会超过最短路径的两倍，所以查找、插入、删除最坏都是 `O(logn)`。

**3分钟详答：**  
如果前一题是在讲“红黑树是什么”，这一题就是更进一层：**它为什么有效**。  
红黑树本质上是一种**近似平衡的二叉搜索树**，它除了满足 BST 的左小右大外，还额外满足几条颜色规则。面试里最常说的核心性质有：  

- 节点要么红要么黑；  
- 根节点是黑色；  
- 红节点不能连续，也就是红节点的孩子必须是黑色；  
- 从任意节点到其所有叶子空节点的每条路径，黑节点数量相同。  
这些规则的作用是限制树的高度。尤其是“**不能连续红”和“黑高相同”**这两条，能防止某一侧无限拉长。最终可以证明：红黑树的高度是 `O(logn)`，更具体地说，不会超过 `2log(n+1)` 量级。  
为什么工程里这么常见？因为红黑树在“平衡程度”和“维护代价”之间做了很好的折中。AVL 更严格，所以查询可能稍快；但红黑树插入删除时旋转和调整通常更温和，整体更适合频繁更新的场景。  
很多语言和库里的有序 map/set 都基于红黑树，原因就是它既保证最坏复杂度，又支持中序有序遍历、范围查询、前驱后继。  
面试里再补一句会更像工程师：**哈希表平均更快，但不保序；红黑树是牺牲一点常数，换来顺序性和稳定的最坏复杂度。**

**可能的追问：**  
1. 红黑树为什么不追求像 AVL 那样更严格的平衡？  
**15秒答：** 因为更严格的平衡会带来更频繁的旋转和维护开销，红黑树选择了更好的综合折中。

2. 红黑树的查找过程和 BST 有区别吗？  
**15秒答：** 查找逻辑本身没有区别，还是利用左小右大；区别主要体现在插入删除后需要通过染色和旋转维持平衡。

3. 为什么说红黑树最长路径不超过最短路径两倍？  
**15秒答：** 因为红节点不能连续，黑高又相同，所以一条路径里红节点最多和黑节点差不多一样多，整体长度被限制在常数倍范围内。

## B 树和 B+ 树有什么区别？为什么数据库索引通常更偏爱 B+ 树？


B 树的非叶子节点也存数据，查找到中间节点就可能结束；

B+ 树把数据记录都放在叶子节点，非叶子节点主要存索引，叶子之间还串成链表。

这样 B+ 树更适合范围查询、顺序扫描，也更适合磁盘/页式存储，所以数据库索引更常用 B+ 树。

**3分钟详答：**  
区别主要有三点：  
第一，**数据放置位置不同**。B 树的内部节点和叶子节点都可以存 key 和对应记录；B+ 树则通常把内部节点当“导航目录”，真正的数据记录都在叶子节点。  
第二，**查询行为不同**。B 树有可能在中间节点就命中结束；B+ 树查找到叶子才结束，因此路径更稳定。  
第三，**范围查询能力不同**。B+ 树的叶子节点通常按顺序用链表连接，做区间查询、顺序扫描、分页遍历时特别高效；而 B 树做范围查询往往没有这么顺。  
数据库为什么更爱 B+ 树？因为数据库索引不只做单点查询，更常见的是**范围查询、排序、扫描前几页**数据。B+ 树内部节点只放 key，不放整条记录，因此单个页能容纳更多索引项，扇出更大，树更矮，磁盘 IO 更少；叶子链表又让范围查询很自然。  
面试里最加分的一句是：**数据库选 B+ 树是因为它更贴合页式存储和范围扫描的访问模式。**

**可能的追问：**  
1. 为什么 B+ 树通常比红黑树更适合数据库索引？  
**15秒答：** 因为 B+ 树是多路树，树高更低，磁盘 IO 更少；红黑树分支少、树更高，更适合内存结构而不是磁盘索引。

2. B+ 树为什么范围查询快？  
**15秒答：** 因为所有数据都在叶子节点，叶子又按顺序链起来，找到起点后顺着叶子链表扫就行。

3. B+ 树内部节点为什么只存索引更有优势？  
**15秒答：** 因为这样单个页能容纳更多 key，分支因子更大，整棵树更矮，访问层数更少。

## 二叉搜索树、AVL 树、红黑树分别有什么特点？


BST 满足左小右大，查找逻辑直观，但可能退化成链表；AVL 树平衡更严格，查询更稳；红黑树平衡稍弱但旋转调整更少，综合插删查性能更适合工程实现。

**3分钟详答：**  
普通二叉搜索树 BST 的性质是：左子树都小于根，右子树都大于根。基于这个性质，查找、插入、删除在理想情况下都是 `O(logn)`。但如果输入数据接近有序，树会退化成链表，最坏复杂度变成 `O(n)`。  
为了解决退化问题，就有了平衡搜索树。AVL 树要求左右子树高度差不能超过 1，所以它非常“瘦高受限”，查询性能很稳，适合读多场景；但代价是插入删除时可能要更频繁地旋转调整。  
红黑树的平衡条件没 AVL 那么严格，它通过颜色和若干规则保证“不会太歪”，从而把高度控制在对数级。它的好处是插入删除的调整相对温和，所以很多工程库会用红黑树实现有序 map/set。  
面试里答到这里还可以再进一层：  

- 需要有序遍历、范围查询、前驱后继，就该想到 BST/红黑树这类有序结构；  
- 只查单点、去重、计数，哈希表可能更快；  
- 只要取最大最小，堆更合适。  
  也就是别把树只当“会背定义”，而要把它放回整个数据结构选型体系里。

**可能的追问：**  

1. 为什么 BST 会退化？  
   **15秒答：** 因为如果按有序或近有序数据插入，新的节点总往同一侧长，树高会接近 `n`，就像链表一样。

2. AVL 和红黑树谁查找更快？  
   **15秒答：** 理论上 AVL 更平衡，查询可能更快；但红黑树插删调整更少，工程上综合表现更好，所以更常用。

3. 红黑树为什么工程里常见？  
   **15秒答：** 因为它在有序性、最坏复杂度和更新代价之间做了比较好的平衡，插入删除不像 AVL 那么“爱旋转”。
