# LPRC面试

## RPC框架核心运行流程
1. **客户端**：调用 stub.Login() → Protobuf 自动转到 LrpcChannel.CallMethod()
2. **服务发现**：首次调用时从 ZooKeeper 查询服务地址
3. **建立连接**：通过 socket 建立 TCP 连接
4. **序列化**：将请求对象序列化为字节流
5. **协议封装**：打包成  [total_len] [header_len] [header] [args] 格式发送
6. **服务端接收**：Muduo OnMessage 回调，循环处理解决粘包
7. **解包路由**：解析协议头，根据 service_name + method_name 查找方法
8. **反射调用**：通过 Protobuf 反射动态创建请求/响应对象并调用业务
9. **业务处理**：执行业务逻辑 UserService::Login()
10. **返回响应**：done->Run() 触发 SendRpcResponse 发送响应
11. **客户端接收**：读取响应，反序列化，返回结果

## 1. 架构设计类
**这个框架和gRPC/brpc的区别？你的框架的优缺点？**

优点：

1. **简洁易懂**：代码量小，适合学习 RPC 原理
2. **自定义协议**：纯 TCP + Protobuf，性能高
3. **Muduo**：基于 Reactor 模式，高并发网络模型
4. **ZooKeeper**：完整的服务发现机制
5. **Protobuf 反射**：动态路由，框架不依赖具体方法名

缺点：

1. **无连接池**：频繁建连开销大
2. **无负载均衡**：单连接，无多节点支持
3. **无超时重试**：可靠性差
4. **无线程池**：业务阻塞 IO 线程
5. **无流式调用**：只支持 unary
6. **单语言**：只支持 C++
7. **无鉴权**：安全性弱



**问：为什么做C++版本的RPC？C++的优势体现在哪里？**
答：C++内存管理可控、底层优化空间大（如非阻塞IO）、性能高，适合金融/游戏/嵌入式等高性能场景；避免跨语言调用的性能损耗，适配C++编写的底层基础设施（数据库/中间件）。

## 2. 核心技术细节类
### （1）Protobuf相关
问：为什么选Protobuf做序列化？对比JSON/XML的优势？
答：二进制序列化体积小（比JSON小50%+）、解析快、类型安全、跨语言、可扩展；JSON是文本格式，解析开销大，不适合高性能RPC。

问：自定义RpcChannel的核心作用？CallMethod方法如何实现？
答：RpcChannel是Protobuf的核心接口，封装远程调用的通信逻辑；CallMethod中会：① 解析调用的服务/方法名；② 序列化请求参数；③ 拼接RPC协议头；④ TCP发送请求；⑤ 阻塞等待响应并反序列化。

问：Protobuf生成的代码结构是什么？如何基于它实现业务服务？
答：生成抽象服务类（含纯虚方法）、Stub类（客户端调用）、Message序列化方法；业务服务继承抽象服务类，实现纯虚方法，注册到Lrpcprovider即可。

### （2）网络通信相关
问：如何解决TCP粘包/拆包？
答："我使用**长度字段**的方式：在每个消息头部增加 4 字节的 total_len 和 4 字节的 header_len，接收端先读取长度字段，判断数据是否完整再处理。这样可以解决粘包和拆包问题。"

问：框架用的是阻塞IO还是非阻塞IO？为什么？如何优化？
答："框架使用的是**非阻塞 IO**，基于 Muduo 的 Reactor 模式。Muduo 使用 epoll 等待事件，事件就绪才处理，所以一个线程可以同时处理多个连接。这适合高并发场景。

### （3）Zookeeper服务治理
### Q1: ZK在框架中的作用？

> "ZK 在框架中充当**服务注册与发现**的角色。
>
> 服务端启动时将服务地址注册到 ZK，
>
> 客户端调用前从 ZK 查询服务地址。
>
> 这样实现了服务地址的动态管理，解耦了客户端和服务端。"

### Q2: 为什么用临时节点？

> "方法节点使用临时节点（ZOO_EPHEMERAL），当服务端进程崩溃或断开连接时，临时节点会自动删除，
>
> 客户端下次查询时就不会拿到失效的地址，保证了服务可用性。"

### Q3: ZK集群挂了怎么办？

> "可以通过以下方案保证可用性：
>
> 1. 本地缓存服务地址，ZK 不可用时使用缓存
> 2. 配置本地备用地址作为兜底
> 3. 定期心跳检测，失效后自动剔除并重新查询
> 4. 生产环境部署 3 节点 ZK 集群保证高可用"

### Q4: ZK节点类型？

> "ZK 有 4 种节点类型：永久节点、永久顺序节点、临时节点、临时顺序节点。
>
> 我的项目中服务节点用永久节点，方法节点用临时节点。"

### （4）C++特性与工程化
问：框架中用到了哪些C++11特性？
答：① 单例模式（Lrpcapplication，用std::mutex保证线程安全）；② 智能指针（std::unique_ptr/std::shared_ptr管理内存，避免泄漏）；③ 多线程（std::thread+std::mutex/condition_variable）；④ lambda表达式（线程池任务封装）。

问：如何保证线程安全？举例说明。
答：服务注册表（Lrpcprovider中的服务映射）是共享资源，用std::mutex加锁；线程池的任务队列用mutex+condition_variable实现生产者-消费者模型，避免忙等。

问：CMakeLists.txt的核心配置有哪些？
答：① 指定C++标准（set(CMAKE_CXX_STANDARD 11)）；② 链接依赖库（target_link_libraries(xxx protobuf zookeeper_mt)）；③ 指定源码/输出目录（aux_source_directory(src DIR_SRCS)）；④ 生成可执行文件（add_executable(server src/server.cc)）。

## 3. 问题排查与性能优化类
**问：开发过程中遇到的最大问题是什么？如何解决？**
答：示例：① 问题：客户端调用时出现数据解析错误，排查发现是TCP粘包导致；解决：添加RPC协议头，解析数据长度后再读取完整请求；② 问题：ZK注册失败，排查发现是节点路径写错+未处理ZK连接异常；解决：修正路径，添加ZK连接重试逻辑。

## 4. 扩展与场景类
**问：如何扩展框架支持异步调用？**
答：修改RpcChannel的CallMethod，支持回调函数；底层用epoll监听socket可读事件，响应返回后触发回调，而非阻塞等待。

**问：如何实现负载均衡？**

## 1. 简单描述项目、实现了怎么样的功能？用了那些技术栈？

### 项目描述

Lrpc 是一个**简化版的 C++ RPC 框架**，实现了让客户端像调用本地函数一样调用远程服务器上的函数的功能。核心目标是理解 RPC 的工作原理，包括序列化、协议封装、网络传输、服务发现和动态路由。

### 实现功能

- **远程过程调用**：客户端可以通过 Protobuf 生成的 Stub 调用远程服务
- **服务注册与发现**：服务端启动时注册到 ZooKeeper，客户端查询服务地址
- **自定义协议**：解决 TCP 粘包/拆包问题
- **动态路由**：利用 Protobuf 反射，不硬编码方法名
- **高并发网络**：基于 Muduo 的 Reactor 模型

## 2. 为什么要使用Zookeeper作为服务器注册与发现中心？

### 为什么选择 ZooKeeper

**1. 成熟的分布式协调服务**

ZooKeeper经过大量生产环境验证，提供**分布式一致性**保证。

2**. 临时节点机制**

```cpp
// 服务端创建临时节点
zkclient.Create("/UserServiceRpc/Login", "127.0.0.1:9000", ..., ZOO_EPHEMERAL);
```

临时节点会在会话断开时**自动删除**，这意味着：
- 服务进程崩溃 → 节点自动消失 → 客户端不会拿到失效地址
- 不需要额外的心跳检测和清理机制

**4. Watcher 通知机制**

```cpp
// 客户端可以监听节点变化
zoo_get(handle, path, 1, buf, &len, watcher);
```

当服务地址变化时，ZK 会主动通知客户端，实现**服务动态感知**。

**5. 高可靠性**

ZK 使用 ZAB 协议保证分布式一致性，半数以上节点存活即可工作，适合作为分布式系统的"枢纽"。

## 3. 消息传输为什么用Protobuf,而不用JSON?

### Protobuf vs JSON 对比

Protobuf 的优势

**1. 高性能（核心原因）**

- 二进制格式，无需解析字符串
- 字段使用数字编号，而非字段名
- 编码紧凑，网络传输开销小

**2. 强类型检查**

编译时就能发现类型错误，而不是运行时。

**3. 自动生成代码**

自动生成：
- `Service` 基类
- `Stub` 客户端类
- 序列化/反序列化方法

**4. 向前向后兼容**

- 通过字段号而非字段名实现兼容
- 新增字段不影响旧版本
- 旧版本忽略新字段

### 为什么不用 JSON

- **性能差**：文本格式，解析开销大
- **体积大**：字段名重复传输
- **类型弱**：运行时才能发现错误
- **无 IDL**：需要手动约定格式

## 4. 在实现 RPC 服务时遇到了哪些难点？你是如何解决的？

### 难点1：TCP 粘包/拆包

**问题**：TCP 是字节流协议，可能一次收到多个包或半个包。

**解决**：自定义协议，使用长度字段。

```cpp
// 发送端
uint32_t total_len = 4 + header_size + args_size;
send(len);           // 4字节总长度
send(header_len);   // 4字节头长度
send(header);       // 变长头
send(args);         // 变长参数

// 接收端 - 循环处理
while (buffer.size >= 4) {
    uint32_t total_len = read_len();
    if (buffer.size < 4 + total_len) break; // 数据不完整
    // 处理完整包...
}
```

### 难点2：动态路由

**问题**：服务端如何根据请求找到对应的方法？

**解决**：利用 Protobuf 反射机制。

## 5. 如何处理分布式系统中的故障？

### 故障类型和应对策略

**1. 服务端故障**

- **检测**：通过 ZooKeeper 临时节点自动感知（节点删除）
- **处理**：客户端缓存失效地址，尝试重新查询
- **恢复**：服务重启后重新注册

**2. 网络超时**

- **处理**：添加超时控制
```cpp
struct timeval timeout = {3, 0}; // 3秒超时
setsockopt(fd, SOL_SOCKET, SO_RCVTIMEO, &timeout, sizeof(timeout));
```

**3. 服务不可用**

- **处理**：多节点负载均衡
- **容错**：重试其他节点

**4. ZooKeeper 故障**

- **处理**：本地缓存作为兜底
```cpp
std::string getAddress() {
    // 先查缓存
    if (cache.exists(method)) return cache.get(method);
    // 缓存未命中，查询ZK
    std::string addr = zkClient.GetData(method);
    cache.set(method, addr);
    return addr;
}
```

**5. 流量洪峰**

- **处理**：请求队列 + 背压
```cpp
if (taskQueue.size() > MAX_QUEUE_SIZE) {
    return error; // 拒绝请求，防止雪崩
}
```

## 8. 请简要解释一下RPC(远程过程调用)是什么，它是如何工作的？

### RPC 定义

RPC（Remote Procedure Call，远程过程调用）是一种**让程序调用另一个地址空间（通常是远程服务器）上的函数或方法**的技术，对调用者来说就像调用本地函数一样。

### RPC 工作原理

**核心思想**：把网络通信透明化

```
// 本地调用
result = user.Login("zhangsan", "123456");

// RPC 调用（看起来一样）
result = stub.Login(&controller, &request, &response, nullptr);
// 底层实际发生了网络通信
```

### 工作流程

**1. 客户端侧**

```
调用 stub.Login() 
    ↓
Protobuf 框架将调用转发到 LrpcChannel.CallMethod()
    ↓
序列化请求对象为二进制
    ↓
查询服务地址（ZK）
    ↓
建立 TCP 连接
    ↓
发送协议包：[len][header][args]
    ↓
等待响应
    ↓
反序列化响应，返回结果
```

**2. 服务端侧**

```
Muduo 监听到达的数据
    ↓
OnMessage() 回调被触发
    ↓
解决粘包，拆解出完整请求
    ↓
解析协议头，获取 service_name + method_name
    ↓
从 service_map 查找对应的服务和方法
    ↓
通过 Protobuf 反射创建请求/响应对象
    ↓
调用业务代码（UserService::Login）
    ↓
业务完成后，通过回调发送响应
    ↓
序列化响应，发送回客户端
```

### RPC 框架的核心组件

| 组件           | 作用                             |
| -------------- | -------------------------------- |
| **Stub**       | 客户端代理，把调用转发到 Channel |
| **Channel**    | 负责网络通信、序列化、协议封装   |
| **Protocol**   | 定义网络协议格式                 |
| **Registry**   | 服务注册与发现                   |
| **Dispatcher** | 服务端请求分发                   |

## 9. 你如何看待微服务架构与传统单体应用架构的区别？

### 单体架构的特点

传统的单体应用。所有模块共享同一个数据库，部署的时候整个应用一起部署。扩展的时候只能对整个应用进行水平扩展，如果某个模块是瓶颈，只能扩展整个应用而不是那个瓶颈模块。

优点是开发和部署简单，测试方便，没有网络延迟。缺点是技术栈单一，一个模块的技术选型会影响整个项目；可靠性差，一个模块的bug可能导致整个应用崩溃；扩展性差，无法针对瓶颈模块单独扩展。

### 微服务架构的特点

每个微服务独立开发、独立部署、独立数据库。服务之间通过RPC或者HTTP进行通信。可以通过对单个服务进行水平扩展来解决性能瓶颈。技术栈可以多样化，每个服务可以选择最适合的技术。

优点是扩展灵活，一个服务可以部署多个实例；技术多样，不同服务可以用不同技术；独立开发，不同团队可以并行开发不同服务。缺点是分布式复杂性，需要处理网络延迟和失败；数据一致性困难，跨服务的事务难以保证；运维成本高，需要容器编排、服务发现等基础设施。

### 实际选择

选择哪种架构取决于项目规模和团队情况。小项目用单体架构更简单高效；大项目用微服务可以更好地解耦和扩展。我的RPC框架就是微服务架构的基础设施。

## 10. TCP粘包与拆包问题解决方案

### 问题原因

TCP 是**面向字节流**的传输协议，没有消息边界：

```
发送： [消息1][消息2][消息3]
接收： [消息1部分] [消息1其余 + 消息2部分] [消息2其余 + 消息3部分]
```

### 解决方案

**方案1：固定长度**

每个消息固定长度，不足补空。

```cpp
const int MAX_MSG_SIZE = 1024;
char buffer[MAX_MSG_SIZE];

// 发送
send(buffer, MAX_MSG_SIZE); // 总是发送固定长度
```

- 优点：简单
- 缺点：空间浪费

**方案2：分隔符**

消息之间用特殊字符分隔（如 `\n`）。

```cpp
// 发送
std::string msg = "hello\nworld\n";
send(msg.c_str(), msg.size());

// 接收 - 按分隔符切分
std::stringstream ss(buffer);
std::string line;
while (std::getline(ss, line)) {
    // 处理每条消息
}
```

- 优点：直观
- 缺点：特殊字符处理复杂

**方案3：长度字段（推荐）**

在消息头部加上长度字段。

```
[4字节总长度][4字节头部长度][变长头][变长参数]
```

```cpp
// 发送端
uint32_t body_size = data.size();
uint32_t total_size = 4 + body_size; // 4是头长度字段本身

uint32_t net_total = htonl(total_size);
uint32_t net_body = htonl(body_size);

send(&net_total, 4);
send(&net_body, 4);
send(data.data(), data.size());
```

```cpp
// 接收端
while (buffer.size() >= 4) {
    uint32_t total = ntohl(*(uint32_t*)buffer.data());
    if (buffer.size() < 4 + total) break; // 不够，暂不处理
    
    // 完整包，取出处理
    std::string msg = buffer.substr(4, total);
    buffer.erase(0, 4 + total);
}
```

- 优点：通用、可靠
- 缺点：需要额外 4-8 字节

**方案4：HTTP 协议**

利用 HTTP 协议的 Content-Length 头。

```
HTTP/1.1 200 OK
Content-Length: 123

{"name": "zhangsan"}
```

- 优点：通用、跨语言
- 缺点：开销大，性能低

### 你的框架采用的方案

采用长度字段方案。发送端把消息打包成[4字节总长度 ] [4字节头部长度] [变长头部] [变长参数]的格式，接收端用while循环从缓冲区读取数据，每次先读取4字节得到总长度，判断数据足够后再处理完整包。

## 11. Zookeeper的Watcher机制

### 什么是 Watcher

Watcher 是 ZooKeeper 的**事件通知机制**，可以让客户端在节点发生变化时**主动收到通知**，而无需轮询。

### 工作原理

客户端调用zoo_get时传入一个watcher函数指针，同时设置第四个参数为1表示注册watcher。当节点数据发生变化或者节点被删除时，ZooKeeper会调用客户端注册的watcher函数。

需要注意的是Watcher是一次性触发，每次触发后会自动删除，如果需要持续监听需要在回调中重新注册。

### 在RPC中的应用

服务端的临时节点被删除时，客户端可以收到通知，从而知道服务端不可用。客户端可以重新查询地址或者使用缓存。

## 12. Muduo Reactor网络模型

### 什么是Reactor模型

Reactor模型是一种事件驱动的并发模型，核心思想是把所有IO操作统一到一个事件循环中。当某个IO事件发生时，调用对应的回调函数处理。

### Muduo的核心组件

EventLoop是事件循环的核心，通过epoll_wait()等待事件就绪。Channel是每个文件描述符的封装，管理这个fd上可读、可写、错误等事件的回调。TcpServer是服务器封装，管理监听socket和连接。

### 工作流程

创建TcpServer并设置connection和message回调，然后调用start()启动，最后调用loop()进入事件循环。EventLoop会调用epoll_wait()等待事件，当有连接到达或者有数据可读时，epoll返回就绪的fd列表，遍历这些fd调用对应的回调函数处理。

### 相比多线程模型的优势

传统多线程模型是每个连接一个线程，线程切换开销大，内存消耗高。Reactor模型是单线程处理所有IO，线程开销小，效率高。对于IO密集型的RPC服务，Reactor模型更加合适。

## 13. 服务注册与发现的高可用性保障

### 高可用目标

服务注册中心故障不影响已有服务调用，服务下线后客户端能及时感知，客户端能处理服务地址变化。

### 保障方案

**ZooKeeper集群**：生产环境部署3节点以上的ZK集群，半数以上节点存活即可工作，单节点故障不影响服务。

**本地缓存**：首次查询ZK后缓存到本地，即使ZK不可用也能使用缓存地址。实现方式是建立一个unordered_map，key是方法路径，value是地址。

**配置文件兜底**：在配置文件中写一个备用地址，当ZK和缓存都不可用时使用。实现方式是在配置文件中添加fallback_server配置项。

**定期心跳检测**：客户端定期检查连接有效性，发现连接失效后从缓存移除并尝试重新获取地址。

### 故障切换流程

客户端调用服务时，首先查询本地缓存，有缓存则直接返回；缓存没有则查询ZK，成功则更新缓存并返回；查询失败则使用配置文件中的兜底地址。

### 故障切换流程

```
客户端调用服务
        │
        ▼
┌───────────────────┐
│  1. 查询本地缓存   │──有──► 返回地址
└───────────────────┘
        │无
        ▼
┌───────────────────┐
│  2. 查询 ZooKeeper│──成功──► 更新缓存，返回地址
└───────────────────┘
        │失败
        ▼
┌───────────────────┐
│  3. 使用兜底地址  │──────────────► 返回地址
└───────────────────┘
        │
        ▼
   调用服务
```



## 14. 高性能日志系统的实现

### 高性能日志的关键点

**缓冲写入**：不是每条日志都写文件，而是先写到内存缓冲区，缓冲区满了或者达到一定时间后才批量写入文件。这减少了系统调用次数，大幅提升性能。

**异步日志**：写日志的操作不阻塞主线程。日志先写到队列就返回，由专门的日志线程负责从队列取出并写入文件。

**日志分级**：编译时决定日志级别，DEBUG级别的日志在生产环境不编译进去，避免无效的性能开销。

**格式优化**：时间格式化等操作优化，避免每次都重新格式化。

### Glog的实现原理

Glog**每个线程有自己的本地缓冲区，先写到本地缓冲区，满后刷新到全局缓冲区。全局缓冲区由单独的日志线程异步写入文件。这样大部分时间只是内存操作，只有缓冲区满时才涉及IO。**
