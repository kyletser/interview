## 1. 同步 vs 异步

- **同步**：程序发起 I/O 后，等待操作完成才能继续执行后续代码。内核通知就绪后，**应用自己执行 IO 读写**。
- **异步**：程序发起 I/O 后立即返回，不等待 IO 完成；IO 由内核后台完成，完成后内核通过**回调函数**通知程序处理。

## 2. 阻塞 vs 非阻塞

- **阻塞**：调用 read/recv，无数据则线程挂起等待，直到数据到达。
- **非阻塞**：调用 read/recv，无数据立刻返回错误，不阻塞线程，需轮询 / 事件监听。

## 3. IO 多路复用

单线程监听多个 fd，内核通知就绪事件，避免单连接单线程。

属于**同步非阻塞 IO**，实现：select/poll/epoll。

## 4. epoll 工作原理

epoll 是 Linux 高效**事件通知机制**的 IO 多路复用，不是事件驱动。

线程阻塞在`epoll_wait`等待就绪事件；事件就绪后，**应用线程主动执行 read/recv**，因此 epoll 是**同步 IO**。

epoll 仅通知就绪，不遍历全量 fd，性能远高于 select/poll。

## 5. IO 模型总结表

|      I/O 模型       | 同步 / 异步 | 阻塞 / 非阻塞 |              特点 & 场景               |
| :-----------------: | :---------: | :-----------: | :------------------------------------: |
|     阻塞同步 IO     |    同步     |     阻塞      |               简单低并发               |
|    非阻塞同步 IO    |    同步     |    非阻塞     |                轮询场景                |
| IO 多路复用 (epoll) |    同步     |    非阻塞     |   高并发：Nginx、Redis、Node.js 底层   |
|    原生异步 AIO     |    异步     |    非阻塞     | 内核完成 IO + 回调，Linux 极少原生使用 |

## 6. epoll LT/ET 触发模式

- LT 水平触发（默认）：缓冲区有数据就持续通知，**可配合阻塞 fd**，简单稳定。
- ET 边缘触发：仅状态变化通知一次，**必须配合非阻塞 fd**，需循环读完所有数据，高性能高并发。

## 7. 最终核心结论

1. epoll 是**同步非阻塞多路复用 IO**，是**事件通知机制**，不是事件驱动、不是异步 IO。
2. epoll_wait 阻塞等待事件，socket fd 设置为非阻塞。
3. 真正异步 AIO：内核完成 IO 读写，完成后回调通知应用，天生单线程异步。
