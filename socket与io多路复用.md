# 一、面试官最先拷打的：你到底理解 socket 是什么吗

## 1. 什么是 socket？

### 面试回答

socket 本质上是**操作系统提供的网络通信抽象接口**。
应用程序并不是直接操作网卡发数据，而是通过 socket 这个接口，把数据交给内核，再由内核协议栈完成 TCP/UDP 通信。

如果是面向 TCP，socket 可以理解为：

- 一端通信的抽象描述
- 包含协议、IP、端口、收发缓冲区、连接状态等信息
- 应用层通过文件描述符 fd 来操作它

在 Linux 下，socket 也可以看成“**一种特殊的文件描述符**”，所以能配合 read、write、close、epoll 这一套一起用。

### 继续追问：为什么 socket 也是文件描述符？

### 回答

Linux 里很多资源都统一抽象成文件描述符，例如：

- 普通文件
- 管道
- socket
- epoll 实例
- eventfd

这样做的好处是统一 IO 接口，用户态不需要针对每种资源都设计完全不同的读写方式。
所以网络编程里我们对 socket 调用 read/write，底层其实就是内核根据 fd 找到对应对象，再执行相应操作。

### 容易翻车点

不要说“socket 就是一个端口”或者“socket 就是一条连接”。
更准确地说，**socket 是通信端点的抽象**。TCP 已建立连接的 socket 才对应一条连接状态。

------

## 2. socket 编程的基本流程是什么？

### 服务端回答

TCP 服务端基本流程：

1. `socket()`：创建监听 socket
2. `bind()`：绑定本机 IP 和端口
3. `listen()`：把 socket 变成监听状态
4. `accept()`：接收客户端连接，返回新的已连接 socket
5. `read/recv`：读取数据
6. `write/send`：发送数据
7. `close()`：关闭连接

要注意：

- `listen fd` 只负责接收新连接
- `accept` 返回的 `conn fd` 才负责和具体客户端通信

### 客户端回答

TCP 客户端流程：

1. `socket()`
2. `connect()`
3. `send/recv` 或 `write/read`
4. `close()`

### 面试官常追问：为什么 accept 会返回一个新的 fd？

### 回答

因为监听 socket 和连接 socket 职责不同。

- 监听 socket 用来监听指定端口，等待新的连接请求
- 每个 accept 返回的新 socket 表示一个独立已建立连接，拥有各自的发送接收缓冲区和连接状态

如果不分开，就没法一边继续接收新连接，一边和已有客户端通信。

# 二、TCP 相关会怎么深挖

## 3. bind、listen、accept 分别做了什么？

### 回答

`bind()`：把 socket 和本地地址绑定起来，告诉内核这个 socket 用哪个 IP 和端口。
`listen()`：把当前 socket 从普通主动/被动通信 socket 转成监听 socket，并设置监听队列长度。
`accept()`：从已完成连接队列里取出一个连接，返回对应的已连接 socket。

### 追问：listen 的 backlog 是什么？

### 回答

backlog 可以理解为监听队列长度相关参数，但实际在 Linux 内核中和两个队列都有关系：

- **半连接队列**：保存收到 SYN、还没完成三次握手的连接
- **全连接队列**：保存已经完成三次握手、等待 accept 的连接

更准确地说，`listen(backlog)` 主要影响已完成连接队列长度上限，实际生效还会受到内核参数限制，比如 `somaxconn`。

### 再追问：如果服务端来不及 accept，会怎样？

### 回答

如果全连接队列满了：

- 新完成握手的连接可能被丢弃
- 客户端可能连接超时或重传
- 服务端表现为“端口开着，但连接不上”

这是高并发场景下常见问题，所以需要：

- 提高 backlog
- 调整内核参数
- 提升 accept 处理能力
- 合理设计 reactor / 线程池

------

## 4. accept 是从三次握手哪个阶段开始返回的？

### 回答

当 TCP 三次握手完成后，连接进入已建立状态，并被放入已完成连接队列，
这时 `accept()` 才可能返回。

也就是说，`accept()` 返回时，连接在内核看来已经建立好了。

### 容易翻车点

不要说“收到 SYN 就 accept 返回”，这是错的。

------

## 5. TCP 是面向字节流是什么意思？

### 回答

TCP 不保留应用层消息边界。
发送方连续调用多次 `send`，接收方未必对应收到同样次数的 `recv`。

比如发送方：

- send("hello")
- send("world")

接收方可能收到：

- "helloworld"
  也可能：
- "hel"
- "loworld"

所以 TCP 粘包拆包问题的本质，不是 TCP 出错，而是**字节流协议没有消息边界**。

### 追问：如何解决粘包拆包？

### 回答

常见方案有三种：

1. **固定长度消息**
   - 每个包长度固定
   - 简单，但浪费空间，不灵活
2. **分隔符**
   - 比如 `\r\n`
   - 文本协议常见，如 HTTP 早期风格、Redis 协议部分场景
3. **长度字段 + 消息体**
   - 先读包头得到长度，再按长度读完整包
   - 这是最常用、最稳妥的方法

如果是二进制私有协议，一般我会采用：

- 固定包头
- 头部包含 body length
- 再配合接收缓冲区进行解析

------

## 6. recv 返回 0、-1、正数分别代表什么？

### 回答

- **大于 0**：读到了这么多字节
- **等于 0**：对端正常关闭连接，发送了 FIN
- **等于 -1**：出错，需要看 errno

如果是非阻塞 socket：

- `EAGAIN` 或 `EWOULDBLOCK` 表示当前没数据可读，不是异常
- `EINTR` 表示被信号中断，通常应重试

### 追问：为什么返回 0 表示对端关闭？

### 回答

TCP 是全双工。对端关闭写方向后，本端读到缓冲区清空，再继续读就会返回 0，这表示“流结束”，类似文件 EOF。

# 三、阻塞、非阻塞、同步、异步，这是最容易被拷烂的地方

## 7. 阻塞 IO 和非阻塞 IO 的区别？

### 回答

阻塞和非阻塞，描述的是**调用某个系统调用时，当前线程会不会被挂起等待**。

以 `recv` 为例：

- **阻塞 IO**：如果内核接收缓冲区里没数据，线程会睡眠等待
- **非阻塞 IO**：如果没数据，不会睡眠，直接返回 `-1`，errno 为 `EAGAIN`

所以非阻塞不是“不需要等待”，而是“**不会在这次调用里阻塞住线程**”。

### 追问：非阻塞是不是更快？

### 回答

不一定。
非阻塞的价值主要不在单次调用更快，而在于**避免线程卡死在单个 fd 上，提高并发处理能力**。
如果只处理一个连接，非阻塞不一定比阻塞快，甚至可能因为频繁轮询更浪费 CPU。

------

## 8. 同步 IO 和异步 IO 的区别？

### 回答

这个问题很多人会混。

核心看：**数据从内核拷贝到用户缓冲区，这一步是谁完成、谁等待。**

- **同步 IO**：应用主动发起 IO，数据准备好后，还要由当前线程参与拷贝/完成流程
- **异步 IO**：应用发起请求后立刻返回，内核把数据准备和拷贝都做完，再通知应用

Linux 常见的 `select/poll/epoll` 本质上都还是**同步 IO 模型**，它们解决的是“等事件就绪”的问题，不是“内核帮你把整个 IO 完成”。

### 一句非常适合面试的话

> epoll 是 IO 多路复用机制，不是异步 IO；它只是高效地通知你哪个 fd 就绪了，真正的 read/write 还是要应用自己做。

这句很关键。

------

## 9. 阻塞、非阻塞、同步、异步之间是什么关系？

### 回答

这两组概念维度不同：

- 阻塞 / 非阻塞：看函数调用会不会让线程挂起
- 同步 / 异步：看 IO 完成流程中，应用是否需要主动参与

所以会有组合：

- 阻塞同步 IO
- 非阻塞同步 IO
- IO 多路复用同步 IO
- 信号驱动 IO
- 异步 IO

Linux 服务器编程里最常见的是：

- 非阻塞 socket
- 配合 epoll
- 本质上仍然是同步 IO

# 四、IO 多路复用，面试官最爱深入问的部分

## 10. 什么是 IO 多路复用？

### 回答

IO 多路复用的核心思想是：

**一个线程同时监听多个 fd 的事件状态，一旦某些 fd 就绪，再对这些 fd 进行读写。**

这样做的目的不是“同时真正并行处理很多 IO”，而是：

- 避免每个连接一个线程
- 避免阻塞在单个 fd 上
- 提高线程利用率
- 降低线程切换和内存开销

比如一个线程用 epoll 监听 1 万个连接：

- 平时大多数连接没数据，不需要处理
- 哪些连接可读了，epoll 通知你
- 你再只处理活跃连接

## 11. select、poll、epoll 的区别？

这个问题几乎必问。

### 标准回答框架

可以从四个角度答：

1. 数据结构
2. fd 数量限制
3. 事件通知方式
4. 时间复杂度/性能特点

### 回答

#### （1）select

- 用位图 `fd_set` 表示关注的 fd 集合
- 每次调用都要把整个集合从用户态拷到内核态
- 内核遍历所有 fd，找出就绪的
- 返回后用户还要再遍历一遍，检查哪些 fd 就绪
- 一般有最大 fd 数限制，常见是 1024

#### （2）poll

- 用 `pollfd` 数组代替位图
- 没有 select 那种 1024 的位图硬限制
- 但本质上仍然要每次把整个数组拷到内核
- 内核仍然线性扫描全部 fd
- 性能瓶颈本质没变

#### （3）epoll

- Linux 特有
- 先通过 `epoll_ctl` 把 fd 注册到内核中的红黑树等结构里
- `epoll_wait` 时不用每次重新传全部 fd 集合
- 内核把就绪 fd 放到 ready list
- 用户只拿到真正就绪的 fd，不用扫描全部 fd
- 在大量连接、少量活跃的场景下性能优势明显

### 高质量总结句

> select 和 poll 更像“每次都拿着全名单去问谁准备好了”，
> epoll 更像“先把名单交给内核，谁准备好了内核主动记下来，等你来取”。

这句很适合口语表达。

------

## 12. epoll 为什么比 select / poll 高效？

### 回答

主要有两个原因：

#### 第一，不需要每次重复拷贝整个 fd 集合

select/poll 每次调用都要把关注的 fd 集合从用户态传到内核态。
epoll 通过 `epoll_ctl` 事先注册，后续 `epoll_wait` 不需要重复传整个集合。

#### 第二，不需要每次线性扫描所有 fd

select/poll 内核和用户都要遍历所有 fd。
epoll 内核维护就绪队列，`epoll_wait` 直接返回已就绪 fd，避免了大规模无效扫描。

### 再补一句更严谨的

严格说，epoll 不是在所有场景下都一定更快。
它在“**连接很多，但同时活跃连接比例不高**”的场景下优势最明显，这也是高并发服务器常见场景。

------

## 13. epoll 的底层数据结构是什么？

### 回答

常见说法是：

- interest list：保存关注的 fd，内核里通常用红黑树管理
- ready list：保存已经就绪的 fd，通常用链表等就绪队列组织

`epoll_ctl` 负责增删改 interest list 中的关注项。
当 fd 状态发生变化时，内核回调把它加入 ready list。
`epoll_wait` 从 ready list 中取就绪事件返回给用户态。

### 面试提醒

不同内核实现细节可以变化，你不用把“红黑树 + 双链表”背成唯一真理。
你可以说：

> 常见实现中，关注集合会用便于快速增删查的数据结构维护，比如红黑树；就绪事件会放在 ready queue 里，便于 epoll_wait 直接取出。

这样更稳。

------

## 14. epoll 的 LT 和 ET 是什么？

### 回答

LT 是 **Level Triggered，水平触发**。
ET 是 **Edge Triggered，边缘触发**。

#### LT

只要 fd 仍然处于可读/可写状态，epoll_wait 每次都可能继续通知。

比如 socket 里还有数据没读完，下次 epoll_wait 还会告诉你它可读。

#### ET

只在状态从“不可读变成可读”这类边沿变化时通知一次。
如果你这次没把数据读完，后面状态没再变化，可能就不会再次通知。

### 面试回答建议

> LT 更像“只要还有货，就反复提醒你”；
> ET 更像“货刚到时提醒一次，之后你自己要一次搬空”。

------

## 15. 为什么 ET 必须配合非阻塞 IO？

### 回答

因为 ET 模式下，你通常需要在一次事件通知后**循环把数据读到不能再读为止**。
如果 fd 是阻塞的，当你读到一半没数据了，`read` 可能阻塞住，线程就卡死了。

所以 ET 的典型写法是：

- 把 socket 设成非阻塞
- 收到可读事件后循环 `read`
- 直到返回 `EAGAIN`

### 面试官追问：ET 模式如果不把数据读完会怎样？

### 回答

可能丢失后续通知。
因为 ET 只关心“状态变化”，如果可读状态已经出现过，而你没读空缓冲区，那么之后状态没有新的边沿变化，epoll 可能不再提醒你，这个连接看起来就“卡住了”。

------

## 16. LT 和 ET 怎么选？

### 回答

工程上大多数场景：

- **LT 更容易写对，容错更高**
- **ET 理论上减少重复通知，适合极致性能优化**

如果团队没有特别强的网络库积累，优先 LT。
如果是成熟高性能框架，可能会用 ET + 非阻塞 + 一次性读空/写尽。

### 一句稳妥表达

> LT 偏保守、易用，ET 偏高性能、实现复杂。是否选 ET，要看团队对事件循环和缓冲区管理的掌控能力。

------

# 五、非阻塞 connect、accept、read、write 会怎么问

## 17. 非阻塞 connect 会发生什么？

### 回答

对非阻塞 socket 调用 `connect`：

- 如果立即建立成功，直接返回 0
- 如果连接还在进行中，返回 -1，errno 是 `EINPROGRESS`

这不是失败，而是表示连接正在异步建立。
后续应该把这个 fd 注册到 epoll，监听可写事件。
当 socket 可写时，再通过 `getsockopt(fd, SOL_SOCKET, SO_ERROR, ...)` 检查连接是否真正成功。

### 容易翻车点

不要说“可写就一定 connect 成功了”。
**可写只表示连接过程结束了，成功还是失败要看 SO_ERROR。**

------

## 18. 非阻塞 accept 有什么注意点？

### 回答

监听 socket 设为非阻塞后，`accept` 在没有新连接时会返回 `-1`，errno 为 `EAGAIN`。
这说明当前没有连接可接，不是异常。

如果使用 ET 模式监听 listen fd，收到可读事件后，通常要循环 `accept`：

- 一直 accept
- 直到返回 `EAGAIN`

否则可能漏掉队列里剩余连接。

------

## 19. 非阻塞 write 一次写不完怎么办？

### 回答

这是高频问题。

非阻塞 socket 的发送缓冲区可能放不下所有待发送数据。
`write/send` 可能：

- 返回正数：写入了一部分
- 返回 -1 且 `EAGAIN`：当前暂时不能写

所以必须维护**应用层发送缓冲区**。

标准做法：

1. 尝试发送
2. 如果只发出一部分，剩余数据放到发送缓冲区
3. 注册 EPOLLOUT
4. 下次可写事件到来时继续发送
5. 全部发送完后取消 EPOLLOUT，避免空转

### 面试官追问：为什么发完后要取消 EPOLLOUT？

### 回答

因为 socket 大多数时候往往是可写的。
如果一直监听 EPOLLOUT，会不断收到可写通知，导致事件循环空转，浪费 CPU。

所以一般只在“**确实有未发完数据**”时才关注写事件。

------

# 六、面试官开始问你服务器模型了

## 20. 单线程 epoll 能不能处理高并发？

### 回答

可以处理很多连接，但是否足够取决于业务逻辑。

单线程 epoll 的优点：

- 结构简单
- 无锁或少锁
- 避免线程切换
- 适合 IO 密集型场景

但如果业务处理很重，比如：

- JSON 解析复杂
- 数据库查询慢
- 大量加解密
- 图片处理

那单线程事件循环容易被耗时任务阻塞，导致其他连接延迟升高。

所以常见架构是：

- **IO 线程负责事件分发和收发**
- **工作线程池负责业务计算**
- 必要时把网络层和业务层解耦

------

## 21. Reactor 和 Proactor 是什么？

### 回答

#### Reactor

- 应用主动等待事件就绪
- 就绪后自己调用 read/write 完成 IO
- epoll + 非阻塞 socket 属于典型 Reactor

#### Proactor

- 应用发起异步 IO 请求
- 内核完成读写后再通知应用
- 应用处理的是“已完成”事件，而不是“可读/可写”事件

Linux 传统 epoll 更多是 Reactor。
Windows IOCP 更接近 Proactor。

### 高质量一句话

> Reactor 处理的是“就绪事件”，Proactor 处理的是“完成事件”。

------

## 22. 主从 Reactor 模型是什么？

### 回答

常见高性能服务器会用主从 Reactor：

- **主 Reactor**
  - 负责监听 listen fd
  - accept 新连接
  - 把连接分配给某个从 Reactor
- **从 Reactor**
  - 各自维护自己的 epoll 实例
  - 负责已连接 socket 的读写事件处理

好处：

- 把接入和通信分离
- 减少单个 epoll 压力
- 能利用多核 CPU

Nginx、Netty 等都有类似思想，只是实现细节不同。

------

## 23. 为什么高并发服务器不用一个连接一个线程？

### 回答

主要因为成本太高。

#### 第一，线程数量多会占用大量内存

每个线程默认栈空间通常是 MB 级别。
几千几万个线程会非常浪费内存。

#### 第二，线程切换开销大

连接多但大多数不活跃，如果每个连接一个线程，CPU 会浪费在调度切换上。

#### 第三，可扩展性差

线程数不能无限增长，系统调度也会成为瓶颈。

所以高并发网络服务器更常见的是：

- IO 多路复用
- 少量线程处理大量连接
- 必要时线程池分担业务计算

------

# 七、socket 细节追问，特别容易被问死

## 24. close 和 shutdown 有什么区别？

### 回答

`close` 是关闭文件描述符，减少引用计数。
如果这个 fd 是最后一个引用，内核才真正释放 socket 资源。

`shutdown` 是关闭连接的某个方向：

- `SHUT_RD`：关闭读
- `SHUT_WR`：关闭写
- `SHUT_RDWR`：都关

TCP 里常见的是 `shutdown(fd, SHUT_WR)`，表示“我这边不再发数据了，但还能继续接收对方数据”，会触发半关闭。

### 面试官追问：半关闭有什么意义？

### 回答

适用于“请求发完，但还等对方返回结果”的场景。
比如某些协议里客户端发送完所有数据后，用 `shutdown(SHUT_WR)` 告诉服务端“我发完了”，服务端仍可继续处理并回数据。

------

## 25. 什么是 TIME_WAIT？为什么会出现？

### 回答

主动关闭连接的一方，在发送最后一个 ACK 后，会进入 TIME_WAIT 状态，等待一段时间，通常是 2MSL。

目的主要有两个：

1. **确保最后的 ACK 能让对方收到**
   - 如果对方没收到，会重发 FIN
   - TIME_WAIT 状态可以重发 ACK
2. **防止旧连接的延迟报文影响新连接**
   - 等网络中的旧报文自然消失

### 追问：TIME_WAIT 太多怎么办？

### 回答

先分析原因，不要一上来乱调参数。

可能原因：

- 短连接太多
- 服务端主动关闭过多
- 压测或爬虫场景频繁建连断连

优化方向：

- 优先使用长连接、连接池
- 让客户端主动关闭，减少服务端 TIME_WAIT
- 合理调内核参数，比如端口复用、回收策略，但要谨慎
- 结合业务与网络环境评估，不能只靠“内核魔改”

### 面试提醒

不要上来就背 `tcp_tw_recycle`，这玩意早就不推荐了，而且有 NAT 问题。

------

## 26. 什么是粘包？UDP 会不会粘包？

### 回答

TCP 会有“粘包拆包”问题，因为它是字节流。
UDP 一般不会有 TCP 意义上的粘包，因为 UDP 是报文边界保留的，一个 `recvfrom` 对应一个数据报，要么收到完整报文，要么被截断，但不会把两个独立 UDP 报文无边界拼成一个。

------

## 27. 什么是惊群问题？

### 回答

惊群是指多个进程或线程同时等待同一个事件，当事件发生时它们都被唤醒，但最终只有一个真正处理成功，其余又白白睡回去，造成上下文切换浪费。

在网络编程里，经典场景是多个进程/线程同时监听同一个 listen socket 的 accept。

### 追问：epoll 能解决惊群吗？

### 回答

不绝对。
传统实现里还是可能出现。
Linux 后续引入过一些优化机制，比如 `EPOLLEXCLUSIVE` 来减轻这类问题，但要看具体场景和内核版本。

所以更完整的回答是：

> epoll 可以帮助优化事件通知机制，但惊群是否出现，还和监听模型、内核机制、是否共享 accept 等有关，不是“用了 epoll 就完全没有惊群”。

# 八、工程设计题，面试官会从“会不会 API”升级到“你会不会做系统”

## 28. 如果让你设计一个高并发 TCP 服务器，你会怎么设计？

### 参考回答

我会先按职责拆分：

### 1）接入层

- 一个或多个监听 socket
- 非阻塞
- 使用 epoll 监听 accept 事件

### 2）IO 层

- 每个连接维护：
  - fd
  - 读缓冲区
  - 写缓冲区
  - 连接状态
  - 心跳/超时信息
- 事件循环只负责：
  - 接收连接
  - 读取数据
  - 协议解包
  - 投递业务任务
  - 发送待写数据

### 3）业务层

- 线程池处理耗时逻辑
- 避免在 IO 线程里直接做慢操作
- 处理完成后把响应投递回 IO 线程发送

### 4）协议层

- 自定义长度头解决粘包
- 设置最大包长防止恶意大包攻击
- 做协议合法性校验

### 5）连接管理

- 心跳机制
- 空闲连接超时回收
- 异常断开处理
- 限流、防刷、黑名单

### 6）系统优化

- 合理设置 backlog、rcvbuf、sndbuf
- 使用 SO_REUSEADDR，必要时考虑 SO_REUSEPORT
- 打日志、监控 QPS、连接数、事件延迟、发送队列积压

### 一句很加分的话

> 高并发服务器的核心不只是“用了 epoll”，而是围绕事件驱动模型，做好连接管理、缓冲区管理、协议解析和业务解耦。

------

## 29. 为什么要把业务线程和 IO 线程分开？

### 回答

因为 IO 线程的职责应该是“快进快出”。
如果 IO 线程里做了耗时操作：

- 会阻塞事件循环
- 导致其他连接得不到及时处理
- 增加延迟，甚至导致连接堆积

所以通常做法是：

- IO 线程只做网络事件处理
- 业务线程处理计算/数据库/缓存等逻辑
- 通过队列或任务调度机制解耦

------

## 30. 如何处理连接超时？

### 回答

常见做法：

1. 给每个连接记录最近活跃时间
2. 使用定时器结构管理超时连接
3. 周期性扫描或用时间轮/小根堆检查
4. 超时则关闭连接并清理资源

如果连接很多，简单遍历所有连接成本高，
工程上常用：

- 小根堆
- 时间轮
- 分层时间轮

### 面试官追问：时间轮为什么高效？

### 回答

因为它把大量定时任务按时间槽分桶，
每次只处理当前槽位，避免每轮都扫描全部定时器。
在大量连接、统一粒度超时场景下效率很高。

# 九、你简历写“熟悉 IO 多路复用”，面试官很可能这么连续追问

下面我给你一套**拷打链条 + 标准回答思路**。

------

## 链条 1：概念链

### 问

你说熟悉 IO 多路复用，那它解决了什么问题？

### 答

它解决的是**单线程/少线程高效管理多个 fd 的等待问题**。
传统阻塞 IO 中，一个线程往往被一个 fd 卡住；IO 多路复用把“等待多个 fd 是否就绪”集中交给内核，一旦某个 fd 就绪，再由应用处理，从而提高并发能力和资源利用率。

------

### 问

它为什么叫“多路复用”？

### 答

因为本质上是一个线程复用来监听多路 IO 事件。
不是说一个 fd 被复用，而是**一个等待机制同时服务多个 IO 通道**。

------

## 链条 2：select / poll / epoll 链

### 问

select 的缺点是什么？

### 答

- fd 数量通常受限
- 每次都要拷贝 fd 集合到内核
- 内核要线性扫描全部 fd
- 用户态还要再扫描一遍结果
- 大量 fd 场景性能差

------

### 问

poll 比 select 好在哪里？

### 答

主要是数据结构更灵活，没有 select 的位图硬限制，
但本质仍要线性扫描，性能瓶颈并没有根本解决。

------

### 问

epoll 的本质提升是什么？

### 答

本质是把“关注哪些 fd”和“哪些 fd 已经就绪”分离：

- 关注集合长期驻留内核
- 就绪事件单独组织
- `epoll_wait` 直接返回活跃 fd

所以减少了重复拷贝和无效遍历。

------

## 链条 3：epoll 细节链

### 问

epoll_create 是干什么的？

### 答

创建一个 epoll 实例，返回一个 fd。
这个 fd 本身也代表一个内核事件管理对象，后续用它来调用 `epoll_ctl` 和 `epoll_wait`。

------

### 问

epoll_ctl 三种操作是什么？

### 答

- `EPOLL_CTL_ADD`：注册新 fd
- `EPOLL_CTL_MOD`：修改监听事件
- `EPOLL_CTL_DEL`：删除 fd

------

### 问

什么时候用 MOD？

### 答

比如连接初始只监听读事件；
当发送缓冲区有未发完数据时，需要增加写事件监听；
写完后再去掉写事件。
这时候就需要 `EPOLL_CTL_MOD` 动态修改监听掩码。

------

## 链条 4：ET / LT 实战链

### 问

ET 一定比 LT 更好吗？

### 答

不一定。
ET 通知次数更少，理论上更高效，但实现更复杂，容易因为没读空/写尽而出 bug。
LT 实现简单、鲁棒性更高，很多业务系统优先用 LT。

------

### 问

ET 模式下 read 怎么写？

### 答

要循环读：

- 反复 `read`
- 直到返回 `EAGAIN`
- 或者返回 0 表示对端关闭
- 或者真实错误

因为一次通知后不保证还会再次提醒，所以要尽量把内核缓冲区读空。

------

## 链条 5：工程链

### 问

如果一个连接发来一个 10MB 的包，你怎么处理？

### 答

不能无脑收。要做协议层保护：

- 包头声明长度
- 设置最大包长限制，比如 1MB 或按业务定
- 超过上限直接关闭或拒绝
- 接收缓冲区按需扩展，避免恶意内存占用
- 业务线程也要做流量控制

这既是正确性问题，也是安全问题。

------

### 问

一个连接一直不发完整包怎么办？

### 答

这是典型慢连接/慢速攻击问题。
处理方式：

- 设置读超时
- 限制包头和包体接收时长
- 分阶段超时控制
- 长时间不完整就断开
- 必要时配合限速和连接数限制

# 十、面试官会特别喜欢的“容易问死人的点”

## 31. epoll 监听的是“文件描述符”还是“文件”？

### 回答

更准确地说，epoll 监听的是**fd 所对应的内核对象上的事件状态**。
fd 是用户态句柄，真正有意义的是背后的 file/socket 对象。

如果多个 fd 引用同一个底层对象，行为会涉及引用关系和内核实现细节，不能简单理解成“只是数字相同/不同”的问题。

------

## 32. 一个 socket 可读，是否代表一定能读到业务完整包？

### 回答

不代表。
可读只说明内核接收缓冲区中至少有一些数据，
可能只是半个包、一个半包、多个包拼在一起。
所以还必须由应用层协议解析来判断是否拿到了完整消息。

------

## 33. 一个 socket 可写，是不是说明你写多少都行？

### 回答

不是。
可写只表示发送缓冲区当前有可用空间，
不代表足够容纳你所有待发送数据。
非阻塞模式下一次 write 仍可能只写部分数据，甚至返回 EAGAIN。

------

## 34. epoll 会不会重复返回同一个 fd？

### 回答

会，尤其在 LT 模式下。
只要条件一直满足，比如 socket 里一直有未读数据，就可能反复返回。
ET 模式则更依赖状态变化。

------

## 35. 为什么 epoll 适合高并发，但不一定适合所有场景？

### 回答

因为它的优势主要体现在：

- 连接很多
- 活跃比例低
- 事件驱动明显

如果连接数量很少，或者业务主要瓶颈不在 IO，而在 CPU/磁盘/数据库，那 epoll 不一定带来显著收益。
真正决定性能的往往是整体架构，而不是“用了 epoll”这一个点。

# 十二、给你一份“面试官最可能连环问”的完整版模拟

------

## 模拟问答 1

### 问

你说熟悉 socket 编程，那服务端建立 TCP 连接的流程说一下。

### 答

服务端一般是先 `socket` 创建监听 fd，然后 `bind` 绑定 IP 和端口，再 `listen` 进入监听状态。
客户端发起三次握手后，服务端通过 `accept` 从已完成连接队列中取出一个连接，返回新的已连接 fd。
之后监听 fd 继续负责接入新连接，已连接 fd 用来做具体收发。

------

## 模拟问答 2

### 问

为什么 accept 不直接在原来的 fd 上通信？

### 答

因为监听 fd 和连接 fd 的职责不同。
监听 fd 是面向端口的接入入口，而每个连接都需要独立的收发缓冲区和状态管理，所以 accept 会返回一个新的连接 fd。

------

## 模拟问答 3

### 问

你说熟悉 IO 多路复用，那 epoll 为什么比 select 好？

### 答

我觉得核心有两点。
第一，select 每次都要把整个 fd 集合从用户态拷到内核，epoll 通过 epoll_ctl 事先注册，避免重复拷贝。
第二，select/poll 每次都要线性扫描全部 fd，而 epoll 维护就绪队列，epoll_wait 直接返回活跃 fd。
所以在大量连接但少量活跃的场景下，epoll 更有优势。

------

## 模拟问答 4

### 问

epoll 是异步 IO 吗？

### 答

不是。
epoll 是 IO 多路复用机制，本质还是同步 IO。
它只是通知应用哪个 fd 已经就绪，真正的数据读写还是应用自己调用 read/write 完成，不属于内核帮你完成整个 IO 的异步模型。

------

## 模拟问答 5

### 问

ET 和 LT 区别是什么？为什么 ET 要配非阻塞？

### 答

LT 只要条件仍然满足就会持续通知，ET 只在状态变化时通知一次。
ET 下如果收到可读事件却没有把数据读空，后续可能因为没有新的状态变化而收不到提醒。
因此 ET 通常配合非阻塞 socket，收到事件后循环读取直到返回 EAGAIN，避免线程卡住，也避免遗漏数据。

------

## 模拟问答 6

### 问

非阻塞发送时 write 只写了一半怎么办？

### 答

需要维护应用层发送缓冲区。
把没发完的数据暂存起来，并监听 EPOLLOUT，等 socket 可写时继续发送。
全部发完后要取消写事件监听，否则会因为 socket 经常可写而导致事件循环空转。
